## 要約

Claude Code を長期運用する上で直面する2つの課題（何を書くべきか不明・ルールが守られない）を解決するための実践的な設計パターン集。前提として、CLAUDE.md は LLM へのソフトな指示であり約70%の遵守率しか期待できない。一方 Hooks はシェルコマンドとして100%決定論的に実行される。この差異を理解した上で、CLAUDE.md と Hooks の役割を明確に分離することが運用の基本となる。

CLAUDE.md は3つのスコープで読み込まれる：グローバル（~/.claude/CLAUDE.md）、プロジェクト（プロジェクトルート）、ローカル（サブディレクトリ）。グローバルにはセキュリティポリシーや言語設定、プロジェクトにはフォルダ構造や検索規則、ローカルにはサブシステム詳細を書くのが基本設計。

紹介されるパターンは9つ。①優先度階層の明示：ファイル冒頭に参照優先度を宣言し、複数 CLAUDE.md 共存時の判断精度を上げる。②Mapパターン：フォルダ構造と検索除外ディレクトリ（例：03_Archive/）をテーブル形式で冒頭に宣言することで、Claude の不要ファイル参照によるトークン浪費と誤回答を防ぐ。③検索フローチャート：Grep/Glob（高速・正確）→ RAG検索（概念・広域検索）→ find_similar_notes → Glob広範囲というフォールバックチェーンを定義し、Claude がループしないよう誘導する。④イベントトリガーテーブル：ツールエラー・ユーザー修正・設計判断・外部API使用などのイベントごとに書き込み先とアクションを定義し、セッション中にリアルタイム記録することでコンテキスト圧縮後もナレッジを保持する。⑤コード変更トリガーテーブル：CLAUDE.md や agents/ 変更時に連動更新すべきドキュメントを対応表化し、整合性チェックを自律化する。⑥クエリキーワード辞書：頻出操作を自然言語ショートカット（例：「朝刊」で morning-briefing スキル実行）として定義し、UX を自然言語インターフェース化する。⑦哲学セクション：細かいルール羅列より行動原理を先に定義する。「ルールを追加したくなったら人格で代替できないか考える」という指針により CLAUDE.md の肥大化を長期的に防ぐ。⑧シークレット管理の明文化：os.environ.get のフォールバック値への実値記述禁止、.env* の .gitignore 登録、gitleaks pre-commit hook の導入などをコード例付きで明記し、コード生成時のセキュリティミスを予防する。⑨段階的参照：CLAUDE.md を「地図」として機能させ、詳細仕様は .claude/docs/ 配下の別ファイルに委譲することで肥大化を防ぐ。

監査エージェント開発への示唆：イベントトリガーテーブルとコード変更トリガーテーブルのパターンは、LangGraph や ReAct ベースのエージェントにおける状態遷移ログの設計と構造的に類似している。エージェントの判断記録・フィードバックループ設計に直接応用可能。

## アイデア

- CLAUDE.md の遵守率70% vs Hooks の100%決定論的実行という非対称性を設計原則として明示し、ソフト指示とハード実行を意図的に分離するアーキテクチャ思想
- イベントトリガーテーブルによりセッション中リアルタイムに記録することで、Claude のコンテキスト圧縮（compact）後もナレッジを保持するメモリ永続化戦略
- 「ルールを追加したくなったら人格で代替できないか考える」という哲学セクションの設計：ルール増加→遵守率低下のスパイラルを、行動原理の定義で構造的に断ち切る手法
