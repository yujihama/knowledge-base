## 要約

京都vs大阪の妖怪をテーマにしたX（旧Twitter）参加型の自走型合戦ゲームの開発事例。技術スタックはBun/TypeScript、SQLite＋Drizzle ORM、React 19、Tailwind v4、Railway上のDockerデプロイ構成。LLMにはClaude Code CLIの`claude -p`コマンドを使用し、画像生成にはOpenAI gpt-imageを採用する。

設計の核心は「LLMは文章生成のみ担当し、勝敗・数値・進行はすべてコードが決定する」という役割分担にある。例えば技名「ぶぶ漬け責め」は内部的に`{ kind: magic, power: 1.4, mpCost: 8, status: しびれ20% }`というデータ構造を持ち、キャラクター性は文章とステータス振り分けで表現しつつ、ゲームバランスはエンジンが保証する。

Xとの連携は2軸。①いいね=育成（1いいねで妖怪のランダムなステータスが1%上昇、複利・上限なし）、②リプライ=投票（卵ツイートへの返信で翌日の妖怪の職業を決定）。職業（job）は固定リストを持たず、LLMがリプライを解釈して重みテーブルを更新し、未知のjobが自然発生する設計。

プロンプトインジェクション対策として多段防御を実装。リプライ原文を読む処理はメイン生成プロンプトと完全に隔離し、jobのlabelは20文字・descは60文字にクランプの上、かな・カナ・漢字・英数のみに文字種制限。LLMへのデータ渡しはフェンス付きJSONブロック構造を使用。出口側ではLLMが書いた投稿文を投稿前にURL・@メンション除去するコードを通す。仮にインジェクションが成功しても「変な文章が表示される」止まりで、勝敗・数値・X読み書き権限はコードが持つため被害が限定される設計。

永続化はイベントソーシング方式を採用。`events`テーブルに`(tick, seq)`で順序付けしながら発生事象を追記専用で記録し、状態テーブル（妖怪・job・ゲージ等）はすべて投影として再構築可能にする。1ステップ＝1トランザクションでイベント＋投影を同時コミットし、途中失敗でも進行カーソルは前進済み分を保持するため再開は残り段から続行できる。`bun run rollback --to-event N`または`--to-tick N`で任意時点へのロールバックが数秒で完了し、LLM再呼び出し不要・課金ゼロで再構築できる。X投稿済み記録は`post_dispatches`として投影とは別に保持し、ロールバック後の二重投稿を防止する。

画像生成においては、AI画像の左右向きが不安定な問題をベクトル検索＋多数決で解決。無料モデルで画像をベクトル化し、人間が左右ラベルを付与した少量サンプル＋反転ペアを用意し、未知画像に対してベクトル検索上位数件の多数決で向きを判定するkNN的アプローチを採用。CPU処理で十分な精度を実現している。

## アイデア

- LLMを文章生成に限定し勝敗・数値判断をコードに委ねることで、LLMの非決定性をゲームの権威から切り離す設計パターン——監査エージェントでも「LLMは所見文の生成のみ、判断基準の適用はルールエンジン」という分界に直接応用できる
- イベントソーシング＋1トランザクション1ステップの構成により、LLM出力そのものをイベントログに保存することで非決定なLLMを含むシステム全体の再現性を担保しつつ課金ゼロでロールバック可能にする——LLMを使ったバッチ処理の冪等性・監査ログ設計の参考になる
- 外部入力（Xリプライ）をuntrusted inputとして扱い、越境点を最小化（jobのlabel/descの2フィールドのみ）＋文字種・文字数クランプ＋隔離呼び出し＋出口後処理の多段防御でプロンプトインジェクションリスクを構造的に低減する手法
