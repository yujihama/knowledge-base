## 要約

Claude Code のメモリ機能において MEMORY.md を「メモ帳」ではなく「索引（インデックス）」として設計することで、コンテキスト消費を大幅に削減できるという設計論。

Claude Code のメモリは2階層構造になっている。MEMORY.md は会話開始時に毎回全文読み込まれるインデックス層であり、個別メモリファイル（`<type>_<topic>.md`）はインデックスのトリガー記述を契機として必要時のみ読み込まれる本文層だ。この構造を活かさず MEMORY.md に本文を書き込むと、60本のメモリを持つリポジトリでは「60本 × 500字 ≒ 30,000字」が毎回送信される。インデックスに150字要約のみを置けば「60本 × 150字 ≒ 9,000字」に収まり、3倍以上の差が生じる。

150字ルールの根拠は3つの制約から逆算される。①索引としての役割を保つ（本文に踏み込まない）、②50行超のインデックスでも1画面で走査できる一覧性、③150字 × 1.5トークン/字 ≒ 225トークン/行 × 60行 ≒ 13,500トークンという毎回の初期コスト許容範囲。

読み込み順序は「上位＝常時参照 / 下位＝特殊条件」で設計する。プロファイルや口調設定を上位に、プロジェクト固有の短命なメモリを下位に配置する。セクション見出しによる分類も有効だが、50本程度ならフラット構造で十分とのことだ。

各エントリの「—」右側はトリガー文であり、「内容の要約」ではなく「いつ参照すべきか」を書くことで個別メモリの呼び出し精度が上がる。

プロンプトキャッシュ（Anthropic の「先頭からの完全一致」で効く仕組み）との相性も重要で、①既存行を頻繁に書き換えない、②並べ替えはセッション開始前にまとめて行う、③新エントリは末尾追記を優先する、の3点でキャッシュ無効化を抑制できる。

陳腐化対応として user/feedback/project/reference の4種別ごとに削除判定基準を設け、定期的に prune することで「使われないエントリが残り続ける」問題を防ぐ。監査エージェント開発では長期セッションが多くなりがちなため、この設計によるコンテキスト軽量化は応答速度・コスト両面で直接的な効果をもたらす。

## アイデア

- MEMORY.md を2階層（インデックス＋個別ファイル）に分離することで、メモリ数が増えても初期コンテキスト消費量をO(1)に近い形で抑えられる設計は、RAGのチャンク戦略と本質的に同じ発想
- Anthropicのプロンプトキャッシュが「先頭からの完全一致」で効く仕様を前提に、MEMORY.mdの書き換えタイミングと順序を設計するという、キャッシュ戦略をメモリ管理に組み込む視点
- トリガー文に「最終参照: YYYY-MM-DD」を付与することで陳腐化エントリを目視発見可能にする一方、自動更新するとキャッシュが落ちるというトレードオフの管理手法
