## 要約

外国人の友人から「東京・京都の次はどこに行けばいい？」と繰り返し聞かれた経験と、移住・多拠点生活を考える中で「その土地が何者か」を物語として知りたいという個人的動機から、日本全1,741市区町村＋176有人離島の散文テキストを生成するWebサービス「AURA Poetry」を半年かけて開発した。

技術的な核心はClaude APIを使ったバッチ生成で、対象は合計約1,900件。プロンプト設計で最重視したのはトーンの制御で、「〇〇が有名です」式の観光紹介文ではなく、場所の空気感・歴史の重さ・住民の気配を散文として書かせるよう細かく文体指示を入れた。日本語版は短い断言と体言止め、英語版はゆったりした観察者の視点と、言語ごとに異なるトーン設定を採用。入力データはWikipedia記事要約に加え、文化財・温泉・山・湖・港・鉄道などの独自収集構造化データを組み合わせた。データが薄い過疎地ほどLLMの生成に依存するが、かえって質の高い文章になるケースもあったと報告している。全件レビューは行わず、スポットチェックと品質フィルタリングを組み合わせた。

インフラはHeroku Basic dyno（512MB RAM）＋Postgres Essential-1で月約2,650円。512MBの制約に対し、重い処理はローカル実行してJSONエクスポート→管理コマンドでインポートするパターンで対処。地図表示は当初GeoJSON（16.5MB）で配信していたが、MapLibre GL JS＋pmtiles形式に移行することで1リクエストあたり17KB程度に圧縮、FCPを4.4秒から3.2秒に短縮した。タイル生成にはtippecanoeを使用し、文字列feature IDを数値変換する必要があるなどのハマりポイントも経験した。

コンテンツとして現在、全1,741市区町村の物語・地図・文化財等データ、176離島の個別ページ、252件の地方イベントを統合済み。英語版も全ページで提供しており、Xでの英語発信後に海外アクセスが増加した。監査エージェント開発への直接的な示唆は薄いが、大量コンテンツのバッチ生成においてプロンプトによるトーン制御と品質フィルタリングを組み合わせるアプローチは、エージェントが大量ドキュメントを生成・評価するパイプライン設計に応用できる。

## アイデア

- 1,900件規模のバッチ生成で全件レビューせずスポットチェック＋品質フィルタリングを組み合わせる品質管理手法は、大量ドキュメント生成パイプラインの実用的なアプローチとして参考になる
- 同一データを入力しても言語ごとに異なるトーン指示を与えることで読み心地の差異を意図的に設計できる点は、多言語エージェント出力の品質設計に応用できる
- データが薄い過疎地ほどLLMの生成依存度が上がりかえって良い文章になるという観察は、RAGにおける検索結果の有無とLLM生成品質の関係を考える上で興味深い逆説的知見
