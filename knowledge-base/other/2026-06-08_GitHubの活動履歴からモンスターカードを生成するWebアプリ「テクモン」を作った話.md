## 要約

「テクモン（techmon.app）」は、GitHubのコミット数・PR数・レビュー数・スター数・使用言語・活動時間帯・継続年数などの公開情報を読み取り、ステータス付きモンスターカードに変換するWebアプリ。GitHubログインのみで10秒で召喚でき、privateリポジトリを含む「完全召喚」モードも選択可能。

カードの構成要素は4ステータス（HP/ATK/SPD/SKL）、属性（光/闇/無）、称号、LLM生成のフレーバーテキスト、レア度★（1〜6）、総合力PWRの6要素。HPは活動継続性、ATKはPR・コミット量、SPDは直近の活動テンポ、SKLは使用言語の幅と希少性から算出。属性は活動時間帯（昼→光、夜→闇、分散→無）を基本とし、レビューやissueクローズなどコラボ活動の比率も加味。称号とフレーバーテキストはLLMが生成し、ユーザー情報は抽象化した上でLLMに渡すことでプライバシーを担保している。

設計上の重要な判断として、同一アカウントは何度召喚しても完全に同一のカードが出る「決定論的設計」を採用。ガチャのリセマラを排除し、「GitHubから見た現在のあなた」を固定的に表現する思想を貫いている。

レア度はPWR（活動量）だけでなく外部評価（自リポジトリへのStar/Fork取得、著名OSSへのPRマージ）が強く影響し、レア5以上は外部評価なしでは到達不可能な設計。これにより活動量と社会的評価の両軸でレア度を決定している。

開発者自身がステータスの基準値を自分のGitHub活動量を中央値として設定し、上限を999に設定したところ、本番デプロイ後すぐに複数ユーザーから「召喚時にエラーが出る」との報告が入った。調査すると一部ユーザーの活動量が999の上限を超えておりシステムが落ちていたため、上限を9999に引き上げる「限界突破」機能を緊急追加した。この経験から、自分を中央値と仮定して設計することの危険性と、世の中には想定外の活動量を持つエンジニアが実在することを実感した開発者の体験記でもある。監査エージェント開発への直接的示唆は薄いが、システム設計における初期仮定の検証不足がバグを生む典型例として参考になる。

## アイデア

- ユーザー情報をLLMに渡す前に抽象化することでプライバシーを守りながら個性的なテキスト生成を実現する設計パターン
- 同一入力から常に同一出力を返す決定論的設計により、ガチャ的なランダム性を排除し「現在地の鏡」としてのシステムを構築する思想
- 活動量（量的指標）と外部評価（質的・社会的指標）を別軸で扱い、最高レア到達に両方を必須とするゲームバランス設計
