## 要約

Claude Code のスキル機能を使い、AI生成文から「AIくささ」を取り除く日本語特化リライトスキル「humanizer-ja」を作成した記事。英語版 blader/humanizer を骨格として参照し、中国語版（op7418/humanizer-zh）から運用ルールの整理手法を、韓国語版（daleseo/korean-skills/humanizer）から言語固有の不自然さを前提に置く姿勢を取り込み、単なる翻訳ではなく日本語向けに再設計している。インストールは `npx skills add https://github.com/vibe-bear/humanizer-ja` で行う。

設計上の主な判断は4点。①対象テキストを300〜700字程度の説明文・ブログ文に限定し、SNS・小説・業務メール・法務文書は初版スコープ外とした。範囲を広げると「自然さ」の定義が崩れるためである。②出力は分析レポートではなく本文のみを標準とした。humanizer を使う場面では結果の方が分析より優先されるという判断。③単語単位の言い換えではなく、段落構成の変更・論の順序入れ替え・前置きの圧縮・まとめ句の整理まで許可した。日本語の不自然さは単語ではなく段落の流れにも現れるためである。④ガードレールを明確化し、事実・主張・結論の変更禁止、新情報の付加禁止、固有名詞・専門用語・数値の改変禁止、元のトーン維持を徹底した。

評価サンプルを作る中で日本語固有の観点として有効だったのは、「近年では」「現代では」等の一般論からの書き出しを避けて要点から入ること、「理解の促進」「効率化の実現」等の抽象名詞を動詞に戻すこと、接続詞を一律削減するのではなく重複して機械的に見える箇所だけを取ること、の3点。また、一見自然に見えるが均質な文に大きく手を入れすぎると「エディタ感」が出るという問題を発見し、改善幅が小さい場合は構成変更より言い回し整理を優先するルールを追加した。

評価セットは難易度別（露骨に硬い説明文／一見自然だが均質なブログ文／段落の流れが弱い文／トーンを崩しやすい個人ブログ文）に構成されており、スキルが「不自然な文を強く叩く」ではなく「自然な文を壊さず少し良くする」方向に調整すべきだという設計指針を導いた。監査エージェント開発への直接的な示唆は薄いが、LLMのアウトプット品質を制御するためのガードレール設計（何を変えてよいか・何を変えてはならないかを明示的にルール化する手法）は、エージェントの出力制約設計に応用可能なパターンである。

## アイデア

- 「何を変えてよいか・何を変えてはならないか」を明示ルール化するガードレール設計パターンは、エージェントの出力制約設計に汎用的に応用できる
- 評価用サンプルを難易度別に構造化することで、スキルの調整方向（強く直す vs 壊さず少し良くする）を客観的に決定できるという設計プロセスの知見
- 出力形式を「分析レポートではなく本文のみ」に絞ることでユースケース適合性を高める、スキル設計における出力インターフェースの意思決定例
