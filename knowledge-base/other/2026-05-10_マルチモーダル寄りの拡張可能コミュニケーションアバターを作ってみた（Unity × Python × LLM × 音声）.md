## 要約

修士1年の学生が趣味で開発した、音声・視覚・言語を統合したローカル動作可能な対話アバターシステムの設計・実装レポート。

システム構成はPythonがTCPサーバーとしてオーケストレーションを担い、Unityクライアントと双方向通信する構造。音声認識にはfaster-whisper（オフライン動作）、会話生成にはGemini API、音声合成にはVOICEVOX（ローカルエンジン）を使用。Unity上ではVRM形式のアバターが表示され、LLM出力に基づいて6種類の表情が切り替わり、ランダムな口形状で口パクを実現している。アバターは2種類、背景シーンは7種類から起動時に選択可能。

処理フローは「マイク入力 → faster-whisperで音声認識 → Geminiで応答生成 → VOICEVOXで音声合成 → Unityアバターへのコマンド送信と音声再生」という一連のパイプラインで構成される。

技術的課題として、音声認識→応答生成→音声合成の合計レイテンシが大きく、リアルタイム性に欠ける点が挙げられている。faster-whisperとVOICEVOXはチャンク/バッチ処理ベースであり、ストリーミング処理への移行・VOICEVOX GPU版の活用・外部サーバーへのオフロードを改善策として挙げている。一方、音声認識の誤認識はLLMが文脈補完することで実用上は許容範囲に収まったとしている。

設計思想として、このシステムを特定ドメインに閉じた実装ではなく「対話の核（コア）」として位置づけ、プロンプトやツール連携、Unityの表現層を差し替えることで面接練習・キャラクター配信・受付案内など多様なユースケースへ派生できる拡張性を重視している。

監査エージェント開発への示唆として、音声・視覚・テキストを統合したマルチモーダルインターフェースはエージェントの出力チャネル設計の参考になる。特に「応答内容（LLM）・表現媒体（Unity）・入出力方式（音声）」を独立したコンポーネントとして分離し、TCPで疎結合に接続するアーキテクチャは、監査エージェントにおける複数サブシステム統合の設計パターンとして応用可能。

## アイデア

- Pythonをオーケストレーターとして中央に置き、TCPでUnityクライアントと疎結合に接続する構成により、音声・視覚・言語の各モジュールを独立して差し替え可能にした点
- LLM出力に表情IDを含めてUnity側で表情切り替えを制御するアプローチ：応答テキストと表現媒体を同一LLM出力で制御することでモーダル間の同期を実現
- faster-whisperの誤認識をLLMの文脈理解で補完する設計：認識精度の低さをパイプライン下流のLLMで吸収し、システム全体の頑健性を高める実装方針
