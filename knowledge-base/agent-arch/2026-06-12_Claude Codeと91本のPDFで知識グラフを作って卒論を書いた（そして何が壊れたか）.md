## 要約

非エンジニアの著者が卒論執筆のため、中国の社会信用システム（SCS）に関する91本のPDF文献（約460万字）からClaude Code（Claude Opus 4.5）とPythonスクリプト17本を用いて知識グラフを構築した実践報告。最終的に622概念・846エッジのグラフを生成し、Literature-Based Discovery（LBD）のSwanson ABCモデルを適用して50件の研究ギャップ候補と15件の異国間比較仮説を生成、卒論の分析に使用した。

パイプラインは7フェーズ構成：PDF→pdftotext変換、約3,000字単位のチャンク分割（1,795チャンク）、Claude Codeによる概念抽出（646概念）、知識グラフ生成、マルチタスク予測、ギャップ検出、可視化。コンテキスト管理の原則として「即時永続化」「固定サイズチャンク（3,000字・200字オーバーラップ）」「progress.jsonによるステートレス再開」「1文献1セッション」の4点を採用し、複数回のセッション死亡を乗り越えた。

主な失敗と修正は3点。①初期グラフのノード40%（248件）が孤立：コンテキスト非持ち越し設計が文献横断エッジを構造的に欠落させた。paraphrase-multilingual-mpnet-base-v2による重複検出で183ペア候補から24件をマージし、手動エッジ定義（+38本）・定義文ベース自動追加（+162本）・人間承認付き候補提案（+100本）の3段階でエッジを補強。孤立率40%→16.4%、平均次数1.7→2.72に改善。②ギャップ候補50件全部が「ブラックリスト」（次数54のハブ）経由：次数15超のノードをBノードから除外、同一Bからの採用を最大5件に制限、異国間ペアを優先する制約を追加して再生成。③JSONキー名の不一致（書くスクリプトと読むスクリプトを別セッションで生成したため）：Task Bの「scores」vs「four_axis_scores」等3タスクすべてで食い違い、dict.get()が例外を出さずデフォルト値を返すため「全カテゴリ0件」のレポートが5ヶ月間正常終了し続けた。

著者の役割はコードレビューではなく「理屈のレビュー」に徹した点が特徴的。監査エージェント開発への示唆：LLMを用いた非構造文書の構造化において、スキーマ定義の一元管理とセッション間でのJSON契約の明示的検証（Pydantic等）が必須。また「何を候補から除外するか」の設計（ハブ除外ロジック等）がLLM出力品質と同等以上にアウトプット品質を左右する。

## アイデア

- progress.jsonによるステートレス再開設計：セッション死亡を前提としたチェックポイント管理は、長時間LLMパイプラインの必須パターン。監査エージェントの大量文書処理でも同様の設計が有効
- ハブノード除外によるギャップ検出の品質制御：Swanson ABCモデルの機械的適用はハブ経由の自明ペアで埋まる。次数閾値・多様性制約・ドメイン優先フィルタの組み合わせが実用的なギャップ検出に必要
- 別セッション生成スクリプト間のJSONスキーマ不一致問題：dict.get()のサイレント失敗が長期間気づかれない典型的事故。Pydanticによるスキーマ検証をパイプラインの境界ごとに挟む設計の重要性
