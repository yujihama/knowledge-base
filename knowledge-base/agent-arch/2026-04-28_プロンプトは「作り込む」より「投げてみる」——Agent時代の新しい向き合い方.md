## 要約

2025年後半以降のLLMの能力向上を背景に、プロンプトエンジニアリングの実践的な使い分けを提唱する記事。従来は「Role/Task/Instructions/Output Format」を丁寧に書き連ねた構造化プロンプトが必要だったが、現代のモデルはコード・ログ・設計書・仕様書を横断して処理し、意図を推論できるため、日常業務では簡潔なシンプルプロンプトから対話で深掘りする方が速く、精度も高いと主張する。

具体的な使い分けは4ケースで示される。①障害分析：「このログの異常箇所と推定原因を時系列の表でまとめて」という一行から始め、「このエラーが頻発している時間帯に共通点はあるか」と対話的に深掘りする。②コードレビュー：「バグ・パフォーマンス・可読性の観点で重要度順に指摘して」という初手から「このパフォーマンス問題、データ量が増えたときにどう影響するか」へ展開。③技術調査：パターンの概要・メリデメを初手で取得し、自社システムへの適用方法という本質的判断に集中。④設計レビュー：機能・非機能・セキュリティ・運用保守の整理を初手で行い、認証リスクや障害影響範囲を選択的に深掘り。

作り込むべき場面の判断基準も明示されており、SOPとして繰り返し使用するケースや、ツール・エージェントのシステムプロンプトとして組み込む場合は精緻な設計に価値があるとする。ファイルをそのまま添付して渡す運用（ログ・設計書・コードを説明なしで渡す）も推奨される。

監査エージェント開発への示唆として、エージェントに組み込む固定プロンプト（SOPや判断基準）は作り込むべきだが、人間とエージェントのインタラクション設計においては「シンプルな初手＋対話的深掘り」のパターンをUXとして採用することで、監査人の思考リソースをリスク判断本質に集中させられる。また、LLM-as-judgeや多段階エージェントの設計においても、完全な仕様を一度に渡すより、段階的な問いかけで精度を高めるアーキテクチャ検討に通じる視点を含む。

## アイデア

- 「使い回すものほど作り込む価値がある」という原則：SOPやエージェント組み込みプロンプトは精緻化すべきで、日常の一時的な対話は簡潔でよいという明確な分離基準が実践的
- プロンプトエンジニアリングの知識を「何を足すか」より「何を削るか」の判断眼として活かすという逆転の発想——過去の知識が不要になるのではなく、削る判断に転用される
- ファイルをそのまま渡す運用（コンテキストはファイルが語る）：以前必要だった形式説明の手間が不要になったという具体的な変化が、モデルの文脈理解能力の向上を端的に示している
