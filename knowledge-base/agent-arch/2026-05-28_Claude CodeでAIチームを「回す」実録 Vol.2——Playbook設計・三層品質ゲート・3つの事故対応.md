## 要約

Claude Codeで9体のAIエージェントチーム（COO・writer・researcher・reviewer・reader等）を構築した著者が、「作った」後の「回す」フェーズを記録したZenn Book Vol.2の公開予告記事。

Vol.1がエージェントの役割定義ファイル（.claude/agents/<name>.md）による設計図作成を扱ったのに対し、Vol.2は翌朝から直面した「どう指示を出すか分からない」という運用の空白から始まる。著者はこれを「名詞（設計図）から動詞（稼働）への移行」と表現し、組織図を描いただけでは工場は動かないというアナロジーで説明する。

本書は3部構成。第1部「チームが動く組織設計」では、Playbookと委任プロトコルの設計を扱う。タスクをエージェントに委任する際に毎回同じ説明を繰り返さないための仕組みで、writer/researcher/reader/reviewerの4体の分業境界線の定義、新エージェント（architect）追加時の4軸評価フレームなどを含む。

第2部「品質はチームに担保させる」では三層品質ゲートを解説する。reviewer（事実確認）× reader（感情評価）× 著者ペルソナ（一人称確認）の3軸スコアリングを組み合わせ、Human in the loopを「監視」ではなく「信頼の設計」として位置づける。AIにAIを採点させても精度が上がらないという初期の壁を経て、reviewerとreaderを分離することで解決した経緯も含む。

第3部「事故の記録」では実際に起きた3件を実名・時系列で記録する。①AUTO_SYNC_ZENN=trueのエージェント独断デフォルト化によるZenn記事5本の一時非公開、②blogsync直接実行による4記事のURL書き換え、③awkコマンドの改行バグがdry-run確認をすり抜けた26ファイル空ファイル化。これらをtriggerリスト（「この操作前には必ず確認する」の言語化一覧）に変換し、事故が仕組み改善のサイクルに接続される設計を提示する。

監査エージェント開発への示唆として、三層品質ゲートの構造（独立した検証エージェントによる多軸スコアリング）はLLM-as-judgeパターンの実装例として参照価値が高い。また、事故ログからtriggerリストを生成するサイクルは、監査における発見事項→コントロール改善のPDCAと構造的に同型であり、エージェントチームのガバナンス設計に直接応用できる。価格は900円、序章は無料公開。

## アイデア

- 三層品質ゲート（reviewer×reader×著者ペルソナ）による役割分離は、単一エージェントによる自己採点の精度限界を回避するアーキテクチャ的解法であり、LLM-as-judgeの実装パターンとして汎用性が高い
- 事故ログをtriggerリストに変換するサイクルは、エージェントチームのガバナンスをポストモーテム駆動で進化させる仕組みであり、監査のPDCAサイクルと構造的に同型
- Playbookと委任プロトコルによる「タスクの所在追跡」は、マルチエージェント環境でのタスク状態管理問題（誰が何を持っているか不明になる問題）への組織設計レベルの解答
