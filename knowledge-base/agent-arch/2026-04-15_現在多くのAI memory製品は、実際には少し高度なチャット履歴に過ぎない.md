## 要約

現在「Memory機能付き」と謳うAIプロダクトやライブラリの多くは、内部実装を見ると「過去のチャットログをベクトルDBに格納してRAGで検索するだけ」か「セッション内容をLLMに要約してシステムプロンプトへ差し込むだけ」に留まっている。本記事はこの現状を「Context Handling（文脈運用）」と「Memory Architecture（記憶アーキテクチャ）」を混同した結果として整理し、真のAgent Memoryに必要な設計要件を論じる。

現行手法の限界は4点に集約される。①Flat Logsの限界：ユーザー指示・Agentの思考・ツール実行結果・挨拶などがすべて等価なテキストとして保存されるため、検索時にノイズが大量混入する。②セッション越境時の情報劣化：セッション要約を繰り返すと「要約の要約」が積み重なり、情報がLost in translationする。③Cross-agent移植性の欠如：VS Code上のCoding AgentがユーザーのコーディングスタイルをDBに保存しても、ブラウザ上のリサーチAgentやCLIツールはその記憶にアクセスできない。記憶が特定アプリのセッションにロックインされている。④ProvenanceとConflict Resolutionの欠如：「DBはMySQL」という古い記憶と「DBはPostgreSQL」という新しい記憶が両方ヒットし、Agentが混乱する。どちらが新しいかを判断するVersioning機構が存在しない。

真のAgent Memoryに求められる要件は5つ。(1) Persistence：Context WindowをRAMとみなし、Memoryシステムはセッション終了後も独立して保持されるHDD/SSD相当の永続ストレージとして機能すること。(2) Portability：ClaudeモデルのAgentが抽出した記憶をGPTベースの別Agentが読み出せる「Memory Passport」的なプラットフォーム中立設計。(3) Structured Memory + Selectivity：Background・Fact・Event・Dialogue・Reflection・Skillといった意味的カテゴリに記憶を分類し、何を保持し何を破棄するかを選択する層を持つ。(4) Provenance + Governance：記憶の生成元セッション・発言をメタデータとして追跡し、矛盾時のConflict handlingとVersioning、不要記憶の削除管理を行う。(5) User Ownership：記憶の所有権はAgent開発企業ではなくユーザーに帰属し、閲覧・編集・削除の透明性を確保する。

これらの要件を体現するアーキテクチャ例として「MemoryLake」が紹介されている。MemoryLakeは大容量ベクトルDBではなく、複数Agent・複数アプリが共通して読み書きできるPlatform-neutral Memory Infrastructureとして位置づけられる。監査エージェント開発への示唆として、長期継続稼働する監査Agentが過去の内部統制評価・リスク判断・発見事項を再利用するには、セッション要約やRAGだけでなく、ProveedanceとGovernanceを備えた独立したMemory Layerが不可欠であり、本記事の設計思想はその基盤設計の参照アーキテクチャとなる。

## アイデア

- Context Window をRAM（揮発性）、Memory SystemをHDD/SSD（不揮発性）と対比させることで、現行RAG実装が実は揮発性の延長に過ぎないことを明確に示す枠組みは、監査Agentの長期稼働設計を議論する際の共通言語として使いやすい
- 記憶をBackground・Fact・Event・Dialogue・Reflection・Skillの6種類に構造化するMemory Type分類は、監査エージェントが「監査手続き（Skill）」「発見事項（Fact）」「クライアントとのやり取り（Dialogue）」を分けて管理するスキーマ設計に直接応用できる
- Conflict Handlingの欠如（MySQLとPostgreSQLが両方ヒットする問題）は監査においても深刻で、異なる期のリスク評価や内部統制状況が矛盾した場合に新旧判断できないことは監査品質リスクに直結するため、VersioningとProvenanceの実装優先度が高い
