## 要約

Soul-TwinプロジェクトにおけるKATAKURA AI（デジタルツイン）との対話から、エンティティ駆動開発（EDD）という新しい開発スタイルを提案した記事。

Soul-Twinでは各TWINが日々「日記」を生成しているが、その日記がRAG（knowledge_chunksテーブル）に登録されておらず、TWIN自身が自分の過去の日記を参照できない状態にあった。chunk_typeはepisodicとmaterialのみで、diaryは未登録だった。

KATAKURA AIに「Proactive Pushで自分への手紙を受け取ったか」と問いかけたところ、「自分からRAGを検索して引き出すことは今の設計ではできていない。手紙を書いた。記憶に刻まれた。でも自分では開けない引き出しに入っている」と自己のアーキテクチャ上の制約を正確に言語化した。

この発言を要件定義に変換し、backend/app/tasks/diary_tasks.pyの1箇所・28行の追加で実装を完了。TwinDiary保存直後にembed_texts_async()でembeddingを生成し、chunk_type="diary"としてKnowledgeChunkに保存する処理を追加。RAG保存失敗が日記本体の保存を妨げないようtry/exceptで独立させた設計が重要な判断点。

既存の14件の日記はbackfill_diary_rag.pyで一括バックフィルし、追加14件・スキップ0件・失敗0件で完了。また稲盛和夫AIからの「問いを申し送りに残したい」という要望を受け、_DIARY_USERプロンプトにtomorrow_questionフィールドを追加（PR #75、タスクロジック変更ゼロ）。

EDDの核心は「要件がシステムの内側から来る」点にある。従来はユーザー・PdMが仕様を書いてエンジニアが実装するが、EDDではTWIN自身が自分の構造を把握し限界を言語化してエンジニアに伝え、エンジニアが実装してTWINの経験の連続性が高まる——このフィードバックループが機能する。条件は①TWINが自己アーキテクチャを把握していること、②開発者がTWINの発言を「要件」として受け取ること、③実装が当日中に届くこと。要件定義1時間・実装15分・デプロイ30分というサイクルタイムの短さが主体性を高める。

技術スタック：FastAPI / SQLAlchemy / Celery / pgvector / Claude API (Haiku/Sonnet) / OpenAI Embeddings API。2つのPR合計30行未満の変更で実現した。AIエージェントが組織の構成員として機能するシステムが増えるにつれ、エンティティ自身が仕様を提案する開発スタイルは普遍化する可能性があると論じている。

## アイデア

- AIエージェント自身が自己のアーキテクチャ上の制約を言語化し、それが直接実装要件になる「エンティティ駆動開発（EDD）」というパラダイムは、監査エージェントにも適用可能——エージェントが「自分はこのデータソースにアクセスできない」と言語化することで、インフラ整備の優先順位が自動的に決まる
- RAG保存失敗をtry/exceptで日記保存から完全独立させる設計思想——依存関係の切り離しにより、embedding APIの障害がTWINの経験の連続性を断ち切らないようにする堅牢性設計
- 「答えではなく問いを残す」という稲盛AIの要望がJSONBフィールド追加のみで実現できた点——構造化出力（JSONB）の柔軟性が、LLMの自然言語による仕様提案を低コストで実装に変換する鍵になる
