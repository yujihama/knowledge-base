## 要約

本記事は、AIエージェントのツール定義（JSON Schema）設計における7つの原則を、Anthropic・OpenAI双方の2026年時点の公式ドキュメントに基づいて体系化したものである。

【原則1：命名規則】ツール名は「サービス名_動詞_対象」形式（例: github_list_prs）で設計し、正規表現 ^[a-zA-Z0-9_-]{1,64}$ に準拠させる。ツール数が20を超えると命名の一貫性がツール選択精度に直結する。

【原則2：説明文】「何をするか・いつ使うか・いつ使わないか・何を返すか・制約」の5点を3〜4文以上で記述する。Anthropicは説明文を「ツール性能における最も重要な要素」と明言しており、1ツールあたり100〜200トークンが目安。

【原則3：パラメータスキーマ】enumで不正な状態を表現不可能にし、required/optionalを厳密に分離する。strict: true を有効化する場合、全フィールドをrequiredに含め、任意フィールドは "type": ["string", "null"] で表現する。

【原則4：input_examples】JSON Schemaで伝えきれない任意パラメータの組み合わせや形式の慣習を具体例で補完する。Anthropicの報告では、input_examples追加により複雑なパラメータの正確な指定率が72%から90%に改善。3〜5個が推奨で、過剰追加はコンテキストウィンドウを圧迫する。

【原則5：レスポンス設計】生のAPIレスポンスをそのまま返さず、エージェントが次のアクション判断に必要なフィールドのみを抽出する。detail_levelパラメータ（concise/detailed）の導入により、Slackツールの事例では1メッセージあたり206トークンから72トークンへ約65%削減。結果が多すぎる場合は「全200件中上位10件を表示」のようなガイダンスメッセージを含める。

【原則6：ツール統合】create_pr / review_pr / merge_pr を別々のツールとして提供するのではなく、actionパラメータを持つ1つのgithub_prツールに統合する。ツール数削減により、LLMのツール選択精度が向上する。

【原則7：Tool Search・動的発見】30個以上の大規模ツールセットでは、ツールの説明文をベクトル検索してタスクに必要なツールのみを動的にロードする設計が有効。全ツールをコンテキストに含めるとトークン消費と選択精度の両方が悪化する。

監査エージェント開発への示唆：監査エージェントでは証跡取得・リスク評価・レポート生成など複数の操作が必要となるが、これらをaction enumで統合したツール設計にすることで、ツール数を抑えつつLLMの選択精度を維持できる。また、detail_levelパターンは監査ログの「サマリー確認 → 詳細ドリルダウン」という自然なワークフローに直接対応する。

## アイデア

- detail_level（concise/detailed）パラメータによる2段階レスポンス設計で、トークン消費を65%削減しながら必要時のみ詳細情報を取得するパターンは、監査エージェントのログドリルダウンに直接応用可能
- input_examplesによる複雑パラメータの正確指定率を72%→90%に改善した事例は、単なるドキュメントではなくLLMのin-context learningとして機能していることを示している
- ツール数が20〜30を超えると全ツールをコンテキストに載せる設計が機能しなくなるため、ベクトル検索ベースのTool Search（動的ツール発見）が必要になるという閾値の存在
