## 要約

LLMや AIエージェントを業務システムに組み込む際、単なるメトリクス表示（処理件数・成功率・latency）だけでは不十分であり、人間が「確認・承認・差し戻し・rollback」できるワークスペース設計が必要であるという設計論。

ダッシュボードは「何が起きたか」を見る場所であるのに対し、ワークスペースは「次に何をすべきか」を判断・作業する場所として明確に区別される。ワークスペースの中心的オブジェクトは `WorkspaceCase` であり、`CaseStatus` として `new / generating / parsed / limited / needs_review / degraded / approved / rejected / executed / failed / repair_needed / rolled_back / compensated / closed` の14状態を持つ。業務単位（1チケット＝1case等）でLLM呼び出し複数回をまとめて追える設計が重要。

一覧画面では成功件数より「詰まっている理由」を見せることが運用上重要とされ、`missing`（足りない情報）・`reversibility`（reversible/compensatable/irreversible）・`owner`（次に見るべき担当者）の列を持つ `WorkspaceRow` 型を定義する。

詳細画面では AI の提案テキスト単体でなく、InputReceipt・ParseGuardReceipt・RetrievalReceipt・ReviewReceipt・Boundary・Reversibility を並べて根拠（grounds）と提案を分離表示することで、「実行してよい条件が揃っているか」を人間が判断できる構成にする。

Proposal の versioning では `active / superseded / approved / rejected` の状態を持ち、古い proposal を `superseded` として保持しつつ、backend の review gate でも `canApproveProposal` チェックを実施することで UI の disabled のみに依存しない二重防護を実現する。

DEGRADE ケースは単なる `failed` ではなく `reason / missing / next_actions / retry_after` を持つ `DegradeState` 型で表現し、「何をすれば再開できるか」を明示する。さらに `WorkspaceNotification` による通知・escalation（SLA残時間付き）と組み合わせることで、queue に放置されるリスクを防ぐ。

レビュー画面では Effect の before/after diff（例：`user.permissions: ["read"] → ["read", "admin"]`）を表示し、approve と execute を明示的に分離する。approve 後は effect gate → idempotency key 確認 → receipt 生成 → 実行という順序を経ることで、「承認したつもりが即実行」になる事故を防ぐ。

監査エージェント開発への示唆として、本記事の WorkspaceCase・ReceiptBundle・RollbackPlan・CompensationPlan の設計は、内部監査AIにおける調査結果の承認フロー・証跡管理・差し戻し機能に直接転用可能であり、特に effect-bearing な処理（是正指示の発行等）における approve/execute 分離と rollback 計画の事前作成要件は、監査コントロールの観点から重要な設計パターンとなる。

## アイデア

- Proposal に `superseded` 状態を持たせ backend でも承認不可とする二重防護パターンは、監査証跡の改ざん防止にも応用できる
- DEGRADE を `next_actions` 付きの作業可能状態として表現することで、エージェントの停止が運用ブラックボックスにならない設計思想
- approve と execute を明示分離し effect gate を介在させるアーキテクチャは、LangGraph の human-in-the-loop ノードと組み合わせた監査エージェント設計に直接マッピングできる
