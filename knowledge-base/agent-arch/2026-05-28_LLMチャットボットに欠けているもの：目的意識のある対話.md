## 要約

本記事は、現在のLLMベースのチャットボットが持つ根本的な限界として「目的意識のある多ターン対話（Purposeful Dialogue）」の欠如を指摘し、その評価・改善手法を論じる。

現行のベンチマーク（MMLU、HumanEval、MATH等）は単発応答の品質を測るものであり、長期的・反復的な対話の一貫性を評価していない。例えばSWE-benchのようなGitHub Issue解決タスクも、実際には人間との複数回のやり取りが必要であるにもかかわらず、ワンパス生成で評価される。

システムプロンプトは現在の主要な動作制御手段だが、研究により敵対的条件下では脆弱であることが示されている。著者らはこの問題を定量化するため、2つのLMエージェントを長期間対話させる実験環境を構築し、「コンシステンシー（一貫性）」を測定した。具体的には、8ターン対話の途中（例：5ターン目）に挿入プロンプトを入れ、元の指示をどれだけ維持できるかを評価した。GPT-4oでも挿入後に一貫性が低下することが確認された。

対話システムの歴史的な文脈も整理されている。1970年代のSchankの「レストランスクリプト」やELIZA・PARRYから始まり、現代のLLMは事前学習（次トークン予測）→対話フォーマット導入（apply_chat_template等）→RLHF（KLペナルティ、LoRA）という3段階で対話能力を獲得する。しかしRLHFは事前学習コーパスに比べて小規模であり、Lecunのケーキ比喩では「上に乗った小さなチェリー」に過ぎない。

目的意識のある対話をゲーム理論的に定式化する試みも紹介される。不完全情報ゲームとして対話を捉え、一方が情報を隠しつつ目標達成を図る構造（例：不正な払い戻しを引き出そうとするユーザー対顧客サービスBot）を扱う。この枠組みにより、エージェントの目的追跡能力・明確化質問能力・話題維持能力が評価可能になる。

解決策として著者は、(1)合成対話データを用いたファインチューニング、(2)ルールベース報酬を用いたRLの適用、(3)目的追跡・ユーザーモデリングのためのモジュラー設計、(4)評価者LMとしての活用を挙げる。監査エージェント開発においては、内部監査の問答プロセス（証拠収集・確認・交渉）が多ターン構造を本質的に持つため、目的一貫性の担保と明確化質問の自動生成は直接的な設計課題となる。

## アイデア

- 2つのLMエージェントを自動対話させることで任意長の合成対話データを生成し、システムプロンプト一貫性をストレステストする手法は、監査エージェントの長期インタビューシミュレーションに応用できる
- 対話を不完全情報ゲームとして定式化する枠組みは、ユーザーが隠れた意図（例：不正申告）を持つ場面での監査エージェント設計に直接適用可能
- RLHFは対話フォーマットが初めて訓練データに登場するステップであり、事前学習との乖離が一貫性低下の根本原因である可能性がある—これはLoRAターゲット層の選択戦略に影響する
