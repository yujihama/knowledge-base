## 要約

Cladeはクロードコードをベースにしたマルチエージェント開発フレームワーク。v1.4.0ではプランナーエージェントが計画を独立グループに分割し、各グループを専用のGitワークツリー上で並列実行する機能が追加された。

並列化の核心は2つの仕組みにある。第1に、Gitワークツリー（git worktree）による完全分離。各worktree-developerエージェントは別ブランチ上のリポジトリのフルコピーで動作するため、他エージェントの変更が見えず上書き衝突が構造的に発生しない。第2に、pre-toolフック（check-group-isolation.js）によるファイルオーナーシップの自動強制。Write・Edit・Bash rmの各操作をフック時点でインターセプトし、グループ外ファイルへのアクセスをブロックする。

ワークフローは4ステップで構成される。①plan-reportのYAMLフロントマターにparallel_groupsを定義（各グループはid・担当ファイルパス・タスクIDのリストを持つ）、②agent-developerがグループ数分のworktree-developerをバックグラウンドで一斉起動、③各エージェントがEnterWorktreeで自分のワークツリーに入り、ファイルオーナーシップを守りながらタスクを実装、④全エージェント完了後にmergerエージェントがベースブランチへ順次マージ。コンフリクト発生時はAskUserQuestionでユーザーに解決を確認し、成功後にワークツリーブランチを削除。

性能面では、6タスクを直列処理していた機能開発が3タスク×2グループの並列実行に変わり、所要時間は全タスクの合計ではなく最長グループの時間に近づく。ユーザーはplan-reportを承認するだけで、並列化の詳細はフレームワーク側が担う。

v1.4.0ではその他にも、バックグラウンドエージェントがダイアログを表示できない問題（承認フローを親コマンドへ移動）、エージェントファイル名の命名不整合（reviewer.md→code-reviewer.md）、マイルストーン制御の親コマンドへの集約など、バックグラウンド実行に関わる構造的な問題が修正された。

## アイデア

- Gitワークツリーをエージェント間の分離単位として使うアイデアは、共有ファイルシステムによる競合を構造的に排除する非常に実用的なアプローチ
- pre-toolフックによるファイルオーナーシップ強制は、エージェントへの「信頼はするが検証する」設計パターンの具体例——ガイドラインではなくハードな境界として機能する
- plan-reportのYAMLフロントマターでparallel_groupsを宣言する方式は、プランナーが依存関係を明示的に構造化し、オーケストレーターがそれを読み取るclean interfaceを形成している

## Yujiの取り組みへの示唆

監査エージェントシステムでLangGraphを使ったマルチエージェント設計を進めている場合、独立したサブタスク（例：証跡収集・リスク評価・レポート生成）を並列実行する際の干渉防止パターンとして直接参考になる。特にファイルオーナーシップをpre-toolフックで強制する設計は、LangGraphのノード間データ競合を防ぐためのガードレール設計に応用できる。mergerエージェントがコンフリクト時にユーザーへエスカレーションする仕組みは、監査エージェントにおける「人間のレビューが必要な判断」を構造的に組み込むhuman-in-the-loopパターンとしても参考になる。
