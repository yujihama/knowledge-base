## 要約

本記事は、AIエージェントを複数の役割に分割してチームとして運用する「AI チーム化」について、どの規模・条件から割に合うかを実践的視点で整理したポエム形式の考察。著者の結論は「チーム化は仕事の量と文脈が1つの会話から溢れたときに、処理と責任範囲を分けるための設計」であり、AI を賢くする手段ではない、という点にある。

**チーム化の固定コスト**：サブエージェントの作成には、役割定義・プロンプト設計・ツール権限の分離・出力フォーマットの統一・失敗時の切り分けロジック・メインによる統合ロジックという複数の設計コストが伴う。これはモデル切り替えのコストと構造的に同じであり、この固定費を回収できる仕事量がなければ割に合わない。

**チーム化が効く3つの具体例**：(1) Slack夜間ログの回収・分類・下準備：入力量が多く、夜間バッチ処理が可能で、タスク候補抽出・担当推定・優先度仮置き・朝サマリ生成と役割が明確に分解できる。(2) 大規模マイクロサービス群（100以上）の横断調査：コードベース・ログ・仕様・PR・監視情報を一つの会話に詰め込めないため、専門サブエージェント（探索/ログ/仕様/PR/影響範囲/妥当性）による文脈分割が必要になる。(3) 複雑なE2Eテスト運用：「実装バグか/テスト不備か/fixture不整合か/API契約変更か/環境不安定か/flaky か」という判断を繰り返す必要があり、テスト実行・エラー判断・修正・統合の4エージェント構成が有効。特にエラー判断エージェントの定型化に最大の価値がある。

**Claude Code Agent Teams との対比**：従来のsubagentsはメイン→サブの親子型でサブ同士は会話不可。Agent Teams（`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`で有効化）はteam leadと複数teammateが共有タスクリストを参照しながら進め、teammate同士が直接メッセージ可能なチーム型。ただし公式は実験的機能として、3〜5人規模・ファイル競合回避・research/reviewのような境界明確な仕事から始めることを推奨。

**チーム化が割に合う条件の整理**：(1) 入力が多く1会話に詰めると重い、(2) 毎日・毎週反復する、(3) 役割分割しても成果物が壊れない、(4) サブ出力を機械的に比較・統合しやすい、(5) 最終判断を人間かメインが握れる、(6) 失敗しても即座に大きな副作用が出ない。著者の現在地は「AI相棒」（単一エージェントとの対話）であり、まだチーム化の必然性はないと総括している。

## アイデア

- 「サブエージェントは無料の部下ではなく増えたインターフェース」という視点：エージェントを増やすほど入力仕様・出力フォーマット・ツール権限・統合ロジックの設計が必要になり、反復量がなければ回収できない。監査エージェント設計でも同様に、サブエージェント追加前に回収可能な反復量を試算する設計指針として有効
- エラー判断エージェントの先行価値：E2Eテスト失敗の修正より先に「これは直してよい失敗か」の分類を定型化するエージェントが最大価値を持つという指摘は、監査AIにおける「異常検知→原因分類→対応判断」の段階分離設計に直接応用できる
- チーム化の閾値を「並列にしたい量があるか」×「統合できる形で返せるか」の2軸で判断するフレームワーク：LangGraphでのマルチエージェント設計時に、各サブグラフの出力スキーマを事前に固定し、統合ノードで機械的にマージできる構造かどうかを設計基準にできる
