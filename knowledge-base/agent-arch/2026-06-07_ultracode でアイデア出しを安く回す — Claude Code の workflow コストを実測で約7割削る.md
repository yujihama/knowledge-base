## 要約

Claude Code の dynamic workflows（ultracode）は2026年5月にClaude Opus 4.8と同時公開されたプレビュー機能で、JavaScriptの取りまとめスクリプトを自動生成し、最大1,000体・同時16体のサブエージェントを並列実行できる。通常セッションより大幅にトークンを消費するため、本記事では架空の観葉植物ケアアプリを題材に「調査→発明→選別→採点→設計」の5段パイプラインを用い、コスト削減の効果を実測した。

最大の知見は3つのコスト削減レバーの優先順位だ。①工程ごとのモデル単価変更（最大効果）、②採点体数の削減（33%トークン削減）、③キャッシュの温め（実測で最大3.2倍の価格差）。全体のベースライン（全15体がOpus 4.8）をOpus換算332,327トークン・$2.07とすると、合わせ技ではOpus換算68%減・実測74%減の$0.53まで圧縮でき、実行時間も174秒→93秒（46%短縮）となった。

モデル単価はOpus:Sonnet:Haiku=5:3:1（入力単価$5/$3/$1 per 100万トークン）。調査・発明・採点をHaikuに落とし選別・設計のみOpusに残すと、コスト61%削減（換算）を達成。採点を6体並列から1体バッチに変えると固定コスト床（1体あたり約20〜23kトークン）を5体分節約でき、33%トークン削減・39%実測削減を得た。

重要な落とし穴として、並列同時発火するエージェントには「プレフィックス共有キャッシュ」が機能しないことを実測で確認。冷えた状態では6体それぞれが約21,400トークンのキャッシュ書き込みを実施し、共有は起きない。有効なのは同一ワークフロー内で近い時刻に同種エージェントをまとめ「温めて反復」する運用のみで、逐次実行時のみプレフィックス整列が効く。

品質面では、採点にHaikuを使うと「既存機能の組み合わせに過ぎない非自明な重複」の検出に失敗することを確認。明白な逐語重複は検出できるが、非自明な重複はルーブリック指示があってもHaikuでは見抜けなかった。SonnetはOpus並みの検出精度を持ちながら単価が0.6倍と安定しており、発明・採点ともにSonnetを推奨構成とする。コスト計測にはtotalTokens（変動係数0.7%未満と安定）ではなくOpus換算トークンを横断比較の主指標として使うべきで、混成モデル構成では新トークナイザの影響でtotalTokensが最大35%増加し作業量比較に使えない。

## アイデア

- サブエージェント1体あたり約20〜23kトークンの固定コスト床が存在するため、体数削減がトークン数削減の最大レバーになるという構造的洞察
- 並列同時発火するエージェントではプレフィックス共有キャッシュが機能せず、各体が個別にキャッシュ書き込みする——この制約を実測で定量化した点
- モデル混成構成での横断比較にはOpus換算トークン（入力単価比5:3:1の重みづけ）を使うことで、トークナイザ差異の交絡を除去できるという計測手法
