## 要約

サブスク枠内（Claude/ChatGPT月額固定）で個人開発を回す前提のもと、著者がLLMを使った並列開発構成（サブエージェント分担・複数worktree・PR経由マージ・自律エージェント運用）を試みた経験の記録。結論として、並列開発の構成自体よりも「並列に流せるタスクが手元にない」ことが本質的なボトルネックだったと指摘する。

並列化を試みた際の問題点として、(1) API従量課金が発生するためGitHub Actions等への組み込みが困難、(2) コンフリクト解消・PR管理・worktree進捗追跡など並列化のための認知コストが実装コストを上回る、(3) そもそも実装可能な粒度に分解済みのタスクが不足している、の3点を挙げる。

著者が現状採用しているのが「Ralph Loop」と呼ぶシンプルなローカルチケット駆動ループ。`docs/tickets/`ディレクトリ配下にMarkdownチケット（背景・ゴール・スコープ・非ゴール・受け入れ条件・検証方法を記載）を置き、シェルスクリプトが`claude -p`または`codex exec`に`prompts/ralph-loop.md`を渡して「チケット選択→最小実装→lint/test/build→完了処理→CHANGELOG追記→コミット」を1ループとして回す。ブロックされたチケットは`docs/tickets/blocked/`に退避してループ停止を防ぐ。GitHub Issue・ラベル運用・WBS・自律エージェントは使わない。

タスク生成側のフローとして、`product-map.md`に未整理アイデアを集積し、別LLMに`CHANGELOG.md`と`product-map.md`を渡して課題候補を生成させ、人間が最終判断でチケット化または棄却する。チケット化・設計判断は週末にまとめ、平日夜はRalph Loopによる実装に集中することで判断疲れを排除している。

監査エージェント開発への示唆：エージェントに渡すタスクの粒度設計（独立性・受け入れ条件の明文化・ブロック状態の分離）はReActエージェントのツール設計と同じ原則であり、エージェントが自律的に進めるためのタスク仕様書フォーマットとして直接転用できる。また、ボトルネックが「何をやるか決める」工程にあるという観点は、監査エージェントにおける手続き生成（要件定義フェーズ）の自動化が実装フェーズより優先度高いことを示唆する。

## アイデア

- LLMの実装速度向上により相対的に「タスク生成速度」がボトルネック化するという非対称性の発見：処理側を最適化しても供給側が詰まれば全体スループットは改善しない（リトルの法則的観点）
- Ralph Loopにおけるチケットの独立性要件（そのファイルだけ読めば着手可能）が、エージェントへのコンテキスト設計原則として機能している点：依存関係を事前に人間が解消することでエージェントの判断負荷を下げる
- 別LLMにCHANGELOG.mdとproduct-map.mdを渡して課題候補を生成させるメタループ構造：実装LLMと戦略生成LLMを分離することで、実装コンテキストの汚染なしにバックログを動的生成できる
