## 要約

OSSエージェントフレームワークである Agent 0 と Hermes のウェブ検索ルーチンを、実際のプロンプトおよびツール定義レベルで比較・分析した記事。両者ともに「検索→本文取得」という骨格は共通しており、Agent 0 は search_engine（バックエンド: SearXNG）＋document_query、Hermes は web_search（Exa/Parallel/Firecrawl/Tavilyから自動選択）＋web_extract という構成をとる。

最大の設計差はクエリ整形の扱いにある。Agent 0 はツール説明文（agent.system.tool.search_engine.md）に「自然文の質問にしない」「3〜10語の高シグナル語を使う」「フィラー語を落とす」という明示ルールを記述し、bad/good の例示対まで提供している。これにより、モデルが検索エンジンに投げるクエリが構造的に短く絞られ、ヒット精度が上がりやすい設計になっている。一方 Hermes の web_search 説明は site:、filetype:、-term といった演算子が「バックエンドによっては使える」と案内する程度で、整形ルールや例示はなく、クエリ品質はモデルの裁量に委ねられる。

デフォルト取得件数にも差があり、Agent 0 は SEARCH_ENGINE_RESULTS=10 で10件、Hermes は limit のデフォルト5件と2倍の開きがある。これが「Agent 0 のほうが当たりやすい」という体感の定量的な根拠の一つとなる。

Hermes はその代わり、システムプロンプト側でルーチンの堅牢性を担保している。mandatory_tool_use（現在の事実は記憶で答えず web_search を使う）、tool_persistence（空や部分結果なら別クエリで再試行）、missing_context（取得可能な情報は推測せず取りに行く）という3ブロックを明記しており、失敗時のフォールバック設計も整っている。web_extract は5000字未満はそのままMarkdown化、大きいページはLLM要約で5000字に圧縮、2M字超は拒否、失敗時はbrowserへフォールバックという詳細な分岐仕様を持つ。

監査エージェント開発への示唆として、ツール説明文にクエリ作法（整形ルール＋bad/good例示）を埋め込む手法は、Agent 0 の設計から直接移植可能。特に内部監査システムで「会計基準のバージョン検索」「判例・規制文書の取得」といった高精度検索が求められる場面で、クエリ整形の強制はモデル交換より先に試すべきチューニングポイントとなる。また、limit を5→10に引き上げ、web_search→web_extract の連結をシステムプロンプトで明文化するだけで Hermes 側の精度を Agent 0 水準に近づけられるという具体的改善策も提示されている。

## アイデア

- エージェントのウェブ検索精度差はモデル性能よりも「ツール説明文に書かれたクエリ整形ルールと例示の有無」で説明できるという実証的知見
- Agent 0 の bad/good 例示対をツール説明文に埋め込む手法は、任意のエージェントシステムの検索精度チューニングに即座に転用可能
- 検索デフォルト件数（5件 vs 10件）という設定値の差が体感精度に直結するという定量的な視点
