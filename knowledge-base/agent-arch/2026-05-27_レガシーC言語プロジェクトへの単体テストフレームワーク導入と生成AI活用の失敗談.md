## 要約

2025年末〜2026年初頭にかけて、稼働中のレガシーC言語プロジェクトへUnity/CMockによる単体テストフレームワークを導入し、ローカルLLMを活用したテストコード自動生成を試みた実践記録。

環境はWindows上でCygwin+NetBeans IDE 8.2という制約が強く、追加パッケージのインストール不可という縛りがあった。Unity/CMockによるMock自動生成は機能したが、カバレッジ出力にはlcovが必要であり、CygwinにはlcovがないためMSYS2を迂回路として採用した。しかしMSYS2でビルド不可なソースコード（Cygwin専用ヘッダを使用）が発覚し、最終的にlcov相当のHTMLレポート生成プログラムを生成AIと共同で3日かけて自作する羽目になった。

LLM活用の初期構成はVSCode+RooCode/Clineと、Qwen3-Coderやgptなどのモデルをllama.cppで動かす形だったが、tool callが不安定で無限修正ループに陥った。終盤はThinkStation PGX 2台クラスター構成+vLLMでgpt-oss-120bを動かし、エージェントをClaude Codeに切り替えた。Claude CodeはRooCode/Clineと比べtool callの動作が安定しており、設計書→コードという順序立てたフローに変更することでループ回数は減少した。

成果としては設計書生成の精度はそこそこ達成できたが、設計書→コード変換の品質が低く人手修正が必須となり、ビルドも人手対応が続いた。最終的に不具合3箇所を発見したところで作業中断。

反省点として、(1)AI活用開始前にルール策定を徹底すること、(2)設計書・指示書をAIとも共同レビューすること、(3)設計書生成とコーディングでモデルを使い分けること（gpt-oss-120b単独では限界）が挙げられている。当時MiniMax M-2.7があればtool call安定性の面で結果が変わった可能性があると著者は考察している。監査エージェント開発への示唆：制約環境下でのAIコーディング支援は、まずツールチェーンの動作保証を確立してからエージェントを投入しないと、デバッグ工数がAI活用メリットを上回るリスクがある。

## アイデア

- tool callの安定性がエージェントフレームワーク選定の最重要基準になるという実証：RooCode/Clineで失敗しClaude Codeで改善した具体的な事例
- 設計書→コードという段階分離アプローチにより1-shotプロンプトの無限ループを回避できる可能性があるが、それでもモデルの能力上限が品質ボトルネックになるという現実
- ローカルLLM環境でlcov相当ツールを生成AIと共同自作するという、OSS非依存のカスタムツール開発事例
