## 要約

AutoCAD用AIエージェントを個人開発している筆者が、国内建築・土木・電気分野で広く使われるフリーCADソフト「Jw_cad」をMCP（Model Context Protocol）サーバー経由でAIと連携させようと試みた記録。

Jw_cadには「外部変形」という拡張機構があり、ユーザーがメニューからバッチを起動すると、JWC_TEMP.TXTというテキストファイル経由でPythonなど外部プログラムと図形データをやりとりできる。ファイル単位の読み書き自体は問題なく動作した。

問題は、外部変形のトリガーがJw_cad側のユーザー操作に固定されており、外部から「外部変形を実行せよ」と命令するAPIが存在しない点にある。MCPの本質は「AIから外部システムへ能動的に働きかける」ことだが、Jw_cadはその「受け口」を持たない。

回避策として「予約型アーキテクチャ」を検討した。MCPツール呼び出し時にpending_job.jsonへ処理内容を書き出し、ユーザーが後から外部変形を手動起動した際にディスパッチャが読み取って実行する設計だ。論理的には成立するが、毎回ユーザーが手動でバッチを選ぶ必要があり、直接バッチを起動するのと実質変わらない。AI主導の自動操作という目標を達成できない。

もう一つの選択肢はRPA的アプローチで、WinAPIやUI Automationを使ってJw_cadのウィンドウにキーストロークやマウス操作をプログラム的に送り込み、外部変形メニューを擬似的に開かせる方法。技術的には実現可能だが、ウィンドウ位置・フォーカス状態・バージョン差異への依存が大きく、途中でモーダルウィンドウ等の想定外挙動が発生するとシーケンスが破綻しリカバリが困難になる。AutoCADの.NET API TransactionのようなロールバックをUI操作ベースで実装するのは非常に難しく、本番品質のプロダクトには不向きと結論づけている。

筆者の結論：Jw_cad外部変形によるファイル連携は動作するが、能動的トリガー手段の欠如がMCPサーバー化の根本的障壁。RPA方式は個人利用レベルなら許容できるが、配布・運用前提のプロダクトには品質面で厳しい。CADエージェント開発には「外部から能動的に呼べるAPI」と「失敗時の安全なロールバック機構」の2点が必須要件であることが、今回の試みで改めて明確になった。

## アイデア

- 「能動的トリガー手段」と「ロールバック機構」がCADエージェント設計の必要条件という整理は、監査エージェントにおける外部システム連携設計（例：ERPへの書き戻し操作）にも直接応用できる汎用的な視点
- 予約型アーキテクチャ（pending_job.jsonパターン）は非同期・人間介在型ワークフローの設計パターンとして、Human-in-the-loopエージェントの中間状態管理に転用できる
- 公式APIを持たない既存ソフトウェアをエージェントに組み込む際のアーキテクチャ選択肢（ファイル仲介・RPA・API拡張）とそのトレードオフが体系的に整理されており、レガシーシステムとのエージェント統合問題の典型例として参照価値が高い
