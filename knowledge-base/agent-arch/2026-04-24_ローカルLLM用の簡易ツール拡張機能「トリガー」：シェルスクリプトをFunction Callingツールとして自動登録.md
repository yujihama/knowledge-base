## 要約

DGX Spark上でのローカルLLM活用実験から生まれたアイデアとして、「トリガー」という機能が提案されている。Claude CodeやCodexのスキルはローカルLLMでは動作しないことが多い課題に対し、より単純な代替手段として設計された。

仕組みはシンプルで、ワークスペース内の `triggers/` ディレクトリにサブディレクトリを作成し、各サブディレクトリに `trigger.yaml`（ツール名・説明・ハンドラ定義）と `handler.sh`（実行スクリプト）を配置するだけ。AIアシスタントフレームワーク「xangi」起動時に `triggers/` 全体をスキャンしてFunction Callingのツール定義を自動生成し、LLMに登録する。ユーザーの発話がトリガー条件に合致すると、LLMがFunction Callingで対応するツールを呼び出し、`handler.sh` がワークスペースルートをcwdとして実行される。stdoutの内容がLLMに返され、最終応答が生成される。

xangiはClaude Code / Codex / Gemini CLI / ローカルLLMをバックエンドとして選択可能で、Discord・Slack経由での利用もサポートするAIアシスタントフレームワーク。Dockerによる環境隔離でのクイックスタートも提供しており、デフォルトモデルはGemma4:26b。

実例として2つのトリガーが紹介されている。天気予報トリガーは `wttr.in` にcurlでアクセスするだけの最小実装。テックニューストリガーはRSSフィードをcurlで取得し、`uv run` でPythonパーサを呼び出して上位5件を返す構成で、仮想環境なしでPythonを実行できる点もポイント。

スキルとの違いとして、スキルはプロンプトベースで柔軟性が高い一方、ローカルLLMでは安定しない。トリガーはFunction Callingを直接使うため再現性が高く、性能の低いモデルでも確実にツールを呼び出せる。ただし著者自身が認めるように、Qwen3.6-35B-A3Bのような高性能モデルの登場でスキルが十分機能し始めており、このアプローチの必要性は短命になる可能性がある。

監査エージェント開発への示唆：エージェントの信頼性を高めるために「LLMが誤りにくいツールインターフェースを設計する」という発想は監査ユースケースにも重要。ReActエージェントでツール定義のdescriptionを充実させることでモデルの能力に依存せず安定したツール選択を実現できる点は、LangGraphベースの監査エージェント設計でも応用可能。

## アイデア

- スキル（プロンプトベース）とFunction Calling（コードベース）の中間を埋める設計レイヤーとして、シェルスクリプトをFunction Callingツールに自動変換する仕組みは、モデル性能に依存しない安定したツール実行を実現する実用的アプローチ
- trigger.yamlのdescriptionフィールドがLLMのツール選択精度を左右するという設計は、プロンプトエンジニアリングをスクリプトレベルで分離・管理できる構造で、メンテナビリティが高い
- 性能の低いモデルに合わせてツールのインターフェースを単純化するという逆転の発想は、エッジ・オンプレ環境で高性能APIを使えない制約下での監査エージェント設計に直接応用できる
