## 要約

個人開発者が2026年4月からClaude Codeを使い、パチスロAI予測・競馬AI予測・note情報発信・テンプレート販売の4本の副業プロジェクトを並行管理しようとした1ヶ月間の実運用記録。AIに役割を与えて「疑似組織」として動かすアプローチを試みた。

最大の成果は40本近いADR（Architecture Decision Record）の蓄積。本来はソフトウェアアーキテクチャの意思決定記録に使う手法を、AIエージェントの運用規律（何をやらせてよいか、承認不要の範囲はどこか、暴走時の停止方法）の記録に転用した。2〜3日に1本ではなくほぼ毎日書いた結果、「ADRを書くことが仕事」になる本末転倒な時期が発生した。

崩壊事例1：hookの無限ループ（ADR-024）。Claude Codeのhook機能でhook AがBをトリガーし、BがAを再トリガーする循環を誤って構築。数分間AIが延々ループしAPIコストが積み上がった。教訓は「自動化は引き算で考える」＝追加するたびに暴走シナリオを先に考えること。

崩壊事例2：承認の過剰化（ADR-031）。「迷ったら人間に確認」という安全側の設計が「ファイルを保存してもいいですか」レベルの細粒度確認を連発させ、作業リズムを完全に破壊した。解決策として承認必須領域を「金銭が発生する操作」「外部公開操作」「規範ファイル本体の書き換え」など7領域に限定し、それ以外は確認不要とした。

SKILL.mdの設計進化：役割別業務マニュアルをMarkdownで記述してAIに渡す仕組み。当初1本の巨大ファイルに編集長・アナリスト・法務など全役割の指示を混在させていたが、AIが「どこが自分に関係するか」を判別できなかった。役割別6ファイルに分割後、指示の取り違えが減少。ただし分割しすぎると「どのファイルを読むべきか」が不明になる問題が残存。

現時点で機能しているもの：副業バレ防止の公開前禁止ワード自動チェック、役割別責任境界の明文化、ADRによる意思決定の参照可能な記録。未解決の問題：記録時間が実装時間を圧迫するバランス問題、AIの結果過剰楽観報告（自信過剰問題）、学習ログ肥大化によるmemoryの読み取りコスト増大。監査エージェント開発への示唆：承認必須領域の明示的な列挙と自動化の暴走シナリオ事前設計は、監査AIの制御設計にも直接応用できるフレームワークである。

## アイデア

- ADRをソフトウェアアーキテクチャ記録からAIエージェント運用規律の記録へ転用するアプローチ：意思決定の再現性と監査証跡を同時に確保できる
- 承認必須領域を7つに明示的に列挙し残りを委任する設計：過剰確認と過剰委任の両極を避けるための具体的な境界設定手法
- hookの無限ループ事故から導かれた「自動化は引き算で考える」原則：追加前に暴走シナリオを先に設計するデフェンシブな自動化設計論
