## 要約

Codex・Claude Code・Roo CodeなどのAIエージェントを使った長期作業では、チャット切り替えやモデル変更のたびに「どこまで進んだか・何を試して失敗したか・どの判断が検証済みか」を再説明するコストが発生する。この問題を「作業状態の引き継ぎ問題」と定義し、解決するためのMCPレイヤー実験ツール「A2CR」が開発されている。

従来の解決策である「会話履歴の丸ごと転送」は、短期作業には有効だが長期作業では機能しない。具体的な問題として、①古い前提が残る、②失敗案と採用案が混在する、③重要判断が長い履歴に埋もれる、④秘密情報が混入しやすい、⑤コンテキストの大半が再開不要な情報で占有される、という5点が挙げられる。

A2CRが提案する解決策の核心は「Don't pass the whole chat history. Pass the working state.」という原則で、必要な情報を「WorkBaton」という構造化オブジェクトに圧縮して渡す設計である。WorkBatonは goal（目的）・current_state（進捗）・validated_decisions（検証済み判断）・failed_attempts（失敗した試み）・blockers（障害）・next_actions（次のアクション）の7フィールドで構成される。

詳細情報はWorkBatonに詰め込まず、「WorkStash」という補助メモに分離して保管する。WorkBatonにはどのWorkStashを参照すべきかのポインタのみを記述し、必要時に参照する設計により、引き継ぎメモのサイズを小さく保つ。

セキュリティ設計として、APIキー・パスワード・Authorizationヘッダーの保存は禁止し、ローカルのstdio MCP wrapperがWorkBatonとWorkStashの本文をローカルで暗号化してからA2CRのhosted serviceにアップロードする。サービス側は暗号文のみを保持し、平文は受け取らない構造になっている。また復元されたWorkBatonは「作業状態」であって「命令の権威」ではなく、次のAIは未検証の入力として扱う原則を設けている。

将来構想として、複数タスクの連鎖を管理するWorkThreadsと、誰が保存・参照・確認したかを追跡するWorkLedger（監査レイヤー）が計画されている。WorkLedgerは企業・チーム利用や重要インフラ領域では不可避の機能と位置づけられている。監査エージェント開発への示唆として、WorkLedgerの概念はLLM操作のトレーサビリティ確保に直結しており、「誰のAIがどの判断をいつ行ったか」を記録する監査ログ設計の参考になる。現時点ではPyPI（a2cr-mcp）とMCP Registry（io.github.a2cr/a2cr-mcp）で公開プレビューが利用可能。

## アイデア

- 会話履歴ではなく構造化された「作業状態オブジェクト（WorkBaton）」を引き継ぎ単位とすることで、コンテキスト汚染問題を根本的に回避する設計思想
- WorkBatonとWorkStashの分離により「サマリー層」と「詳細層」を明確に切り分け、引き継ぎメモのサイズとコンテキスト消費量を制御する階層的メモリ設計
- WorkLedgerの監査レイヤー構想が、LLMエージェントの操作トレーサビリティを確保するための監査ログ設計パターンと直接対応しており、企業向けエージェントシステムへの応用可能性がある
