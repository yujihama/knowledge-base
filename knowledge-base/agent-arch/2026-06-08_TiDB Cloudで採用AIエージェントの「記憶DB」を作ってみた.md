## 要約

採用業務における情報の分散問題（求人票・候補者メモ・面接ログ・判断ルールが別々に存在）を解決するため、TiDB CloudをバックエンドとしたAIエージェント用記憶DBを構築した実装記事。

TiDB Cloudを採用した核心的な理由は、構造化データへのSQL検索と非構造テキストへのベクトル検索を同一DB上で実行できる点。従来の構成では「RDB＋ベクトルDB＋メモリストア」の3分割が必要だったところを、TiDB Cloudの`VECTOR(1536)`型カラムによって統合できる。

設計したテーブルは4つ：`jobs`（求人票）、`candidates`（候補者メモ）、`interview_notes`（面接ログ）、`agent_memories`（採用担当者の判断ルール）。各テーブルに`embedding VECTOR(1536)`カラムを持たせ、embeddingはOpenAI互換APIで生成してDB保存する。

`agent_memories`テーブルの設計が特徴的で、`importance`（重要度1〜5）と`expires_at`（有効期限）カラムを持たせている。エージェントの記憶は増えるほど良いわけではなく、古いルールや低重要度の記憶がノイズになるため、重要度・有効期限によるフィルタリングで「使える記憶」のみを検索対象とする。

検索フローは2段階のハイブリッド検索：(1) SQL絞り込み（職種・経験年数・勤務地・希望年収等の構造化条件）、(2) ベクトル類似検索（`VEC_COSINE_DISTANCE`関数で曖昧な志向性マッチング）。SQLとベクトル検索の結果を組み合わせてLLMに渡し、推薦理由・懸念点・次回確認質問・スカウト文面を生成する。

エージェントメモリの反映は、agent_memoriesテーブルから重要度上位・有効期限内のルールをベクトル検索で取得し、LLMのシステムプロンプトに動的に注入することで実現。これにより「未経験者より実務経験1〜3年を優先する」「スカウトでは一次面接から案内する」といった採用担当者固有の判断ルールが次回以降の回答に自動反映される。

監査エージェント開発への示唆としては、エージェントメモリのimportance/expires_at管理パターンが参考になる。監査エージェントでも「過去の判断基準」「リスク評価ルール」をメモリとして保持する際に、重要度と有効期限を設けることでメモリの鮮度管理が可能。また、TiDB CloudのようなHTAP（Hybrid Transactional/Analytical Processing）型DBは、構造化された監査データとテキスト形式の調書・指摘事項を同一DBで扱う構成に応用できる。

## アイデア

- agent_memoriesテーブルにimportanceとexpires_atを持たせ、重要度・有効期限でフィルタリングすることで「ノイズになる古い記憶」を排除するメモリ管理パターン
- SQL検索（構造化条件の絞り込み）とVEC_COSINE_DISTANCE（意味的類似検索）を同一DBで組み合わせるハイブリッド検索により、従来の3DB構成を1DBに統合できる設計
- 採用担当者の判断ルール（スクリーニング方針・スカウト方針等）をDBに永続化してLLMプロンプトに動的注入することで、エージェントの「個人化された記憶」を実現する手法
