## 要約

役割別AI（営業AI、法務AI、人事AIなど）を導入する際、システムプロンプトで役割名を変えるだけでは不十分であるという設計上の問題を論じた記事。核心的な主張は「AIの役割はキャラクター設定ではなく参照空間の設計によって決まる」という点にある。

多くの企業でのAI導入パターンとして、全社ドキュメント（Notion、Confluence、CRM、Slackログ等）をそのまま全AIに渡しつつ、プロンプトだけ「あなたは営業担当です」と変える実装が行われている。この場合、各AIは役割名と口調だけが異なり、参照空間は同一の未整理ドキュメント群となる。結果として、営業AIは法務の内部リスクメモを顧客向け回答に混入させ、法務AIは営業の希望的観測を契約判断に使い、CS AIは過去の特別対応を標準対応として案内するリスクが生じる。

著者はこの問題を「参照汚染」と呼び、RAGが接続する参照空間に古いドラフト、承認前文書、個人メモ、部門固有の例外対応が混在することで発生すると説明する。重要な区別として「検索可能性」と「使用許可」を分離する必要性を挙げており、文書が見つかることと、その文書を業務判断の根拠として使用してよいことは別の問題だと指摘する。

対策として、RAGの検索結果をそのまま推論に流すのではなく、役割・権限・目的に照らして使用可能な文書を選別するAIを中間に挟むアーキテクチャを提案している。これにより参照汚染を抑制できるとする。

設計思想としてBRM（Branching Reference Model）を提示し、役割別AIを「参照空間の枝分け」として捉える。各AIにどのNodeを参照させ、どのEdgeをたどることを許可し、どのDecision Linkを根拠として採用してよいかを明示的に設計することが本質だと論じる。

役割別AIを導入前に確認すべき事項として、責任範囲の定義、参照してよい正本の特定、過去資料の参照条件、ドラフトの扱い、例外対応の一般化防止、他部署情報の参照条件、人間確認が必要な判断の特定、出力の記録・追跡可能性、生成物を参照空間に戻す際のルール整備を列挙している。

監査エージェント開発への示唆：監査AIにおいても同様の問題が発生しうる。承認済みの監査手続き、過去の例外対応、ドラフト段階の内部通達が同一の参照空間に混在すると、監査判断が汚染される。内部統制評価AIや証拠収集AIでは特に、文書の権威レベル（正本/ドラフト/廃止済み）と適用範囲（全社/特定部門/個別案件）を参照制御の設計に組み込む必要がある。

## アイデア

- 「検索可能性」と「使用許可」の分離：文書がRAGで検索可能であることと、その文書を業務判断の根拠として使用してよいことを別次元として設計する概念は、アクセス制御とは異なる新しい粒度の権限管理を示している
- RAGの中間選別AIアーキテクチャ：検索結果を推論に直接流さず、役割・権限・目的に照らして使用可能文書を絞る専用AIを挟む二段階構成は、LangGraphのルーティングノードとして実装可能な具体的パターン
- BRM（Branching Reference Model）による参照空間の枝設計：AIの役割をNode・Edge・Decision Linkの組み合わせとして定義し、部門横断時には接続条件を明示する設計思想は、マルチエージェントシステムの認可モデルとして応用できる
