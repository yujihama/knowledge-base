## 要約

本記事は、ファッションドメインをテストケースとして、RAGシステムの設計思想を「要約器」から「ルール抽出エンジン」へと再定義する実験「trend-to-rule」の記録である。

出発点は水野学の著書『センスは知識からはじまる』の思想：センスとは才能ではなく、王道・普通を知ったうえで流行との差分を取り出す知識構造である。この考えをRAGアーキテクチャに落とし込むため、retrieval段階でドキュメントを「canonical（王道・定着知識）」と「emerging（流行・兆し）」の2種類に分離する設計を採用した。

通常のRAGは query → retrieve → summarize → answer というパイプラインで、出力は「今季はこの色が人気」のような短い要約に留まる。この形式は情報を短縮するが、「何が普通で何が新しいのか」「どこまで取り入れると自然か」という構造的判断材料を失う。

emergingクエリには「recency boost」を適用し、単なる最新順ではなく「relevance × recency」の複合スコアで、王道に対して最近ズレ始めている兆しを検出する設計とした。canonical側はrelevanceを重視して定着知識を確保し、この2レーンを分けることでLLMが両者を混合した自然文に溶かしてしまう問題を防いだ。

取り出す構造は common_rule（両者にまたがる再利用可能なルール）、tradeoff（条件によって揺れる部分）、suggestion（普通に少し良い差分の提案）の3層で、実装上はこれらをstructured intermediate artifactsとして保持してからuser-facing文章（解釈ルール）に変換する。RAGの出力をそのままfinal answerにせず中間表現を経由することで、構造が prose に溶けて失われることを防ぐ。

Vector DBは後に廃止しTavilyへ移行したが、canonical/emerging分離とrecency lensという設計原則は実装変更後も維持されている。この点が「実装は変わるが設計は残る」という本記事の主張の根拠となっている。

監査エージェント開発への示唆：本手法の「canonical（既存基準・過去判例）とemerging（新たなリスク事象・規制変化）を分離してretrieval → 共通ルール抽出」という構造は、内部監査のナレッジベース設計にそのまま転用可能である。監査基準（IIA基準、COSO等）をcanonical、最新のリスクトレンドや規制改正をemergingとして扱い、両者から再利用可能な監査判断ルールを抽出するパイプラインとして応用できる。

## アイデア

- retrieval段階でcanonical/emergingの2レーンに分離し、LLMへ渡す前に構造を保持することで、要約による情報損失を防ぐアーキテクチャパターン
- recency boostを「最新順ランキング」ではなく「王道に対する最近のズレを検出するレンズ」として定義し直す発想は、検索スコアに意味的役割を与える設計手法として汎用性がある
- RAG出力をfinal answerに直結させず、structured intermediate artifacts（common_rule / tradeoff / suggestion）を経由させることで、人間が判断するためのフレームを返す設計思想
