## 要約

本記事は、CLAUDE.md・Skills・MCPといったClaude Code固有のツール支援が使えない環境でも、LLMの構造的制約を理解することで同等の開発品質を維持できる「プロンプト駆動開発」手法を解説する。

著者が特に重視するLLMの構造的制約は3つ。(1) Context Rot：コンテキストが長くなるほど回答精度が低下する現象（Chroma Researchの研究に基づく）。(2) Instruction Decay：長いセッションで最初の指示の影響力が薄れる現象。(3) Sycophancy：明確な合格基準がないとLLMが「動いているから大丈夫」と甘い判断をする傾向。

これらに対処するため、著者は開発を4フェーズに分割する。準備フェーズでは「指示書（instructions.md）」を作成してプロジェクト文脈を明示的に注入（CLAUDE.mdの手動代替）。実装フェーズでは計画書を先に生成・レビューしてからコーディングを依頼し、Context Rotを防ぐ。テスト・品質フェーズでは実装前にE2Eテスト項目書を作成してSycophancyを防止。完了フェーズではチェックリストを外部化してInstruction Decayに対処する。

核心的な手法は「成果物をファイルとして永続化し、次ステップでパス指定してLLMに参照させる」こと。計画書を作ったチャットとは別の新しいチャットで実装を依頼することで、不要なコンテキストを排除してクリーンな状態で作業を開始できる。これはClaude Codeの/clearコマンドと同等の効果をもたらす。

MCPが使えない場合の外部情報取り込みは、Backlog・GitHub IssueのチケットをMarkdownに手動変換してローカルに保存する方式で対応。この手動変換はノイズを除去できる利点もあり「手動のContext Budget管理」と表現されている。

コミットメッセージの品質もLLM活用効率に直結する。`fix bug`のような曖昧なメッセージではgit logがコンテキストとして機能しないが、Conventional Commits形式（例: `feat(auth): add login flow (#123)`）で書かれていればLLMがIssue番号を辿って背景を把握できる。

AddyOsmaniのagent-skills（Plain Markdownで定義された開発ワークフロー）との対応関係も整理されており、手動プロセスをSkillとして定義することで、Claude Code・Cursor・Copilotなど複数ツールで再利用可能になる点も指摘されている。

## アイデア

- 成果物ファイルをチャット間の「コンテキスト転送媒体」として使う設計：計画書・指示書・チェックリストをファイルに書き出してパス指定することで、Skillsや/clearと同等の効果を手動で再現できる点が実用的
- E2Eテスト項目を実装前に作成することでSycophancyを構造的に封じる：合格基準を先に明示することで、LLMが「一応動いているから大丈夫」と判断する余地をなくすアンチパターン対策
- コミットメッセージ・ブランチ命名・PRテンプレートという「開発プロセスのメタデータ」がLLM活用効率に直接影響するという視点：git logをLLMのコンテキストとして機能させるためには、Conventional Commits形式とチケット番号の紐付けが前提条件になる

## Yujiの取り組みへの示唆

監査エージェント開発においてLangGraphのワークフローをフェーズ分割する際に、本記事の4フェーズ手法（指示書→計画書→テスト項目→チェックリスト）が設計指針として直接応用できる。特にSycophancy対策として「エージェントの判断基準を実装前に外部ファイルで定義する」アプローチは、LLM-as-judgeの評価基準設計にも転用可能。MCPが使えない組織環境（Backlog・CodeCommitなど）での監査ドキュメント取り込みを手動Markdown化で代替する方法も、Deloitte内部の制約環境での実務に参考になる。
