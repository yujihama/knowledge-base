## 要約

本記事は、著者が公開したAAEF（Agentic Authority & Evidence Framework）v0.1.3の設計思想と技術的構成を解説する。AAEFの核心は「Model output is not authority（モデル出力は権限ではない）」という原則で、LLMが「この操作を実行すべき」と出力しても、それは組織としての実行許可を意味しないという考え方に基づく。AIエージェントがDBを更新したりメールを送ったりファイルを削除したりと外部システムへ副作用を与える行為を扱う場合、従来の「モデルが安全か」という問いに加え、「その行為は認可・境界づけ・帰属・証跡検証が可能か」という問いが必要になる。AAEFはこの問いに答えるための4つの中核要素を定義する。①Delegated Authority：権限の委任元を明確にし、委任は縮小のみ許容（操作種別・対象リソース・有効期限・最大金額・再委任可否などを制約）。②Policy-Enforced Action Boundaries：モデル出力とツール実行の間にAuthorization LayerとTool Dispatch Layerを分離し、プロンプトインジェクション由来の命令が実行に到達しないよう防御。③Verifiable Evidence：単なるログではなく、agent_id・principal・authority_scope・policy_id・人間承認の有無などを含む構造化されたEvidence Eventを高インパクト行為に記録。④Revocable Trust：エージェントのidentity・instance trust・delegated authority・tool accessなどを動的に失効できる仕組みを必須とする。脅威モデルはT1〜T9の9カテゴリを定義し、「Delegation Escalation（委任連鎖で権限が膨張）」「Task Splitting to Avoid Approval（承認閾値を小分けで回避）」「Indirect Prompt Injection（外部コンテンツへの命令埋め込み）」などを具体例で説明。Control CatalogはGOV・ID・PRN・DEL・AUZ・TOOL・MEM・EVD・HUM・RESの10ドメインで計34コントロールを含み、CSVで管理しvalidatorも提供。NISTのRMF・CSFv2、OWASP LLM Top 10、MITRE ATLASなど既存フレームワークを補完する位置付けで、監査エージェントが外部ツールを呼び出す設計においてAction Boundary設計と証跡スキーマの参照元として直接活用できる。

## アイデア

- Authorization LayerとTool Dispatch Layerを分離することで、LLMの出力がそのままツール実行に変換される経路を断ち切り、プロンプトインジェクションによる実害化を構造的に防ぐ設計は、LangGraphのノード間でのツール呼び出し制御に直接応用できる
- 委任は『縮小のみ』という原則により、親エージェントが持たない権限を子エージェントに渡せないことを保証する。マルチエージェント連鎖でのスコープ爆発を防ぐ設計として、ReActエージェントのツール権限設計に取り込める
- Task Splitting to Avoid Approvalという脅威モデルは、単一アクションではなくAction Sequenceとして評価する仕組みが必要であることを示す。監査エージェントが複数の小操作を集約してリスク評価する機能への実装ヒントになる
