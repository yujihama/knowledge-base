## 要約

AIエージェントのセキュリティリスクは「モデルが悪いプロンプトに騙されるか否か」ではなく、エージェントを動かすハーネス層（設定・権限・ツール・MCP・hooks・外部入力の流入経路）の設計問題として捉えるべきという主張。Anthropic Hackathon Winner として公開された「everything-claude-code」リポジトリを事例に、AgentShield というセキュリティレイヤの設計思想を解説している。

AgentShield は「セキュリティエンジン for the harness layer」と自己定義し、主に3つのリスク面を監査する：(1) Harness configuration risk（CLAUDE.md / AGENTS.md / rules files / workflow prompts に含まれる unsafe defaults や bad permission boundaries）、(2) Foreign-data suspicion（issue・PR・Web・MCP説明文・hooksなど信頼境界外から入ってくるテキストの疑い）、(3) Tooling and MCP exposure（正当な自動化と危険なツール/MCP挙動の切り分け）。

everything-claude-codeリポジトリでは、AgentShieldはskills/security-scan/SKILL.mdとして定義され、/security-scanコマンドから呼び出せるほか、commands/verify.mdのpre-prモードにも「Full checks plus security scan」として組み込まれている。さらにGitHub ActionsのYAML例（affaan-m/agentshield@v1、min-severity: medium、fail-on-findings: true）も提供されており、ローカル確認→slash command→PR前検証→CIという連続した導線を持つ。

具体的な検出対象は、Bash(*)のような過剰なallow list、${file}補間によるhookのcommand injection、shellを実行するMCP、CLAUDE.md内のauto-run instructions、deny list不在、npx -yによる自動インストールなど。これらはアプリ本体のSASTではなく「エージェントがどう世界と接続されているか」を見るものであり、通常のdependency scanとは異なる観点。

セキュリティガイドの核心的主張は「agent security has to be treated as infrastructure」と「never let the convenience layer outrun the isolation layer」。新しいMCP追加・自律ループ・権限拡大・メモリ永続化・マルチエージェント並列化のたびに、隔離性・被害範囲・外部入力の信用判定・audit可能性を確認することが求められる。設計原則として最小権限・deny list・スコープ制限・実行境界分離を挙げており、モデルが判断を誤ることを前提に「その負け方がsurvivableになるように設計する」という姿勢が強調されている。監査エージェント開発においては、CLAUDE.md/hooks/MCPをコードと同等のレビュー対象に含め、AgentShieldのようなスキャナをPR前導線に組み込むことが直接適用できる実践として示されている。

## アイデア

- セキュリティチェックを「気をつける習慣」ではなくpre-PRフローに組み込むことで、忘れても自動的に実行される状態を作る設計思想（behavior by design vs. behavior by intention）
- Foreign-data suspicionという概念：外部テキスト（issue/PR/MCP説明文）を「悪意があると断定」するのではなく「信頼してはいけないかもしれない文脈」として扱うという判断フレームが監査エージェントの入力検証設計に直接応用できる
- スキャナ本体をOSSにして監査可能にし、チーム導入・自動化・ガバナンスを上位レイヤで有償提供するという分離モデルが、セキュリティツール自体の信頼性問題を解決している点
