## 要約

本記事は「ReAct エージェント適用域シリーズ」の第3作。前2作で業務AIを4象限（(1)スクリプト、(2)Algorithmic Search、(3)LLMワークフロー、(4)Autonomous Agentic Loop）に分類し、ReActが正当に必要なのは(4)象限のみという主張を展開してきた。本作はさらに「時間軸」という新しい分析軸を導入する。

業務には「設計フェーズ（柔軟性の最大化・探索的）」と「運用フェーズ（予測可能性の最大化・定型的）」の2フェーズがある。ReActの中核は「次に何をするかをLLMが動的に決める」ことであり、これが正当化されるのは「次に何をすべきか事前に分からない」状況、すなわち設計フェーズや探索フェーズに限定されると論じる。コーディング、Deep Research、未知環境の探索がその典型例だ。

本番運用フェーズでは予測可能性・ログ追跡可能性・コスト安定性・責任帰属可能性が求められる。「顧客対応のように毎回新しい問い合わせが来る業務は動的では？」という反論に対しては、処理経路は有限の型（FAQ対応・専門知識検索・ルーティング・エスカレーション）に収束するため、(3)LLMワークフロー象限の単機能LLM関数や専門chat agentとして事前定義できると反論する。

本番運用中に既存経路に収まらない新パターンが現れた場合、ReAct流の「動的対応」ではなく、一時的に人間判断でエスカレーションしつつ設計フェーズへフィードバックしてワークフローを更新するサイクルが正しいアーキテクチャとされる。

三部作の総括として、エージェントエコシステムの混乱は3層構造になっていると指摘する。①象限の誤適用（(3)象限業務に(4)象限アーキテクチャを被せる）、②語彙の不在（(3)LLMワークフロー象限を肯定形で呼ぶ業界用語がない）、③時間軸の混同（設計と運用を同一システムに圧縮する設計思想）。③が根本原因で、②③が連鎖して①が生じる。

監査エージェント開発への示唆として、本論は重要な設計指針を与える。監査業務において、リスク評価ロジックや証拠収集フローは「設計フェーズで腑分けを完了させ、(1)+(3)象限の組み合わせで運用フェーズを動かす」構成が適切。本番監査フローにReActを直接組み込む設計は、責任帰属（attribution gap）とredict不能性のリスクを抱えており、LangGraphでの明示的なワークフロー定義が推奨される。

## アイデア

- 設計フェーズ（探索的・柔軟性最大化）と運用フェーズ（定型的・予測可能性最大化）を分析軸として導入することで、ReActの正当適用域を「象限」ではなく「フェーズ」で定義し直す点が鋭い。コーディングエージェントが正当なのは『設計フェーズに常駐する道具』だからという説明は、現場のReAct乱用問題を整理する強力なフレームになる
- 本番運用中に新パターンが出現した際の処理を『設計フェーズへのフィードバックループ』として明示的に設計する考え方は、CI/CDにおけるincidentレビューサイクルと同型であり、エージェントシステムの継続的改善プロセスとして実装可能な具体的指針
- エージェントエコシステムの混乱を『象限誤適用→語彙の不在→時間軸の混同』という3層の因果連鎖として構造化した分析は、業界の設計議論をメタレベルで整理するフレームワークとして有用。特に『語彙が無いから消去法でReActに流れる』という観察は、LangGraphのようなワークフロー定義ツールが普及する価値を再定義する
