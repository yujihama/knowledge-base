## 要約

Databricks の Hongwen Song・Song (Vinson) Wei による論文「More Skills, Worse Agents? Skill Shadowing Degrades Performance When Expanding Skill Libraries」（Agent Skills '26 ワークショップ）の解説記事。スキルライブラリの規模拡大がエージェント性能に与える影響を実験的に測定した研究。

実験設計はシンプルで、特定タスクの成功率を向上させると確認済みの小規模ライブラリと、そこにスキルを追加した大規模ライブラリ（最大202本）を比較する。評価データセットには SkillsBench を使用。結果として、スキルを202本まで増やすと成功率が最大21ポイント低下（2モデル平均）した。追加されたスキルはいずれも破損や悪意ある設計ではなく、別のタスクで有効に機能する正常なスキルである。

低下の原因を論文は2種類に分解する。①context overhead（文脈オーバーヘッド）：スキル数増加により名前・説明文がコンテキストを圧迫し、正しいスキルを選択できても実行品質が落ちる。②skill shadowing（スキルシャドウイング）：説明文が類似した別スキルが本来選択すべきスキルを覆い隠し、誤選択が発生する。

直感に反する知見として、21ポイント低下の内訳は shadowing が約68%、context overhead が約32%であり、shadowing が支配的な原因だった。さらに context overhead は信頼区間がゼロをまたいでおり統計的確定には至っていない。

具体例として mario-coin-counting タスクが示される。コインカウントタスクに対し、ライブラリ内の video-frame-extraction スキルが説明文の類似性により26回全試行で誤選択された。スキル選択はスキル本文（SKILL.md の内容）を読む前に名前と description だけで行われるため、中身を丁寧に書いても誤選択は防げない。

モデル別の崩れ方も異なり、Haiku 4.5 はスキルを選択しなくなる方向、Sonnet 4.6 は誤選択し続ける方向に劣化した。

実運用への示唆として、description を互いに重複しないよう設計すること、「入力が動画か静止画か」「成果物がフレーム画像か数値か」「使ってはいけない場面」まで description に明記することが推奨される。また、全スキルをコンテキストに積むのではなく、検索で候補を絞ってから提示するアーキテクチャも次善策として言及される。

監査エージェント開発への示唆：監査手順スキルを複数設計する際、スキルの正しさは単体で決まらず「タスク×ライブラリ全体の構成」の関係性の中で初めて確定する。コントロール評価・リスク識別・証跡収集など説明が近接するスキルを並べる場合、description の段階で適用境界を明示しないと shadowing による誤実行が起きる。

ただし本論文はワークショップ論文で規模は控えめ（202スキル、2モデル、単一プロバイダ）であり、さらに大規模なライブラリや他アーキテクチャへの一般化は未解決と著者自身が述べている。

## アイデア

- スキルの「正しさ」は単体の性質ではなく、タスクとライブラリ全体の構成との関係によって初めて決まるという相対的な正しさの概念
- 誤選択がスキル本文を読む前（名前とdescriptionのみ）で発生するため、中身の品質より選択インターフェースの設計が性能を左右するという逆転
- モデルサイズによって劣化パターンが異なること（小モデルは諦め型、大モデルは誤選択型）は、スキルライブラリの規模設計をモデルに応じて変える必要性を示唆する
