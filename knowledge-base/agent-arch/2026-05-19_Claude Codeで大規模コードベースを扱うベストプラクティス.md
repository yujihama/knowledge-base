## 要約

Claude Codeはファイルの読み書き・コマンド実行・自律的問題解決が可能なエージェント型コーディング環境であり、数百万行規模のモノレポやレガシーシステムでも実運用されている。本記事はその大規模コードベースでの効果的活用法を7つの観点でまとめたものである。

最重要課題はコンテキストウィンドウ管理で、会話履歴・読み込んだファイル・コマンド出力がすべてトークンを消費し、枯渇するとClaude自身の指示理解・記憶能力が低下する。この制約を前提に各プラクティスが設計されている。

①検証手段の提供：「validateEmail関数を書き、実行後にテストを走らせて成功を確認せよ」のように成功条件をプロンプトに組み込み、Claudeが自己検証できる形にする。曖昧な指示は品質劣化に直結する。②探索→計画→実装の3ステップ：Shift+TabでPlan Modeに入り先にファイル読解と変更計画を確定させてから実装に移ることで、見当違いの実装を防ぐ。③具体的コンテキスト付与：ファイルパス・エラー全文・期待値（100ms以下のクエリ等）を明示する。④CLAUDE.md活用：プロジェクト構造・コーディング規約・ビルド/テスト/リントコマンド・主要依存ライブラリをルートのCLAUDE.mdに記述しておくことで、会話ごとに再説明する手間を省く。⑤新規会話の積極的開始：コンテキスト消費が増えレスポンスが遅延したり指示を忘れ始めた時は新会話を開始する。Claude Codeはファイル・Git履歴等のプロジェクト状態を会話をまたいで保持するため作業継続性は損なわれない。⑥サブエージェント活用：「リポジトリ全体から未使用インポートをすべて検出せよ」のような広域調査はサブエージェントに並列実行させることでコンテキスト効率と速度を両立できる。⑦スクリーンショット活用：UI変更作業ではスクリーンショットを添付して視覚的差異を伝達し、変更後の確認も画像比較で行うことで言語的曖昧さを排除できる。

監査エージェント開発への示唆として、サブエージェントによる並列調査パターンはLangGraphのマルチエージェントオーケストレーションと設計思想が共通しており、コンテキスト管理戦略はLLM-as-judgeや長大なログ処理を伴う監査フローの設計にそのまま転用できる。

## アイデア

- コンテキストウィンドウ枯渇を『忘却』ではなく『性能低下』として捉え、管理戦略をツール設計に組み込む発想は、長期タスクを扱うエージェントアーキテクチャ全般に応用できる
- CLAUDE.mdによるプロジェクトコンテキストの外部化は、エージェントのシステムプロンプトを動的に構成するRAGパターンと本質的に同じ構造であり、知識の揮発性問題への解法として汎用的
- サブエージェントへの並列委譲パターンは、監査エージェントにおいてリスク領域ごとにワーカーエージェントを分散させ結果をオーケストレータが統合するLangGraphの設計と直接対応する
