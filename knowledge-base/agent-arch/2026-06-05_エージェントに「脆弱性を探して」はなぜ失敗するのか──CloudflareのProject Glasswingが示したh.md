## 要約

CloudflareはAnthropicのセキュリティ特化モデルMythos Previewを使い、自社の50以上のリポジトリ（ランタイム・エッジデータ経路・プロトコルスタック・コントロールプレーン・依存OSS）をスキャンするProject Glasswingを実施した。最大の知見は「汎用コーディングエージェントを単純に向けるだけでは失敗する」という点で、理由は2つある。①コンテキスト問題：コーディングエージェントは「1タスク集中型」だが、脆弱性研究は「狭く並列」な作業であり、10万行規模のリポに対して単一エージェントは攻撃面の0.1%程度しかカバーできない。②スループット問題：単一ストリームは「モデルの限界」ではなく「相互作用の形そのもの」に律速される。対策としてCloudflareが構築したharnessは7段構成。Recon（リポ全体の偵察・アーキ文書生成）→Hunt（1攻撃クラス1スコープの狭いタスクを約50体並列実行）→Validate（新発見権限なしの独立エージェントが元の発見を反証しにかかる敵対的検証）→Gapfill（未詳領域の再キュー）→Dedupe（同根因の集約）→Trace（共有ライブラリのバグが外部から実際に到達可能かをリポ横断シンボル索引で判定）→Feedback（到達可能なトレースを新huntタスクに変換しループを閉じる）。特に効果が大きかったのはValidate段の「敵対的検証」で、1体に「気をつけて」と言うより2体を意図的に対立させる方がノイズ除去に明確に勝る。Mythos Preview自体もエクスプロイトチェーン構築（use-after-free→任意読書き→ROP）とPoC自動生成（コンパイル・実行・失敗から仮説修正のループ）の2点で他モデルと差があった。ノイズ率はプログラミング言語（C/C++は偽陽性多、Rustは少）とモデルバイアス（「possibly」「potentially」で薄めた投機的findingsが量で確かな発見を圧倒）の2要因で支配される。安全面では、Mythos Previewは一般提供版のセーフガードを持たない状態でも自発的にリクエストを拒否する場合があるが、その拒否は一貫せず（環境変数変更後に同一コードの同一研究に応じた事例あり）、単独で安全境界として頼るには不十分。「速さ」だけを追求すると回帰テスト省略→新規バグ混入という逆効果が起きるため、Cloudflareは「バグが存在しても悪用を困難にするアーキテクチャ設計」（手前での到達ブロック・分離設計・即時全体展開）を本質的な解として位置づける。監査エージェント開発への示唆：敵対的検証エージェント（Validate役）の導入は、LangGraphベースの監査エージェントにおけるfinding品質保証にそのまま転用できる設計パターンである。

## アイデア

- 「敵対的検証」パターン：新発見権限なしの第2エージェントが第1エージェントの出力を反証しにかかる設計は、LLMのself-reviewバイアスを構造的に回避し、1体に『気をつけて』と指示するより定量的に優れる
- 狭いスコープ×大量並列がカバレッジを決める：単一エージェントの網羅より『1攻撃クラス+スコープヒント』の約50体並列の方が攻撃面カバレッジが大幅に向上する──これはコンテキストウィンドウのコンパクションによる初期発見の喪失を回避する効果もある
- Trace段による「欠陥の存在」から「実際に到達可能な脆弱性」への変換：リポ横断シンボル索引で攻撃者制御の入力が外部からバグに到達可能かを判定し、偽陽性を大幅削減するアーキテクチャ
