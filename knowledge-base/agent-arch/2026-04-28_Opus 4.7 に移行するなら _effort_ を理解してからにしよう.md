## 要約

Claude Opus 4.7 は 4.6 から価格据え置きで 14 ベンチマーク中 12 で性能が向上しているが、単純なモデル ID 差し替えでは動かない破壊的変更が複数ある。最大の変更点は `effort` パラメータの厳格化と、新レベル `xhigh` の追加だ。

`effort` は Claude が応答にどれだけトークンを費やすかを制御するパラメータで、`low < medium < high（デフォルト）< xhigh < max` の 5 段階がある。思考トークンだけでなく、ツール呼び出し回数・本文長・コメント充実度まで一括で変化する。`xhigh` は 30 分以上の長時間エージェント・反復ツール呼び出し・詳細な検索処理向けで、Anthropic はコーディングとエージェント用途の推奨開始点として位置づけている。`max` は「真のフロンティア問題のみ」であり、過剰思考により JSON スキーマ逸脱などの品質低下が起きることも明記されている。

主要な破壊的変更は 3 つ。①`temperature` / `top_p` / `top_k` を指定すると HTTP 400 エラーが発生する。②`budget_tokens` による思考制御が廃止され `adaptive` + `effort` への置換が必要。③トークナイザー更新により同一テキストで最大 1.35 倍のトークンを消費する。

ユースケース別の推奨 effort は以下の通り。Playwright E2E の探索的テスト・新規シナリオ作成・デバッグは `xhigh`、安定シナリオ実行は `high`、単純スモークは `medium`、flaky テスト根因調査は `max` を一時的に使用。Amplify ビルド監視ではステータスポーリングに `low/medium`、ログ解析・原因特定に `xhigh`、CFn 循環依存などの難案件に `max` と、フェーズごとに effort を切り替える設計が推奨される。

コスト観点では、軽量タスクを `medium` に落とすだけで月 $1,000 の AI 原価に対し $150〜300/月 の削減余地がある。タスク種別・トラフィック量・品質要求の 3 軸でマッピングし effort を設計する。

Claude Code（Managed Agents）では effort は自動管理され、ユーザー側からの指定・観測は不可能。`/fast` コマンドは effort 調整ではなくモデルを 4.6 に切り替えるだけ。細かく制御したい場合は Messages API を直接呼び出し `output_config: { effort: "xhigh" }` を明示指定するしかない。`task_budget`（β機能）を併用するとエージェントの暴走抑制にも有効。

監査エージェント開発への示唆：LangGraph ベースのエージェントループで反復ツール呼び出しが多い処理（証憑検索・ログ解析・構造化レポート生成）には `xhigh` + `task_budget` の組み合わせが有効。`temperature: 0` による決定性担保のコードが残っている場合は本番移行前に必ず除去が必要。

## アイデア

- effort パラメータが思考トークンだけでなくツール呼び出し回数・本文長まで一括制御する「行動シグナル」である点は、エージェントのループ設計においてコストと品質のトレードオフを統一的に操作できる仕組みとして興味深い
- タスク種別 × トラフィック量 × 品質要求の 3 軸マッピングで effort を設計するフレームワークは、LLM 呼び出しコスト最適化の汎用的な思考ツールとして監査エージェントの各ステップ設計に直接応用できる
- Claude Code（Managed Agents）では effort が完全に自動管理され観測すら不可能という制約は、プロダクション品質が要求されるエージェントは Managed Agents に依存せず Messages API を直接叩く設計を強制するアーキテクチャ上の示唆を含む
