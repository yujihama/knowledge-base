## 要約

Headroom は Netflix エンジニアの Tejas Chopra 氏が開発した OSS で、LLM に渡すコンテキストをモデルへ届く前に圧縮してトークン消費を削減するプロキシ層。Python/TypeScript ライブラリ、プロキシサーバー、MCP サーバーの3形態で導入でき、Claude Code・Codex・Cursor・LangChain 等に対応する。

中核機能は CCR（Compress-Cache-Retrieve）。圧縮のたびに原文をローカルの SQLite へ保存し、圧縮済み出力には参照マーカーを埋め込む。モデルが原文を必要と判断した場合は `headroom_retrieve` ツールをハッシュ付きで呼び出して取り戻せる二段構え設計で、従来の単純な切り捨てや要約と異なり「情報の完全消失」を防ぐ意図がある。

圧縮手法はコンテンツ種別ごとに切り替わる。JSON は Rust 製 SmartCrusher で統計的に配列を間引き（先頭・末尾・エラー・変化点を保持）、100〜500件の配列で 70〜90% 削減。コードは AST 解析（Python/JS/Go/Rust/Java/C++ 対応）でシグネチャとハンドラを残しつつ関数本体・コメント・docstring を削除。テキストは ModernBERT ベースの軽量モデル Kompress で処理する。

公式ベンチマークはビルドログ 93.9%・JSON 配列 90%超という数字を掲げるが、実運用（2026年3〜4月の5万件超プロキシセッション、250以上の環境）での中央値は 4〜5%、平均でも 11% 程度。短い会話が大量に占めるため中央値は低く、ログやツール出力のような重い入力が来たときだけ削減率が跳ね上がる性質。「常時60%減」ではなく「重い入力への防御策」と理解するのが適切。

圧縮の実態は非可逆処理であり、要約的な間引きに近い。CCR の「可逆性」はアルゴリズム自体の可逆性ではなく、SQLite の原文キャッシュと retrieve ツールによる復元可能性を指す。プロキシとして全通信が経由するためセキュリティリスクが存在し、テレメトリはデフォルト ON のため `HEADROOM_TELEMETRY=off` で無効化が必要。また `--host 127.0.0.1` で外部公開しない設定も必須。

プロキシのオーバーヘッドは約 52ms だが、コンテキスト圧縮によりプロバイダ側処理時間が短縮されるため、テスト12シナリオ中11で「圧縮コストは削減で回収可能」とされている。GitHub Copilot が 2026年6月1日から AI Credits 従量課金へ移行し、Claude も同月15日に対話的利用と自律エージェント実行（Agent SDK、`claude -p`、GitHub Actions）でクレジット消費を分離したことが導入検討の直接的背景。一方で Claude Code v2.1.0（2026年1月）以降、重いツール出力を自動でファイル退避する機能が標準実装されており、基盤モデル側の改善によって Headroom の必要性が相対的に下がっている側面もある。

## アイデア

- CCR（Compress-Cache-Retrieve）の設計思想：非可逆圧縮と原文キャッシュの二段構えで「情報消失リスク」と「トークン削減」を両立する発想は、エージェントのメモリ管理アーキテクチャに応用できる
- 圧縮手法をコンテンツ種別（JSON/コード/テキスト）で切り替えるルーティング設計：監査エージェントで扱う構造化データ（ログ、JSON レスポンス）と自然言語（調書、議事録）を別経路で前処理するパイプライン設計の参考になる
- 実運用の中央値 4〜5% vs 公称 60〜95% の乖離：ベンチマーク数値を「重い入力の最良ケース」として読む重要性と、実際の効果測定に匿名テレメトリ（5万件超セッション）を活用した評価手法
