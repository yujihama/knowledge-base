## 要約

RAGやAgentを実装した後の品質評価に関する設計思想を論じた記事。評価軸を「忠実性50%・関連性25%・正確性25%」の3軸に分解することで、スコア低下時に原因を切り分けられる構造を提案する。忠実性（回答が検索文書に基づいているか）を最重視する理由は、企業利用においてハルシネーションが直接的な事業リスクになるためである。

忠実性の計測は単一手法では不十分とし、3手法の重み付け統合を推奨する：文単位の根拠確認（各文とコンテキストの含意関係を照合、重み0.4）、LLM-as-a-Judge全体評価（重み0.4）、エンティティ重複率（固有名詞・数値の文書側への存在確認、重み0.2）。性質の異なる3手法を組み合わせることで各手法の弱点を相互補完するポートフォリオ的アプローチである。

LLM Judgeについては2026年時点の研究（arXiv:2509.24678、arXiv:2601.21817）を引用し、参照なし単発採点はサンプリングによる揺れや較正の甘さが生じやすいと警告。複数Judgeの多数決でも同一偏りが残る問題を指摘し、スコアは絶対値でなく週次差分（例：0.78→0.82）として相対評価することを推奨する。

本番環境では正解データが存在しないという現実的課題に対し、4つの代替指標を提示：自己一貫性（同一質問の複数回実行によるばらつき測定、重み30%）、網羅性（質問意図のカバー率、25%）、明確性（構造・読みやすさ、20%）、外部整合性（既知事実との矛盾確認、25%）。RAGASやDeepEvalも同様のアプローチを標準機能として実装済みである点も言及。

評価をCIのように回す自動パイプラインを推奨し、構成は「テストケース準備→クエリ並列実行→3軸評価→時系列統計分析→改善提案生成」。テストケースには事実質問・複数文書横断質問・回答不能質問（hallucination検出用）の3種を含めることを必須とする。観測性ツールとしてArize Phoenix（OpenTelemetryベース）とLangSmith（実行ツリー記録）を紹介する。監査エージェント開発への示唆として、ハルシネーション検出を忠実性評価の最優先指標とする設計、および自動パイプラインによるデグレード検知は、内部監査領域でのLLMアプリ品質保証に直接適用可能な枠組みである。

## アイデア

- 忠実性を文単位根拠確認・LLM Judge・エンティティ重複率の3手法で重み付け統合するポートフォリオ設計は、各手法の弱点を相互補完し単一指標への過依存を避ける実践的アーキテクチャ
- 正解データなし環境での主役指標として「自己一貫性（同一質問の複数回実行によるばらつき）」を採用する発想は、モデルの確信度を間接測定する手法として監査エージェントのQA設計に応用可能
- LLM Judgeスコアを絶対値でなく時系列差分として評価する運用方針は、較正の甘さや偏りを相殺し、プロンプト変更の改善・デグレードを検知するCI的アプローチとして堅牢
