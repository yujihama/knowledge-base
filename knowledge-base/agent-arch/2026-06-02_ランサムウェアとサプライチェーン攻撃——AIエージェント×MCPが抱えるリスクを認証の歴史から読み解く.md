## 要約

本記事は、認証技術の歴史的進化を軸に、Model Context Protocol（MCP）とAIエージェントが抱えるセキュリティリスクを体系的に分析する。認証の進化は「秘密そのものを流す」から「署名だけを流す」への一本の軸で整理できる。人間向け認証ではパスワード→MFA→FIDO2/パスキーへ、システム向け認証ではシークレットアクセスキー→SigV4＋IAMロールへと進化し、いずれも「秘密鍵を外部に出さず署名のみを送る」設計が現代の正解となった。しかしMCPは現状この進化軸から遅れており、多くの実装が長期有効なAPIトークンを環境変数に保持するシークレットアクセスキー時代と同等の構造にある。MCPの公式仕様はOAuth 2.1を定義しているが、stdioトランスポート（ローカル起動）では環境変数での認証情報受け渡しが仕様上も想定されている。AIエージェントが侵害された場合のリスクは従来のシステム侵害と質的に異なる。通常のAPIキー漏洩では被害がそのシステムの機能範囲に限定されるが、エージェントはファイル操作・メール送信・カレンダー管理・コード実行・クラウドAPI呼び出しを横断的に「考えながら動く」ため、攻撃者がAIの判断力と行動力を丸ごと手にする形になる。Gartnerフェロー Leigh McMullenは2026年3月に「行動制限なしにAIエージェントに権限を渡すことは大惨事のレシピ」と警告している。脅威は2つのベクトルに整理される。脅威A（プロンプトインジェクション）は、正常なMCPと正常なエージェントが外部の悪意ある入力（ドキュメント・Webページ・メール本文）によって乗っ取られるリスクで、OWASPの「Top 10 for Agentic Applications 2026」でASI01（Agent Goal Hijack）として最上位に位置づけられる。脅威B（サプライチェーン攻撃）は、身元未検証のMCPをエージェント環境に追加した場合に、同一エージェントが保持する他の全MCPの認証情報に横断アクセスされるリスクだ。NISTも2025年3月公開の「AI 100-2」で「エージェントへの広範な権限付与は侵害時の被害半径を不釣り合いに拡大させる」と指摘している。筆者はこのギャップへの現実解として、OSのキーストア（Windows Credential Manager / macOS Keychain / libsecret）からトークンを取得して子プロセスの環境変数に注入するMCPランチャーを構築した。設定ファイルへの平文保存・Gitへの誤コミット・チャットログへのトークン漏洩を防ぐが、悪意あるMCPがツールレスポンス経由でトークンを外部送信するケースは防げない。MCPが連動して動く部分への対処はFIDO2的な署名ベース認証をMCP間に持ち込むフェーズ3の実装として後続記事に委ねている。監査エージェント開発への示唆として、複数システムにまたがる権限を持つエージェントでは最小権限原則の徹底と、MCPサプライチェーンの信頼性評価が不可欠であり、GRCフレームワーク上でのエージェント権限管理の設計指針として直接参照できる内容を含む。

## アイデア

- 認証技術の歴史（パスワード→FIDO2、シークレットキー→SigV4）を一本の進化軸「秘密が流れる→署名が流れる」で統一的に整理した分析フレームが、MCPのセキュリティギャップを可視化する強力な診断ツールになっている
- AIエージェントの侵害は従来のシステム侵害と「被害上限の性質」が根本的に異なるという指摘——通常は「そのシステムで何ができるか」が上限だが、エージェントは「AIが何を考えて動けるか」が上限になるという質的変化の概念化
- OSキーストアをトークン管理のバックエンドとして使いMCPサーバー実装を変更せずにラップするアーキテクチャパターンは、既存エコシステムへの非侵襲的なセキュリティ層の追加手法として汎用性がある
