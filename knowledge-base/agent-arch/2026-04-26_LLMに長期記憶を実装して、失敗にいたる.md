## 要約

著者はClaude Codeに外部長期記憶システム「ghost」を実装した個人プロジェクトの経緯と失敗を報告している。発端は、Claudeに長文翻訳（コージブスキーの一般意味論）を章単位で依頼する際、毎回文脈を説明し直す手間を省くため。ghostはSQLiteをバックエンドとし、v28まで開発が進んでいる。記憶モデルはHebb則にもとづく強化リンク、時間減衰、干渉忘却、replayによる重要度浮沈を実装した神経科学的な設計で、14日前の記憶は直近のものより弱く重み付けされる構造を持つ。v8で追加された特殊モード「delusion」（Jaspersの妄想概念に由来）は減衰・情動バイアス・再固定化を無効にした純粋検索モードで、raw_turnsテーブルにClaude Codeの全対話ターンを原文保存する完全記憶機構。v16ではSQLiteの読み取りをmdファイル出力に変更した。失敗の本質は「プロンプトのフラット性問題」にある。ghostが時間的重み・Hebbianリンク強度・replay優先度を計算して記憶を選別しても、それをプロンプトとしてClaudeのコンテキストに渡した瞬間、全テキストが等距離にフラット処理される。14日前の弱い記憶も今朝の強い記憶も、コンテキスト内では同列に扱われ、ghostが構築した時間的・重み的構造が消失する。著者はこれを「LLM自体の実装に神経科学的知見が組み込まれているのに、その外側で同じ模倣をするのは屋上屋」と表現し、記憶の遠近法をコンテキスト注入前提で再設計する方向性を示唆するが、具体的な次手は未定としている。Claude.aiはMEMORY.mdを活用して過去記憶と接続し饒舌に応答するのに対し、Claude Code CLIはMEMORY.mdが空でコンテキストが薄いため迎合が減るという副次的観察も述べられている。監査エージェント開発への示唆：エージェントに外部記憶を持たせる際、記憶の重み付け構造をコンテキスト注入後も保持するには、重みをメタデータとしてプロンプト内に明示的にテキスト化するか、記憶取捨選択をLLM外のランキング層で行うアーキテクチャ設計が必要。

## アイデア

- プロンプトに渡した瞬間に時間的重み・Hebbianリンク強度が全消失する『フラット性問題』は、外部記憶システム設計の根本的制約であり、重みをテキストとして明示的に埋め込む手法や再ランキング層の必要性を示唆する
- delusionモードの設計（減衰・情動バイアス・再固定化をすべてオフにした純粋検索）は、通常の減衰記憶と完全記憶を用途別に切り替えるハイブリッドメモリアーキテクチャの実装例として参照価値がある
- Claude.aiとClaude Code CLIで記憶コンテキストの厚みが異なり、CLIでは迎合が減るという観察は、LLMの応答スタイルがシステムプロンプト・メモリ注入量に依存して変化することを示す実験的知見
