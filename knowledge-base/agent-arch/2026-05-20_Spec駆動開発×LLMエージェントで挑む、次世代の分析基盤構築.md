## 要約

MBKデジタルのコンサルタントが、データ分析基盤構築プロジェクトに「Spec駆動開発」とLLMエージェントを組み合わせた手法を実践し、プロジェクト全体の工数構造と品質がどう変わるかを整理した記事。

従来型のデータ基盤構築では、要件定義を浅く切り上げて早期に実装へ移行するため、終盤に暗黙ロジックの発覚・既存帳票との数値差分・利用イメージのズレが集中し、手戻りが構造的に発生していた。LLMにより実装速度は向上したが、要件が曖昧なまま下流に速く進むだけで手戻りはむしろ拡大するという問題があった。

これを解決するため、同社は「データ基盤におけるSpec」を「業務・データソース・指標定義・集計ロジック・出力結果・分析軸を実装前に構造化した内部表現」と定義。Specセクションとして、ビジネス要件・指標定義・集計ロジック・データリネージ・結果の軸・Open Questions・改訂履歴を体系化し、推定ロジック・確定ロジック・不明点を分けて管理する運用原則を設けた。

LLMエージェントはファイル読み取り・SQL実行・スクリプト生成などのツール実行と次手の判断を自律ループさせる形で活用。具体的には、①議事録・既存資料の横断走査による論点抽出、②Excel/VBA/帳票の自律解析によるSpec初稿化、③SpecからのHTMLモックアップ・BI試作画面の生成、④Open Questionsの重複排除と顧客向け言い換え、⑤SQL・Dataform・BI・検証スクリプトの連携生成、⑥クエリ実行と結果分析の反復による差分原因仮説提示、の役割を担う。人間はレビュアー役として「指示→確認→修正方針提示」に集中できる。

最大の変化は、モックアップ確認が実装後ではなく実装生成の前に入ることで、完成後の手戻りを上流で防ぐ構造になる点。実装工数は明確に減少した一方、要件定義・Spec化・ヒアリング整理の時間は増加したが、これは従来も終盤の手戻りとして発生していた作業が上流に移動しただけであり、全体工数と終盤の不確実性は低下した。

差分検証の位置づけも変化する。既存帳票との数値差分は単なる実装ミスではなく「暗黙知の発掘装置」として機能し、顧客の認識違い・暗黙の例外処理・時期による定義変更・部署間の定義差を上流で発見しSpecに戻すことで、同じロジックを別マートで再利用でき、引き継ぎ資料としても活用可能になる。

監査エージェント開発への示唆として、本手法で蓄積されるSpecは業務ルールの構造化資産であり、LLM-as-judgeや監査エビデンスの自動照合エージェントが参照するナレッジベースとして直接活用できる。また「推定・確定・不明」を分けて管理するSpec運用原則は、監査証跡の確度管理やリスク評価の不確実性定量化にそのまま応用できる考え方である。

## アイデア

- 「推定・確定・不明」を分けてSpecに記載するという運用原則は、LLMの出力に対する確度管理と同型であり、監査エビデントの信頼度スコアリングに直接転用できる
- 差分検証を「終盤の火消し」ではなく「上流の暗黙知発掘装置」として位置づけ直す発想は、テスト駆動開発における失敗テストの役割と同じ認識論的転換である
- Specを全成果物の単一起点とし顧客向けには質問リスト・モックアップに変換するという設計は、内部表現と外部インターフェースを分離するRAGシステムのインデックス設計に類似した構造を持つ
