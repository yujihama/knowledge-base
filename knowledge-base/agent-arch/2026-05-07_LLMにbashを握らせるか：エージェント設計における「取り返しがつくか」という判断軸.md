## 要約

AIエージェント開発において、LLMにbash（シェルコマンド）を直接実行させる設計を採用するかどうかという問題を、リスクと利便性の両面から考察した設計論。

bashをLLMに握らせる最大のメリットは、自然言語指示からファイル操作・プログラム実行・ネットワーク操作・ツールインストールまでを一つのインターフェースで実現できる点にある。ユーザーが「このPDFをExcelにまとめて」と指示するだけで、LLMがpdftotext・Python等を組み合わせて実行できる。一方で、bashは「rm -rf /」のような一行で破滅的な破壊を引き起こせる。LLMが悪意を持っているわけではなく、「ディスク容量を増やすためにキャッシュを広めに消そう」といった小さな判断のズレが連鎖して取り返しのつかない状況を生む、という「悪意なき破壊」が本質的なリスクである。

Anthropicはこの問題に対し、自社ツール「Cowork」においてLinux VM内でコード・シェルコマンドを実行する三段構えのアーキテクチャで対応している。①bashの自由度はそのまま保持、②実行影響範囲をVMという箱に物理的に限定、③VM作業領域をセッション終了後に使い捨て、という設計により「強さと危うさのセット」を切り分けている。これはブラウザのJavaScriptサンドボックス・Docker・OS権限モデルと同系譜の「自由に動かしつつ影響範囲で縛る」パターンのLLMエージェントへの応用である。

ただしCowork方式が機能する前提は「壊れてもやり直せる」領域であること。コード生成・調査レポート・プロトタイプ構築はやり直しが効くが、本番DB・顧客情報・図面・会計データは現物を触らないと意味がなく、かつ間違いに気づけないまま数週間後に発覚するリスクがある。

著者はAutoCADエージェントを開発しており、CAD世界のbashにあたるAutoLISPをLLMに自由生成・実行させることを検討している。しかし業務で使う本物の図面は「寸法値の変化・レイヤ属性の変化がパッと見では分からず、製造工程に入ってから発覚する」性質を持つため、Cowork的な「箱に閉じ込める」発想が成立しない。99.9%の確率で意図通りのLISPが生成されても、残り0.1%の「気づけない壊れ」をどう扱うかという問いに答えが出るまでは踏み切れない、というのが現在の立場である。

設計の判断軸として「取り返しがつくか？」を先に問うことで、Cowork型サンドボックス方式が有効な領域と、動的コード実行の許容範囲を慎重に絞る必要がある領域を区別できる。監査エージェント開発においても、本番データや監査証跡を直接操作するツール設計では同様の「悪意なき破壊」リスクへの対処が必須であり、操作前後の差分記録・承認フロー・ドライランモードの設計が重要な示唆となる。

## アイデア

- 「悪意なき破壊」という概念：LLMは悪意を持たないが、小さな判断のズレが連鎖してrm -rf的な取り返しのつかない結果を生む構造は、監査エージェントが本番データを操作する設計における根本的なリスクモデルとして応用できる
- AnthropicのCoworkにおける三段構え（bash自由度保持・VM物理隔離・作業領域使い捨て）は、LLMエージェントへのDockerコンテナ・権限モデル等の既存セキュリティパターンの応用であり、「自由に動かしつつ影響範囲で縛る」という設計原則は汎用性が高い
- 「取り返しがつくか？」という判断軸でツール設計の方針が分かれる：Git管理下のコードはやり直せるが、図面・本番DB・会計データは気づけないまま書き換えられる可能性があり、この差異に基づいてドライランモードや差分承認フローの要否を決定できる
