## 要約

PerplexityのCTOやY CombinatorのCEOをはじめとするトップ開発者たちが、内部ツールからMCP（Model Context Protocol）を外し、CLIへ回帰しているトレンドを分析した記事。

CLI回帰の主な理由は2点。第1にトークン消費量の差。GitHub向けMCPサーバーは44ツールで約1683行・6.3万文字（約14,268トークン、約0.3円/リクエスト）のメタデータをプロンプトに乗せる必要があるのに対し、CLIはbashツール1つの定義（十数行）で済む。最新LLM（Claude 3.5 Sonnet、GPT-4o等）はgit・grep・gh・curlといった標準的なCLIコマンドを学習済みで、わざわざツール定義を教える必要がない。第2に実行効率。MCPは「LLM推論→ツール呼び出し→結果受信→再推論」の往復ループが必要で待ち時間が長い。CLIはパイプ（|）や論理積（&&）でLLMが一発で複数タスクをまとめたコマンドを生成でき、シェルが最後まで一気に実行するためラグが大幅に減る。著者の実験では処理時間が半分以下になったとしている。

一方でMCPが不可代替な領域も存在する。CLIはファイル名にシングルクォートやスペースが含まれるとエスケープ漏れで構文エラーになるエッジケースに弱いが、MCPはJSONスキーマで引数を構造化するためこの問題が起きにくい。また安全性の面では、bash全開放はrm -rf /相当の権限をAIに与えることと同義であり、企業クラウド環境では到底許容できない。MCPは「ファイル読み取りのみ許可」「DBのSELECTのみ許可」のようなサンドボックスAPIとして機能し、エンタープライズ向けでは引き続き主役となる。

結論として、今後のAgentエコシステムは用途による住み分けが進む。CLIは個人開発・ローカル軽量Agentの主役としてトークン節約・高速・低コストを武器に比重を拡大。MCPはJSONによる堅牢なパラメータ制御と高セキュリティを活かし企業向け・クラウド共有環境の安全弁として存続するが比重は縮小する。監査エージェント開発への示唆：エンタープライズ内部統制・監査ツールでは権限制限の観点からMCPが依然として適切だが、開発・検証フェーズのローカル実行やコスト最適化ではCLIベースのシンプルな設計が有効。トークン消費削減はLLM-as-judgeを多用する監査エージェントのAPIコスト管理にも直結する論点である。

## アイデア

- GitHub MCP Serverの44ツール定義が約14,268トークン（約0.3円/リクエスト）に相当するという定量データは、MCPスタック選定時のコスト試算の基準値として直接使える
- 最新LLMがCLIコマンドを学習済みであることを逆手に取り、ツール定義コストをほぼゼロにする『LLMの既存知識をツール定義の代替として使う』設計思想は、Agent軽量化の重要なアーキテクチャ原則
- CLIのパイプ結合によるバッチ処理とMCPの一問一答モデルの実行効率差は、Agent設計における『LLMとの通信回数を最小化する』原則の具体例として、タスク分解粒度の設計指針になる
