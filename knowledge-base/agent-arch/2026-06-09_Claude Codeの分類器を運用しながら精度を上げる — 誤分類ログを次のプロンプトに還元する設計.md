## 要約

Terraform planの差分（~行）をClaude Codeで危険度分類するシステムを運用しながら精度を向上させるフィードバックループの設計について解説した記事。初期プロンプトで完璧な精度が出ない理由として、①組織固有の文脈（IAM権限設計、タグ運用等）がプロンプトに反映されていない、②カテゴリ境界がリソースや属性値によって揺れる（cidr_blocksへの0.0.0.0/0追加はsecurityだが内部VPC限定のCIDR追加はoperationalに近いなど）、③Haikuモデルの判定揺れがある、の3点を挙げる。

正解データ（ground truth）はPRレビューで人間がカテゴリを訂正した事実から取得する。専用ラベリング作業は不要で、レビューの副産物として蓄積できる。ただし「訂正されなかった＝正解」ではなく、訂正された事実を強いシグナル、されなかった事実を弱いシグナルとして扱う。

誤分類はJSONL形式で構造化して記録する。各レコードにはPR番号、リソースアドレス、AI判定カテゴリ、人間訂正カテゴリ、変更属性名、理由、日付を含む。レビュアーの負担を最小化するため、PRコメント末尾の定型行をCIで拾ってappendする運用を推奨。

ログのプロンプト還元は2段階で行う。第1段階は直近8件の訂正をfew-shotとしてプロンプトに差し込む方法（jqで整形してCI側で置換）。第2段階は同一attr+訂正先カテゴリの組み合わせが3回以上繰り返した誤分類を固定ルールに昇格させる方法で、理由と日付を付与して後の見直しを可能にする。

KPIはsecurityやhidden-destructionの見逃し件数（偽陰性）を最重要指標とし、過剰検知（偽陽性）は許容範囲として許す非対称な設計。見逃しが増えた訂正は最優先でルール昇格させる。

精度向上の手順はプロンプト還元→モデル昇格（危険カテゴリ候補のみSonnetに回す）→ファインチューニングの順で、正解データが数百件貯まりかつ前2段階で頭打ちになった場合のみファインチューニングを検討する。監査エージェント開発への示唆として、誤分類ログをknown-failuresとして資産化し、レビュー副産物を教師データに変換するこの設計パターンは、LangGraphベースの監査判断エージェントの精度改善サイクルにそのまま応用できる。

## アイデア

- レビューの副産物（人間の訂正）を正解データとして構造化収集し、専用ラベリング作業なしにground truthを蓄積する設計
- 誤分類の頻度に応じてfew-shot（試行錯誤）から固定ルール（確定知見）へ段階的に昇格させる2段階還元の仕組み
- 偽陰性（危険見逃し）と偽陽性（過剰検知）のコスト非対称性を認識し、KPIを危険カテゴリの見逃し件数に絞る設計思想
