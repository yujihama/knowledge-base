## 要約

AutoCAD上で動作するLLMエージェントにおいて、エージェントの失敗操作が図面状態に残存する問題とその設計的解決策を論じた記事。

ステートレスなテキスト生成や無状態APIと異なり、CADのようなステートフルな外部環境では、LLMが途中まで操作を実行して失敗した場合、中途半端な図形が図面上に残留する。これが次のObserveフェーズで「正常な図面状態」として誤認識され、LLMが汚染された状態を前提に再計画するという負の連鎖が発生する。

解決策として「操作バッチ」という概念を導入する。LLMが生成した複数ステップの操作計画全体を1つの論理実行単位として扱い、AutoCAD .NET APIのTransaction機構（Commit/Abort）に乗せる。全ステップが成功した場合のみCommitし、1つでも失敗した場合はAbortして実行前の図面状態に巻き戻す（ロールバック）。

実装上のBatchExecutor.csでは、Validate(plan)をTransactionの外側に置いて実行前に検知可能なエラー（操作名不正、必須項目欠損等）を弾き、その後StartTransaction()でバッチ全体を囲む。失敗時はtr.Abort()でロールバックし、LLMにはrolled_back: trueを含む構造化エラー（失敗したステップ番号、操作名、エラー内容）を返す。これによりLLMは汚染された状態ではなくクリーンな実行前状態を前提に再計画できる。

さらに、コマンド実行成功とタスク達成は別物として、Commit前にアプリ側の決定論的なVerifyフェーズを挟む設計を採用。作成系は結果オブジェクトの存在確認、削除系は対象の消失確認など機械的に判定できるものをLLMに委ねず確定する。Verify失敗時も同様にロールバックして再計画させる。

ロールバックできない操作（ファイル出力、外部API通知、別アプリ操作等）はTransactionの外側・Commit後に実行する原則を設ける。また無限リトライ防止のため、上限回数設定・同一エラー連続での自動修正停止・対象未発見時の再観測・危険操作の実行拒否・人間確認へのエスカレーションを実装している。

設計思想の核心は「プロンプトで失敗率を下げ、実行基盤側で失敗時の被害を抑える二重構え」であり、この考え方はDB更新・ファイル編集・Gitリポジトリ変更・業務アプリ操作など、ステートフルな外部環境を扱うLLMエージェント全般に適用可能。監査エージェントが業務システムに対して書き込み操作を行う場面でも、同様のトランザクション管理層は必須の設計要素となる。

## アイデア

- 操作バッチ全体を1つのDBトランザクションとして扱うことで、Observe-Think-Actループの観測汚染を防ぐ構造が、監査エージェントがERPや業務DBへの書き込みを行う場面にそのまま適用できる
- rolled_back: trueをLLMへの返却に明示することで、LLMが「部分的に成功した状態」を誤前提にした再計画を防ぐ情報設計は、エラーフィードバックのフォーマット標準化として汎用的な知見
- コマンド成功とタスク達成を分離しCommit前にアプリ側決定論的Verifyを挟む設計は、LLM-as-judgeに頼らずに実行結果の正当性を確認できる範囲を最大化するアーキテクチャパターン
