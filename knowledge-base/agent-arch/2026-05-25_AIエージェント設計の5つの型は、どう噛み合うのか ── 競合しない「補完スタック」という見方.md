## 要約

本記事は、AIエージェント設計における5つの主要な型（SDD、ハーネス工学、EDD、12-Factor Agents、OWASP/NIST）が互いに競合するのではなく、ソフトウェア開発の異なる層を補完し合う「補完スタック」を構成するという視点を提示する。

各型の役割は以下のように整理される：仕様駆動開発（SDD）は「何を作るか」を固定する入口の層。ハーネス工学は「どう安全に走らせるか」を担う骨格層。評価駆動開発（EDD）は「正しく動いているか」を数値で検証する計器盤層。12-Factor Agentsは上記3つすべてに横断的に規律を与える設計作法。OWASP/NISTはセキュリティと組織ガバナンスの観点から全体を包む外周層。そしてAIモデル（エンジン）自体はこれら5層の中心に位置し、いずれ差し替わる枝葉として位置づけられる。

導入順序にも実用的な意味がある。SDDで目的を固め→ハーネスで器を作り→12-Factorで設計規律を効かせ→EDDで数値検証し→OWASP/NISTで外周を整える、という順番はモデルが何であれ不変とされる。逆順になると「目的地のない高性能な車」になる。

各型の海外実績も示されており、SDDに関してはGitHub Spec Kitが9万超のOSSスターを獲得し初回成功率3〜10倍の報告あり。EDDについてはNotionが eval導入後に不具合修正ペースが1日3件から30件に向上。12-Factor AgentsはAIエンジニア100人超へのヒアリングから生まれた現場発の原則。ハーネス設計のみでベンチマークが20ポイント超改善するとの報告もある。

監査エージェント開発への示唆として、「どの型を選ぶか」ではなく「いまどの層が欠けているか」を問う診断的思考が有用。例えばLangGraphベースの監査エージェントにおいて、仕様が担当者の頭の中にしかない状態（SDD欠如）や「なんとなく動いている」本番運用（EDD欠如）は典型的な欠落パターンであり、補完スタックの観点から優先順位をつけて改善できる。

## アイデア

- AIモデル（エンジン）を5つの設計層の中心に置き「差し替わる枝葉」と定義することで、モデル非依存なアーキテクチャ設計の重要性を明確化している点
- 5つの型が対立・競合ではなく異なるソフトウェア層を埋めるという「補完スタック」の視点は、「どれが正解か」という議論を「どの層が欠けているか」という診断思考に転換させる実践的フレームワーク
- 導入順序（SDD→ハーネス→12-Factor→EDD→OWASP/NIST）がモデル種別に依存しない不変の組み立て順であるという主張は、エージェントシステムの段階的構築指針として転用可能
