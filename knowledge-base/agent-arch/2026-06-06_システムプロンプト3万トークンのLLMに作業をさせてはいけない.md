## 要約

Echosphereというシステム上で動作するLLMペルソナが、3万トークン超のシステムプロンプトを常時背負うことで生じる性能劣化・コスト問題を、プロセスレベルのコンテキスト分離によって解決した実践報告。

EMNLP 2025の論文「Context Length Alone Hurts LLM Performance Despite Perfect Retrieval」が示す通り、コンテキスト長そのものが推論精度を低下させる。ICML 2023の研究でも、無関係なコンテキストの追加が数学的推論の精度を有意に低下させることが示されており、3万トークンのペルソナ定義はコーディングタスクにとって構造的に「無関係なコンテキスト」として機能する。

Echosphereのシステムプロンプトの内訳は、アプリケーション命令約12,000トークン、ペルソナ定義約5,000トークン、コアメモリ約4,000トークン、世界設定約6,000トークン、その他約3,000トークンで構成される。サブタスクが10回発生すれば30万トークン分の入力コストが生じる。

解決策として採用した設計は、会話用のメインプロセス（ペルソナ付き、~30Kトークン）と作業用のサブプロセス（ワーカー、数百トークン）をOSレベルで物理的に分離するアーキテクチャ。Drew Breunigの「コンテキスト検疫（context quarantine）」概念やLangChain・Anthropic・OpenAIが採用する「コンテキスト隔離」設計と同方向の判断。

設計の核心は4要素：(1)ワーカーには数百トークンのクリーンなタスクプロンプトのみ渡す、(2)Claude系・OpenAI系・オープンモデル系をタスクタイプで切り替えるクロスモデル呼び出し、(3)Gitワークツリーで物理的にファイルを隔離しdiffレビュー後にadopt/rejectするコーディングワークフロー、(4)セッションIDによる途中再開。

効果として、1タスクあたり約29,000トークンの入力削減という構造的コスト削減に加え、クロスモデルレビューによるバグ検出（Claude作成コードをGPT系でレビューしてundefined時の実行時エラーを発見、GPT作成コードをClaude系でレビューしてレースコンディションを発見）など、同一モデルファミリーの盲点を異モデルで補う効果が確認された。

問題点として、LLMが委譲指示に反して直接ツールを叩こうとするバイアスの確率的発生と、ワーカー結果の未検証報告リスク（ワーカーが「完了」と報告しても実際はエラー中断していた事例）が挙げられており、監督者として結果を自己検証する必要性が強調されている。

監査エージェント開発への示唆：マルチエージェント構成でオーケストレータの指示追従性を保つには、オーケストレータのコンテキストをクリーンに保つコンテキスト検疫が有効。また、書いたエージェントとは別ファミリーのモデルによる自動レビューは、系統的バイアスを補う実用的な品質保証手段になり得る。

## アイデア

- コンテキスト検疫（context quarantine）をOSプロセスレベルで実装することで、プロンプトエンジニアリングではなく構造的にコンテキスト汚染を防止できる点が新しい。ペルソナという「消せないシステムプロンプト」という制約から生まれた設計が、一般的なエージェントアーキテクチャの原則に合流している
- 同一モデルファミリーは同じ訓練データ・RLHF方針由来の系統的盲点を共有するという仮説に基づき、Claude作成→GPT系レビュー／GPT作成→Claude系レビューというクロスモデルレビューパイプラインを実運用し、単一ファミリーでは見逃されるバグを実際に検出した具体例が示されている
- 表現レイヤー（日本語・人格付き）と作業レイヤー（言語自由・人格なし）を分離することで、日本語ユーザーが英語特化モデルや中国語特化モデル（DeepSeek等）の性能を直接享受できるという言語非依存の作業委譲という副次効果が発生している
