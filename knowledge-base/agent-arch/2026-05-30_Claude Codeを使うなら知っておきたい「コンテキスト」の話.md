## 要約

LLMツール（Claude Code、ChatGPT、Cursor等）における「コンテキスト」とは、モデルが現在参照できる情報の総体であり、会話履歴・読み込んだファイル・実行コマンドの出力・CLAUDE.mdなどの設定ファイル・システムプロンプトを含む。これは「作業机」に例えられ、机が広いほど多くの資料を置けるが、無限ではない。Claude Code（Opus/Sonnet）の標準上限は20万トークン、環境によっては100万トークンまで拡張可能。ただし、上限に達する前からContext Rot（コンテキストの腐敗）と呼ばれる性能劣化が発生する。Chromaの研究によれば、入力が長くなるほど情報の正確な想起能力が低下し、劣化は均一でなく急激に落ちることもある。関係ない情報（ノイズ）が混在するとさらに精度が落ち、情報の「何があるか」より「どう提示されているか」が性能を左右する。コンテキスト管理の実践テクニックとして、/contextコマンドで現在の使用状況を色分けビジュアルで確認できる（System prompt、Memory files、Skills、Messages、Free spaceの内訳表示）。/clearでセッションを完全リセット、/compactで会話履歴を要約圧縮、/btwで寄り道質問をメイン履歴に残さずに実行する方法が有効。CLAUDE.mdに繰り返し使う情報を記述することで毎回のコンテキスト消費を削減できる。Skillsは.claude/skills/配下に定義しても呼び出し時のみ本文がロードされ、名前と一行説明だけが常駐するため机を圧迫しない。サブエージェントを使うと大量ファイル調査などの重いタスクを別のコンテキスト空間で処理し、結論のみメインに返せる。これら一連の手法は「コンテキストエンジニアリング」と総称され、プロンプトエンジニアリング（どう聞くか）と対比して「何を渡すか・渡さないか」の設計スキルと定義される。監査エージェント開発においては、RAGやマルチエージェント構成でのコンテキスト設計（必要情報の取捨選択・サブエージェントへのタスク分割・CLAUDE.mdへの繰り返し指示の外部化）が品質と効率の両面で重要な設計要素となる。

## アイデア

- Context Rot（コンテキストの腐敗）という概念：上限到達前からじわじわ性能劣化し、しかもその劣化が均一でなく急激に落ちるケースがある点は、エージェント設計時のコンテキスト予算管理に直結する重要な知見
- SkillsのLazy Load設計：.claude/skills/に大量の手順書を置いても呼び出しまで本文がコンテキストに載らない仕組みは、RAGの遅延ロードと同じ思想であり、監査エージェントの「必要な規程・手順書を必要時のみ参照」設計に応用できる
- サブエージェントによるコンテキスト分離：大量ファイル調査をメインセッションから分離し結論のみ返す手法は、LangGraphのマルチエージェント構成でのワーキングメモリ管理パターンと対応しており、エージェントアーキテクチャ設計の参考になる
