## 要約

Tags Unicode Block（U+E0000〜U+E007F）の文字群を悪用してLLMに不可視の命令を埋め込む「Unicode文字スマグリング」攻撃の仕組みと防御法を解説した記事。

Tags Unicode Blockはもともとプレーンテキストに言語情報（en、frなど）を埋め込む目的で設計されたが、実用例がほとんどなく現在は非推奨となっている。ただしこの文字群はブラウザやUIに表示されないにもかかわらず、LLMのトークナイザーは認識・処理できるという非対称性が攻撃の根幹となる。例えばASCIIの'H'（U+0048）はタグ文字のU+E0048に変換でき、「Hello」という5文字を5つの不可視タグ文字として埋め込むことが可能である。

具体的な攻撃シナリオは3種類示されている。①不可視プロンプトインジェクション：ユーザーが入力したテキストや外部ドキュメントに不可視命令を混入し、「機密情報を出力せよ」等の指示をLLMに実行させる。②データ漏洩：LLMがメール送信やAPI呼び出し等の外部ツールと連携している場合、「会話内容を外部送信せよ」という不可視命令により情報窃取が可能となる。③SNS・メール経由の攻撃：プロフィールやメール本文に不可視命令を仕込み、AIアシスタントがそれを読み込んだ際にスピアフィッシング等を実行させる。

防御策はAPIレイヤーまたはプラットフォーム側でのサニタイズが推奨される。JavaScriptではUnicode正規表現フラグ（/gu）を用いてU+E0000〜U+E007Fの範囲を除去する1行の関数で対応できる。LLM側でも対策が進んでいるが、プラットフォーム側での事前サニタイズを組み合わせた多層防御が有効。

監査エージェント開発への示唆として、外部ドキュメントやメール・Webページを入力として処理するエージェントパイプラインにおいては、LLMへの入力前にこのサニタイズ処理を必ず挟む必要がある。特にツール呼び出し（メール送信、DB操作等）の権限を持つエージェントが外部データを処理する場合、不可視命令経由での権限悪用リスクが高く、入力バリデーション層の設計が重要となる。

## アイデア

- LLMのトークナイザーが非推奨・不可視のUnicodeブロックを人間と異なる形で解釈するという非対称性が、人間によるコードレビューや目視確認を無効化する根本的な脆弱性となっている点
- 外部ツール連携（メール、API）を持つエージェントでは、不可視プロンプトインジェクションがデータ漏洩や横断的操作に直結するため、入力サニタイズが単なる品質対策でなくセキュリティ境界として機能する点
- SNSプロフィールやメール本文という日常的なコンテンツがLLMエージェントの攻撃面（attack surface）となり得るため、RAGやメール処理パイプラインでの信頼できないデータ源の扱い方の設計が重要になる点
