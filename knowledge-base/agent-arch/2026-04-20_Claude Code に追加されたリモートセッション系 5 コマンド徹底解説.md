## 要約

Claude Code に新たに追加された 5 つのコマンド（/autopilot, /bugfix, /dashboard, /docs, /investigate）は、いずれもローカルの Claude Code セッションから Anthropic 管理のクラウドインフラ上で動く「Claude Code on the web」セッションをスポーンするという共通の仕組みを持つ。既存の /autofix-pr や /ultraplan も同系統のリモート起動コマンドだが、今回の 5 つはワークフロー単位で目的に特化したプロンプトテンプレートが内蔵されており、曖昧な指示でも所定のステップを踏みやすい設計になっている。

各コマンドの役割は明確に分かれている。/autopilot は特定領域に縛られない汎用オートパイロットで、リファクタリング・依存ライブラリ更新・スキャフォールディングなど手順は明確だが時間がかかるタスクをクラウドに丸投げする用途向け。/bugfix はバグ修正フロー専用で「Reproduce → Root-cause → Fix → Regression-test」の 4 ステップを 1 セッションで完結させる。/dashboard はデータソース（例: PostgreSQL）を読んでダッシュボード UI を設計・実装するフローに特化し、MCP コネクタや環境変数経由でクラウドからデータベースへアクセスする。/docs はコードリポジトリの指定範囲を解析して公開 API・型・設定項目を一覧化し、不足・陳腐化したドキュメントを補完して PR を発行する。/investigate はインシデントの根本原因を特定して「修正案を含むレポート」を成果物とするコマンドで、コードを直接変更するのではなく人間の意思決定材料を提供する点が /bugfix との最大の違いである。

既存の /schedule（routines）との違いは「ワンショット vs リピート」にある。新 5 コマンドはユーザーがその場でコマンドを叩いた 1 回限りの起動であり、定期化・イベントトリガ化が必要な場合は /schedule で routine 化するのが自然な流れ。リモートセッションはデフォルトで claude/ プレフィックスのブランチにしか push できないガードが掛かっており、protected branch を誤って触るリスクがない。また /teleport（/tp）を使えばリモートセッションをローカルに引き取ることも可能で、途中での方向修正にも対応できる。実践上の注意点として、リモートセッションはローカル環境を参照できないため、DB 接続情報や CI ログへのアクセスは MCP コネクタと「クラウド環境」の Environment variables に事前設定が必須。プロンプトには完了条件（失敗を再現するテストが存在する、CI が green になる、PR 説明に再現手順が含まれる等）を明示することでリモート実行の精度が上がる。監査エージェント開発においては、/investigate の「インシデント → ログ・メトリクス横断解析 → 原因仮説 → レポート生成」というフローが、エビデンス収集から指摘事項ドラフトまでの監査ワークフローと構造的に類似しており、同様のパターンを LangGraph ベースの監査エージェントに適用できる可能性がある。

## アイデア

- /investigate が「コードを直接変更せずレポートを成果物とする」設計になっている点は、人間の承認ステップを前提とした監査・インシデント対応ワークフローと構造的に一致しており、LLM-as-judge パターンの実装参考になる
- ワンショット実行（新 5 コマンド）とリピート実行（/schedule routines）を明確に分離し、必要に応じて /teleport でローカルに引き取る設計は、エージェントのステートフルな引き継ぎ機構の実装モデルとして参考になる
- MCP コネクタと環境変数によってクラウドリモートセッションがローカルと同等のデータアクセス権を持てる仕組みは、マルチエージェントシステムにおけるコンテキスト・クレデンシャル共有の設計パターンとして応用可能
