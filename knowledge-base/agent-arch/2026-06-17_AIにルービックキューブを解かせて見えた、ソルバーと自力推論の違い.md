## 要約

AutoCAD向けAIエージェントを開発する著者が、「AIがルービックキューブを解く」動画に疑問を抱き、MCP（Model Context Protocol）を自作して検証した実験記録。

MCPには「盤面読み取り（observe_cube）」「1手回転（rotate_cube）」「完成判定（check_cube_solved）」の3ツールを用意した。Claude（Opus 4.8）に「6面そろえて」と指示したところ、実際には`pip install kociemba`を実行し、既製ソルバーに盤面の54文字表現を渡して解法を取得、返ってきた5手の手順（B' U' R' U' B'）をrotate_cubeで流すだけだった。つまり「解いていたのはClaudeではなくkociembaライブラリ」であることが判明した。

次にソルバー使用・コード実行を禁止し、最大150手・AIの自力推論のみで同一スクランブル（B U R U B、5手混ぜ）を解かせた。結果：ソルバーありで5手・約1.5分に対し、自力では110手・約3時間を要した（同一盤面で22倍の手数）。

観察された自力推論の特徴は3点。①最初の11手を「校正」に使い、各回転で盤面がどう変わるかを実験して回転ルールを自ら確立した。②「予測→1手実行→observe_cubeで照合」を毎手繰り返す層別法（初心者向け）で進め、7手ぶんの暗算シミュレーション中に「対色が1ピースに同居する」物理的にありえない配色を出力した際は自己検知して方針を修正した。③スコア推移（満点54）は22→54へと右肩上がりではなく、最終層処理で大きな谷を形成しながら収束した。

著者はこの構造がCADエージェント開発の課題と直結すると指摘する。ルービックキューブは「面を回す」1種類の操作・54枚のステッカー観察という単純な問題設定でも自力では110手を要した。CADはスケッチ・押し出し・フィレット・ブール演算など多種の操作を正しい順序で積み上げる必要があり、はるかに複雑。そこから導いた設計方針は「①操作ごとにobserveで確定的な状態を取得する仕組み」と「②確実なロールバック機構」の2点。キューブにおけるkociembaのような「確実な処理の核」をCADの各タスク種別に用意し、LLMは意図・判断を担い、幾何計算・状態管理・検証は決定論的な仕組みに委ねる役割分担が必要という結論に至った。

## アイデア

- LLMは確実なツール（ソルバー）が利用可能な場合は自律的にそれを選択する。ソルバーを禁止して初めて「自力推論の実力」が表面化する実験設計が巧み
- 自力推論時にClaudeが最初の11手を「環境校正」に使った点：回転ルールを事前知識から仮定するのではなく、実環境で実測して信頼できるシミュレータを構築してから解法に入る戦略は、監査エージェントの「システム理解フェーズ」設計に転用できる
- 「毎手observe→照合」パターンが長期状態追跡の誤差蓄積を防ぐ補助輪として機能した点：LLMの暗算限界（7手で物理的に不整合な状態を生成）を外部参照で補うアーキテクチャは、監査証跡の逐次照合ループ設計に直接応用可能
