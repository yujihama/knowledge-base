## 要約

本記事はHugging Face Blogにて2025年8月18日に公開されたもので、Model Context Protocol（MCP）を活用して研究発見（Research Discovery）を自動化する手法を解説している。研究発見とは、arXiv・GitHub・Hugging Faceといった複数プラットフォームをまたいで論文・実装コード・モデル・データセットを収集・照合する作業を指し、従来は手動切り替えが必要だった。

記事では研究発見を3層の抽象度で整理する。第1層「手動リサーチ」では、論文をarXivで発見→GitHubで実装を検索→Hugging Faceでモデル・データセットを確認→著者・引用の照合→手動整理という一連の作業をすべて人間が行う。第2層「スクリプト化」では、Pythonスクリプト（research_tracker.py）がscrape_arxiv・search_github・search_huggingfaceを組み合わせてconsilidate_resultsを返す形で自動化する。ただしAPI変更・レート制限・パースエラーにより人間の監視なしでは失敗するリスクがある。第3層「MCP統合」では、これら同一のPythonツールをMCPサーバー経由でAIエージェントに公開し、自然言語の指示（例：「過去6か月に発表されたTransformerアーキテクチャ論文を、実装コードとPretrained Modelありの条件で列挙」）でAIが複数ツールをオーケストレーションする。AIは情報の欠落を推論で補い、他のMCPサーバーとのクロスリファレンスも実行する。

具体的ツールとしてResearch Tracker MCPが紹介されており、huggingface.co/settings/mcpから「research-tracker-mcp」を検索して追加可能。Claude Desktop・Cursor・Claude Code・VS Codeなど各クライアント向けの設定が自動生成される。Hugging Face MCP Serverは、Hugging Face SpacesをMCPツールとして公開する標準的な仕組みとして機能する。実装はGradio MCP Guideを使えばPython関数をそのままMCPツールに変換できる。

記事はこのアプローチを「Software 3.0の類比」として位置づけ、自然言語が実質的なプログラミング言語として機能する新しい抽象層と捉える。ただしスクリプトと同様に、人間の監視なしでは誤りが生じやすく、下位層（手動・スクリプト）の理解が実装品質に直結するという注意点も明示している。

## アイデア

- 研究発見を「手動→スクリプト→MCP統合」の3層抽象度で整理するフレームワークは、監査ドキュメント収集・照合作業の自動化設計にそのまま転用できる
- GradioベースのPython関数を即座にMCPツール化できる仕組みは、既存の監査スクリプト（勘定照合・リスク判定ロジック等）をエージェントに公開する最短経路になり得る
- 自然言語指示がソフトウェア実装に相当するというSoftware 3.0の視点は、LLM-as-judgeやGRPOによる報酬設計を「高レベル仕様の自然言語化」として再解釈する理論的根拠になる
