## 要約

Claude Code の Memory 機能は user / feedback / project / reference の 4 種類に分かれており、それぞれ寿命とトリガーが異なる。user メモリは役割・経験・好みなど変わりにくい属性を保存し、全タスクで呼び出される。feedback メモリは過去に修正された判断やスタイルを残し、再発防止に使う。project メモリは進行中の作業・期日・関係者など短命な文脈情報を扱い、完了後は積極的に削除する。reference は外部リソースへのポインタ専用。

設計上の核心は「寿命の異なる情報を混在させない」点にある。たとえば「来週までにこの PR を出す」を user メモリに書くと、翌月には虚偽の情報になる。分離することで各メモリが「育つドキュメント」として機能する。

各メモリには本文だけでなく Why（理由）と How to apply（適用場面）を必ず記述することが推奨される。Why がないと状況変化時に「このルールをまだ適用すべきか」を AI が判断できず、誤った適用が起きる。たとえばタイトルに「N パターン」を使わないルールに Why として「読者の自己投影を妨げる」と書いておくと、「7 つの落とし穴」という表現も同じ基準で棄却できる。

運用ルールとして4点が挙げられる。①1メモリ1ファイル＋MEMORY.md インデックス構造で管理し、インデックスは1行150字以内に絞る。Claude Code は会話ごとに MEMORY.md を全文読み込むが、個別ファイルはトリガーが当たったときのみ参照する設計になっている。②Why と How to apply を必ず記述する。③月1回を目安に棚卸しして陳腐化情報を削除する（project は期日経過後削除、feedback は修正されなくなったら削除）。④メモリに書くべきは「未来の自分が同じ判断を素早く再現できる材料」のみで、作業ログや一度きりの調査結果は Obsidian などの外部メモリに分離する。

監査エージェント開発への示唆としては、エージェントが繰り返す判断パターン（例：証拠の充足性評価基準、リスク分類のルール）を feedback メモリとして構造化し、Why と適用条件を明記することで、LLM-as-judge の一貫性を高める設計に応用できる。また project メモリの短命管理は、監査フェーズ（計画・実施・報告）ごとのコンテキスト切り替えにそのまま対応可能。

## アイデア

- メモリの種類を「寿命」と「トリガー（呼び出し文脈）」の2軸で分類することで、情報の陳腐化リスクを構造的に制御できる点
- Why と How to apply を必須フィールドとして定義することで、ルールが単なる禁則リストではなくエッジケース判断可能な推論根拠として機能する点
- Claude Code が MEMORY.md を常時全文読み込む一方、個別ファイルはオンデマンド参照するという2層アーキテクチャを意識したインデックス設計で、信号/ノイズ比を最適化できる点
