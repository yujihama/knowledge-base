## 要約

自宅自動化をタスクごとの個別Pythonスクリプトで運用していた著者が、LLM中心アーキテクチャへの移行としてHermes Agent（Nous Research製）を採用した構築事例。OpenClawのようなフルシステムアクセス型エージェントは「何でもできすぎる」リスクがあるとして採用を見送り、より制御しやすいHermes Agentを選択した。

セキュリティ設計の核心は「通信先を2つに絞る」方針で、ファイアウォールでLAN内のllama.cppとSlack APIのみ許可し、それ以外のWAN通信は全遮断している。加えてHermes AgentをProxmox上のVMとして隔離することで、エージェントが暴走した場合でも「VMごと落とす」という明確な非常ボタンを確保している。コンテナや直接プロセス起動と異なり、ネットワーク・リソース分離がVM単位で完結する点を評価している。

LLMエンジンにはEVO-X2（Ryzen AI Max 395 + RAM 128GB）上のllama.cppを使用。中古で約30万円（現行定価40万円超）で入手。モデルはQwen3.6（35B-A3B）で、会話用途で約60 tok/secの応答速度を実現。アイドル時の消費電力の低さも24時間稼働における選定理由の一つ。

キャラクター設定はHermes AgentのSOUL.mdで定義し、学園アイドルマスター「篠澤広」のペルソナを採用。もともとClaude向けに書いたプロンプトをQwenに流用したため、モデル間のシステムプロンプト従順性の違いから口調崩れが発生。改善策として、エージェント自身に「篠澤広をサーベイしてSOUL.mdを作成」させ、不適切な例文を除外する方法を採った。

今後の機能追加はHermes Agentの「Skillを自作してくれる」仕組みを活用して口頭で依頼する方針だが、エージェントが自律的にコードを生成・実行する仕組みがプロンプトインジェクションリスクを内包する点を課題として認識している。SaaSへの移行については、トークン単価が電気代を下回れば検討するものの、個人情報の外部流出への懸念から現状は自ホストを維持する判断。監査エージェント開発への示唆としては、エージェントの権限範囲を「通信先の絞り込み」「VM隔離」「自ホストLLM」の組み合わせで制御するアプローチは、企業内監査エージェントの安全な展開設計にも直接応用可能な考え方である。

## アイデア

- エージェントの権限範囲をファイアウォールレベルで「通信先2つのみ」に絞るという設計は、LLMエージェントのセキュリティ設計における最小権限原則の具体的実装例として参考になる
- SOUL.mdキャラクター定義をエージェント自身にサーベイさせて生成させる手法は、モデル依存のプロンプトチューニング問題をエージェントの自律能力で解決するアプローチとして興味深い
- Hermes AgentのSkill自作機能が「プロンプトインジェクションリスクを内包する」という指摘は、自律コード生成エージェントの本質的な攻撃面であり、監査エージェント設計での入力バリデーション設計に直結する
