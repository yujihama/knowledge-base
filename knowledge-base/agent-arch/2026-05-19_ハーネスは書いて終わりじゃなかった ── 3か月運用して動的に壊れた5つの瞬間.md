## 要約

Claude Codeのハーネス（CLAUDE.md・Hooks・Subagent・Memory・Permission設定の組み合わせ）を3か月運用した実体験から、「静かに壊れる」5つのパターンを分析した記事。

【壊れ方1: CLAUDE.md のstale化】Opus 4.6で機能していた「コミットメッセージを日本語のですます調で書け」というルールが、4.7へのモデル更新後に確率的（10回中3回）に従われなくなった。4.7は4.6より「リテラル」なinstruction-followingになったとAnthropicは説明するが、現場感覚は逆で、4.6が曖昧に拾っていたニュアンスが4.7では拾われなくなるケースがある。

【壊れ方2: Hooks発火順序の変化】PreToolUseでリンター、PostToolUseでテストを走らせる構成が、subagent経由のtool呼び出し時に親プロセスのHookが発火しないケースが発生。Claude Code Hooksの12種類のライフサイクルイベントは、直接呼び出しとTask tool経由のsubagent呼び出しで発火対象が異なる仕様であり、両方のシナリオをテストしていなかったことが原因。

【壊れ方3: Subagentへのcontext非継承】リサーチ・コードレビュー・ドキュメント生成用の3つのsubagentを運用中、「先週議論したcontext engineeringのアプローチを踏まえて」と依頼したところ、子エージェントは「情報がない」として一般論を返した。Subagentは親とは独立したcontext windowを持ち、Task toolに書いたpromptのみを受け取る。CLAUDE.mdの細かいルールも、会話で積み上げた前提も、skillファイルも、明示的にpreloadしない限り継承されない。

【壊れ方4: Memory誤発火（auto-compaction）】長時間リファクタリングセッション中、冒頭30分で説明したクラスの設計意図がauto-compaction後に消失。ClaudeがコードからYという設計意図を推測し、元の意図と異なるリファクタを提案した。/clearと/initの中間にあるauto-compactionはユーザーが完全制御できず、重要な前提はCLAUDE.mdとmemoryファイルの両方に書き込む運用に変更した。

【壊れ方5: Permission allowlistの穴】mcp__myserver__*のワイルドカードで許可していたMCPサーバーがv1.2→v1.3にマイナーアップデートした際、新規追加されたtoolが自動的にpromptなしで呼ばれるようになった。安全なallowlist設計は必要なtoolを個別列挙することであり、サードパーティMCPのCHANGELOGを更新のたびに確認する運用が必要。

共通構造は「エラーが出ない静かな劣化」で、設計時点が最も安全で時間とともに意図から乖離していく。運用ルーチンとして「毎週CLAUDE.mdの見直し」「モデル更新時の全subagentプロンプト再テスト」「MCP更新時のallowlist見直し」「重要決定時の即時memoryファイル書き込み」を実施している。監査エージェント開発への示唆：エージェントの挙動をコントロールするハーネス自体が動的に劣化するインフラであり、設計フェーズと別に「運用・保守フェーズ」の設計が必要。権限管理（Permission）の最小権限原則はMCPのtool粒度でも適用が必要。

## アイデア

- CLAUDE.md は静的ファイルではなく、モデルバージョンという外部変数によって解釈が変わる「生きた契約書」であり、モデル更新をトリガーとした定期的な再テストが必要という視点
- Subagentは「賢い分身」ではなく「毎回オンボーディングが必要な別の新人エンジニア」というメンタルモデルの転換：Task toolのpromptに毎回前提を明示的に書き込むことで対処
- Permission allowlistのwildcard使用は、サードパーティMCPのバージョンアップによって自動的に穴が開く時限爆弾であり、最小権限原則をtool単位の粒度で適用することの重要性
