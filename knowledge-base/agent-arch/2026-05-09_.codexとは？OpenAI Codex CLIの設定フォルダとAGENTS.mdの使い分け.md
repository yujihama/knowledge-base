## 要約

OpenAI Codex CLIが参照する設定フォルダ`.codex`の仕組みと、自然言語指示ファイル`AGENTS.md`との役割分担を解説した記事。`.codex`はモデル選択・承認ポリシー・sandbox設定・MCP・hooksなどのパラメータを`config.toml`で管理するための設定フォルダであり、`AGENTS.md`はCodexに守らせる開発ルールを自然言語で記述するファイルで、両者は明確に分離して管理すべきとされている。

設置場所は2階層あり、グローバル（`~/.codex/config.toml`）はユーザー全プロジェクト共通の標準設定を置き、プロジェクトルート（`<repo>/.codex/config.toml`）はそのリポジトリ専用の設定上書きに使う。`config.toml`の優先順位はCLI引数 > `--profile`指定 > プロジェクト内`.codex` > グローバル`~/.codex` > システム設定 > Codexデフォルトの順で、カレントディレクトリに近いほど優先される。重要な注意点として、プロジェクトローカルの`.codex`はCodexがそのプロジェクトを`trusted`と判定した場合にのみ読み込まれ、`untrusted`の場合はローカルのconfig・hooks・rulesがすべてスキップされる。

`AGENTS.md`の読み込み順はグローバル（`<Codex home>/AGENTS.md`）→プロジェクトルート（`<repo>/AGENTS.md`）→サブディレクトリと階層順に結合され、同階層に`AGENTS.override.md`が存在する場合は`AGENTS.md`より優先される。後から読み込まれるカレントディレクトリに近い指示ほど優先されやすい設計になっている。

Claude CodeにおけるCLAUDE.mdとの類比で捉えると理解しやすい。監査エージェント開発への示唆として、マルチリポジトリ構成でエージェントを運用する場合に、共通のLLM設定やMCPサーバー定義をグローバル`.codex`に集約し、各リポジトリ固有のhooksや承認ポリシーをプロジェクトローカルに分離するパターンは、LangGraphベースのエージェントシステムにおけるコンフィグ管理の設計指針としても応用できる。

## アイデア

- `.codex`（機械可読な設定）と`AGENTS.md`（自然言語の指示）を明示的に分離する設計思想は、Claude CodeのCLAUDE.mdとsettings.jsonの役割分担と対応しており、AIエージェントツール全般に共通するコンフィグ設計パターンとして注目できる
- プロジェクトの`trusted`/`untrusted`判定によってローカル設定を丸ごとスキップするセキュリティモデルは、マルチユーザー・マルチリポジトリ環境でのエージェント設定の権限制御として参考になる
- `AGENTS.override.md`という上書き専用ファイルを設けることで、階層的な指示の継承と局所的な差し替えを両立する仕組みは、階層型マルチエージェントシステムにおける指示伝播の設計に応用できる
