## 要約

LLMエージェントに賃金・産業統計データの自律分析をさせる際、dbtのスキーマ情報だけではドメイン固有の「暗黙知」が欠落するという課題に対し、dbt analyses × Markdownのインサイト知識ベースで解決する手法を解説した記事。

課題の核心は、dbtのtest・description・tagsで「データの定義」は伝えられても、「テーブルの癖（例: 産業カテゴリの年度間定義変更）」「過去の仮説の成否」「棄却された仮説」といった暗黙知は伝えられない点にある。その結果、エージェントは毎回ゼロから基礎集計を繰り返し、深掘り分析に至らなかった。

解決策として、`analyses/insights/`ディレクトリ下にSQLファイルとMarkdownファイルをペアで配置する構成を採用。MarkdownにはYAMLフロントマターを設け、`sources`（参照モデル名）・`columns`（使用カラム）・`tags`（横断検索用）・`status`（verified/exploratory/refuted/partial）の4フィールドでメタデータを機械可読化した。

特に`status: refuted`の活用が価値の中心で、「情報通信業の地域格差は全産業最小（CV=6.3%）」「女性比率が高い産業ほど賃金格差が大きい（逆説）」「複合サービス業は高LQ×低賃金の罠」といった棄却済み仮説を記録することで、エージェントが同じ誤りを繰り返すコストを排除する。

検索はPyYAMLと標準ライブラリのみで実装した`scripts/insights.py`が担う。エージェントの利用フローは「タグ・ソース・ステータスで絞り込み → 必要ファイルのみ`--show`で本文取得」の2ステップで、RAGや埋め込みモデル不要でコンテキスト消費を最小化する。

監査エージェント開発への示唆として、この「verified/refutedインサイトの蓄積」パターンは、監査手続の結果管理にも転用可能。「このリスク仮説は前期に検証済み」「この統制テストは特定条件下で偽陽性を示す」といった知見をstatus付きMarkdownで蓄積すれば、監査エージェントが過去の調書から文脈を引き出しながら深掘りできる構造になる。

## アイデア

- status: refutedで棄却済み仮説を明示的に記録することが、エージェントの重複試行を防ぐ最も価値ある設計判断である点
- RAGや埋め込みモデル不要で、PyYAMLと標準ライブラリのみによる2ステップ検索（絞り込み→詳細取得）でコンテキスト効率を確保できる軽量アーキテクチャ
- SQLファイルとMarkdownをペアで管理することで、再現性（dbt ref()によるクエリ）と暗黙知（インサイト記録）を同一Gitリポジトリで一元管理できる設計
