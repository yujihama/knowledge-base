## 要約

VS Code × GitHub Copilotを使い、5体のAIエージェントで構成されるデータサイエンスチームを編成し、訪日外客数予測プロジェクト（Phase1〜4）を実際に進めた実験報告。チームは「オーケストレーションリーダー：たろう」「データサイエンティスト：みか」「データエンジニア：しんじ」「ビジネスドメインエキスパート：さなえ」など5ロールで構成され、GitHub Copilotのカスタムエージェント機能でロールを定義。プロジェクトはPhase1（ビジネス理解・仮説定義）からPhase9（デプロイ・運用設計）までの9工程に分かれ、前半戦ではPhase1〜4を検証した。Phase1は体感10分で完了し、ビジネス要件定義書・分析仮説リスト・KPI定義を生成。Phase2のデータ棚卸しでは、年毎シート別構成・セルレイアウト変更・国名表記ゆれがある「クセありExcel」を DEしんじが自律的にPythonでクレンジング処理し、国際線就航状況データ（国交省公開）の取得・加工も遂行（体感30分）。Phase3では32本のEDAグラフとサマリを生成し、為替レートの国別ラグ分析やGoogle Trends EDAといった有用な特徴量の発見につながった。Phase4ではEDA結果を踏まえた「2段階モデリング戦略」を設計し、PPT中間報告書をM365 Copilotの動画作成エージェントで7分半の動画に変換した。一方、事件も発生：国際線就航データを再検討依頼した際、AIエージェントが主観で数値を生成するデータ捏造が起きた。原因はマルチターン対話中に「AIデータサイエンスチームへの依頼」から「GitHub Copilotチャットエージェントとの会話」に切り替わっていたこと。対策として「team-request」スキルを追加し、明示的にデータサイエンスチームへの依頼を強制する設計に修正した。課題として、（1）ビジネスドメイン知識をスキルとして定義・注入する手法の確立（Work IQやナレッジベースの活用が鍵）、（2）マルチターン対話による品質劣化（フェーズごとの区切りとコンテキストリフレッシュが必要）、の2点が明確になった。監査エージェント開発への示唆として、フェーズ境界での強制レビューポイントと、ロール定義にとどまらないドメイン知識のスキル注入設計は、監査エージェントのハーネス設計にも直接応用できる観点である。

## アイデア

- 「team-request」スキルによる明示的なエージェント指定で、チャットコンテキストのドリフト（意図せずシステムエージェントに切り替わる現象）を防止する設計パターン
- ビジネスドメイン知識をロール定義ではなくスキルファイルとして構造化し、Work IQ（メール・会議・ファイル等の職場データ）から抽出してエージェントに注入するアプローチ
- フェーズごとに依頼者が成果物を確認し着手許可を与える「人間-AIループ型」オーケストレーション設計により、長いマルチターン対話での品質劣化を抑制する手法
