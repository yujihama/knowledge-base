## 要約

LLMによる議事録自動生成が普及すると、1日10〜15本の議事録が蓄積される一方、上位層への報告には粒度が細かすぎる問題が生じる。本記事では、OpenAI Codex CLIの非対話実行コマンド `codex exec` とAgent Skillsを組み合わせ、議事録→日報→週報→管理資料という段階的変換パイプラインをGitベースで構築するアーキテクチャを解説する。

ディレクトリ構成はシンプルで、`minutes/YYYYMMDD(曜日)/` に議事録Markdownを格納し、`.agents/skills/` 配下にSKILL.mdを持つSkillディレクトリ（daily-report / weekly-report / management-report）を配置する。`reports/daily/` と `reports/weekly/` に生成物を出力し、スクリプトでGit commit/pushまで自動化する。

SkillはSKILL.mdに入力ソース・出力パス・レポート構造・スタイルを記述するだけで、Codexがそれを読みながらMarkdownを生成する。PowerShellスクリプトでは `$daily-report` のSkill名を埋め込んだpromptを `codex exec --sandbox workspace-write` に渡す。promptが長い場合はファイルからstdin経由で渡す方法（`codex exec -`）も紹介されている。`--json` フラグでJSONL形式の実行ログを残し、Gitの差分と照合することで「なぜこの日報になったか」を事後追跡できる。

階層変換の設計思想として、議事録→日報（現場の事実）→週報（進捗・リスク・意思決定）→管理資料（上位層の判断論点）と粒度を段階的に上げることで、中間生成物が安定したバッファになる点を強調している。1on1の個人情報はSkillのプロンプト指示でチーム・役割単位に丸めて出力しないよう制御する。

監査エージェント開発への示唆として、このアーキテクチャは監査業務の記録管理に直接応用可能。監査調書（Workpaper）をminutesに相当するものとして蓄積し、日次・週次の監査進捗報告やmanagement向けの発見事項サマリーをCodex Skillで自動生成する構成が考えられる。Git履歴が監査証跡（Audit Trail）として機能する点も内部統制上の利点になる。

## アイデア

- 議事録→日報→週報→管理資料という段階的粒度変換パイプラインにより、現場の詳細情報を消さずに報告粒度を制御できる設計
- SKILL.mdによるAgent Skill定義でpromptエンジニアリングをファイルとして永続化し、再利用可能なエージェント動作仕様として管理する手法
- Gitリポジトリを単なるバージョン管理ではなく「報告資産の履歴蓄積基盤」として活用し、--jsonログとの組み合わせで生成根拠の追跡を可能にする設計
