## 要約

知識共有プラットフォーム「Brain」にCS（カスタマーサービス）担当AIチャットを導入した事例。リリース後47時間で238件の利用があり、AIがカバーできなかったケースはわずか5件（カバー率97.9%）。人間対応の問い合わせも9.3通/日から3通/日へ約67.7%削減された。

アーキテクチャはAWS上に構築。ALBのpath-based routingによりチャット専用のECS Serviceへリクエストを振り分ける。APIのECS Serviceと分離した理由は、①SSEによる長時間接続とAPIのHTTP属性の違い、②Auto Scalingの条件の違い、③障害分離、④デプロイの独立性確保の4点。チャット用ECSはheartbeatを定期送信してALBのidle timeoutを回避する。入力バリデーションとElastiCacheを使ったrate limiterを組み合わせ、正当なリクエストのみがRAGとLLMと通信する設計。レスポンスはSSEでchunkとして逐次返却し、エラーや個別対応不可の場合はredirectコードを含むJSONを返す。

RAGのベクトルDBにはPineconeではなくOpenAI Vector Storeを採用。保守運用コストの削減と、OpenAIのエコシステムへの依存による長期的最適化を期待してのことで、今回のシンプルなドキュメント検索ユースケースではPineconeの優位性が割に合わないと判断。

探索ロジックは多段階構成。①ユーザー質問から口語・言い換えを考慮して複数の検索クエリ候補を生成、②最大クエリ数と最大並列度の上限内でOpenAI Vector Storeを並列探索、③文字列ベースのdedup・rerankで十分な結果が得られなければfallback検索、④最終的に本文一致・ファイル名一致・正規化一致・Bigram類似度でスコアリング、⑤候補の強さに応じてLLMによるrerankの要否を信頼スコアで判定しスキップ。

ナレッジは1topic=1ファイルのYAML形式で管理し、type・id・tags・questions・related・front_filesなどのフィールドを保持。フロントエンドのreleaseブランチへのマージをトリガーにknowledgeリポジトリへのPRが自動生成され、OpenAI Vector Storeへ同期するCD（継続的デリバリー）パイプラインを整備。解決できない問い合わせはSlackに通知され、CS担当者と開発者が対応後にDevinをスレッドへ招待することでナレッジをリポジトリに自動反映する仕組みも構築されている。

## アイデア

- LLMによるrerankを信頼スコアで条件分岐し、候補が明確に強い・弱い場合はスキップすることでコストと品質のトレードオフを最適化している点
- フロントエンドのCI/CDをトリガーにknowledge→OpenAI Vector Storeまで自動同期するパイプラインで、コードベースとナレッジの鮮度ずれを構造的に防いでいる点
- Devinをコミュニケーションチャンネル（Slack）に参加させ、CS対応のスレッドを読ませてナレッジを自動更新することで、人間の対応をナレッジ化するループを実現している点
