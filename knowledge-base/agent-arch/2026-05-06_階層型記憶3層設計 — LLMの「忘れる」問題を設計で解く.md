## 要約

LLMのコンテキストウィンドウ制約による「忘却問題」を、人間の記憶構造に着想を得た3層アーキテクチャで解決するシステム設計の解説。実装はNext.js 15 + React 19 + TypeScript + Gemini API（gemini-2.0-flash-lite）を使用した長編小説自動生成システム「novel-automation-system」での実践例として示される。

3層の構造は以下の通り。①短期記憶（直近数章のテキストをそのまま保持、72時間TTL）：文体の一貫性と場面の連続性を担保する。要約ではなく原文保持が重要。②中期記憶（セクション・アーク単位の分析結果）：キャラクター感情の変化、伏線の進行状態、物語テンションのパターンを追跡。短期記憶の期限切れ後の空白地帯を埋める。③長期記憶（世界設定・コア関係性を永続保持）：自動学習機能により物語進行に伴い知識ベースが成長。手動介入不要。

4つの補助コンポーネントが3層の実運用を支える。CacheCoordinator（TTLではなく重要度・参照頻度ベースの層間データ移行制御）、DuplicateResolver（同一情報の複数層存在を解決、より具体的な層＝短期を優先）、AccessOptimizer（クエリ性質に応じた参照層の振り分け）、QualityAssurance（生成前の矛盾検出・整合性チェック）。統合アクセスAPIはリポジトリパターンで3層を透過化し、利用側がデータ所在層を意識せず横断クエリを可能にする。

データ永続化にはPostgreSQLではなくYAML/JSON + GitHub（Octokit API）を採用。理由は①人間可読でデバッグ効率が高い、②コミット履歴によるバージョン管理（イベントソーシング代替）、③運用コストゼロの3点。小説生成の書き込みワークロード（1章生成あたり数回）では検索性能の劣位が問題にならないと判断。

設計パターンはカスタマーサポートBot（セッション/問い合わせ履歴/顧客プロファイル）、教育アシスタント（学習セッション/単元理解度/学習者プロファイル）、ゲームNPCなど他ドメインにも転用可能。監査エージェント開発への示唆として、複数セッションをまたぐ監査証跡の継続追跡（短期：現在の監査手続き、中期：当該監査プロジェクトの進捗、長期：被監査先の過去指摘事項・リスクプロファイル）への応用が考えられる。実装上の推奨として、短期記憶のみのプロトタイプから段階的に構築する順序を提示している。

## アイデア

- 「忘却も設計の一部」という思想：短期記憶の72時間TTLは情報を捨てるためではなく中期記憶へ移行して粒度を下げるためであり、ノイズ除去と重要情報へのアクセス効率向上を両立する
- DuplicateResolverの「具体的な層を優先」原則：長期記憶の既定値より短期記憶の最新具体情報を優先することで、物語進行に伴うキャラクター設定の自然な変化を矛盾として誤検出せず正確に管理できる
- YAML + GitHub によるデータ永続化の設計選択：DBを使わずコミット履歴をイベントソーシング代替として活用することで、任意の時点での記憶状態の再現が可能になりデバッグ・監査トレーサビリティを確保する
