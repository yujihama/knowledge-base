## 要約

MCPサーバーを用いたPoCを通じて得られた、AIエージェント向けの知識設計とツール設計に関する実践的な知見をまとめた記事。主な主張は「たくさん与えるより迷わないように絞る」の一点に集約される。

【CLAUDE.mdの設計指針】Claude CodeはセッションのたびにCLAUDE.mdを読み込むため、肥大化すると毎回不要な情報がコンテキストに乗る。推奨構成は「絶対ルール・共通前提・参照先ナビゲーション」のみをCLAUDE.mdに残し、詳細仕様はdocs/knowledge/以下に機能別ファイルとして分割する方法。ファイル名は内容が一見で分かる命名（viewer-app-overview.md、mcp-tool-contracts.md、auth-flow.mdなど）とすることで、AIも人間も読むべきファイルを迷わず特定できる。CLAUDE.mdは情報の本体ではなく「ナビゲーションと絶対ルールの置き場」と位置づける。

【MCPツール設計指針】公開関数は10個前後かつ名前から役割が明確な状態が最も扱いやすい。問題になるのはコンテキスト超過そのものではなく、名前が似ている・役割差が小さい・入出力の差が不明瞭な関数が増えることによる「選択肢の曖昧さ」。Claude CodeのMCP Tool Searchは起動時にツール名のみ読み込み、完全スキーマは必要時に遅延ロードする設計のため、数自体は即座の問題にはならない。それでも「fetch_page_with_retry / fetch_page_with_headers / fetch_page_with_proxy」のように横展開するより、既存関数に引数オプションを追加して吸収する設計のほうがAIにとって選びやすい。PoCフェーズでは特に「似た責務の関数を増やしすぎない」ことを優先すべきとしている。

【監査エージェント開発への示唆】LangGraphなどで多数のノード・ツールを定義する際も同様の原則が適用できる。各ツールの責務境界を明確にし、命名規則を統一することで、LLMが正しいツールを選択する精度が向上する。CLAUDE.mdをエージェントの「システムプロンプト」に相当する位置づけとして設計し、詳細情報は必要時に参照させるRAG的アーキテクチャは、長期セッションの品質維持に有効。

## アイデア

- CLAUDE.mdを「情報の本体」ではなく「ナビゲーションと絶対ルールのみ」に絞り、詳細はファイル分割することでセッション開始時のコンテキスト汚染を防ぐ設計思想
- MCPの公開関数は数より命名・責務の明確さが重要で、似た関数を横展開するより既存関数への引数追加で吸収する設計がAIの選択精度を高める
- Claude CodeのMCP Tool Searchがツール名のみを起動時に読み込み完全スキーマを遅延ロードする仕組みを前提とした上で、それでも「曖昧な選択肢」が問題になるという観点
