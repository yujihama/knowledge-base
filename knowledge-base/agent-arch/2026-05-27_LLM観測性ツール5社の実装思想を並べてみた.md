## 要約

AIエージェントの本番運用で「ループ事故」を経験した著者が、2026年5月時点で本格検討した5つのLLM観測性ツール（Langfuse、LangSmith、OpenLLMetry、Helicone、Arize/Phoenix）を比較した実践的レポート。LLMアプリの観測性は「ログ→トレース→評価」の3層に分かれており、従来のAPMと前提が異なる点が核心。各ツールの思想と特徴は以下の通り。Langfuse（YC W23）はApache 2.0のOSS first設計で、v3でPostgresからClickHouseに移行し月10〜20Mトレースの本番運用が可能。v4 betaではobservation中心データモデルへ移行予定。LangSmithはLangChain純正で評価機能（Dataset/Experiment/LLM-as-judge）が充実するが、2026年からPlus $39/月・Enterprise $100K/年クラスと価格が重い。Self-hostはEnterprise限定。OpenLLMetryはOpenTelemetry GenAI Semantic Conventionsに準拠したベンダー中立ライブラリで、バックエンドにDatadog/Grafana/Jaegerを選択でき、既存APM資産を活かせる。Heliconeはbase_url書き換え1行で導入できるプロキシ型で摩擦が最小だが、2026年3月にMintlifyに買収されCloud機能がメンテナンスモードに移行。PoC専用として位置づけ推奨。Arize/PhoenixはML監視老舗がLLM観測性に展開した構成で、OSSのPhoenix（Elastic License 2.0）とエンプラSaaSのArize AXが分離。OpenInferenceというOTel互換スペックを独自定義し、データドリフト検出など MLOps統合視点が強み。ユースケース別の推奨はPoC=Helicone、本番MVP=Langfuse Cloud Hobby（無料枠50k units）、規制業界=LangSmith Enterprise or Langfuse Cloud Pro（HIPAA BAA対応）、MLOps統合=Arize AX、マルチベンダー回避=OpenLLMetry+Datadog。ツール選定の決め手として「評価ループが回せるか」「既存スタックと競合しないか」「出口戦略（ベンダーロックイン回避）を持てるか」の3点を強調。監査エージェント開発への示唆：LangGraphベースのエージェントにLangSmith or Langfuseを組み合わせ、LLM-as-judgeを評価ループに組み込むことで監査ロジックの品質を継続的に測定・改善するアーキテクチャが直結する。規制業界（金融・監査）ではHIPAA BAA対応のLangfuse Cloud ProまたはLangSmith Enterpriseが現実的な選択肢となる。

## アイデア

- 観測性を「ログ→トレース→評価」の3層に分解する視点は、AIエージェントのデバッグを体系化する上で有効。特に評価層（LLM-as-judge・Dataset/Experiment）がなければ品質改善ループが閉じないという指摘は、監査エージェントの品質保証設計に直結する
- OpenLLMetryによる「計装をOTelで書き、バックエンドを後から選ぶ」設計は、将来のベンダー変更コストを最小化するアーキテクチャパターンとして汎用性が高い。LangfuseやArize PhoenixもOTel準拠を強化しており、業界標準への収斂が進んでいる
- Heliconeの買収（Mintlify）はツール選定における「出口戦略」の重要性を実証した事例。OSS first（Apache 2.0）またはOTel準拠を選ぶことでベンダーリスクを低減できるという教訓は、本番システムの観測性設計原則として汎用的に適用できる
