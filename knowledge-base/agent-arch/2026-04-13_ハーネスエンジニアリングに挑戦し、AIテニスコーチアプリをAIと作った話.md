## 要約

著者がClaude CodeとAIエージェントを主体とした「ハーネスエンジニアリング」スタイルの開発に挑戦し、テニスメモ→AIコーチング変換アプリ「テニラボ」を構築した実践レポート。

【アプリの概要】
テニラボ（tennis-lab.exe.xyz）は、練習・試合後のメモをAIコーチが分析し、原因分析・改善提案・パターン検出を行うWebアプリ。フロントはReact 19 + TypeScript + Vite + Tailwind CSS 4 + shadcn/ui + Zustand、バックエンドはFastAPI + Python 3.12 + uv、LLMにはGoogle Gemini API（gemini-3.1-flash-lite-preview）を使用。認証・DBはSupabase、デプロイ先はexe.dev（月20ドルのVM Hostingサービス）。PWA対応済み。

バックエンドではユーザーのメモ入力と診断結果からコンテキストを構築し、structured outputでコーチ役割の回答を生成。テニスをしない日にもアプリを開く動機として、過去メモを参照したチャット機能を実装。

【開発スタイル】
Boris Tane氏が提唱する「意図→エージェント→コード+テスト+デプロイ→確認」のSDLC圧縮モデルを採用。具体的には、仕様書（docs）を先行作成する仕様駆動開発を実践し、会話エージェント自体には実装させずtasklist生成まで担当させた後、agent teamsやsubagentsを並列呼び出しすることでコンテキストの汚染を防ぎながら多タスクを並行処理。E2Eテストもエージェントに実行させた。

【判明した課題と限界】
一時期「mainブランチへAIが直接commitするスタイル」も試みたが、コードベース肥大化に伴いgit/GitHub PRベースのフローに回帰。ローカルストレージやデータ同期周りの環境差異によるバグはAIへの丸投げが困難で、再現性のある自動テスト環境（E2Eハーネス）の整備が事前に必要と結論。また、体験フロー設計（Hookedモデルの適用、チャット機能による日常接点の創出）やChatGPTとの差別化戦略は人間が担う領域として残ることも確認。

【監査エージェント開発への示唆】
subagents並列化によるコンテキスト分離戦略は、監査エージェントでの複数証跡収集タスクの並列実行設計に直接応用可能。structured outputによるLLM応答の型強制はPydanticとの親和性が高く、エージェント出力の信頼性確保に有効。また、E2Eテスト整備なしにエージェント自律化を進めると品質劣化を招くという知見は、監査ワークフロー自動化においても重要な教訓となる。

## アイデア

- 会話エージェント自体には実装させずtasklist生成までに留め、並列subagentsに実装を委譲することでコンテキスト汚染を防ぐ設計パターン
- Hookedモデルを参照した「テニスをしない日にもアプリを開く理由」の設計：過去メモ参照チャットによる非練習日のエンゲージメント維持
- E2Eテストハーネスが未整備のままAI自律コミットスタイルを採用するとコードベース肥大後に崩壊するという実証的な失敗知見
