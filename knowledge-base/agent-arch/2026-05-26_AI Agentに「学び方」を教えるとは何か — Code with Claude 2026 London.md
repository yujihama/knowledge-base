## 要約

Anthropic主催の開発者カンファレンス『Code with Claude 2026』でのWarp社Petra Donka氏によるセッション「Teaching agents to learn from your team」の解説記事。Warpの社内エージェント「Buzz」の実装・運用事例を通じて、判断力を要するタスクにおけるエージェント設計の実践知を体系化している。

BuzzはTwitter・Reddit・Bluesky・LinkedInの言及を監視し、「返信すべきか」「いいねのみか」「無視か」を判断してドラフトをSlackに流すエージェント。最終投稿は必ず人間が行う設計で、コードは0行・約15個のSkillファイル（Markdown）のみで構成される。Skillファイルはgitで版管理可能なため、既存のPRレビュー文化にそのまま乗せられる点が運用継続性の鍵となっている。

設計過程は3段階の試行錯誤として整理される。第1試行では「バグの言及があればこう返す」式のif-thenルールリストを構築したが、未知ケースで即崩壊する「too brittle」な結果に終わった。第2試行では「Be helpful, not defensive」「プロダクトを作っている人間のように振る舞え」といった原則（principles）への転換を図り、Skillファイルの分量が元の1/5に縮小しながら出力品質は向上した。第3試行では「学び方」そのものをエージェントに教える段階として、フィードバックループの設計に移行。Slackのスタンプ（絵文字リアクション）をフィードバック信号として活用し、チームの既存コミュニケーション動線に学習ループを埋め込むことで、摩擦なく運用知識をエージェントに蓄積する仕組みを実現した。

本記事の核心的な洞察は、エージェントが「動くが任せられない」状態（80%地点）で停滞する原因が、プロンプト磨きという「間違った抽象度のレイヤ」で改善を続けることにある点。コード生成のように外部チェック（ユニットテスト・コンパイル）が定義できる領域と異なり、ソーシャル返信・UX文言・コードレビューコメントのような「判断力勝負の領域」では、外部チェックが存在しないため、人間の判断そのものを学習信号として取り込む観測設計が必要になる。監査エージェント開発への示唆として、監査判断・リスク評価コメント・報告書のトーンといった「外部チェックが困難な仕事」においても、レビュアーのフィードバックをSlack等の既存動線に乗せて収集・蓄積する設計パターンが直接適用可能である。

## アイデア

- 「コード0行・Markdownのみ」構成：Skillファイル約15枚でエージェントを構築し、gitのPRレビュー文化でそのまま版管理・ロールバックできる設計は、エージェント導入の組織的ハードルを大幅に下げる
- ルール→原則→学び方の3段階進化：詳細なif-thenルールリストから原則記述に移行するとSkillファイルが1/5に縮小しつつ品質向上、さらに人間フィードバックを学習信号化する段階へと昇華させる進化パターン
- 既存コミュニケーション動線へのフィードバックループ埋め込み：Slackスタンプ（絵文字リアクション）を学習信号として活用し、チームが新たなツールやワークフローを覚えることなくエージェントを育て続けられる設計
