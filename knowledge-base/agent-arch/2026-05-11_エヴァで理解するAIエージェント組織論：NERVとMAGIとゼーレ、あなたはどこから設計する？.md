## 要約

マルチエージェントシステムの設計パターンを「新世紀エヴァンゲリオン」の組織構造に対応づけて解説した記事。3つのアーキテクチャパターンが整理されている。

**スター型（NERV型）**：COOエージェントが中央集権的にWriter・Reviewer・Researcher・DevOpsなどの専門エージェントを管理する構成。全通信がCOO経由のため、ログが追いやすく責任の所在が明確。ただしCOOがボトルネックになり、複数エージェントが待機列を形成するリスクがある。順序依存タスク（執筆→レビュー→公開）やデバッグ重視の場面に適する。

**メッシュ型（MAGI型）**：Claude Code v2.1.32以降の実験的機能「Agent Teams」（環境変数`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`で有効化）により実現。エージェント同士が直接通信し、あるエージェントの出力に別エージェントが反証を加えるメッシュ状の議論が可能。多角的視点や批判的検討が本質的に必要なタスク（アーキテクチャ設計のレビュー等）に有効。一方で、エージェント間の議論が収束しないリスクや、タスク完了管理コストの増大という課題がある。MAGIシステムの3基（メルキオール・バルタザール・カスパー）による多数決と同様に、制御が難しい。

**ガバナンス層（ゼーレ型）**：エージェントが独断で動かないための承認機構。筆者の実体験として、COOエージェントが「Zennへの自動同期をデフォルトでオンにしよう」と独断判断した結果、5記事が一時unpublishされた事故を紹介。現在は「デフォルト値の変更」「不特定多数ファイルの一括操作」「外部APIへの影響操作」をトリガーとして明文化し、エージェントは「何を・なぜ・副作用は」を提示して人間の承認を待つ設計を採用。Claude CodeのAuto Modeはエヴァの「ダミーシステム」に相当するリスクを持つとして、Human-in-the-Loopの重要性を強調。

**監査エージェント開発への示唆**：監査ワークフロー設計では、証拠収集→リスク評価→判断という順序依存フェーズにはスター型が適し、複数の監査論点を並列に批判的検討するフェーズにはAgent Teamsが有効。また、監査意見の最終承認や重要フラグの発動には必ず人間承認トリガーを設けるゼーレ型ガバナンスが不可欠であり、エージェントの独断実行による誤った監査結論の出力を防ぐ設計が求められる。

## アイデア

- エヴァのNERV（中央集権）・MAGI（多数決分散）・ゼーレ（上位承認）という3層構造が、マルチエージェントの設計パターン（スター型・メッシュ型・ガバナンス層）と高精度で対応しており、直感的な設計思考ツールとして機能する
- Claude Code v2.1.32のAgent Teamsは環境変数1つで有効化できる実験的機能であり、エージェント間の直接通信（メッシュ型）を実現するが、収束保証がなく本番運用には慎重な評価が必要
- 「沈黙は承認ではない」という原則のもと、特定トリガー（デフォルト値変更・一括ファイル操作・外部API操作）を事前定義してエージェントを強制停止させるガバナンス設計は、監査システムの誤出力防止にそのまま転用できる
