## 要約

pi-monoはbadlogic氏が開発したTypeScript製モノレポで、LLM API抽象化層・agent runtime・CLI・UI・Slack連携・GPU pod管理までを含むスタック構成をとる。パッケージは`pi-ai`（LLM provider抽象化）、`pi-agent-core`（statefulなagent loop）、`pi-coding-agent`（対話CLI）、`mom`、`pods`、`tui`、`web-ui`の7つに分かれており、必要な層だけを単独利用できる点が特徴。`pi-ai`は2026年4月時点でOpenAI・Anthropic・Google・Vertex AI・Mistral・Groq・Cerebras・OpenRouter・Bedrock・GitHub Copilot・Gemini CLIを含む23 provider/categoryをサポートし、provider抽象化が中核に置かれている。`pi-coding-agent`のbuilt-in toolsは`read`・`bash`・`edit`・`write`・`grep`・`find`・`ls`の7個に絞られており、最小限のファイル操作と実行のみをコアに残す方針。最大の特徴はPhilosophyセクションに示された「No MCP・No sub-agents・No permission popups・No plan mode・No built-in to-dos」という意図的な非搭載設計で、ワークフローを固定しないために定番機能をあえて外している。拡張面は厚く、`packages/coding-agent/examples/extensions`には94個の関連ファイル・65本のTypeScript例、`packages/coding-agent/docs`には24本のドキュメントが整備されている。extensionでは、custom tool登録・tool callのblock・custom command追加・custom UI追加・session persistence・custom rendering・custom provider登録が可能で、jitiによりTypeScriptをビルドなしで読み込める。記事著者は実際にfaux providerを使い、`rm -rf sandbox`を`/rm\s+-rf/`パターンで検知してblockし、`pwd`は通過させるguard extensionを実装。tool callのinterception・条件付きblock・block後のsession継続・安全なコマンドの通過という4点を実証した。npmの`@mariozechner/pi-coding-agent` v0.70.2のunpacked sizeは約11MBで依存関係は21個。監査エージェント開発への示唆として、permission flowの自作・独自tool混在・セッション制御・provider差し替えが必要な社内エージェント基盤構築に適したアーキテクチャであり、特にtool call interceptionによるguardレイヤーの実装パターンはLangGraphベースの監査エージェントにおける操作制限機構の設計参考になる。

## アイデア

- MCP・sub-agents・permission popup・plan mode・built-in to-doをあえて標準搭載しない「前提を削る」設計思想：現場ごとに答えが異なる機能は固定せずextensionに委ねることで、ワークフローの自由度を最大化する
- tool callのinterceptionによるguardレイヤー実装：extensionがtool_callをフックして危険なbashコマンドをblock後もsessionを継続させる仕組みは、監査エージェントにおける操作権限制御の実装パターンとして直接応用可能
- pi-ai・pi-agent-core・pi-coding-agentの3層分離：LLM provider抽象化・stateful agent loop・対話CLIを独立させることで、monorepo全採用ではなく1〜2層だけを既存システムに組み込む選択肢を提供する
