## 要約

本記事は、現行のAIエージェントエコシステムが「ReActループ（Thought→Action→Observation）」を全ての業務に持ち込もうとしているというカテゴリエラーを指摘し、業務の性質に基づく4象限フレームワークを提唱する。

横軸は「決定論で書けるか／意味判断を要するか」、縦軸は「ワークフローが事前定義可能か／探索的か」の2軸。これにより(1)スクリプト象限、(2)Algorithmic Search象限、(3)LLMワークフロー象限、(4)Autonomous Agentic Loop象限の4つに分類される。

主な主張は「業務AIの主戦場は(3)であり、(4)のReActが正当に必要な領域は極めて狭い」というものだ。(1)はスクリプトで十分、(2)はOR/探索アルゴリズムの領域でLLM不要、(3)は経路が事前定義可能なためworkflow+単機能LLM関数で足りる。(4)が必要なのは、コーディングエージェント・Deep Research・動的なブラウザ操作など、次の行動をモデル自身が実行時に決定しなければ進めない探索的タスクに限られる。

(3)バッチ形の具体例として請求書マッチングを示す。発注書との照合処理の80%は決定論ルール（PO存在確認・期日・重複・金額誤差1%以内）で処理でき、「line itemの表現違いによる実質一致判定」という1ポイントだけに`match_line_items(invoice_lines, po_lines) -> MATCH/PARTIAL/NO_MATCH`という単機能LLM関数を挿入する設計が提示される。次のステップは常にパイプライン側が決定し、LLMが自律的に行動を選択する余地はない。

「LLMは確率的だから業務に向かない」という反論に対しては、「人間の例外判断もそもそも確率的であり、LLM関数は人間が確率的に担っていた役割をそのまま引き受けるに過ぎない」と反論する。汎用エージェントは不要で、50カテゴリあれば単機能関数50個で足りる。

(3)アーキテクチャを取ることでアカウンタビリティも整理される。LLM呼び出しごとに入力・出力が明示され、パイプラインログで処理追跡が完全に可能となり、「次の行動を誰が決めたか」が常に明確になる。責任主体は常にデプロイ者（人間）に帰属し、製造物責任モデルに乗る。

現行エコシステムが(4)の語彙で全象限を覆う背景にはマーケティング問題がある。Thoughtworksの「agentwashing」批判、GartnerによるAgentic AIプロジェクトの40%以上が2027年までにキャンセルされるという予測、Anthropic自身の「Building Effective Agents」における「エージェントを建てないという選択肢もある」という指摘が、本記事の主張を裏付ける業界コンセンサスとして引用されている。

監査エージェント開発への示唆：監査ワークフローの多くは経路が事前定義可能な(3)象限に属する。証跡収集・仕訳検証・異常検知といった処理は決定論パイプライン＋単機能LLM関数で設計すべきで、ReActループの導入はアカウンタビリティを損なうリスクがある。LangGraphを使う場合も、ノード間の遷移ロジックをLLMに委ねるのではなく、確定的なエッジで制御する設計が業務適合性を高める。

## アイデア

- 「判断業務こそ自律エージェントが不要かもしれない」という逆説：判断＝思考＝エージェントという短絡を解体し、判断業務の思考の多くは知識検索・整理であり、ReActループが必要な探索的推論とは別種だと論じる点が鋭い
- LLMの確率性を欠点ではなく「人間の確率的判断の代替」として再定義する視点：例外判断は本来確率的な仕事であり、LLM関数の揺らぎは障害ではなくむしろ適合だという転換は、業務AIの導入議論に対する強力な反論構造を提供する
- アカウンタビリティと設計の対応関係：(3)アーキテクチャを選ぶことで責任帰属が明確化されるという主張は、監査・コンプライアンス文脈でのAI導入判断に直接使えるフレームであり、「なぜReActを使わないか」の説明根拠として機能する
