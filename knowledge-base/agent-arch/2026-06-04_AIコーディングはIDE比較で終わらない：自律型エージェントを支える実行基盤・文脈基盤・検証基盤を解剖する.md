## 要約

本記事は、AIコーディングを「ツール名の比較」ではなく「基盤設計の問題」として再定義する論考である。Cursor・Windsurf・Claude Code・Devin・GitHub Copilotといったツール比較が主流になりがちな議論に対し、著者はそれらを成立させる裏側の構造—AI Coding Infrastructure—を5層で整理している。

L1（Interaction Layer）はpromptやissueで人間の意図をAIへ渡す層。L2（Context Layer）はRAG・AST・GraphRAG・rulesでAIがコードベースを理解する層。L3（Execution Layer）はshell・container・VM・sandboxでAIが変更・実行する層。L4（Verification Layer）はtest・lint・typecheck・CI・SWE-benchでAI出力を検証する層。L5（Governance Layer）はpermissions・review・PR・approval・logsで人間が責任境界を管理する層である。

実行基盤については、ローカル端末（Claude Code、Cline）、クラウドVM（Devin、OpenAI Codex Cloud）、WebContainers（StackBlitz）、CI環境（GitHub Actions）の各形態のリスクを比較し、ファイル削除・環境変数読み取り・DB接続・本番認証情報へのアクセス可否を事前に分類することが不可欠と主張する。サンドボックスはL0（なし）からL4（ephemeral VM）までレベル分けされ、devcontainer・Docker・Nix・Tvixが実用的な選択肢として挙げられる。

文脈基盤では、RAGだけではコードベース理解が不十分であるとし、Tree-sitter（AST）・ast-grep・Sourcegraph Codyのようなcode graph・GraphRAGによる決定論的根拠が必要だと強調する。GitHub Copilot instructions・Claude Code memory・Cursor Rules・Cline Rulesといった「rules/instructions」の設計も文脈管理の重要要素として位置づける。

検証基盤については、TypeScript型検査・ESLint・Playwright・GitHub Actionsを組み合わせた多層検証を推奨し、SWE-benchやAIDev研究などの外部評価指標も参照する。AIが「テスト未実行のまま完了宣言する」リスクを防ぐため、CI統合とhuman merge authorityの維持が不可欠とする。

権限基盤では、Claude Code Permissions・Cline tools・MCPを例に「AIがどこまで操作できるか」の境界設計を論じ、巻き戻し基盤としてGit branch・PR・checkpointの活用を推奨する。統治基盤ではaudit log・policy・review・human approvalの整備が組織運用の前提条件と結論づける。

監査エージェント開発への示唆として、この5層モデルはLangGraphベースのReActエージェント設計に直接適用可能であり、特にL5（Governance Layer）の権限境界設計とL4の検証基盤はAI監査エージェントの信頼性担保に不可欠な要素である。

## アイデア

- AIコーディングの成熟度をツール名ではなく「実行基盤・文脈基盤・検証基盤・権限基盤・統治基盤の整備度」で測るという評価フレームワークは、組織導入時の客観的な判断基準として実用的
- RAGだけでなくAST・型情報・依存グラフなど決定論的根拠を文脈基盤に組み込む設計思想は、LLMの推測ベース誤動作を抑制する構造的アプローチである
- エディタの役割が「書く場所」から「AIと人間が変更を調停し、エージェントを統治する場所」へ移行するという概念的転換は、IDEアーキテクチャの再設計方向性を示唆する
