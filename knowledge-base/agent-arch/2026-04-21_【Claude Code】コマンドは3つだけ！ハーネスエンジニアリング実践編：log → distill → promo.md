## 要約

ハーネスエンジニアリングとは、AIエージェントの出力品質をプロンプト単体ではなく、周辺環境（ルール・コマンド・ワークフロー）の設計で継続的に改善していく手法。本記事はHappy Elements カカリアスタジオによる実践事例で、Claude Codeの運用において `/harness:log` → `/harness:distill` → `/harness:promote` の3コマンドのみでフィードバックループを構成する方法を解説する。

**3コマンドの役割**
- `/harness:log`：セッション内でClaudeが犯したミス（例：lintルール違反、クォートスタイル違反）を「何が起きたか」「どういう条件で起きたか」という構造化ジャーナルとして `.claude/harness/log/` に書き出す。
- `/harness:distill`：蓄積されたジャーナル群を横断分析し、頻発パターンをルールやエージェント定義の候補として `.claude/rules/local/` や `.claude/agents/local/` に出力する。
- `/harness:promote`：個人用（`local/`）として効果を確認したルールをチーム共有の `.claude/rules/` に昇格させ、git管理下に置く。

**ディレクトリ設計の特徴**
`.claude/` 配下を「チーム共有（git管理）」と「個人用（gitignore対象の `local/`）」の2段構成にすることで、個人の試行錯誤がチームのノイズにならない。ジャーナル自体もgitignoreとすることで気軽に記録できる。

**具体的なサイクル例**
1. Claudeがシングルクォート規約に沿わない文字列リテラルを3回連続で出力
2. 開発者が指摘・修正後に `/harness:log` を実行 → `date / event / context / frequency` 形式のジャーナルが生成
3. `/harness:distill` を実行 → 「文字列は式展開がない限りシングルクォートを使う」という候補ルールが自動抽出
4. `/harness:promote` でチーム共有ルールに昇格 → 以降のセッションでClaudeの前提知識に組み込まれる

**得られた効果**
Birgitta Böckelerの「keep quality left」（品質チェックを上流に寄せる）の考え方が自然に実現される。「テストで気づく→lintで気づく→そもそもClaudeがそう書かなくなる」という前倒しが達成され、同じ指摘の繰り返しが激減。会話の中心が指摘作業から設計議論へとシフトする。

**補助コマンド**として `/harness:status`（ハーネスの育ち具合を俯瞰）、`/harness:score`（カバレッジ・健全度を数値化）、`/harness:rebuild`（ルール群が肥大化した際の再構築）、`/harness:clean-local`（ローカル設定の整理）も存在するが、必須ではなく段階的に導入可能。

**監査エージェント開発への示唆**：監査ルール違反の繰り返しパターン（例：証跡確認の抜け漏れ、リスク評価基準の不統一）をジャーナル化→蒸留→ルール昇格するサイクルに応用できる。チェックリスト的な監査手続きをエージェントの前提知識として構造化することで、同じ監査品質の問題が再発しない自己改善型エージェントを構築できる。

## アイデア

- 「log → distill → promote」の3段階により、その場の口頭指摘をgit管理可能なチームルールへと昇格させる構造化パイプラインを実現している点—暗黙知の形式知化を自動化するメカニズムとして応用範囲が広い
- 個人用（local/）とチーム共有の2段構成により、実験的なルールが本番ハーネスを汚染しない設計になっている点—ソフトウェアの「ステージング環境」概念をエージェント設定管理に適用した発想が興味深い
- Böckelerの「keep quality left」をAIエージェント文脈に翻訳し、品質チェックの前倒しをルールの蓄積で自動実現する点—CI/CDのシフトレフト思想をLLM出力品質管理に応用したフレームワークとして参照価値がある
