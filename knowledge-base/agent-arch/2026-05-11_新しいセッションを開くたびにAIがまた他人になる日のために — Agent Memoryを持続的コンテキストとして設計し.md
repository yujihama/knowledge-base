## 要約

LLMアプリケーション・Agentシステムにおける「クロスセッション記憶問題」を、Memory Architectureの欠如として診断し、設計上の解決策を提示する記事。現状のAIツールは「シングルセッション内のIQ」は高いが、セッションをまたぐと知性がリセットされる。ユーザー側がクリップボード経由でコンテキストを手動同期（「人間API」）せざるを得ない状況は、UIの不便さではなくMemory Architectureの構造的不足である、と著者は主張する。

チャット履歴の直接埋め込みは会話の長期化とともにノイズが蓄積し、ハルシネーションの温床になる。RAG（Vector DB類似度検索）は静的な外部事実の検索には優れるが、「ユーザーの意思決定履歴」や「なぜそのアーキテクチャを選んだか」という動的な状態・文脈管理には不向きである。要約圧縮（Summary Compression）は重要なニュアンスやコードの細部がロッシーに削ぎ落とされ、反復により情報が劣化する。これら3つはすべて「記憶の代替品としての局所パッチ」に過ぎない。

真の連続性を持つAgentには、4層の設計が必要とされる。①Session Layer：揮発性の短期記憶（現コンテキストウィンドウ）、②Retrieval Layer：RAGが担う静的外部知識の検索、③Memory Layer：ユーザーの好みや意思決定背景など時間とともに進化し続ける長期文脈の永続化、④Orchestration Layer：「今どの記憶を呼び出すか」「新情報で過去記憶をどう更新するか」を調停する層。多くのシステムでは③Memory Layerがすっぽり抜け落ちている。

開発者が求めるMemoryの要件として、Persistent（セッション切断後も保持）、Portable（Agent・モデルをまたいで共有可能）、Governed（誤った記憶の明示的な上書き・削除が可能）の3点が挙げられる。これを実現するために、Vector DBに会話Embeddingを詰め込む自前実装では記憶のコンフリクト解決・時間軸管理・エンティティ更新・忘却曲線への対応が困難になる。

解決策としてMemoryLakeというプロダクト/概念が紹介される。「AIのためのMemory Passport」として、クロスセッション・クロスエージェント・クロスモデルで一貫したコンテキストを持ち運ぶ基盤であり、CRUDライクに記憶を扱えるインフラ抽象化レイヤーとして位置づけられる。監査エージェント開発への示唆として、複数の専門Agent（リサーチャー・コーダー・レビュアー）が協調するマルチエージェントシステムにおいてAgent間で「前提知識」を共有させる際、このMemory Layer設計は直接適用可能な参照アーキテクチャになり得る。

## アイデア

- 記憶をRAGプラグインやDBの1テーブルとして扱うのではなく、独立したInfrastructureとして設計するパラダイムシフト（Memory as Infrastructure）
- Chat History・RAG・Summary Compressionの3手法をすべて「記憶の代替品としての局所パッチ」と整理し、それぞれの失敗モードを具体的に示した分類フレームワーク
- Persistent / Portable / Governed の3要件で記憶インフラを評価する基準は、監査エージェントのトレーサビリティ設計（誰がいつ何を判断したかの記録管理）に直接転用できる
