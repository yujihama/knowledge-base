## 要約

LLMエージェントをローカルから本番環境へ移行する際に頻発する4種類の障害パターンと、それを「物理的制約」で防ぐ実装手法を解説した記事。AOS（AI Operating Standard）v0.2の仕様更新として、§10 Implementation Examplesを新設し、動作確認済みの公開リポジトリ physical-agent-patterns と仕様を接続した。

障害パターンは以下の4つ。①「黙って止まる」：例外を飲み込んで正常終了を偽装する。②「痕跡がない」：会話上は完了を宣言するがディスクに証拠が残らない。③「再起動で消える」：対話セッション内でしか動かず永続性がゼロ。④「違反を自己申告しない」：ルール違反を犯したエージェント自身が報告しない。

これらに対応する4つの物理パターンを実装例付きで提供する。

§10.1 Manifest宣言：ツールが自身のゾーン境界（oracle_paths／permitted_output_paths）をmanifest.jsonで機械可読に宣言する。宣言単独では準拠とならず、PreToolUseフックによる実行時強制とセットで初めて有効。

§10.2 物理証拠パターン：完了宣言の前提としてevidence JSONファイルをディスクに書き込む順序を強制する。呼び出し側はファイル存在確認のみで完了を検証でき、会話上の「やりました」に依存しない。

§10.3 免疫ループ：常駐エージェントがAOS違反をスキャンし、検知レポートをJSONとして書き出した後に修復プランナーへ渡す。検知と修復を別プロセスに分離することでロール分離（§4.3）を実現し、設計判断が必要な場合は人間（Sovereign）へエスカレーションする。

§10.4 systemdランタイム：agent.serviceとagent.timerでエージェントをOS supervisor管理下に置く。Persistent=trueにより再起動後の初回起動時に未実行スケジュールを補完し、冪等ガード（output_path.exists()チェック）で重複実行を防ぐ。出力ファイルが実行証拠として再起動をまたいで残存する。

v0.2の更新方針は「規範（MUST/MUST NOT）は変更せず、§10の実装例を追加して仕様をコードに接続する」という完全後方互換。監査エージェント開発においては、物理証拠パターンと免疫ループは監査トレイルの自動生成・改ざん防止・異常検知の設計に直接応用可能であり、エージェントの自己申告に依存しないガバナンス構造を実現する参考実装として価値が高い。

## アイデア

- 「証拠ファイルが存在して初めて完了を名乗れる」というphysical-firstパターンは、会話ターン上の自己申告を信頼せず、ファイルシステムを唯一の真実の源泉（Single Source of Truth）とする設計思想で、監査証跡の自動生成に直接転用できる
- 検知（読取専用スキャン）と修復（書き込み）を別プロセスに分離する免疫ループは、エージェントが自分の違反を自分で隠蔽できない構造を作り、内部統制の「職務分離」原則をコードレベルで実装したものと見なせる
- manifest.jsonによるゾーン境界の宣言と、PreToolUseフックによる実行時強制を「宣言と強制のセット」として要求する設計は、ポリシーをコードで検証可能にするPolicy-as-Codeアプローチをエージェントに適用した事例
