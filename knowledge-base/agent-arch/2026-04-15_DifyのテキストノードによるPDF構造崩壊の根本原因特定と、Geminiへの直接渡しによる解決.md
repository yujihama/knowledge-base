## 要約

採用候補者の履歴書・職務経歴書をDifyワークフローで解析し、資格取得年月や卒業年月をJSON形式で構造化出力するシステムで、日付が実際の書類と異なる値で出力されるバグが発生した。具体的には、資格Aの正しい取得日2024-06が2022-11（1行下の資格の日付）として出力されたり、学歴の卒業年月が系統的に1行ずれて紐づけられるという症状が見られた。

最初のアプローチとしてプロンプト改善を試みた。「文書に明記されている日付のみ使用」の強調、和暦→西暦換算表（令和・平成・昭和）の明記、Chain-of-Thought的な段階的手順の明示、セクションをまたいだ日付流用禁止の明記を実施した。またGemini 3 FlashからGemini 3.1 Proへのモデル切り替えも試みたが、いずれも根本解決に至らなかった。

根本原因の特定はDifyの実行ログでテキスト抽出ノードの出力内容を直接確認することで行った。その結果、Excelベースで作成された複雑な表形式PDFに対して、テキスト抽出ノードがPDFの列構造を正しく認識できておらず、テーブルのラベル（「年 月入学」等）と実際の値（2021, 2023等）が完全に分離・散乱した状態でGeminiに渡されていたことが判明した。左列（資格名・年月）と右列（語学情報）が混在し、LLMはどの数値がどの項目に対応するか判断不能な状態だった。

解決策として、PDFの場合はテキスト抽出ノードを経由せずGeminiにPDFを直接渡すアーキテクチャに変更した。GeminiはPDFをビジョン機能でネイティブに読めるため、表のレイアウトをそのまま認識できる。ただし実運用ではExcel・CSVなど複数形式が想定されるため、IF/ELSEノードでファイル拡張子（.pdf）を判定し、PDFはGemini直接渡し・それ以外は従来のテキスト抽出経由とするブランチ構成を採用した。

分岐後の出力は Variable Aggregator で集約し、実行されなかったブランチのnull値がOUTPUTノードに渡らないよう対処した。さらにLLMノードが2つになったことでプロンプトの二重管理問題が生じたため、DifyのEnvironment Variables機能を使いシステムプロンプトを1変数（PARSE_PROMPT）で一元管理し、両LLMノードから参照する構成とした。

監査エージェント開発への示唆として、PDFや複雑なドキュメントを処理するパイプラインでは「LLMへの入力品質の保証」が最優先事項であり、プロンプトエンジニアリングの前にデータ抽出層の正確性を検証することが重要。監査証跡や財務書類など表形式PDFを扱う場面では、テキスト抽出を介さずマルチモーダルモデルに直接渡す設計が有効な選択肢となる。

## アイデア

- プロンプト改善よりも上流のデータ抽出品質を先に検証するというデバッグ順序の原則：LLMの出力がおかしいとき、モデルやプロンプトの問題と決めつける前に実行ログでLLMへの実際の入力を確認することが根本原因特定の近道
- Dify IF/ELSE + Variable Aggregator パターン：ファイル形式ごとに処理ブランチを分岐しつつ、後続ノードがnullを受け取らないようにVariable Aggregatorで出力を集約する設計はDifyワークフロー設計のベストプラクティスとして汎用性が高い
- Dify Environment Variables によるプロンプト一元管理：複数のLLMノードで同一プロンプトを共有する際、環境変数を参照させることで保守コストを削減し、片方の更新漏れによる挙動不一致を防ぐ手法
