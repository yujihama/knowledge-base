## 要約

AIエージェント基盤を「Agent Harness」として捉え、L1〜L10の10層と2つの横断層（X: Observability、Y: Supply Chain & Repo Contract）に分解するフレームワークを提示する記事。チーム内でOSS/SDKを比較・議論する際に「見ているレイヤーが違う」問題を解消するための共通地図として設計されている。

10層の構成は次の通り：L1 Model（LLM本体）、L2 LLM Gateway/Router（LiteLLM等によるルーティング・フォールバック）、L3 Governance/Policy（実行前のポリシー判定）、L4 Execution Sandbox（Docker/E2B/Daytonaによる隔離）、L5 Memory & State（セッション横断の知識・状態保持）、L6 Capability Layer（MCPサーバー/Agent Skillsによる外部接続）、L7 Agent Runtime/Loop SDK（Claude Agent SDK・OpenAI Agents SDK等の実行ループ）、L8 Multi-Agent Coordination（LangGraph・CrewAIによる役割分担・並列実行）、L9 Durable Execution（Temporal等による長時間タスクのdispatch・再試行）、L10 Task Source（GitHub Issues・Linear・Slack等の仕事の起点）。

後半ではDocker Agentをこの地図にマッピングする実例を提示。L4 Sandbox（Docker Sandboxでネイティブ完結）とL7 Runtime（YAML/CLI駆動・OCI registry配布）は✅、L9 Durable Execution（停滞検知・workflow replayは外部補完が必要）は⚠️という評価となっている。

この地図の実用的な使い方として、Claude Agent SDKをL7として採用した場合にL1（Anthropicモデルへのロックイン）が発生するという具体例を挙げ、「どの層のロックインを、どの時間軸で許容するか」を議論するフレームとして活用できると説明している。

監査エージェント開発への示唆：L3 Governance/Policyは内部監査ユースケースで特に重要な層で、Microsoft Agent Governance ToolkitやOWASP Agentic Top 10を参照した設計が必要。また、L9 Durable Executionは長時間・複数ステップの監査タスク（証跡収集→リスク評価→レポート生成）を安定運用する上で不可欠な層であり、Temporalなどの採用検討が有効。

## アイデア

- L7（Runtime）だけでなく10層全体で基盤を評価するフレームワークは、OSS選定時の「見ているレイヤーが違う」問題を構造的に解消する。監査エージェント設計でも同様に、PoC段階ではL7中心になりがちだが、本番運用にはL9（再実行）やX（トレース）の設計が必須という気づきを与えてくれる
- L3 Governance/PolicyをL4 Sandboxと明示的に分離している点が重要。L3は「実行してよいか」の事前判定、L4は「実行されても閉じ込める」事後隔離であり、監査AIにおけるリスク管理の2段階防御として直接応用できる
- Y Supply Chain & Repo Contract（AGENTS.mdのような指示書・スキル・依存管理）を横断層として定義しているのは新しい視点。エージェントが毎回同じ前提で動けるようにリポジトリ側に契約を持たせるという考え方は、監査手続きの標準化・再現性確保と概念的に一致する
