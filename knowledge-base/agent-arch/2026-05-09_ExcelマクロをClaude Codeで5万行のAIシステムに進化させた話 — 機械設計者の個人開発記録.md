## 要約

機械設計エンジニア（本業3年目）が、2年前にExcelマクロで構築した競馬予想システム「Diva」を、Claude Codeを活用して半年で約5万行のAIシステムに再構築した記録。

旧Excel版の限界は明確だった：(1)過去走の手動貼り付け、(2)複雑な計算式による再計算の重さ（数分の固まり）、(3)芝・ダート・馬齢等のレース条件パターンへの非対応。新DIVA（AI版）ではこれら全てを解消した。

技術スタックはバックエンドにPython＋FastAPI、フロントエンドにPWA（Progressive Web App）、データベースにSQLiteを採用。コード規模はフロントエンド（HTML/JavaScript）が33,183行、バックエンド（Python）が19,279行、合計約5万行（ライブラリ・自動生成コードを除く）。主要機能は5つ：出走表データの自動取得・解析、33の独自特徴量を組み合わせたスコアリング、過去結果から重みを自動最適化する自己学習、券種別（単勝/複勝/3連単等）の購入条件学習、PWAによるスマホオフライン対応。特徴量は当初22個から改善を重ねて33個に拡張。半年運用で回収率200%超の券種が複数生まれた。

6ヶ月の開発で得た3つの気づきが本記事の核心。第1に「AIエージェントの真価は実装速度より設計判断の質にある」：ボーナスlearner機能を数時間で実装したが評価スコアが悪化し、7往復のコミット格闘の末にrevertした経験から、書く前の設計判断の重要性を学んだ。第2に「個人で作れる範囲は予想以上に広い」：フロント+API+DB+デプロイ+CI/CDという従来なら小規模チームが担っていた範囲を、AIエージェントと2人で動かせる時代になっている。なおPWAのService Workerによる過剰キャッシュ問題はService Worker無効化で根本解決した。第3に「ドメイン知識が決定的に重要」：「展開予測」特徴量を6回作り直した経験から、脚質×ペース変化のインパクトというドメイン理解なしにはコードだけでは解決できないことを実感。

監査エージェント開発への示唆：AIによる高速実装が設計ミスも高速に拡大させるリスクは監査システムでも同様。「何を検証するか」の判断軸（監査ロジック・内部統制の概念）は人間側のドメイン知識として保持し続ける必要がある。また、時系列データからスコアリングロジックを設計・自己学習させるアーキテクチャは、監査証跡の異常スコアリング設計に直接応用可能な発想を含む。

## アイデア

- 「AIは設計ミスも高速に実装する」という逆説：実装速度が上がるほど、誤った設計判断のコストも比例して上がる構造。設計レビューフェーズの重要性が増す
- ドメイン知識×AI実装の役割分担モデル：AIがコード生成を担い、人間がドメイン固有の判断軸（脚質×ペース変化等）を持ち込む分業構造が、複雑な予測システムで有効
- 33特徴量の自己学習ループ：レース条件パターン別に重みを最適化し、過去結果から自動更新するサイクルが半年で安定稼働した事実は、強化学習的なフィードバックループの実用性を示す
