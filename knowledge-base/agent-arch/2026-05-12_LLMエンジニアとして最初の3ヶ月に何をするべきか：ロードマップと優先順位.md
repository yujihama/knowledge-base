## 要約

Python経験者がLLMエンジニアとして実務レベルに達するための3ヶ月ロードマップを、月ごとの具体的なコードと行動指針で解説した実践記事。

【スキル4層構造】習得には依存関係があり、「基礎層（API呼び出し・プロンプト設計・埋め込み）→実装層（RAG・チャンキング・ベクトルDB）→評価層（RAGAS・LLM-as-judge）→応用層（エージェント・Fine-tuning・本番化）」の直列順が最短経路と主張する。

【1ヶ月目】OpenAI APIの`chat.completions.create()`を動かすことから始め、temperature・max_tokensの挙動実験、few-shotプロンプト設計の感覚習得へ進む。後半はChromaDB + `text-embedding-3-small`を使った最小構成RAGを実装し、さらにBM25とベクトル検索を`EnsembleRetriever`（RRF統合）で組み合わせるハイブリッド検索まで到達する。固有名詞・型番検索にはキーワード検索が不可欠であるため、実務では最初からハイブリッド構成を推奨。

【2ヶ月目】RAGASライブラリで`faithfulness`（コンテキストへの忠実度）・`answer_relevancy`（質問適合性）・`context_recall`（検索網羅率）の3指標を定量計測し、プロンプト改変のたびにJSONログとして差分を記録する改善サイクルを確立する。Golden Set作成コストが高い場合はGPT-5.5をjudgeとして使うLLM-as-judge手法で代替する。

【3ヶ月目】OpenAI Agents SDKを使ったReActエージェント実装（tool_choice, handoffs）、Fine-tuningの要否判断（基準：few-shotで改善しない・出力形式が複雑・プロンプトが肥大化）、本番化（FastAPI + Pydanticによるスキーマ検証・リトライ・コスト計測・観測可能性）まで扱う。

監査エージェント開発への示唆：評価層（RAGAS＋LLM-as-judge）の設計は、監査証跡のRAGシステムにそのまま適用可能。faithfulnessスコアの低下はハルシネーションの定量的証跡となり、LLM-as-judgeは正解ラベル不要で大量の監査結果を自動品質チェックするユースケースにも転用できる。

## アイデア

- スキル習得に依存関係グラフを設定し「今のレイヤーを動かしてから次へ」という直列原則を徹底することで、理論先行による3ヶ月空振りを防ぐ設計思想
- RAGASの3指標（faithfulness・answer_relevancy・context_recall）をプロンプト改変ごとにJSONログで記録し、スコアの変化量を追う評価サイクルが「試作品作れる人」と「プロダクト作れる人」の分水嶺という指摘
- Golden Set作成コストが高い場面でGPT-5.5をjudgeとして使うLLM-as-judgeを評価パイプラインに組み込み、ラベリング工数を削減しつつ定量評価を維持するアプローチ
