## 要約

複数のワーカープロセスから LLM API を並列呼び出しする際、各プロセスが独立した asyncio.Semaphore を保持するため、サービス全体の RPM（Requests Per Minute）・ITPM・OTPM が合算で API 上限を超過し、429 エラーが連鎖する問題を段階的に解決する設計パターンを整理した記事。

問題の核心は、asyncio.Semaphore(100) がプロセス内同時実行数しか制御しない点にある。ワーカーを 3 台に増やすと最大 300 並列となり、全体上限を超える。また、同時実行数を絞っても 1 分あたりのリクエスト数やトークン数は別次元の制約であるため、RPM・ITPM・OTPM を独立して管理する必要がある。

推奨される段階的アプローチは 5 ステップ。(1) 静的配分：総上限をワーカー数で割り、環境変数 LLM_WORKER_CONCURRENCY で各ワーカーに割り当てる。ワーカー数が固定なら最もシンプルな初手。(2) Retry-After 追従：429 受信時に固定待機でなく、Anthropic API のレスポンスヘッダー retry-after や anthropic-ratelimit-requests-reset を解析して正確な待機時間を算出する。(3) Jitter 付与：全ワーカーが同時に再送する「429 の連鎖」を防ぐため、待機時間に ±20% 程度のランダム幅を加える（AWS が推奨する Exponential Backoff And Jitter パターン）。(4) 429 発生率の観測：バッチ処理では 429 率 1% 未満なら静的配分継続、1% 以上なら分散レートリミッター検討という判断基準を設ける。(5) 分散レートリミッター導入：ワーカー数が動的スケールする場合や静的配分で収まらない場合、Redis/Valkey/etcd 等の共有ストアで全ワーカー共通の利用枠を管理する。

Redis を使う実装では、残量確認と更新を Lua スクリプトで原子的に実行（read-decide-update の一括処理）し、競合を排除する。また LLM 呼び出しごとに Redis へアクセスするとボトルネックになるため、ワーカーが一定量（lease）をまとめて借り、消費はローカル状態で管理するバルク取得方式を採用する。TTL（PEXPIRE）でワーカー停止時の枠の取り残しを自動回収する設計も重要。

監査エージェント開発への示唆：LangGraph を用いた監査エージェントが複数ノードで並列 LLM 呼び出しを行う構成では、この問題が直接発生する。静的配分と Retry-After 追従を先行実装し、本番負荷を観測してから Redis ベースの分散レートリミッターへ移行するアプローチは、複雑性を段階的に導入できる点で実用的。RPM・ITPM・OTPM を別カウンターで管理する設計は、長文プロンプトを多用する監査ユースケースで特に重要。

## アイデア

- 429 の連鎖を「全ワーカーが同時に再送する thundering herd 問題」と捉え、Jitter でタイミングを分散させる視点は、LangGraph マルチエージェントの再試行ロジック設計に直接応用できる
- Lease（利用枠のバルク取得）パターンは Redis アクセス頻度を下げるだけでなく、ローカルキャッシュで高速に同時実行数制御できる点で、ネットワーク遅延が支配的な分散環境で効果的
- 静的配分 → Retry-After 追従 → 観測 → 分散レートリミッターという段階的移行戦略は、過剰設計を避けながら実際の負荷データで判断できる実務パターンとして汎用性が高い
