## 要約

Databricks CTOのMatei Zahariaが2026年6月のData + AI Summit 2026で発表したOSS「Omnigent」を軸に、エージェントの階層構造を整理した記事。

「ハーネス」とはLLM（モデル）をファイル・ターミナル・MCP・ツール呼び出しなど外部環境に接続するソフトウェア層を指す。Claude Codeであれば「Claudeモデル＋Claude Codeハーネス」という構造になる。「メタハーネス」はその上位層で、複数の異なるハーネス（Claude Code、Codex、Cursorなど）を横断的に束ねる役割を担う。

メタハーネスが必要とされる背景には、ハーネスの乱立がある。OpenAI互換APIとMCPの普及によりモデル・ツールの差し替えが容易になった結果、各社・各個人が独自ハーネスを量産。3つの共通課題が生じた：①合成（composition）＝異なるハーネス間でのエージェント組み合わせ、②協働（collaboration）＝複数ユーザーが別エージェントを使いながら同一作業を進める手段の欠如、③統制（control）＝プロンプトインジェクション対策・誤操作防止・コスト管理。

OmnigentはApache 2.0ライセンスのOSSでalpha段階。コアはRunner（個々エージェントのラップとサンドボックス）とServer（履歴・ポリシー・成果物・MCP・Skillsの集約）をCommon APIで接続した設計。`omnigent claude`のようにコマンド一つで既存ハーネスを包み、Terminal UIとWeb UI（localhost:6767）を同時提供する。エージェント定義はYAMLで記述し、`executor.harness`フィールドにclaude-sdk・codex・openai-agentsなどを指定することで同一定義のまま実行ハーネスを差し替え可能。ポリシーはコスト上限（max_cost_usd）やOSツール実行時の承認を宣言的に設定でき、コードを変更せず後付けで適用できる。セッションURLを共有することで複数ユーザーが同一エージェントセッションを参照・操作できる協働機能も持つ。

比較対象として、CrewAI（ロールベースのチームオーケストレーション）、Claude Squad（tmux＋gitワークツリーで複数エージェント並列実行）、Vibe Kanban（Kanbanボードで並列実行・差分レビュー、縮小中）、Conductor（決定論的ワークフロー）、Microsoft Agent Framework（逐次・並列・ハンドオフ・グループチャットパターン対応、2026年4月1.0 GA）が挙げられている。Omnigentが「異なるハーネス間の相互運用と統制」を主眼とするのに対し、CrewAIやMicrosoft Agent Frameworkは同一フレームワーク内の手順オーケストレーションに特化しており、力点が異なる。

監査エージェント開発への示唆：統制ポリシーの宣言的管理（コスト上限・承認フロー）はエージェントのガバナンス設計に直接応用可能。複数の専門エージェント（データ収集・リスク評価・レポート生成）を異なるハーネスで実装しメタハーネスで束ねるアーキテクチャは、監査業務の段階的自動化に適している。

## アイデア

- 「ハーネス→メタハーネス」という階層分離の概念により、モデルとツール接続層と調整層が独立して進化できる設計思想が明確になった点。Claude Codeのような既存ツールをコード変更なしに包んで相互運用できるアーキテクチャは、ベンダーロックイン回避の実践的手法として注目に値する
- コスト上限をポリシーとして宣言的に記述し実行時に強制する機能（max_cost_usd: 5.00、ask_thresholds_usd: [3.00]）は、既存エージェントフレームワークにほぼ存在しない特徴。エージェントのガバナンス要件を満たす上で重要な差別化要素
- YAMLのtools定義でfunctionとagentを同列に記述できる設計は、エージェントをツールとして再帰的に組み合わせる「エージェントのコンポーザビリティ」を実現しており、LangGraphのグラフ構造とは異なるアプローチで合成問題を解決している
