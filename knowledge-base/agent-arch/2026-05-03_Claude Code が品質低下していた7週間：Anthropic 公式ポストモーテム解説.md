## 要約

2026年3月4日から4月20日の約7週間、Claude Code（Sonnet 4.6 / Opus 4.6 / Opus 4.7）の応答品質が低下していた。Anthropic が2026年4月23日に公開したポストモーテムによると、原因は独立した3つの問題が時期をずらして重なったものだった。

原因①（2026-03-04発生）：Opus 4.6 リリース時に推論努力度（reasoning effort）のデフォルト値を high から medium に引き下げた。背景は応答レイテンシ増大への苦情対応だったが、モデルが十分な思考時間を使わなくなり「思考が浅い」と感じられる症状を招いた。2026-04-07 に Opus 4.7 は xhigh、その他モデルは high に戻された。

原因②（2026-03-26発生）：推論キャッシュ最適化の実装バグ。1時間以上アイドル後のセッション再開時に1度だけ実行すべき推論キャッシュクリア処理が、バグにより毎ターン実行されてしまった。その結果、Claude が前ターンの思考プロセスを毎回失い、「忘れる」「繰り返す」「奇妙なツール選択をする」症状が頻発。さらにキャッシュミス恒常化により通常より使用量トークンを多く消費した。2026-04-10 の v2.1.101 で修正。

原因③（2026-04-16発生）：Opus 4.7 発表に合わせてシステムプロンプトに語数制限（ツール呼び出し間25語以下・最終応答100語以下）を追加した。冗長性削減が目的だったが、内部評価で Opus 4.6 / 4.7 ともに約3%の知能低下が検出された。出力長を単純に絞ることで思考プロセスそのものを削ってしまうという、プロンプトエンジニアリング上の重要な教訓を示す事例となった。2026-04-20 の v2.1.116 で語数制限を全面撤回し、3問題すべてが解消。影響ユーザーの使用量制限は2026-04-23 に遡及リセットされた。

Anthropicの再発防止策は「検知・管理・透明性」の3軸：社内ドッグフーディング比率の拡大、複数リポジトリ横断のコードレビュー強化、プロンプト変更の段階的カナリアリリースと事前評価スイート適用、および @ClaudeDevs アカウントによるリアルタイム透明性確保。

監査エージェント開発への示唆：LLM ベースのエージェントにおいて「簡潔に答えよ」系のシステムプロンプト制約は推論品質を損なうリスクがある。ReAct パターンで思考ステップを重視する監査エージェントでは、出力長制限よりも reasoning effort の明示的な制御を優先すべき。

## アイデア

- 「出力長を削ると知能が約3%低下する」という定量的な証拠：冗長性と思考深度には相関があり、システムプロンプトで単純に語数制限をかけると思考プロセスそのものを切り捨ててしまうという、プロンプト設計の逆説的な落とし穴
- 独立した3つのバグが時期をずらして重なることで症状が複合化し、原因特定を困難にしたアーキテクチャ上の問題：単一障害点よりも複合障害の方が検知が遅れるという運用上の教訓
- 推論キャッシュのクリア粒度（セッション単位 vs ターン単位）の設計ミスが、使用量消費の増大と文脈喪失という2つの問題を同時に引き起こした点：キャッシュ最適化ロジックの副作用の見落とし方
