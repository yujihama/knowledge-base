## 要約

本記事はバーでの会話形式を用いて、RAGの仕組みを平易に解説した技術エッセイ。LLMはプロンプト外の情報を参照する手段を持たないという根本的な制約から出発し、RAGがその問題を解決するための実用的なアーキテクチャであることを説明する。

RAGの動作原理を「体育館に本を裁断して配置する」比喩で説明：ドキュメントをチャンクに分割し、埋め込みモデル（Embedding）によってテキストを数百〜1500次元以上の数値ベクトルに変換、ベクトルDB（Chroma、Pinecone、pgvector等）に格納する。クエリもベクトル化し、コサイン類似度によるk-NN検索で意味的に近いチャンクを取得、それをプロンプトに挿入してLLMに渡す。1500次元以上の高次元空間を使う理由は「図鑑」のように表面的な共通点と本質的な意味の違いを複数の軸で区別するためであり、次元数が多いほど微妙な意味的差異を捉えられる。

重要な点として、RAGはLLMが知識を「吸収」するわけでも賢くなるわけでもなく、関連テキストを機械的に探してプロンプトに貼り付けるだけの仕組みであることを強調。ベクトルDBに格納するデータはPDFに限らず、Slackメッセージ・議事録・メール・GitのIssueコメントなど「後でLLMに見せたいテキスト全般」であり、入力パイプを自動化することで知識ベースを自律的に育てる設計が可能。

MCPとの組み合わせについても言及：Model Context ProtocolをRAGのベクトル検索に組み込むと、エージェントがMCP経由で自律的に検索を呼び出して回答を構築できる。設計上の調整パラメータとして、チャンクサイズ（小さいと精度向上だが文脈が切れる）、取得件数（多いとノイズ増加）が挙げられる。コンテキストウィンドウの拡大でRAGが不要な場面も増えているが、大量ドキュメント・高更新頻度・コスト制約・ハイブリッド検索が必要な場面ではまだ現役。

監査エージェント開発への示唆：大量の監査証跡・内部規程・過去の指摘事項などをベクトルDBに自動投入するパイプラインを構築し、MCPツールとして検索機能を公開することで、監査エージェントが必要な文脈を自律的に取得できるアーキテクチャが実現可能。チャンクサイズや取得件数の設計が回答品質を直接左右するため、チューニング戦略を事前に設計することが重要。

## アイデア

- RAGは「LLMが賢くなる仕組み」ではなく「プロンプトへの自動コピペ機構」に過ぎないという本質的な整理が、過大評価を排除した正確な設計判断につながる
- 格納対象を『PDF』に限定せず『後でLLMに見せたいテキスト全般』と定義し直すことで、Slack・Git・メール等を自動パイプで流し込む設計思想に転換できる
- MCPをRAGのフロントエンドとして使うと、エージェントが検索タイミングを自律判断できるようになり、ユーザー側がベクトルDBの存在を意識しない設計が実現する
