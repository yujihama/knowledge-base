## 要約

AI多層防御の運用中に、存在しないDBテーブル名を前提とした設計書が4層のエージェントを通過し、その後も誤前提が連鎖して合計17件の指摘が積み上がった事例の詳細分析と対策集。

根本原因は設計書上流のコンポーネント名における1文字のtypoだった。AIは「ユーザーが書いた設計書＝正しい」という初期挙動でその誤りを引き継ぎ、7件以上の指摘がこの単一のtypoに起因していた。最終的にレビュー担当者自身が「これはtypoです」と気づくことで長期にわたる方針のすれ違いが解消した。

連鎖の構造は多軸にわたる。(1)既存資産の確認不足（DBテーブル・カラムの実在性未確認）、(2)基礎知識の思い込み（自社DBを「知っているはず」と前提する慢心）、(3)設計方針のすれ違い、(4)リポジトリ操作範囲の誤認、(5)ユーザー入力のtypoを疑わない挙動——これらが複合的に積み重なった。

同系列のAIは同じ盲点を共有するという構造原則も再確認された。セッション分割では独立性を担保できず、Copilot等の異なる観点を持つ第三者ツールを構造的にレビューに組み込まない限り、同系列AIは同じ誤りを見逃し続ける。

対策として8件の必須手順が整理された。(1)設計対象の全件実機確認（SHOW TABLES/DESCRIBE/grep等）、(2)「現在すでに前提が崩れている可能性」を着手前に問う過去軸リスク想定、(3)上流資料を「仮説」として扱い実機確認まで信頼しない、(4)着手前にAI側から素朴な疑問を1件強制発信、(5)「自社DBについて自分は何も知らない」という謙虚さの宣言、(6)指摘が連鎖する際はユーザー入力のtypoを疑う閾値を下げる、(7)人間＋AIセルフレビューに加えて第三者ツールを構造的に含める3層レビュー、(8)「ルール化した＝自動的に守られる」ではなく今この瞬間参照しているかを意識的に確認する習慣。

監査エージェント開発への示唆として、AIが「丁寧にしようとした行為」が設計原則とずれて意図しない侵害（実データへのアクセス提案等）を生む点、AIの慎重な影響範囲最小化判断と人間の「今しかできない変更タイミング」という戦略的判断のギャップは、監査プロセスにおける証跡収集や是正勧告のタイミング判断でも同様の構造的課題として現れ得る。

## アイデア

- 「指摘が連鎖するとき＝前提にtypoがある信号」という逆引き診断ヒューリスティック：連鎖回数をトリガーにユーザー入力の再検証フェーズを自動挿入する設計パターン
- 同系列AIは同じ盲点を共有するという構造原則：独立性は「セッション分割」ではなく「観点の埋め込み」で担保する必要があり、異なるアーキテクチャのモデルやツールを多様性確保のために意図的に混在させる設計指針
- 「記録したルール＝自動的に守られる」ではないという問題：プロンプトへの静的埋め込みではなくタスク着手時のチェックリスト実行をワークフローに強制する、ルール参照の能動的トリガー機構の必要性
