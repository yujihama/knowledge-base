## 要約

現行のLLMチャットボットはMMLU・HumanEval・MATHなどのベンチマークでスコアを伸ばし続けているが、これらは非インタラクティブな1ショット評価であり、実際のユーザー体験の向上と必ずしも比例しない。本稿は「目的意識のある対話（Purposeful Dialogue）」という概念を提唱し、複数ターンの会話を通じて目標を達成する能力こそが重要だと主張する。

対話設計の歴史的背景として、1970年代のSchankの「レストランスクリプト」やELIZA、PARRYなど、当時はすべての発話が手動設計されていた。現代のLLMは次トークン予測で学習されており、(1)事前学習（Reddit・Stack Exchangeなど対話風データを少量含む巨大コーパス）→(2)チャットテンプレート導入（<system>/<INST>タグで構造化）→(3)RLHF（KLペナルティ・LoRAで微調整）という3段階で対話能力を獲得する。ただしRLHFはLeCunの「ケーキのチェリー」に相当する微調整に過ぎない。

問題の核心は一貫性（Consistency）の欠如である。著者らはシステムプロンプトを与えた2つのLMエージェント同士を長期間対話させ、途中でシステムプロンプト違反を誘発するインジェクション発話を挿入する実験環境を構築した。8ターン程度の対話でも、多くのモデルは途中でロールから逸脱することが確認された。MT-BenchやAlpaca-Evalは1ターン評価であり、この問題を捉えられない。

解決の方向性として、(1)Dialogue State Tracking（DST）：目標・制約・対話履歴を明示的に追跡する構造的アプローチ、(2)長期記憶と動的ユーザープロファイル更新、(3)協調ゲーム理論的フレームワーク（ナッシュ均衡・交渉理論の応用）が提示される。SWE-benchのようなGitHub Issue解決タスクでも、単一アクションではなくエンジニアとの多ターン対話が不可欠であり、ペアプログラミングに相当する協調的コーディングエージェントの実現には目的意識ある対話が前提となる。

監査エージェント開発への示唆として、ReActエージェントや監査手続きの実行においても、要件確認・追加情報収集・承認フローなど複数ターンの人間との対話が本質的に必要であり、単一プロンプトでの完結を前提としたベンチマーク設計や評価基準はエージェントの実力を過大・過小評価するリスクがある。LangGraphのような状態管理フレームワークでDSTを実装し、対話一貫性を定量評価する仕組みを組み込むことが重要。

## アイデア

- 2つのシステムプロンプト付きLMエージェントを対話させ途中にインジェクション発話を挿入することで、既存ベンチマークが捉えられない長期一貫性の劣化を定量評価できる手法は、監査エージェントの耐性テストにそのまま応用可能
- ナッシュ均衡・交渉理論など協調ゲーム理論を対話設計に導入する視点：エージェントと人間が「隠れた目標」を持つ協調ゲームとして相互作用をモデル化することで、要件引き出し・合意形成フローを設計できる
- Dialogue State Tracking（DST）を明示的な構造として実装することで、LLMが対話中にロール・制約・目標を忘れる問題を防ぎ、長期エージェントセッションの信頼性を高められる
