## 要約

Claude CodeにKaggleコンペの提出自動化を依頼したところ「直接Kaggleにアクセスできません」と返答され、検索結果にも「Code CompetitionへのAPI完全自動提出は不可能」とする記事が上位に並んだ。しかし公式リポジトリ `Kaggle/kaggle-cli` のドキュメントを確認すると、`kaggle competitions submit` コマンドに `-k`（ノートブック指定）、`-v`（バージョン）、`-f`（出力ファイル）オプションが正式に用意されており、CLI一発での提出が可能であることが判明した。この「できない」情報の原因は、旧リポジトリ `kaggle-api` 時代の制約をLLMの事前学習データが引きずっていること、および成功体験より失敗談のほうがSEO的に上位表示されやすいという情報バイアスにある。著者が最終的に辿り着いた解決策は、①Kaggle CLI公式ドキュメント（認証・データセット取得・Notebook push・Code Competition submitまでの手順）と②参加コンペのOverviewページ（GPU/TPU可否・外部データ可否・実行時間制限・提出形式）の2つをClaude Codeのコンテキストに読み込ませてから実装を依頼するというシンプルな手順だった。途中の詰まりポイントとして、ローカル生成コードがKaggle Notebook実行時のパス `/kaggle/input/<コンペ名>/` と食い違う問題、GPU禁止コンペでGPUコードを生成してしまう問題も発生したが、いずれもコンテキストへの明示的な情報追加で解決した。このパターンは「最新の公式仕様」と「プロジェクト固有のローカルルール」という2層のドキュメントをAIに渡す手法として一般化でき、GitHub Actions、Hugging Face Hub、社内APIなどほぼ全てのツール統合タスクに適用可能。監査エージェント開発への示唆として、外部APIやツールをエージェントに扱わせる際は最新仕様ドキュメントとドメイン固有の制約（規制・ルール等）を明示的にコンテキストインジェクションする設計が不可欠であり、エージェントの「知識の鮮度」問題への対処として実行時ドキュメント注入パターンの有効性を示す実例となっている。

## アイデア

- LLMの事前学習データの陳腐化と検索結果の失敗バイアスという2重の情報劣化を、一次情報の直接注入で突破するパターンは、エージェントが外部ツールを操作する際のコンテキスト設計原則として定式化できる
- 「汎用仕様ドキュメント」＋「ローカルルール」の2層注入パターンは、監査エージェントが規制文書（法令・基準）とクライアント固有ポリシーの両方を参照して判断する構造と同型であり、そのまま設計テンプレートとして転用可能
- AIに「できるか？」と能力を問うのではなく「このドキュメントを読んで実装せよ」と実行を指示するというインタラクション設計の転換は、エージェントへのタスク委譲インターフェース設計全般に通じる原則
