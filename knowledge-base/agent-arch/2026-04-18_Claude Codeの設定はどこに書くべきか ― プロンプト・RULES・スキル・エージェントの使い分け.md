## 要約

Claude Codeを本格運用する際に直面する「指示をどのレイヤーに書くか」というコンテキスト設計の問題を、4つの機構（プロンプト・RULESレイヤー・スキル・エージェントファイル）に整理して解説する記事。

前提として、LLMはコンテキストが長くなるほど情報を見落としやすい（Liu et al. 2023「Lost in the Middle」）という特性があり、Anthropicも「Effective Context Engineering」でコンテキスト肥大化による指示追従性劣化（context rot）を警告している。つまり設定の置き場所は「どの情報をいつ・どの優先度でモデルに渡すか」の設計問題である。

**RULESレイヤー（CLAUDE.md / .claude/rules/）**は常時ロードされるため「絶対に読み飛ばされない」保証があるが、関係のない会話でもコンテキストを消費する。「just testでテスト実行」「Conventional Commitsで書く」などプロジェクト固有の恒常ルールに絞るのがコスパ最良。.claude/rules/ではpaths指定による条件付きロードも可能で、特定ファイルパターンにアクセスした時だけ追加ルールを読み込める。

**スキル**の本質はProgressive Disclosure（段階的開示）で、必要な時だけ読み込まれるためコンテキスト常時負荷はdescriptionメタデータ分のみ。descriptionで発動条件を明示できるため、モデルが「今このスキルを使うべきか」を適切に判断できる。GitHub操作・レビュー・リリース手順といった場面特化の知識をプロジェクト横断で再利用できる点も強み。disable-model-invocation: trueで自動発動を抑制し、ユーザー呼び出し専用にもできる。スキル間の責務重複は「どれを使うか判断できない」問題を引き起こすため、排他的設計が必要。

**エージェントファイル**の本質的価値は「コンテキストの分離」と「ツールセットの分離」。スキルでは対応しにくい長時間・多段タスクのワークフロー固定（調査→設計→実装→テスト→PR作成など）や、MCP サーバーを含むツールセット制御に強みがある。MCP サーバーはdescriptionだけでコンテキストをかなり消費するため、専用エージェントに切り分ける価値が大きい。日常使いのメインエージェントのカスタマイズ（必ずWeb検索する、出典付きで返すなど）もエージェントファイルで対応するのが適切。

**設計の要点**: RULESは汎用・恒常ルール最小限、スキルは場面特化・再利用可能な知識の遅延ロード、エージェントはコンテキスト/ツール分離とワークフロー固定、という住み分けが基本原則。監査エージェント開発においても、監査手順・判断基準・証跡収集ツールをこの3レイヤーで整理することで、コンテキスト効率と指示追従性を両立したエージェント設計が可能になる。

## アイデア

- 設定の置き場所問題を「いつ・どの情報を・どの優先度でモデルに渡すか」という情報設計問題として捉える視点は、単なるファイル配置の好みを超えたアーキテクチャ思想であり、LLMの「Lost in the Middle」特性と直結している
- スキルのProgressive Disclosure原則：descriptionメタデータのみ常時ロードし、本体は必要時だけ読み込む設計は、APIの遅延ロードやUIのアコーディオン設計と同じ情報開示哲学をプロンプト工学に適用したものである
- .claude/rules/のpaths付きルールによって「特定ファイルパターンにアクセスした時だけルールをロードする」という条件付き動的ロードが実現でき、モノレポや複数技術スタック混在環境でのコンテキスト最適化に直接応用できる
