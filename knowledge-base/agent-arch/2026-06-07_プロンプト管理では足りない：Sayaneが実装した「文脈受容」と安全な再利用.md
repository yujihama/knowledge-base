## 要約

Sayane v1.1は、LLMとの対話で生まれた文脈を「プロンプト文字列の保存」ではなく「候補としての受容・判断・来歴管理」として扱うローカルファーストな文脈管理基盤。

従来のプロンプト管理では、ChatGPT→Claude→Cursorのようにマルチ環境を横断する際に、文脈がどこから来たか・どの範囲で有効か・誰がどう判断したかが失われる。Sayaneはこの問題に対し以下のアーキテクチャで対応する。

**Semantic Review**: 外部から戻った候補について、同一概念の複数箇所出現（semantic_overlap）や不安定な配置（unstable_placement）を検出し、警告フラグを付与。ただし自動修正はせず、人間の判断を必須とする。

**4種類の判断**: approve / reject / modify / deferの選択肢を設け、rejectされた候補も監査来歴として保存。採用されなかった候補も「文脈の輪郭」として後から検証可能にする。

**Bundle検証**: context bundleにSHA-256ハッシュとmetadataを付与し、Transfer path（例: Sayane→ChatGPT→Claude→Sayane）・LLM memory falseフラグを明示することで、外部プロファイルとLLM内部記憶を区別する。

**Import Policy Profiles**: strict（CI/本番）・standard・legacy_compatible・developmentの4段階ポリシーで、hash_mismatchやsignature_invalidはハードコンストレイントとして常時ブロック。

**scoped_accept（v1.1新機能）**: 「Zenn第2回の導入部では強い問題提起を許容する」のような局所的文脈を、accepted_scope・conditions・negative_constraints・promotion_policy・reuse_policyと一体で保存。全体プロファイルへの条件なし昇格（「トロイの木馬的文脈混入」）を防ぐ。

**MCP出力の派生文脈扱い**: Cursor/Claude Desktop向けMCP出力は「正本プロファイルではなく派生文脈」と明示し、scoped_accept情報（有効範囲・条件・禁止解釈）を落とさずに含める。

監査エージェント開発への示唆：判断来歴の保存・棄却候補の保持・policy profileによる運用条件の切り替えは、LangGraphベースの監査エージェントにおける証拠管理・判断根拠の追跡設計に直接応用可能。特にscoped_acceptの「条件付き許容＋昇格制約」モデルは、監査判断の適用スコープ管理と同型の構造を持つ。

## アイデア

- 「トロイの木馬的文脈混入」の概念：局所的に有効だった文脈がscopeを失って全体方針を汚染する現象を命名・構造化した点が独自
- 棄却された候補を捨てずに監査来歴として保存する設計：採用しなかった判断も文脈の輪郭を構成するという逆説的な情報設計
- MCP出力を「正本ではなく対象ツール向けにコンパイルされた派生文脈」と明示的に区別する設計原則：AIツール間の文脈移送における情報劣化を防ぐアーキテクチャ上の立場
