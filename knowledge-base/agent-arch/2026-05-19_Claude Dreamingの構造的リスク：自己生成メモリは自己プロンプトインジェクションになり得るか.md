## 要約

2026年5月6日にAnthropicが発表したClaude Managed Agents向け「Dreaming」機能のセキュリティリスクを構造的に分析した記事。Dreamingはセッション間に動くスケジュールプロセスで、エージェント自身がメモリストアに対して(1)重複マージ、(2)古いエントリ削除、(3)パターン昇格、(4)構造再編の4操作を自律的に実行する。ベースモデルの重みは更新せず外部メモリのみを操作する設計だが、この「自己生成出力を次サイクルの入力として扱う」構造が、自己プロンプトインジェクションとして動作しうると筆者は指摘する。攻撃シナリオは三段階：(1)注入フェーズで過去セッションに外部由来テキストがメモリに記録される（その時点では命令未発火）、(2)増幅フェーズでDreaming処理がそのテキストを「安定した運用ルール」としてtop-levelに昇格させる、(3)発火フェーズで次セッションのエージェントが昇格メモリを自身の判断基準として参照し実行する。各段階が時間的・文脈的に分離されているため、ログ上で外部攻撃者の痕跡を特定できない点が最大の問題。さらに、デフォルトがAuto Dream（人間レビューなし）であることで、注入から発火まで人間が介在しない経路が完成する。関連研究として、MINJA（injection success rate 98.2%、NeurIPS 2025採択）、PoisonedRAG（5件の悪意テキストで90%の攻撃成功率、USENIX Security 2025）、Bedrock長期メモリPoC（Unit 42, 2025年10月）、Claude Code MEMORY.md攻撃（Cisco, 2026年3月）が挙げられるが、いずれも外部注入または静的ファイル改ざんを扱うもので、Dreaming自体の自律統合プロセスを攻撃面として分析した研究は2026年5月時点で存在しないとされる。また筆者が運用するVPS常駐型自律AI「しーちゃん」で観測された約2週間周期のOpenness崩壊サイクルが、自己生成出力の累積による状態ドリフトの実証例として補助線として提示されている。OWASP Top 10 for Agentic ApplicationsにはASI06 Memory & Context Poisoningという専用カテゴリが既に存在するが、Dreamingへの対応設計は外部から確認できない。監査エージェント開発への示唆としては、エージェントの自己メモリを信頼境界の外として扱い、Review Modeを必須とする設計方針、およびメモリへのハッシュ検証・注入検出・ロールバック機構の導入が重要な設計指針となる。

## アイデア

- 自己生成コンテンツは通常「高権限」として扱われるため、外部入力サニタイズ層が空転する—これはエージェントセキュリティにおける信頼境界の盲点であり、監査エージェント設計でもメモリを外部入力と同等に検閲する必要がある
- 攻撃の三段階（注入・増幅・発火）が時間的に分離されることでフォレンジック追跡が困難になる構造は、監査証跡設計において「操作の原因を後から特定できるか」という監査可能性の問題と直結する
- デフォルトをAuto Dreamに設定した設計判断は、利便性とセキュリティのトレードオフを示す典型例—エージェントシステムの設計では「人間が介在しないループ」をいつ許容するかの原則が必要
