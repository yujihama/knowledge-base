## 要約

2025年3月31日、Anthropicがnpmに公開した@anthropic-ai/claude-codeに.npmignoreの*.map記述漏れがあり、TypeScript約1,906ファイル・51万2,000行超のソースコードが流出した。この記事はその流出コードの分析から導出したAIエージェント設計の5原則を解説する。

核心的な発見は「エージェントループ自体は20行のwhileループに過ぎず、51万行の大半はハーネス（コンテキスト管理・パーミッション・ツールスキーマ・エラー回復・メモリ・圧縮）に費やされている」という点。Sebastian Raschkaはこれを分析し「ハーネスの質がモデルの体感品質を決める」と結論づけた。

原則1はツールの専用化・型付け・制約埋め込み。Claude Codeは約52個の専用ツールを持ち、汎用Bashツールのdescription内にも「find/grep/catには使うな」と明記。USE WHEN / DO NOT USE形式でLLMが必ず参照する場所に制約を書くことで、誤使用を構造的に防ぐ。

原則2はクエリ分類による処理分岐。全クエリを同一パスに通さず、simple（LLM直答）・moderate（ハイブリッド検索1回）・complex（Agentic RAG）の3段階に分類して処理経路を変える。これにより応答速度とAPIコストを最適化する。

原則3はコンテキスト肥大化防止。システムプロンプトを静的部分（ツール定義・行動ルール・安全制約）と動的部分（カレントディレクトリ・Gitブランチ・CLAUDE.md）に分離してキャッシュし、APIコストを最大72%削減。ツール出力が長い場合は先頭60%＋末尾30%のクリッピング、古いメッセージのローリング圧縮、軽量モデルによる会話サマリー生成という3段階圧縮を実施。自動コンパクション機能のバグ1件で1日25万回の無駄なAPIコールが発生した実例が示す通り、コンテキスト管理の設計ミスは深刻なコスト増につながる。

原則4はメモリの3層階層化。第1層MEMORY.md（常時ロード、500トークン程度の軽量インデックス）、第2層トピック別ファイル（オンデマンドロード）、第3層過去会話ログ（grep検索時のみ参照）という構造。アイドル時にautoDreamと呼ばれるサブエージェントが起動し、過去観察結果のマージ・重複除去・矛盾解消を自動実行する。

原則5はセキュリティの使用箇所への埋め込み。ツールdescriptionに制約を直接記述し、判断と実行を分離してパーミッションチェックを介在させる。CVE-2025-59828では「トラスト確立前にフック経由でコード実行が可能」という順序ミスが脆弱性となった実例が示す通り、エージェントの自律性が高いほど順序設計が致命的な差を生む。監査エージェント開発においても、ツール定義への制約埋め込みとパーミッション層の明示的設計は直接応用可能な原則である。

## アイデア

- エージェントの体感品質はモデルではなくハーネス（コンテキスト管理・ツール設計・メモリ・セキュリティ）で決まるという逆説的な知見。51万行のコードがwhileループではなく周辺インフラに費やされている事実が実証している
- ツールdescriptionにUSE WHEN / DO NOT USEを埋め込む設計パターン。セキュリティルールや使用制約をLLMが必ず参照する場所に置くことで、外部設定ファイルへの依存や見落としを構造的に排除できる
- autoDreamによるアイドル時メモリ整理機構。エージェントが非稼働中にサブエージェントが記憶の重複除去・矛盾解消を自動実行する仕組みは、人間の睡眠中の記憶固定化と類似した設計思想であり、長期稼働エージェントの記憶劣化問題への応用が期待できる
