## 要約

現在のLLMチャットボットはMMLU、HumanEval、MATHなどのベンチマークで急速に性能向上しているが、これらは非インタラクティブな一回限りの評価であり、実際のユーザー体験の向上と必ずしも比例していない。本稿はその根本的な原因として「目的指向対話（Purposeful Dialogue）」の欠如を指摘する。

目的指向対話とは、旅行計画・心理療法・カスタマーサービスなど、特定の目標を持つ複数ターンの会話を指す。交渉理論における反復交渉がtake-it-or-leave-it型より優れた結果をもたらすように、情報の段階的交換は一括伝達より効率的だ。Terry Winogradの言葉を借りると「言語の使用はすべて話者の世界モデルを変更する手続きの活性化」であり、発話は相手の状態を変えるための意図的な行為として捉えられる。

対話システムの変遷として、1970年代のSchankの「レストランスクリプト」、Rogerian療法士を模したELIZA、偏執的人格を模倣したPARRYから始まり、現代のLLMは（1）大規模コーパスでの次トークン予測事前学習、（2）チャットテンプレートによる対話フォーマット導入、（3）RLHFによる微調整という3段階で構築されている。RLHFはLeCunのケーキ比喩で言えば「上に乗せた小さなチェリー」に過ぎず、事前学習データに対話フォーマットは存在しない。

定量評価として、著者らは2つのシステムプロンプト済みLMエージェントを長期間対話させる環境を構築し、ロールの一貫性をストレステストした。対話の特定ターンに「ユーザー」役エージェントがロール崩壊を誘導する発言（ジェイルブレイク、役割転換要求等）を挿入し、「アシスタント」役モデルがロールを維持できるかを測定。GPT-4も含む複数モデルで対話が長くなるほど一貫性が低下し、多くのモデルが8ターン程度でロールを放棄することが判明した。

評価軸として「主観的スコア」（LLM-as-judgeによる1〜5点評価）と「客観的スコア」（ロールキーワードの存在確認）を組み合わせて使用した。後者はバイナリだが文脈を考慮できず、前者は文脈を考慮できるがハルシネーションリスクがある。

システムプロンプトの問題点として、（1）大規模コーパスにsystem promptという概念自体が存在しない、（2）コンテキストウィンドウが長くなるにつれsystem promptへのアテンション重みが薄れる（LLMの「ロスト・イン・ザ・ミドル」問題）、（3）RLHF段階でのsystem promptへの依存度が不明確、の3点が挙げられる。

将来のLLMに必要な能力として「turn-taking」すなわち能動的な質問・情報要求・対話の積極的制御が挙げられ、コード生成においてもSWE-benchのようなGitHub issue解決には単一パスでなく双方向対話が必要となる。監査エージェント開発においても、監査人との反復的な情報交換を通じた要件の段階的明確化という観点で直接適用可能な示唆を含む。

## アイデア

- 2つのLMエージェントを長期間対話させてロール一貫性をストレステストする評価フレームワーク：特定ターンに敵対的発言を挿入し、アシスタント役のロール維持率をLLM-as-judgeと文字列マッチングで二重評価する設計は、監査エージェントの堅牢性評価に直接転用できる
- system promptはRLHFで初めて登場するが事前学習データには存在せず、コンテキストが長くなるほどアテンション重みが分散して効果が薄れるという構造的欠陥：この問題を解消するためperiodicな要約注入や構造的リマインダー挿入が実践的対策となる
- 交渉理論の反復交渉アナロジー：一括情報伝達より段階的情報交換が優れた結果をもたらすという原理は、LLMエージェントが監査証拠収集フェーズで監査人と反復的にやり取りする設計の理論的根拠として機能する
