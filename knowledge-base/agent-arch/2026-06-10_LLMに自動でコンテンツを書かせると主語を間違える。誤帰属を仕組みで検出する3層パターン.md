## 要約

全自動X（Twitter）運用パイプライン（RSS収集→LLM要約→品質ゲート→自動投稿）において、LLMが記事の主語・立場を取り違えた投稿を人間のチェックなしに公開してしまう事故が発生した。具体的には、CEOが「AIへの依存計画」を否定したニュースに対し、LLMが「CEOが依存させたいと考えている」と立場を逆転させて投稿した。原因は、パイプラインの品質ゲートが「形式」（長さ・NGワード・出典実在）しか検査せず、「意味の正しさ」を検査する層が皆無だったこと。この記事では、人間の承認を増やさずに誤帰属を抑制する3層の実装パターンを共有している。

①予防（生成プロンプト側）: 誰が何を主張し誰がどの立場かを要約通りに保つ、対立構図（推進vs否定）を混同しない、主語が確定できない場合は個人名で断定せず粒度をぼかす（例:「◯◯社の内部文書によると」）、最も目立つ人名を見出し主張に安易に結びつけないというルールをプロンプトに追加する。ただしLLMは指示を守らないことがあるため、1枚目の壁にすぎない。

②検出（生成後の照合パス）: 生成された案を別のLLM呼び出しで「要約（唯一の真実）」と照合する。判定軸は（a）主語・立場が要約と一致しているか、（b）要約にない事実を捏造していないか、の2点のみ。出力フォーマットを`案1: ok / 案2: ng — 理由`のように固定してパース可能にする。また、検証LLMが落ちたり出力が読めない場合の挙動としてfail-open（通す・可用性優先）かfail-closed（止める・安全優先）かを事前に設計する必要がある。著者はコンテンツが雑談的であるためfail-openを選択。

③自動リカバリ: 検出でNGになった案は理由を添えて1回だけ再生成し再検証する。OKなら採用、まだNGなら静かに隔離（公開しない）。再生成を1回に制限することで無限ループによるコスト爆発を防ぐ。また過剰防御で良案まで隔離され発信が枯れないようNG率をログで監視する。

検証においては、ユニットテストだけでなく実運用と同じモデルで実ケース（悪い案がngと判定、正しい案がokと判定）を再現することを重視している。監査エージェント開発への示唆として、自動化パイプラインにおける「意味的正確性」の検出層設計、fail-open/fail-closedの判断基準、1回限りの再生成による暴走防止パターンは、LLMを使った監査レポート自動生成や証跡要約処理にそのまま適用可能な設計原則である。

## アイデア

- LLMは「文中で最も目立つ固有名詞」を「見出しの主張」に結びつける傾向があり、これが否定・反論構図のあるニュースで立場の逆転を引き起こす具体的なメカニズムが示されている
- 検出層の設計においてfail-open/fail-closedの明示的な判断をパイプライン設計の必須要件として位置づける考え方は、LLMを使った自動化全般に適用できる安全設計の原則
- 再生成を「1回だけ」に制限し、NG率をログ監視することで過剰防御による発信枯渇を防ぐという、守りすぎもリスクとする双方向のバランス設計が実用的
