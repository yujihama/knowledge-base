## 要約

Claude Codeのサブエージェント機能を実運用した観察レポート。サブエージェントとは「メインのAIエージェント（親）がさらにAIの部下を持つ」二段構えの仕組みで、部下は独立したコンテキスト（作業場）で動き、結果だけを親に返す。これにより親のコンテキストに中間作業が積み上がらず、長いセッションでも精度低下を抑えられる。

主な用途は3つ。①並行作業：設計議論と影響範囲調査を同時進行するなど、複数の独立タスクを並列処理。②雑務委託：大量ログからのエラー行抽出・複数ファイルの要約など、手数はかかるが結論は短い作業を部下に逃がす。③独立レビュー：文脈を引き継がない部下は親のバイアスを共有しないため、事実確認・論理検証・読者目線といった別々の観点での並列レビューに有効。本記事自体もこの手法で複数部下にレビューさせて修正している。

コスト面では1部下あたり80k〜124k tokensを消費した実測値が示されており（数十ファイル読み込みのコードレビューや外部リポジトリ調査時）、3人同時起動するとその分が重なる「二度払い」問題が生じる。推奨は1〜3本・独立した仕事のみ。

定義ファイルは `.claude/agents/` に配置するYAML frontmatter形式で、`name`・`description`・`tools`・`model`の4フィールドを持つ。`description`が親の振り先判断に使われるため、一文で職分を明確に書く必要がある。`tools`でRead/Grepのみに絞ることで、書き換え権限を持たせない最小権限設計も可能。定義ファイルなしでも組み込みサブエージェント（汎用作業・コード探索・計画立案）が最初から利用できる。

Claude Codeには類似機能として agent team（lead-teammate間の相互通信、実験的）とagent view（複数独立セッションのダッシュボード監視）もあるが、サブエージェントは「結果だけ集めたい」ケースに使う最もシンプルな形。部下から部下を呼ぶネストは不可で、階層は親→部下の一段のみ。最終判断は常に人間が握る設計思想が強調されている。

監査エージェント開発への示唆：複数の独立した検証役（事実確認エージェント・論理整合性エージェント・規制適合チェックエージェント）をサブエージェントとして並列実行するパターンは、監査手続きの多角的検証に直接応用できる。ただしトークンコストが大きいため、検証の優先度と対象範囲の絞り込みが実務上の設計課題になる。

## アイデア

- 「文脈を持たない」という弱みが独立レビュアーとしての強みに転化する逆説的設計——親のバイアスを共有しないからこそ客観的な反証・セカンドオピニオンが成立する
- サブエージェントの真価は「速度向上」ではなく「人間の判断リソースの解放」——中間作業を部下に逃がし、親のコンテキストには結論のみを積むことで、人間が判断に集中できる構造を作る
- 1部下あたり80k〜124k tokensという実測値から導かれる「二度払い」問題——親のコンテキスト節約のために部下を雇うと、トータル消費はむしろ増える可能性があり、使いどころの費用対効果計算が必要
