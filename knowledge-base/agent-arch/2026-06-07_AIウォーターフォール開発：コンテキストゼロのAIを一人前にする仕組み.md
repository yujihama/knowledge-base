## 要約

Claude Codeはセッションのたびに記憶がリセットされるため、毎回ゼロから文脈を説明し直す必要があり、影響範囲の抜けや仕様矛盾が頻発する。本記事はその問題をウォーターフォール開発のトレーサビリティ概念をAI開発に持ち込むことで解決しようとした実践報告。

設計の核心は「ドキュメントに関係性を持たせること」。ADR（なぜ）→REQ（何を）→SPEC（どのように）→BF（業務フロー）→TC（テストケース）→src（実装）という階層に最小件数（ADR:2件、REQ:5件、SPEC:7件、BF:4件、TC:14件、src:3件）を設定し、各ドキュメントをノード、参照関係をエッジとしてNeo4jグラフDBに登録する。サンプルのイベント予約システムで38ノード・44エッジを構築し、MOTIVATES/DEFINED_BY/COVERED_BY/VERIFIED_BY/IMPLEMENTED_BY/VALIDATESの6種エッジで関係を表現。本文は持たず「何と何がつながっているか」だけを保持する地図として機能させる。

検索にはGraphRAGを採用。意味的類似検索（pgvector、384次元のparaphrase-multilingual-MiniLM-L12-v2）とNeo4jのグラフトラバーサルを2段階で組み合わせる。「キャンセルポリシーに関連するテストケース」のようなクエリに対してSPEC-004・TC-BOOK-008〜010を返せる。全ドキュメントを読ませず必要な数件に絞ることでコンテキスト長を節約する。

Claude Code Skillsとして4つのコマンドを定義。/impact-reportはコーディング前に変更影響をADR→REQ→SPEC→TCの連鎖で自己評価させる。/spec-checkで要件と仕様の整合性を確認し矛盾を実装前に検出。/trace-testは実装後に仕様ID（REQ-002-02, SPEC-004等）付きのテストを生成。/cross-review+/waterfall-prはPRレビュー前に要件・影響・テスト・コード・回帰の5軸チェックリストとPR本文を生成する。

GitHub Actionsで仕組みを強制化。doc-lint.ymlはdocs/の変更時に命名規則・必須セクション・クロスリファレンスの実在をCIで検証。pr-impact.ymlはPR作成時にボットが影響するADR・BFのリストと確認チェックリストを自動コメントする。

インフラはDocker上のNeo4j＋PostgreSQL/pgvectorで完結しOpenAI API不要。既存プロジェクトには/reverse-docでコードからドキュメントを逆生成し「⚠ コードから逆生成」タグを付けて人間確認を挟む導線を用意。監査エージェント開発への示唆として、設計判断（ADR）から実装・テストまでのトレーサビリティチェーンはLangGraph等のエージェントフローにおける証跡管理と親和性が高く、変更影響の自動評価ロジックはエージェントの自律的影響範囲判断に転用できる。

## アイデア

- ドキュメントを「本文なしの関係グラフ」として設計し、意味検索とグラフトラバーサルの2段階でコンテキストを最小化するGraphRAGアーキテクチャ
- ドキュメント件数を「グラフ検索が自明でない結果を返せる最小量」として理論から逆算して設定する設計アプローチ
- /impact-reportによる「コード前に理解レビューを挟む」仕組みで、AIのコーディング前に影響範囲の自己評価を強制する手法
