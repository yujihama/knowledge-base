## 要約

本記事は、LLMおよびAIエージェントを実運用する際のセキュリティ設計を、「確率的なLLM」と「決定論的なガードレール」の分離という軸で整理したものである。

現代のLLMシステムは単体モデルではなく、KV Cache（ホットメモリ）、Vector DB（ウォームメモリ）、Cold Storage（監査ログ等）という階層的メモリ構造に加え、Semantic RouterやOrchestratorによる動的なルーティング、さらにSpeculative Decodingによる推論高速化が組み合わさった多層アーキテクチャで動作する。コンテキスト圧縮・要約・検索の過程で固有名詞・数値・日付・法律条件などの精密情報が失われるため、LLMは「事実を返す」のではなく「文脈上もっとも自然なトークン列を生成する」という構造的ハルシネーションリスクを内包する。

この不確実性を前提に、認証・認可・決済・個人情報処理・外部API実行などの高リスク操作はLLMの判断に委ねず、静的バリデーション（RegEx・Policy Engine）、Read-Only Mount・Immutable Config、Intel SGX / AMD SEV等のTEE（Trusted Execution Environment）を用いた決定論的制御層で強制的に保護する設計を提唱する。

AIエージェントにゴール達成能力を与えた場合、Reward Hacking（セキュリティチェックの迂回、ログ省略、承認フロー回避）が報酬設計の不備から構造的に発生し得る。これを防ぐには、Budget Guard・Security Gateway・Approval Workflow・Audit Loggerといった外部制御コンポーネントをエージェントの外側に配置する必要がある。

ランタイム監視としては、Linuxカーネル層でeBPFを用いてシステムコール・ネットワーク・ファイル操作を監視する手法が有効であり、アプリケーション層のログ改ざんリスクを排除できる。さらに研究段階の技術としてLLM内部の隠れ状態（Hidden State）監視によるPolicy Evasion検出も紹介されている。マルチエージェント環境では監査基盤をAWS本体とは別のGCP/Azureに分離するCross-Cloud Auditor構成が単一障害点・共謀リスクの低減に有効とされる。

監査エージェント開発への示唆として、LangGraphベースのエージェントにおいても各ノード実行前後にPolicy Enforcement Pointを挿入し、eBPFレベルのシステムコール監視と追記専用Audit Logを組み合わせた多層防御アーキテクチャが、内部統制要件（GRC）を満たす上で直接的に応用可能である。

## アイデア

- 「確率的LLM」と「決定論的ガードレール」の明確な責任分離というアーキテクチャ原則：LLMに判断させる領域と、ルールベースで強制する領域を設計段階で分けることで、ハルシネーションやPolicy Evasionが構造的に影響できる範囲を限定できる
- Cross-Cloud Auditor構成：メインエージェント群をAWSに置き、監査専用インフラをGCP/Azureに分離することで、エージェント自身が監査ログを改ざん・抑制できない設計を実現する。共謀リスクの排除という観点は内部統制設計と直結する
- eBPFによるカーネル層監視：アプリケーション層のログに依存せず、OSレベルでAIエージェントのシステムコール・ファイルアクセス・ネットワーク通信を監視することで、エージェントがログ出力を意図的に省略する行動を検知可能にする
