## 要約

著者は3か月間、Claude CodeのSkillを47個運用した実体験をもとに、5個への絞り込みプロセスと判定軸を記録した実践記。

最初の動機は「使えそうなものは全部入れておきたい」という発想で、Anthropic公式のfrontend-design Skillが277,000インストールを超えるなど選択肢が豊富なため、コードレビュー系7個・Git/PR操作系6個・ドキュメント系5個・業務系5個など計47個を導入した。

転機はPostToolUse/PreCommit/SaveFileの3つのhookが同時発火し、ファイル保存のたびに8秒待たされ、さらにformat→lint→test→formatのループが発生してTPMが通常の4倍に跳ねた夜。47個のSkillが競合して機能不全に陥った。

絞り込みは2段階で実施。一次絞り込みでは、セッションログからjqコマンドで30日間の起動回数を集計し、0回だった21個を一括削除（47→26個）。起動0回の中にはaws-deployerも含まれており、理由はGitHub Actionsに既存のデプロイ機構があったため。二次絞り込みでは機能80%重複するSkillを統合し（code-review系7個→1個など）、12個に圧縮。さらに1か月運用して「なくても困らない」7個を削除し最終5個に到達。

残った5個の判定軸は：①週5回以上の使用頻度（code-review）、②失敗時の致命度（secrets-scan：Stripeキー流出事件を機に導入）、③不可逆操作の安全弁（pre-push-guard：git push --forceを3回阻止した実績）、④認知コスト削減（meeting-notes-summarizer：月128分節約）、⑤競合不在の自作ドメインSkill（香料業務向けkaori-knowledge）。

効果として、起動時のコンテキスト消費が47個時代の約15,000トークンから5個で約2,000トークンに削減。Skill動作の全把握が可能になりデバッグが容易になった。月1でSKILL.mdをレビュー・改善できるメンテナンス体制も確立した。

2026年5月時点のClaude Code本体の改善として、/plugin marketplace browseでコンテキスト消費の予測トークン数が表示されるようになり、公式側も多数Skill導入を牽制する方向に動いている。

監査エージェント開発への示唆：複数エージェント/ツールの競合によるループ問題はマルチエージェント設計全般に共通する課題であり、ツール選択のノイズを減らすことがcontext engineeringの基本という知見は、LangGraphベースのエージェント設計にも直接適用可能。

## アイデア

- hooks間の競合によるfeedback loopはLangGraphのノード設計でも発生しうる問題であり、ツール実行イベントの依存関係グラフを事前に設計する必要性を示唆している
- セッションログのjq集計による使用頻度の定量測定→ゼロ起動削除という機械的な足切り手法は、監査エージェントのツール選定にも応用できるROI評価パターン
- 自作ドメインSkillを「競合不在の資産」として位置づける発想は、汎用LLMに独自知識を付加するRAG/Skill化戦略として、特定業務ドメイン（監査・内部統制）への応用価値が高い
