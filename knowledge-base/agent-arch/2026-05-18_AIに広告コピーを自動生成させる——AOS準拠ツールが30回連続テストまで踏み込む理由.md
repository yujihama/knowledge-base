## 要約

本記事は、AOS（AI Operating Standard）v0.1 に準拠した ROI アドコピー生成 CLI ツール（ID: 1027）を題材に、LLM を使った自動化ツールをどう「自動化インフラ」として安定させるかを解説する実務記事。

ツールは広告プラットフォーム・想定読者・商材の3入力を受け取り、複数の広告文案と最優秀案をスコア付き JSON で出力する。重要な設計として、`--mock` オプション使用時は外部 LLM を呼ばず、入力のハッシュから決定論的なスタブ出力（`copies` と `best`）を返す構造になっており、デモと自動テストを同一コードパスで賄える。

評価（evals）は3系統で構成される。①`--mock` 経由で標準出力に `"copies"` と `"best"` が含まれ `"status": "success"` が返ること、②敵対的または過剰に甘い assertion を禁止する静的チェック、③pytest で `@pytest.mark.adversarial` タグ付きテストが意図的にスキップされないこと。特に③は「CI の緑を儀式化しない」ための実務的な柵として機能する。

さらに Playwright を用いて同一シナリオ（支払いなし拒否・モック成功・キーワード存在確認など5項目）を30サイクル連続実行し、全サイクル成功を門番条件としている。単発の PASS では環境ゆらぎによる偶発的成功を排除できないため、反復回数を実績の単位として積み上げる設計思想。

設計書（DESIGN.md）では Oracle / Permitted / Prohibited のゾーン分けで、許可された出力パスへの書き込みや外部サービス契約を静的に保護。評価ロジックと生成ロジックを切り離し、モデルを後から差し替えても評価器の「契約」は安定させておくアーキテクチャが核心。

実務への転用ポイントとして4点が提示される：（1）生成出口を JSON に固定、（2）モデル非依存モックで再現検証、（3）CI 外からもブラウザ＋プロセスの両経路で叩ける、（4）評価に「見逃しにくさ」の軸を持つ。監査エージェント開発への示唆としては、LLM 出力を構造化 JSON に固定し評価器を先に安定させるパターンは、監査ログの自動検証や根拠付き判定ルーティングにそのまま適用できる設計思想。

## アイデア

- 入力ハッシュから決定論的スタブを生成する `--mock` パターンにより、デモ・単体テスト・回帰テストを同一コードパスで賄える設計は、LLM ツールの再現性確保として汎用性が高い
- Playwright で同一シナリオを30サイクル反復し全成功を条件とすることで、環境ゆらぎによる偶発的 PASS を排除し、テスト回数を信頼性の定量指標として使う発想
- Oracle / Permitted / Prohibited のゾーン分けで外部サービス連携や決済ゲートウェイを静的に保護する AOS の物理ガバナンス概念は、エージェントの権限境界設計に直結する
