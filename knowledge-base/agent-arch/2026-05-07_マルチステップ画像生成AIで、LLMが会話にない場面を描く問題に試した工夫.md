## 要約

会話内容をアニメ調イラストに変換してXにシェアするサービスの個人開発において、LLMが会話に登場しない場面を生成する問題に対処するため4つの工夫を実装した記録。技術スタックはReact Router 7 + Hono + Mastra + Replicate（Flux 1.1 Pro）。

パイプラインは「moment extraction → scene composition → flux prompt → 画像生成 → text layout」の4ステップで構成され、1段目で誤った場面が抽出されると後段すべてに伝播する構造的リスクがある。

工夫1は「禁止形プロンプト」の優先配置。推奨形（〜を描いてください）ではLLMが物語的補完を行い無関係な場面を追加するが、禁止形（〜を創作しないこと）を最上部に置くことで出力が安定した。LLMは何を描いてはいけないかを明示しないと把握しないという知見。

工夫2はStructured OutputとJSON出力例の二重拘束。MastraのZodスキーマでkeyMomentsの型と個数（min:2, max:4）を強制するが、型適合だけでは中身の質（action: "散歩する" も型合法）を担保できない。プロンプト本文に具体的なJSON出力例（焦げた唐揚げを見て笑う場面など）を追加することで「粒度」と「意図」も伝達できた。

工夫3は前段の抽象値をコードで具体的な数値制約に変換してから後段に渡す方法。キャラ位置"left"をそのまま渡すのではなく、「x: 0〜400は配置禁止、テキストはx: 700〜1100に配置」というピクセル座標の禁止ゾーンに展開してtext layoutに渡す。LLMは抽象的な概念より数値範囲の方が制約を守りやすいという観察に基づく。

工夫4は生成画像の事後計測によるフィードバックループ。scene上の想定キャラ位置と実際の画像生成結果が一致しない（3〜4割は意図した位置に収まらない）問題に対し、SharpでRGBを16×16グリッドにダウンサンプルし、明度220以上のセルを余白（.）、それ以外をキャラ領域（#）としてマップ化。このASCIIマップをプロンプトに埋め込んでAIにテキスト配置座標を再問い合わせすることで、テキストとキャラの重なりを削減した。

最終的にgpt-image-2の登場により多段パイプラインの存在意義が薄れたとして開発を一区切りにしているが、4つの工夫は汎用モデルの外側に薄い制御層として応用可能と著者は述べている。監査エージェント開発への示唆として、マルチステップエージェントにおける「前段の抽象出力を後段が機械的に従える形に変換する」設計パターンは、LangGraphのノード間データ受け渡し設計にも直接適用できる。

## アイデア

- 禁止形プロンプトを最上部に配置する設計：LLMは「何を描いてはいけないか」を明示しないと物語的補完を行うという特性を逆手に取った制御法で、マルチエージェントシステムの出力制約全般に応用できる
- 抽象値→数値制約への前処理変換："left"→"x:0〜400禁止"のように前段の出力をコードで機械的に解釈可能な形に変換してから後段LLMに渡すパターンは、LangGraphのノード間データ変換設計に直接応用可能
- Sharpによる16×16グリッド余白マップの事後検証ループ：生成AIの出力不確実性を「計測→マップ化→再問い合わせ」で吸収する構造は、エージェントが外部ツール出力を検証するReActパターンの一形態
