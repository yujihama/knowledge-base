## 要約

本記事は、LLM が生成する技術ブログが均質な文体になる問題（「LLM の重力場」）を、永続的なキャラクター人格ファイルと append-only の「澱（dregs）」蓄積機構で解決するアプローチを解説している。著者は Claude Code の write-blog スキルに personalities/<name>.md という「声の契約」ファイルを読み込ませることで、LLM が記事生成前に特定の人格（ろくでなし海賊船長）を内面化してから執筆するよう設計した。人格ファイルには文体・一人称・語彙・アンチパターンを明示的に定義し、丁寧語・慇懃な前置き・教科書的列挙などをブロックリスト化している。核心的な仕組みが「澱」だ。記事を公開するたびに personalities/dregs/<name>.md へ日付付きの沈殿層（効いた一文、浮いた箇所、新出比喩、勝ちタイトルの型）を append-only で追記する。古い層は削除しない。次回の記事生成時にこの dregs ファイルを personality ファイルと合わせて読み込むことで、モデルの重みを更新せずにコンテキスト追加だけでキャラクターが「成長」する設計となっている。タイトル生成では Codex・Gemini・Claude の 3 モデルを並走させ 15 候補を生成後、同一の評価基準でスコアリングして勝者を決める。この際、3 モデル全員に声の契約（voice contract）を prepend することで評価の一貫性を担保している。実装はシェルスクリプト 2 本（personality.sh と append-dregs.sh）で完結する。personality.sh は $XDG_CACHE_HOME 配下の sticky cache にアクティブなキャラクター名を書き込み、次回以降も明示的な切り替えなしに同一人格を維持する。append-dregs.sh は stdin を受け取り、日付と記事スラグをヘッダにして dregs ファイルへ末尾追記するだけのシンプルな設計だ。監査エージェント開発への示唆として、この「append-only コンテキスト蓄積による振る舞いの持続化」パターンは、エージェントが判断ログや失敗パターンを自己の記憶として蓄積しながら次のタスクに活用する長期記憶アーキテクチャの実装に直接応用できる。LangGraph 等でエージェントの判断ルールを外部ファイルに append していく設計と構造が同一であり、モデルファインチューニング不要でエージェントの振る舞いを漸進的に調整する手法として参考になる。

## アイデア

- モデルの重みを更新せず、append-only のコンテキストファイル蓄積だけでエージェントの振る舞いを漸進的に「成長」させるパターンは、ファインチューニングコストゼロの長期記憶実装として監査エージェントのルール蓄積にそのまま転用できる
- 3 モデル（Codex・Gemini・Claude）に同一の voice contract を prepend して並走させ merit ベースで勝者を決める「タイトル競合」は、LLM-as-judge パターンの実装例であり、複数エージェントによる出力品質評価フレームワークとして汎用化できる
- プロンプト単体では 3 段落目で素の優等生に戻る「LLM の重力場」問題を、事前の人格内面化ステップと実績ログ（澱）の読み込みで抑制する設計は、エージェントのペルソナ保持や一貫した判断スタイルの維持に応用できる
