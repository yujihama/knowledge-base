## 要約

本記事は、エンタープライズAIシステムにおけるLLMの推論精度劣化を自律的に検知・修復する「/LPプロトコル」を提唱する。背景として、長大なコンテキストウィンドウ運用時に発生する「Lost in the Middle（中間欠落）」と「コンテキスト汚染の累積」という2つの構造的問題がある。前者はコンテキスト中盤への注意力（Attention）が低下して制約が無視される現象、後者は対話履歴の蓄積によりノイズが増大しハルシネーション頻度が上昇する現象である。

解決策として、分散システムのLiveness Probe概念をLLMに応用する。APIゲートウェイはセッション開始時のシステムプロンプト先頭（Primacy Effect活用）に、XMLタグで囲んだ算術演算プローブ（例：`<system_probe action="calc">14+23</system_probe>`）を埋め込む。XMLタグによる意味空間の隔離でユーザーコンテキストとの干渉を防ぎ、決定論的な算術タスクでトークン境界依存の誤検知も排除する設計である。

応答結果に基づき3段階診断を実施する。「正常（Healthy）」は正しいタグ構造と正解計算（`<probe_response>37</probe_response>`）を返した状態でセッション継続。「警告（Warning）」はタグ崩れや計算ミスが発生した状態で、重要タスクをブロックしHard Reset（システムプロンプトのみ残す完全初期化）または別モデルによるコンテキスト再構築を実行。「危険（Critical）」はコマンドを通常会話として処理しハルシネーション開始した状態で、セッション破棄・新規インスタンス起動を行う。

JIT（Just-In-Time）投入として、DBクエリ生成や契約書レビューなどハルシネーション許容不可タスクの直前にプローブを差し込み、Healthy確認後のみ本命タスクを実行させる。実装上の留意点として、JITプローブは長大コンテキストを2回送信するため入力トークン課金が2倍になるリスクがあり、プロンプトキャッシング（KVキャッシュ再利用）が前提条件となる。また偽装対策としてリクエストごとに乱数A＋乱数Bの動的数式を生成する。Claude 3.5 HaikuなどのモデルではHealthy状態でも計算ミスする偽陽性リスクがあり、モデル能力に応じた難易度チューニングが必要。APIゲートウェイはステートフル化により乱数管理・応答インターセプト・オーケストレーションの責務を負い、TTFT（Time To First Token）分のレイテンシが追加される点もトレードオフとして存在する。

監査エージェント開発への示唆として、本プロトコルはLangGraphベースのマルチエージェントシステムにおけるノード間品質ゲート（Quality-Gated Pipeline）として転用可能。高コストな推論ステップへの入力前にヘルスチェックを挟むことで、監査レポート生成における不正確な推論結果の連鎖を水際で防止できる。

## アイデア

- LLMの認知状態をネットワーク監視のPing/Liveness Probeとして捉え、決定論的算術タスクをXMLタグで意味空間を隔離しながら投入することで、外部リソース不要・低トークンコストで推論精度を定量的に計測する設計思想
- JIT（Just-In-Time）プローブ投入によりハルシネーション許容不可タスク直前にHealthy確認を義務化するPre-flight Checkパターンは、監査AIや医療AIなどの高信頼性要求システムにそのまま応用可能なアーキテクチャパターン
- プロンプトインジェクション対策として、固定値の偽装を防ぐためにリクエストごとに動的乱数を生成して計算させるNonce的アプローチは、LLMをセキュリティ境界として利用するゼロトラストアーキテクチャの実装例として興味深い
