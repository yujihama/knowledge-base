## 要約

エージェント開発の文脈で頻出する「最後は人間が責任を取る」という言い回しの空疎さを指摘し、エンジニアの真の役割を「目的の翻訳業」として再定義する論考。

著者の主張の核心は「目的設定の非対称性」にある。エージェントは与えられた目的の下で最適化する能力を持つが、目的そのものを生成する能力は構造的に持ち得ない。目的を内部化させようとすると「目的を選ぶための目的」が再帰的に要請され、その最上位には必ず人間が置かれる。これはモデルの賢さの問題ではなく、構造的な問題である。

「責任」という言葉が曖昧な理由として、法的責任・倫理的責任・運用上の責任が混在して使われている点を挙げる。現場のエンジニアが日々向き合っているのは実質的に「原因追究と改善策提示のループ」（post-mortem、retrospective、CI、monitoring）に過ぎないが、この原因追究自体が「あるべき姿」の定義なしには成立しない。例として、エージェントがテストを緩めてコードに合わせる行為を「逸脱」と判定するには「テストは仕様である」という規範が先に必要であり、この規範は技術的事実から自動導出されるものではなくソフトウェア工学の文化的価値判断だと論じる。

改善策の選択も同構造で、「再発防止優先→リリース速度犠牲」「影響最小化優先→ロールバック投資」「学習機会→post-mortem全社共有」「開発速度維持→技術的負債計上」のいずれを選ぶかは組織フェーズや競合状況に依存し、技術的事実だけでは決まらない。

「目的の翻訳業」として再定義すると、エンジニアの作業が具体的に見える。システムプロンプト設計（目的を自然言語に翻訳）、ツール定義の絞り込み（目的の境界条件設定）、evaluationの設計（目的適合度の測定装置）、ログのトレーシング設計（逸脱検出粒度の決定）、stop conditionの定義（達成判定基準）、guardrailの設計（目的外逸脱の遮断）がすべて「目的を実装可能な形に変換し、適合性を検証可能にする」一貫した作業の別側面として統合される。

GitHub Copilot in VS CodeのHarness設計（VSC-Bench、PR単位評価フロー）を「ハーネスは目的との接続装置」と解釈し直す点も実践的示唆として有用。

監査エージェント開発への直接的示唆：監査エージェントにおける「逸脱の判定基準」（内部統制上のあるべき姿）はエージェントが自動生成できず、監査基準・組織ポリシー・規制環境を人間が翻訳してシステムプロンプトや評価基準として実装する必要がある。evaluationの設計こそが監査AI品質の核心であり、「人間がレビューする」だけでは設計として不十分。

## アイデア

- 「目的を内部化させようとすると上位目的が再帰的に要請される」という構造的議論は、AGIの能力論とは独立した問題であり、エージェントに目的生成を期待することが原理的に誤りである根拠として有効
- evaluation設計・stop condition・guardrailをすべて「目的適合度の検証装置」として統一的に把握する枠組みは、エージェントシステムの設計レビューチェックリストとして実用的
- マルチエージェント・マルチテナント・マルチステークホルダー構造での「目的設定者の分散」問題は未解決として明示されており、LangGraph等での複数エージェント協調設計における目的整合性の研究課題として重要
