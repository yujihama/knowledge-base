## 要約

ContinueとFilesystem MCPを使い、ローカルLLM（Gemma-4-26B Q5_K_S、llama.cpp）でTailwind CSS v4のフロントエンド開発を試みた実験レポート。8192トークンのコンテキスト制限を持つローカルモデルに対し、MCPの富豪設計が致命的な問題を引き起こした。

具体的な失敗メカニズムは以下の通り：Continueはデフォルトで出力予約枠として4096トークンを確保し、Filesystem MCPのツールスキーマが2761トークン、システムメッセージ（エージェントプロンプト＋.rulesファイル）が2131トークンを消費した。合計9151トークンとなり、8192トークンの上限を超えた状態でエラー「Message exceeds context limit」が発生。AIは質問に一文字も答えず、ファイルを一枚も開かない段階で頓死した。さらにAIは同一ツールを無限ループで叩き続けるという副次障害も発生した。

根本原因はContinueとFilesystem MCPが数十万〜数百万トークンのクラウドLLMを前提とした「富豪設計」であること。8K〜32K制限のローカルLLMにそのまま適用すると初期化時点でトークン予算が破綻する構造的欠陥がある。

解決策として、ContinueのdefaultCompletionOptionsを削除しllama.cpp側に設定を一本化することでパフォーマンスが安定した。また、より根本的な改善案として3点を提示：①必要なコードスニペットとその前後数行のみをgrep抽出して返す自作ツール、②ファイル全体上書きではなく特定要素のクラス名のみをピンポイント差替するツール、③プロンプトチェーンによるステップバイステップ処理。

結論として「既製の全自動エージェントモードへの丸投げをやめ、情報粒度をコントロールする自作の仕組みを設計すべき」という知見を得た。コンテキストへの無駄なテキスト混入を防ぐ人間側の設計思想が、既製MCPより実用的であるという主張。監査エージェント開発への示唆として、ローカルLLMを使ったエージェントシステム構築では固定トークン消費（ツールスキーマ＋システムプロンプト）を最小化する設計が必須であり、LangGraphのようなワークフロー制御と組み合わせる場合も同様のトークン予算管理が重要になる。

## アイデア

- MCPツールスキーマ自体が2761トークンという固定コストを持つことで、8Kコンテキストモデルでは初期化だけで予算の34%を消費するという定量的な罠
- クラウドLLM前提の「富豪設計」ツールをそのままローカルLLMに適用した場合の失敗パターンとして、無限ループ＋コンテキストオーバーフローが同時発生するという複合障害
- 自作軽量MCPの方向性として「ファイル全体ではなくgrepで必要箇所のみを切り出して返す」という情報粒度制御の原則は、RAGのチャンク設計と同じ問題を別レイヤーで解いている点
