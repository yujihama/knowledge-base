## 要約

Claude Code v2.1.172 でサブエージェントが最大5階層まで入れ子で別サブエージェントを呼び出せるネスト機能が追加された。株式会社GENDAのエンジニアが、既存の役割ベースサブエージェント設計（code-explore/implementer/heavy-implementer/test-runner）を運用中にこの機能を体験した事例報告。

サブエージェントは ~/.claude/agents/ 配下に Markdown ファイルで定義され、frontmatter の model: フィールドでモデル（Opus/Sonnet/Haiku）を固定できる。メインセッションは計画・設計判断を Opus が担い、実装系タスクを heavy-implementer（Opus）、調査を code-explore（Opus）、テスト実行を test-runner（Haiku）へ委譲する構成だった。

v2.1.172 以前は「サブエージェントはサブエージェントを呼べない」という制約があり、2層構造に限定されていた。この制約下では1つのサブエージェントが複数役割を抱え込むか、メインセッションが中継役を担う必要があった。新機能により、heavy-implementer が自律的に code-explore（参照追跡）と test-runner（テスト実行）を孫として呼び出す3層構造が実現した。

具体的な観察例：heavy-implementer に「ある関数の呼び出し箇所を一通り直す」タスクを委譲したところ、内部で code-explore が grep による参照列挙を行い、実装完了後に test-runner が pnpm exec test を実行して全PASS を確認。メインセッションはその旨を受け取るのみで、100 tool uses・100k tokens・10分の作業が自律完結した。

ネストが機能する技術的根拠は、.claude/agents/*.md の定義が Agent ツールに name/description/model/システムプロンプトを含む構造化宣言として登録され、メインセッションだけでなく子エージェントからも同じ subagent_type 選択肢として見えるためだ。CLAUDE.md/rules はテキスト指示のため強制力がなく長セッションで埋もれる。Skill はモデル継承のため役割ごとのモデル固定ができない。サブエージェント定義のみが「モデル固定＋別コンテキスト実行＋全階層から再利用」を同時に満たす。

設計判断の更新点として、エージェント定義の判断軸に「サブエージェントから孫として呼ばれそうな汎用ロールは早めに定義する」が加わった。test-runner はメインセッションより implementer/heavy-implementer からの孫呼び出しの方が多くなり、長いテスト出力を呼び出し元のコンテキストに取り込まない「二段重ねの文脈隔離」が機能している。監査エージェント開発への示唆として、LangGraph 等での階層的エージェント設計においても、汎用ツール系サブエージェント（ログ取得、検証実行等）をあらかじめ定義しておくことで、任意の中間エージェントから再利用されるパターンが有効と考えられる。

## アイデア

- サブエージェント定義は「単発の委譲先」ではなく「全階層から再利用される共通部品」であり、定義コストは固定でも呼び出し回数はネストで倍増するため、早期定義のROIが高い
- CLAUDE.md（テキスト指示）・Skill（モデル継承）・サブエージェント定義（構造化宣言）の三者を比較すると、モデル固定＋別コンテキスト実行の両立はサブエージェント定義のみが満たす
- test-runner を孫として使う「二段重ねの文脈隔離」により、長大なテスト出力が実装エージェントのコンテキストを汚染せず、各層が自分の担当に集中できるアーキテクチャが自然に成立する
