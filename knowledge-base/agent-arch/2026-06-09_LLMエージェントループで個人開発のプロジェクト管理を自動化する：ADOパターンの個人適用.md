## 要約

Azure DevOps（ADO）向けに設計された5フェーズのエージェントループアーキテクチャを、Markdownファイル＋Pythonスクリプトで個人開発に移植した実践記録。

元のADOアーキテクチャは「Init → Content Reconciliation → Health Monitoring → Spec Aggregation → SWE Handoff」の5フェーズで構成される。個人開発版では各フェーズを以下のように代替する：

- **Init**：MEMORY.mdからProjectStateデータクラスにデシリアライズ。決定事項・リスク・ブロックタスク・未完了タスクをregexで抽出。
- **Content Reconciliation**：セッションログ内のバッククォート記法（例：`src/api/main.py`）からファイルパスをregexで抽出し、実ファイルの存在チェックと`git status --porcelain`による未コミット検出を組み合わせて、ドキュメントと実装の乖離を自動検出。
- **Health Monitoring**：LLMを使わない決定論的スコアリング式を採用。`HealthScore = 決定完了率×0.4 + (1－リスク率)×0.3 + (1－ブロック率)×0.3`で算出し、A〜Dのグレードを付与。LLM呼び出しを排除することで毎回のAPIコストをゼロにし、結果の揺れも防ぐ。
- **Spec Aggregation**：Claude Codeや他のLLMエージェントへのハンドオフ用に、プロジェクト名・ヘルススコア・未完了決定事項・ブロックタスク・高リスク項目・成功基準を含むYAMLフロントマター付きMarkdown仕様書を生成。
- **SWE Handoff**：生成したSpecファイルをClaude Codeセッションのコンテキストとして渡し、実装エージェントへ引き継ぐ。

エントリーポイントとなる`agent_loop.py`がこれら5フェーズをオーケストレーションし、LLMは実際の実装タスク（SWEフェーズ）にのみ投入する設計。監査エージェント開発への示唆として、プロジェクト健全性のスコアリングロジックを決定論的に実装するパターンは、監査チェックリストの充足率算出や内部統制評価スコアの自動計算にも直接転用可能。また、ドキュメント（仕様書・証跡）と実装ファイルの乖離検出ロジックは、監査証跡の完全性チェックへの応用が見込める。

## アイデア

- ヘルススコアをLLMではなく決定論的数式（重み付き線形結合）で算出することで、APIコストゼロ・結果安定を両立するアーキテクチャ設計
- セッションログ内のバッククォート記法を正規表現で抽出してファイル存在チェックする『Content Reconciliation』パターンは、ドキュメント駆動開発の整合性維持に汎用的に使える
- LLMへのハンドオフをYAMLフロントマター付きMarkdown Specファイルとして構造化することで、コンテキスト注入を標準化し再現性を高める『Spec駆動ハンドオフ』パターン
