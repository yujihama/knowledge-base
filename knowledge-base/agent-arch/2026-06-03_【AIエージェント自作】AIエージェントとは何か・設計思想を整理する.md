## 要約

本記事は「AIエージェントを自作することでAIエージェントを理解する」をテーマにしたシリーズA（全8回）の第1回。エージェントを「自律的にツールを使いながら、目標を達成するまでループし続けるLLMの応用」と定義し、通常のチャット（入力→LLM→回答で終了）との違いを「ループ」と「ツール」の2要素で整理する。

作成するのはCode Maintenance Agent：ソースコードの改良・保守を支援するエージェントで、Claude Codeのミニマム版を目標とする。read_file / write_file / list_files / search_text / run_command / show_diff / patch_fileの7本のツールを装備し、ターミナルREPLとして動作する。

設計のポイントは3つ。①LLM抽象化：抽象基底クラスLLMClientBaseにchat()メソッドを定義し、OllamaClientとClaudeClientを同一インターフェースで切り替え可能にする。起動引数--llm ollama / --llm claudeで切り替え。デフォルトはOllama（qwen2.5-coder:7b）、サブはClaude API（claude-sonnet-4-20250514）。ローカルLLMを使う理由はコスト・速度・学習効果・比較の4点。②人間確認フロー：write_file / patch_file / run_commandなど副作用を持つツールは必ずユーザー承認を取得してから実行。差分・コマンド内容を表示した上で確認を挟む設計。③段階的実装：Phase A（土台：API呼び出し・REPL）→Phase B（ツール化）→Phase C（書き込み・実行）→Phase D（統合）の4フェーズ・13ステップで実装を進める。

技術スタックはPython 3.11以上、OllamaのOpenAI互換APIにopenaiライブラリで接続、anthropicライブラリでClaude API接続、difflib・subprocessを依存ライブラリとして使用。リポジトリはGitHub（pekopugu/agent01）。

監査エージェント開発への示唆：LLMクライアントをファクトリパターンで抽象化する手法は、監査エージェントのLLM切り替え（ローカルLLM→Claude API）にそのまま適用可能。特に書き込み系ツールで人間確認を必須とする設計思想は、監査証跡の整合性を担保する内部統制エージェントの設計原則と親和性が高い。

## アイデア

- LLMクライアントを抽象基底クラスで統一し、ファクトリ関数で切り替えるパターンは、監査エージェントのLLMバックエンド差し替えにも直接転用できる
- 副作用を持つツール（書き込み・コマンド実行）にのみ人間確認を挟む設計は、Human-in-the-loopの粒度設計として参考になる
- ローカルLLM（Ollama）で開発・検証してからClaude APIで品質確認するワークフローは、エージェント開発のコスト管理手法として実践的
