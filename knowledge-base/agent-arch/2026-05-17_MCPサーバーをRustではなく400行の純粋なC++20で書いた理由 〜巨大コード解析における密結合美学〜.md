## 要約

著者はLLVM級の巨大コードベースをAIエージェントに解析させるためのMCPサーバー基盤「mcp-cst-core」を、RustやPythonの公式SDKではなくC++20の400行シングルソースで実装した。その技術的根拠と設計思想を解説する記事。

技術的な核心はTree-sitterの活用にある。Tree-sitter（C99製）はソースコードのノードを「開始バイト・終了バイト」という座標インデックスのみで表現する超軽量設計を採用している。この設計に忠実に従うため、著者はRust製ラッパーが必然的に発生させるメモリ二重コピーや抽象化オーバーヘッドを一切排除し、C++20のSTLと直接結合する方針を選択した。

並列処理にはstd::jthread（C++20の自動joinスレッド）とstd::counting_semaphoreを使用。10コア環境でLLVMのCST（Concrete Syntax Tree）化を並列実行した際、仮想アドレス空間（VIRT）は一時的に274GBまで膨張し、CPU使用率は989%（10コア全開）に達する。解析完了後はRAIIによる決定論的リソース解放によって即座に1.1GBまで収縮し、残存するのはstd::move済みのソーステキストファイルキャッシュのみとなる。mmapなどのOS固有機能は不使用で、Linuxのオンデマンドページング（遅延割り当て）をSTLだけで最大活用している。

依存ライブラリはjsoncppとtree-sitterのみに絞り込み、サプライチェーン攻撃リスクを最小化している点もセキュリティ設計として強調されている。

著者はAPIを「①自然言語型（NLP/RAG）」「②仕様駆動型（OpenAPI/SDK）」「③解析型（座標駆動型）」の3パラダイムに分類し、本実装は③の究極系と位置づける。現在のClineやRoo CodeはこのインデックスドリブンAPIに対応したコンテキスト管理機能を持たず、LLVM級のデータを渡すとコンテキスト破綻が起きる。解決策として「AIが自分のコンテキスト空き容量に応じて切り出しバイト数を指定し、C++基盤がrow境界でチャンクを返し、AIがNextメソッドを自律呼び出しするインクリメンタル・ストリーミング」アーキテクチャを提案しており、これが監査エージェント等の大規模コード解析AIへの直接的な応用可能性を示している。

## アイデア

- 274GB→1.1GBのVIRT劇的収縮：RAIIとLinuxの遅延ページング割り当てを組み合わせることで、OS固有APIなしに仮想メモリを最大活用しつつ決定論的クリーンアップを実現する設計パターン
- 座標駆動型APIパラダイム：Tree-sitterのバイトオフセット座標だけでコード空間を支配し、文字列実体をメモリに持たない設計が、LLVM級巨大コードベースのAI解析で唯一スケールする根拠
- AIドリブン・インクリメンタルストリーミング：AIが自己のコンテキスト残量を把握して切り出しサイズを動的指定し、C++基盤がrow境界でチャンク返却、AIがNextを自律呼び出しするループ構造は、監査エージェントでの大規模ソースコード走査に直接転用可能な設計思想
