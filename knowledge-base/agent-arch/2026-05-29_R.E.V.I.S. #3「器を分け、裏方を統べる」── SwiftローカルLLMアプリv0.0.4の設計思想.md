## 要約

SwiftでローカルLLMアシスタント「R.E.V.I.S.」を開発するeNIGM4氏と、ClaudeがコードをインタビューするZenn連載の第3回。対象はv0.0.4（2025年2月25〜26日、LLMに触れて3日目）。メイン側23ファイル・2,330行と前版から倍近い規模に拡大した。

本版で導入された主な前進は4点。

1. **横断的ベクトルRAG**: ChatThreadにsummaryVectorを持たせ、アイドル時にarchiveToLongTermMemoryでベクトル化。新規入力時にgenerateEmbeddingで埋め込みを生成し、cosineSimilarityで過去スレッドと突き合わせて関連する会話を呼び戻す。v0.0.3のスレッド内圧縮から、スレッドをまたぐ長期記憶検索へ進化した。開発者の言葉を借りれば「どんなに圧縮してもブラックホールではない」ため、容量上限への根本解として設計された。

2. **自動スレッドルーティング**: OrchestratorAnalysisがcommandロールにanalyzeThreadContextを投げ、返りJSONのactionをparseThreadActionで読み取る。NEW_THREAD（話題転換→新スレッド生成）とRECALL（過去スレッド呼び戻し）の2択で、handleThreadTransitionが直前1往復を該当スレッドへ付け替えてactiveThreadIDを切り替える。ユーザーが「新規スレッドボタンを押す」操作を不要にするための横着起源の機能であり、「意味的に尖った文脈を汚染から守る」ことが本命の動機。

3. **文脈注入による孤立防止**: buildComprehensiveContextが他スレッドの直近往復やUserProfile（UserProfileFact/ProfileExtractionResult）を注入し、「直接関係はないが話したことは覚えている」状態を維持。buildShortTermMemoryとの使い分けにより、スレッドを尖らせつつ孤立させない設計になっている。

4. **優先度付きTaskQueue（Falqの祖）**: バックグラウンド処理（圧縮・ベクトル化・スレッド振り分け・プロフィール抽出）が同版で4種まで増えたことへの管理解として導入。QueuePriority（idle=0/normal=1/high=2/critical=3）でユーザー応答直結処理を.critical、裏方整理を.idleに振り分け、「裏処理で返事が出ないポンコツ」を防ぐ。後から土台を入れ替えるコストを嫌い、最初から優先度制御を組み込んだ点が特徴。

さらに2/26付けでVoiceManagerが追加され、v0.0.3時点でspeakerプロンプトに既存していた「音声合成前提・カタカナ表記・記号禁止」の縛りの受け皿が実装された。名称R.E.V.I.S.はアイアンマンのJARVISを意識しており、「話しかけたら勝手に全部やってくれるAI執事」という一本の設計思想がv0.0.1から通底している。監査エージェント開発への示唆としては、TaskQueueの優先度制御パターンが複数の非同期バックグラウンドタスク（ログ分析・リスク評価・レポート生成等）の管理に直接応用可能。

## アイデア

- 「新規スレッドボタンを押すのが面倒」という横着を動機に自動スレッドルーティングを設計した点：UX上の怠惰が意味的文脈汚染防止という本質的な推論品質問題の解決策と一致している
- スレッドを意味的に尖らせつつbuildComprehensiveContextで他スレッドの要約を注入する「尖らせるが孤立させない」二層設計：RAGとコンテキスト注入の役割分担として監査エージェントの複数ワークフロー間情報共有に転用できる
- 後処理が4種に増えた時点でQueuePriorityを先回りして導入したアーキテクチャ判断：critical/idle分離により裏方処理の増加が応答レイテンシに影響しない構造を早期に確保している
