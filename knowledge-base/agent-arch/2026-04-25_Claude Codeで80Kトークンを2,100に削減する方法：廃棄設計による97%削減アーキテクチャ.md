## 要約

Claude Codeを自律エージェント（autonomous-brain）として運用した結果、1セッションあたり78,000〜92,000トークンを消費するようになった。応答遅延・文脈忘却・同一ファイルの重複読み込みが頻発し、コンテキスト肥大化が問題となった。Pineconeやembeddingsの導入を検討したが追加インフラを避けるため、3本柱のアーキテクチャで解決を図り、最終的に2,100トークン前後（削減率97%）を達成した。

問題の本質はメモリ不足ではなく「廃棄の設計不在」にある。同一ファイルを複数回読むと読み込み内容が重複してスタックに積まれ、解決済みエラーの往復議論も全履歴として保持される。この冗長な蓄積が肥大化の原因であり、単純にembeddingsで検索精度を上げても解決しない。

3本柱の内容は以下の通り。①セッション要約（Session Summary）：会話が一定のトークン閾値（40K）を超えた時点でLLMが会話全体を200トークン以内に圧縮し、memory/sessions/に保存する。次セッション冒頭でロードして履歴を代替する。この単体で履歴部分が60〜70%削減された。②差分メモリ（Delta Memory）：project_state.mdに「変化した事実」だけを追記するgit差分型の記録方式。スナップショットではなくdiffのみ保持する。③優先度スタック（Priority Stack）：skillファイル冒頭に「ロードする／しない」ファイルリストを明示し、タスクに無関係な文脈を初期ロード時点から除外する。この明示がなければClaudeは全ファイルを読もうとする。

実装はPostToolUseフックで行い、settings.jsonのhooksセクションにcontext_monitor.pyを登録する。同スクリプトがセッションコストを監視し、40Kを超えた時点で自動的に要約を生成する。要約プロンプトには「新しい担当者が読む引き継ぎメモ」という文脈を与えることで、プロセス記述ではなく決定事項中心の出力を誘導する。

捨てていい情報はエラーと修正の往復、試行ログ、ファイル内容（再読み可能）、確定済み議論。絶対に残すべきは設計理由（Why）、外部制約（「このAPIは使えない」等）、未解決の保留事項、ユーザーのフィードバック。特に外部制約を欠落させると、制約の再発見に多大な時間を要する。

興味深い実測結果として、要約ありのセッション継続とゼロから再開の品質差がほぼ消滅し、むし�要約によるコンテキストリセットが誤認識の蓄積を防いで精度向上に寄与するケースも確認された。監査エージェント開発への示唆として、長期実行エージェントのセッション設計においてもこの「廃棄設計＋優先度ロード」パターンは直接適用可能であり、PostToolUseフックによる自動要約トリガーはLangGraphのステート管理と組み合わせることで運用コスト削減効果が期待できる。

## アイデア

- コンテキスト肥大化の根本原因を「記憶不足」ではなく「廃棄設計の不在」として定式化した点が本質的。embeddingsで解決しようとするアプローチへの反論として明確
- 要約ありセッション継続がゼロ再開と品質でほぼ同等になるという実測結果は、長いコンテキストが必ずしも精度向上に寄与しないことを実運用で示した反通説的エビデンス
- LLMのコンテキスト管理と人間の作業記憶管理が構造的に同一問題として収束するという観察は、認知アーキテクチャとAIシステム設計の対応関係を示す概念として応用範囲が広い
