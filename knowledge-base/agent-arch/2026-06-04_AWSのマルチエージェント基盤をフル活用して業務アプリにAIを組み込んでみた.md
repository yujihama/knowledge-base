## 要約

本記事は、AWS AgentCoreを中心としたマルチエージェント基盤を用いて、インシデント自動対応システム「Agora」を構築した実装レポートである。AWS FISで障害（RequestLimitExceeded）を注入すると、CloudWatch → EventBridge → SQS → Lambda → DynamoDB Streams という一連のイベント駆動連鎖が起動し、Triage・Diagnosis・Resolutionの3エージェントが順次実行される。既存のTicket Serviceには一切手を加えず、後付けでエージェントを統合できる構成が特徴的。

エージェントの実装にはAWS OSSフレームワーク「Strands Agents」を採用。@toolデコレータでツールを定義するだけでよく、フロー制御はLLMが自律判断する。Diagnosis AgentではGraphBuilderによる動的並列実行を活用し、Judgment AgentがRegistry上のランブックをセマンティック検索で選択（3件中2件を選択、1件はLLMが「診断フェーズ不要」と自律除外）。選ばれた2ノードがT+227s〜T+393sの166秒間、CloudWatch・Lambda・Stack Overflow・GitHub Issues等のMCPツールを並列実行した。

AWS AgentCoreは3サービスで構成される。RuntimeはエージェントとMCPサーバーをコンテナとしてサーバーレスでホストし、セッション管理・IAM認証・プロトコルスタックを担う。GatewayはMCPの統一アクセスポイントとして、Runtime上のMCPサーバー・Lambda関数・既存REST API・外部エンドポイントを同一プロトコルで公開する。RegistryはARNでエージェントを動的発見し、セマンティック検索でスキルを選択する。エンドポイントのハードコードが不要なため、ランブック追加・削除時のエージェント再デプロイが不要。

Bedrock Prompt Managementでシステムプロンプトをコード外に分離し、開発者以外でもコンソール編集・バージョン管理が可能。Prompt CachingではCacheConfig(strategy="auto")により自動キャッシュポイント挿入が実現され、Diagnosis WorkersでTurn3〜5の新規input tokensが3トークンまで激減。Claude Sonnet 4.6全体でキャッシュ再利用率91%（cache_read 74,795 / 総input 82,429トークン）を達成した。ベクトルストレージにはS3 Vectorsを採用し、OpenSearchやベクトルDBクラスタ不要でBedrock Embeddings（Titan Embed v2）と組み合わせた類似検索を実現している。コード一式はmahitotsu/moiraiで公開されており、CDK 1コマンドでデプロイ可能。監査エージェント開発への示唆として、既存業務システムへのエージェント後付け統合パターン・セマンティックランブック選択・イベント駆動による非侵襲的統合が直接応用可能である。

## アイデア

- AgentCore RegistryによるセマンティックランブックのLLM自律選択：コードを一行も書かずに動的スキル選択を実現し、ランブック追加・削除時の再デプロイが不要な拡張性の高いアーキテクチャ
- Prompt Caching（strategy=auto）でTurn3以降の新規inputを3トークンまで削減、キャッシュ再利用率91%達成：長いシステムプロンプト＋ツール結果を持つエージェントループへの効果が特に大きい
- 既存REST APIをOpenAPIスペックからMCPへ自動変換してGateway経由で公開するパターン：既存サービスを無改修のままエージェントから利用可能にする非侵襲的統合手法
