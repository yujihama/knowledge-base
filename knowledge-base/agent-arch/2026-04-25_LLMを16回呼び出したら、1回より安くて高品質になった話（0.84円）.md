## 要約

57分の会議議事録をGemini 2.5 Flash-Liteで生成する際、1プロンプトで全セクションを出力させる手法から、タスク分解＋Context Cachingの組み合わせへ移行した設計事例。1回呼び出しでは「話者サマリーが全員同じ内容」「状況報告がアクションアイテムに混入」「議論の経緯が欠落」といった品質劣化が発生した。原因はモデルの限界ではなく、1プロンプトに複数の役割を詰め込んだタスク設計の問題と分析。解決策として「構造はコードで管理、LLMには1タスクのみ渡す」原則を採用。Step1（並列）で概要・決定事項・話者別サマリーを6回、Step2（逐次）でアクションアイテム候補抽出→レビューを2回、Step3a/bで議論タイトル抽出と背景生成を計8回、合計16回に分解した。コスト削減の核心はGeminiのContext Caching機能で、文字起こし（11,978トークン）をキャッシュ登録しIDで参照することで、16回分のトランスクリプト再送信をなくす。キャッシュ読み取り料金は通常入力の1/10（$0.01/1M tokens）。実測コストはキャッシュ作成$0.0012＋キャッシュ読み取り$0.0019＋追加入力$0.0009＋出力$0.0013＝合計$0.0052（約0.84円）。上位モデル1回呼び出しより安く高品質という逆転現象が成立する理由は、下位モデルのキャッシュ読み取り単価が十分低いため入力を潤沢に使い回せる点にある。並列実行により全体レイテンシも削減可能。開発背景はiOSアプリ「爆速文字おこし」の完全無料・広告モデル運用で、広告単価1〜2円/インプレッションに対しLLMコストを1円以下に収めることがビジネス上の制約条件。Anthropic Prompt Cachingでも同様の設計が適用可能。残課題はアクションアイテムの精度で、LLMが「言っていないことを書かない」制約に弱く、候補抽出→レビューの2ステップで部分対処している。

## アイデア

- 「モデルを上げる」前に「タスク設計を見直す」という発想転換：上位モデルへのアップグレードで解決しようとする前に、1プロンプト多タスクという構造的欠陥を先に解消することで、下位モデルでも上位モデルを超える品質が達成できる
- Context Cachingによるコスト逆転の算数：同一文脈を16回渡す一見無駄な設計が、キャッシュ読み取り単価1/10によって成立する。入力コストが全体の76%を占める構造の中でキャッシュが効くほど、分割数が増えてもコストが上位モデル1回を下回り続ける
- 候補抽出→レビューの2ステップによるLLMの生成バイアス制御：LLMが「もっともらしいアウトプットを生成する」方向に最適化されている性質（アクションアイテムを無理やり作る）を、広め抽出→フィルタリングという構造で補正するパターン
