## 要約

Factory社のLuke Alvoieroが提唱するマルチエージェントシステム「Missions」の設計原則を解説した記事。Slackクローン構築ミッションで16日間・人間介入なしの自律稼働を実現した実績に基づく。

核心となるアーキテクチャはOrchestrator（計画・検証コントラクト生成）・Worker（単一機能実装）・Validator（独立検証）の3役割分離。各役割の責務境界を厳密に定め、「構造化ハンドオフ」の5フィールド（completed / not_completed / commands_executed / issues_found / procedure_compliance）で状態を引き継ぐ。

最大の設計上の知見は「検証コントラクトを実装前に書く」こと。実装後に書かれたテストは認知バイアスによる追認になるため、OrchestratorがWorker着手前にアサーション集を確定し、Validatorが独立評価する2フェーズ分離を徹底する。Validatorは実装コードの事前参照を禁じる「敵対的設計」を採用し、WorkerのコンテキストをValidatorに渡さないことでドリフトを防ぐ。

Validatorは2系統に分離される：Scrutiny Validator（静的解析・コードレビュー）とUser Testing Validator（UI操作・E2Eテスト）。並列実行については「書き込みは逐次、読み取りは並列」の原則を採用。100ステップでエラー率1%なら成功率36%、0.1%なら90%以上という数理的根拠から逐次実行の優位性を示す。

「Droid Whispering」と呼ぶ役割別LLM選択も特徴的。Orchestratorには推論能力優先、Workerにはコーディング特化、Scrutiny ValidatorにはWorkerと異なるプロバイダーを使い、訓練データバイアスによる検証スルーを構造的に排除する。

オーケストレーションロジックはハードコード状態機械ではなく約700行のプロンプト＋スキルで定義され、4文の変更で実行戦略を大幅に変更可能。是正ループは例外処理ではなく通常フローとして設計され、実装に60%・検証と是正サイクルに残り40%の工数配分が実績値。最終コードベースのテストコード比率50%・カバレッジ90%を達成。

監査エージェント開発への示唆：Orchestrator-Worker-Validatorの役割分離はLangGraphのサブグラフ設計に直接対応し、検証コントラクトの「事前確定＋独立評価」パターンはLLM-as-judgeの設計原則と一致する。是正ループを通常フローとして組み込む設計は、監査証跡の自動生成ワークフローにも応用できる。

## アイデア

- 検証コントラクトを実装前に確定し生成者（Orchestrator）と評価者（Validator）を分離する2フェーズ設計：同一エージェントが生成と評価を兼ねると認知バイアスが再発するという問題を構造的に解決している
- エラー率の複利効果による逐次実行の正当化：100ステップでエラー率1%→成功率36%、0.1%→90%以上という数理的根拠で「遅く見えるが正確性が積み上がる」逐次戦略を説明する設計思想
- オーケストレーションロジックを約700行のプロンプトで定義し4文の変更で実行戦略を変えられるプロンプト駆動アーキテクチャ：状態機械をハードコードしないことでモデル更新への追従コストを最小化する
