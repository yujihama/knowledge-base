## 要約

MCP（Model Context Protocol）はAnthropicが2024年11月に公開したオープン標準プロトコルで、AIアプリケーション（Claude Desktop、Cursorなど）が外部ツールやデータソースと通信するための共通規格。従来はN個のAIアプリ×M個のツールでN×Mのコネクタが必要だったが、MCPにより「AIのUSB-C」として一元化された。

アーキテクチャはホスト・MCPクライアント・MCPサーバーの3層構成で、外部から取得したデータはAIのコンテキストに直接流れ込む。これが従来ツールとの根本的な違いであり、「命令（コード）とデータの分離」という前提が崩れる。データの中に命令が混在し得るため、構造的にサニタイズ・バリデーションだけでは防御できない。

具体的な攻撃パターンは3種類。①間接的プロンプトインジェクション：Google DriveのドキュメントやGitHubのREADMEなど「AIが読むファイル」に悪意ある命令を仕込み、Slack MCPなど別のツールを経由してデータ送信させる。②ツールポイズニング：MCPサーバーのツール説明文（メタデータ）自体に悪意あるプロンプトを埋め込む。AIは応答計画時にすべてのツール説明文を処理するため、一度も呼び出さないツールでも攻撃が成立する。Invariant Labsが2025年4月に実証済み。さらに接続時は無害として承認を得た後でロジックを差し替える「rug-pull」手口も存在する。③身代わり攻撃（Confused Deputy Attack）：AIがユーザーから付与された権限で動くため、悪意あるissueコメント等に誘導されてリポジトリの別ファイルを書き換えるなどの操作が発生し得る。

承認制（Human-in-the-Loop）は必要条件だが十分条件ではない。MCP仕様書は「SHOULD（推奨）」にとどまり「MUST（必須）」ではなく、自動承認するクライアントも存在する。Claude Desktopでは「常に許可する」がデフォルト位置に表示され、2025年5月のアップデート以降取り消し不可の問題も指摘されている。これは承認疲れ（Approval Fatigue）を加速させる設計だ。

実務上の対策は4点：①素性不明のMCPサーバーをインストールしない（2025年9月にpostmark-mcpという悪意あるサーバーが約300組織の本番環境に混入した事例あり）、②読み取り系と外部送信系のMCPを同時に有効にしない、③権限スコープを最小限に絞る（Google Driveはdrivでなくdrivelfileスコープ等）、④AIがアクセスするデータに機密情報が含まれないか事前確認する。監査エージェント開発においても、複数MCPを組み合わせた自律エージェント構成ではプロンプトインジェクションの影響範囲と速度が増大するため、HITL設計とスコープ制限が特に重要。

## アイデア

- ツールポイズニングは「ツールを一度も呼び出さなくても成立する」点が盲点——AIが応答計画時に全ツール説明文を処理するという仕様上の特性が攻撃面になっている
- rug-pull（承認後の差し替え）は「一度承認したから安全」という静的な信頼モデルの限界を突いており、動的な信頼検証の仕組みが必要であることを示唆する
- 読み取り系MCPと外部送信系MCPの同時有効化がインジェクション成立の必要条件という整理は、最小権限原則をMCPアーキテクチャに適用する際の具体的な設計指針として使える
