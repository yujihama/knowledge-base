## 要約

Web版ChatGPTのSkill機能を使って、複数サイトを比較した固定フォーマットのExcel評価表を出力する仕組みを構築する過程で直面した問題と解決策をまとめた実践報告。

目標としたExcelは、「基本情報（No・カテゴリ・評価項目）」＋「評価対象ごとに評価・理由・参照の3列セット」＋「横断比較列」という1シート構成の評価表。しかし実機検証では、ChatGPTがモデル内部の「Excelらしい分析フォーマット」パターンに引っ張られ、1シート指定なのに複数シートが生成される、参照列が末尾にまとめられる、評価項目が独自に再生成される、評価記号が数値スコアに置き換わるといった問題が繰り返し発生した。

根本原因は、指示ファイルの制約よりもモデルの既存パターンの方が強力だったこと。「フォーマットを守る」という抽象的な指示では不十分で、モデルが過去に学習した「分析Excelの典型形」を上書きできなかった。

有効だった設計は5点。①役割分担の明確化：LLMには調査・採点・理由づけのみ担当させ、シート数・列順・列幅・罫線・色といったExcelの構造と見た目は固定処理で制御する。②禁則の明文化：「フォーマットを守れ」ではなく「作業用シートを追加しない」「参照列を末尾にまとめない」「独自の評価項目を作らない」のように、実際に発生した失敗パターンを具体的な禁則として記述する。③読了証跡の設置：外部資料ファイルに短い識別子を埋め込み、採点前にモデルに識別子を出力させることで「読まずに進む」ケースを検出可能にする。④参照列の配置：列数が増えてもファクトチェックしやすさを優先し、参照列は各評価対象の理由列の右に配置する。⑤生成後の機械検証：シート数が1つか、ヘッダーが指定通りか、最終列が横断比較か、集計行があるか、各対象に参照列があるかを、Skill実行手順内で自動検証する。

監査エージェント開発への示唆として、LLMを使った構造化出力全般に適用できる原則が含まれている。出力フォーマットの固定が重要な業務用途では、モデルの自由度を「調査・推論・言語化」に限定し、出力スキーマの強制と事後検証を機械的な処理として分離することが安定性の鍵となる。これはPydanticによる出力バリデーションや、LangGraphのノード設計でのツール実行と構造化出力の分離と同じ思想であり、監査エージェントにおける証跡収集・評価・レポート生成の役割分担設計に直接応用できる。

## アイデア

- 「正解を書くだけでなく、過去に起きた失敗パターンを禁則として明示する」という指示設計は、LLMの既存パターンを上書きする実践的なテクニックであり、few-shot negativeの考え方に近い
- 外部資料に識別子を埋め込んでモデルに読了を宣言させる「読了証跡」の仕組みは、ハルシネーション検出の軽量なチェックポイントとして監査エージェントのツール実行ログ設計にも転用できる
- LLMの担当範囲を「推論・言語化」に絞り、スキーマ強制と事後検証を固定処理で行う役割分担は、Pydantic + LangGraphによる監査レポート自動生成の設計パターンと同型であり、出力品質の安定化に普遍的に有効
