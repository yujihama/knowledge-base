## 要約

製造業向けACL-aware RAGシステムを、Claude（claude-haiku-4-5）・OpenAI（gpt-4o-mini）・Cohere（command-r-plus-08-2024）の3プロバイダーで同一条件で実装・実測した比較実験レポート。Embeddingはそれぞれ Cohere Embed v3 / text-embedding-3-small / embed-multilingual-v3.0、Vector StoreはClaude版・Cohere版がローカルChromaDB、OpenAI版のみクラウド（File Search API）を使用。

レイテンシ実測では、ACL通過ケースでClaudeが3.02秒、OpenAIが5.81秒、Cohereが2.66秒（レート制限時は12.80秒）。OpenAIの遅延はVector Storeのクラウドホスト起因のネットワークレイテンシによるもので「データ主権 vs レイテンシ」のトレードオフが明示された。コストはGPT-4o-miniが$0.000379/クエリで最安、Claudeが$0.001544、Cohereが$0.001275。OpenAIはinput tokenが1,576と最多だがモデル単価の低さで逆転する。

最も重要な知見は「セキュリティ境界をLLMの手前に置く設計」の有効性。ACLフィルタ・Fail Closed判定をすべてPythonのアプリケーション層（LLM呼び出し前）で処理することで、3プロバイダー全9テストケース（tanaka:ACL通過 / yamada:ACL拒否 / unknown:Fail Closed）で同一のセキュリティ動作を確認。Fail Closedケースは全プロバイダーで0.00秒（LLMへのリクエストが発生しないためコストもゼロ）。LLM側にACL判断を委ねる設計はPrompt Injectionで無効化されるリスクがあるため、Python層でフィルタリングしてからLLMに渡す構造が正解とされる。

回答品質では、temperature=0を指定しても出力フォーマットはモデル固有のスタイルに依存し、Claudeは見出し付き文章形式、OpenAIは番号付きリスト形式（保管期間3年まで言及し情報量最多）、Cohereは簡潔な箇条書き（保護具・点検頻度が欠落）となった。エンタープライズ用途では system prompt で出力フォーマットを明示的に指定する必要がある。

マルチプロジェクト統合の実装上の注意点として、sys.modulesキャッシュ衝突（同名モジュール・パッケージの競合）とChromaDBの相対パス問題が発生。解決策はimportlib.util.spec_from_file_location()によるパス直指定とキャッシュクリア、および__file__基準の絶対パス指定。

監査エージェント開発への示唆：ACLやアクセス制御ロジックをLLMの外側（アプリケーション層）に実装する設計原則は、監査システムにおける証跡の改ざん不能性・アクセス制御の確実性を担保する上でそのまま適用可能。LLMプロバイダーの切り替えやマルチプロバイダー構成においてもセキュリティポリシーが破綻しない設計の具体例として参照価値が高い。

## アイデア

- セキュリティ境界をLLMの手前（アプリケーション層）に置くことで、プロバイダーを変えてもACL・Fail Closedロジックが壊れない『ベンダーロックインしないセキュリティ設計』が実現できる点
- Fail Closedを最初のステップで処理することでレイテンシ0秒・コスト0円を達成できる——セキュリティ強化とコスト削減が同時に成立する設計パターン
- temperature=0はフォーマット統一ではなく内容確定性のための設定であり、出力スタイルのモデル依存性は system prompt で明示的に制御しなければエンタープライズ用途では問題になるという実測による確認
