## 要約

2026年2月以降、Claude Codeの出力品質が低下したという報告がReddit・X・Hacker Newsで相次いだ。IREE/MLIRコンパイラ開発者のstellaraccidentは自身の6,852セッションログを定量分析し、17,871のthinkingブロックと234,760のツールコールから品質低下を数値で裏付けた（GitHub Issue: anthropics/claude-code#42796）。

検出された変化は3点。第一にthinking深度の低下：1月末のベースライン（中央値〜2,200文字）に対し、2月末は〜720文字（-67%）、3月以降は〜600文字（-73%）まで低下した。第二にRead:Edit比率の悪化：以前はファイルを読んでから編集する「Research-First」スタイルだったが、比率は6.6→2.0に低下し、「読まずに編集」する割合が6.2%→33.7%、全ファイル書き換え（Write）率が4.9%→10.0%に増加した。第三に行動指標の悪化：自作のstop hook（怠惰フレーズ検知スクリプト）の発火が0→173回（17日間で約10回/日）、ユーザーEscape（中断）率が0.9→5.9（+556%）に跳ね上がった。

AnthropicのBoris Cherny（bcherny）がIssueに直接回答し、同時期の2つの変更を説明した。①2月9日から導入したAdaptive Thinking（Opus 4.6で固定thinking budgetからモデル自律決定方式に切り替え）が特定ターンでthinkingトークンをゼロにし、その箇所でStripe APIバージョンやGit SHAのハルシネーションが発生していた。②3月3日にeffortパラメータのデフォルトをhigh相当からmedium（85）に変更し、初回告知ダイアログを見逃したユーザーが気づかず使い続けた。なお、3月12日に100%展開されたthinking redaction（UI上でthinking内容を非表示化）自体はthinkingに影響しないとbchernyは述べているが、これにより外部検証が困難になった点には批判が残る。

対処法として、effortを`/effort high`または`settings.json`で`high`に固定すること、`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1`でadaptive thinkingを無効化すること、CLAUDE.mdに「編集前に必ずReadで対象ファイルを確認する」ルールを明記することが有効。また、benvanikが公開したstop-phrase-guard.shを参考に怠惰フレーズ検知hookを実装することで品質低下の先行指標として活用できる。

## アイデア

- thinking signatureフィールドとthinking内容長の間にPearson r=0.971という強相関があり、thinkingが非表示化（redact）された後もsignatureからthinking深度を間接推定できる点は、エージェント品質モニタリングの代替指標設計として応用可能
- Read:Edit比率やResearch:Mutation比率という「ツール使われ方の比率指標」は、LLMの出力テキスト品質ではなく行動パターンから劣化を検出する手法として、エージェントシステムの継続的品質評価フレームワークに組み込める
- 怠惰フレーズ（'should I continue?', 'good stopping point'等）を正規表現でリアルタイム検知してセッションを阻止するstop hookアーキテクチャは、エージェントの「自発的放棄」挙動を抑制するガードレール設計として汎用的に適用できる
