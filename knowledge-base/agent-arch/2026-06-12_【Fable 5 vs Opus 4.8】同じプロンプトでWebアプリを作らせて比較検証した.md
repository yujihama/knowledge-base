## 要約

Fable 5（Mythos クラス初の一般公開モデル）のトークン消費が Opus 4.8 の2倍であることを背景に、「Fable を指示役に徹させ、実装を下位モデルに委譲すればトークンを節約できる」という仮説を検証した実験記事。検証題材は「ブラウザで動くスーパーマリオブラザーズ風ゲームをゼロから作る」という共通プロンプト。体制 A は Fable 5 が直接実装、体制 B は Fable 5 が指示役（タスク分解・brief作成・レビュー・統合）に徹し、実装は headless の Cursor CLI で動かす Composer 2.5 Fast（並列2プロセス）が担当する設計。体制 B の手順は6ステップの自作 skill「cursor-impl」として固定化し、再現性を確保した。結果は仮説と逆の方向に出た。体制 A はトークン約5万、所要時間6分18秒で 1,142行の index.html を一発生成。体制 B は Fable のトークン消費が約14万（3倍）、所要時間約25分となり、Composer 側でも入力11万・出力3万トークンを消費した。品質面でも体制 A が圧倒的に優れており、体制 B では当たり判定の不具合やスプライト描画の不完全さが残った。失敗の原因として著者は3点を挙げる。①brief を介すことで「ジャンプの重力感」など言語化できない実装知識が脱落する。②レビューは「壊れていないか」しか保証できず、高品質への収束は保証できない。③手触りの修正を差し戻しループで行うと①のロスが乗算される。一方で、delegation が有効になりうる条件として「参照実装がある反復作業」「typecheck/lint/テストで受け入れ基準が機械検証できる仕事」「判断は軽いが出力量が膨大なcodemodやi18n文字列外出し」を挙げるが、いずれも「見極めコスト」と「見誤った際の品質低下+3倍トークン損失」を考慮すると、現時点では一旦すべて Fable に書かせる方がトータルのコスパが良いという結論を示す。監査エージェント開発への示唆として、LLM同士の委譲パターンはコンテキスト転送コストと情報損失が大きく、タスク境界の設計が不適切だと節約どころか逆効果になる点は、マルチエージェント構成を検討する際の重要な設計原則となる。

## アイデア

- 「強いモデルを指示役・弱いモデルを実装役」という分業パターンは、brief経由の情報損失とレビューのコンテキストコストにより、単一強モデルより3倍トークンを消費した
- 言語化できない暗黙知（ゲームの手触り・物理挙動）はどれだけ詳細なbrief（100行超）を書いても伝達できず、直接実装との品質差として顕在化する
- 委譲が有効なのは「判断量が少なく出力量が多い」タスク限定であり、その見極めコスト自体がオーバーヘッドになるため、現段階では全タスクを強モデルに直接投げる方が合理的
