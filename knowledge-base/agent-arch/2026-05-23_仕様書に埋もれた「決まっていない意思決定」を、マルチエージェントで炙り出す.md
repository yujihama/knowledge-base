## 要約

仕様書レビューを通過した後も実装中に仕様確認が止まらない問題（空状態の表示未定義、権限による振る舞いの曖昧さ、二重送信防止の漏れ等）を解決するため、Microsoft Agent Framework 1.0 と Azure OpenAI を用いたマルチエージェントツール「Specify」を開発した。

Specify は5つのエージェントで構成される。Planner Agent が仕様書を読んで後続エージェントの観点を整理し、Decision Agent（UI/UXの意思決定漏れ抽出）・Edge Case Agent（異常系・連続クリック等の抽出）・Past PRD Agent（過去仕様書との矛盾検出）が ConcurrentBuilder により並列実行される。最後に Reviewer Agent が重複排除・優先度調整・論点補完を行う。

単一 LLM では観点が偏る・過去の意思決定を知らない・抽出と優先度判断が混在するという3つの限界があったため、役割分担のマルチエージェント構成を選択した。特に重要な実装上の工夫は Past PRD Agent の出力を Reviewer Agent へ合流させる設計で、「Decision Agent が該当 decision を出力しなかった場合でも、Past PRD Agent が検出した矛盾を新規 decision として補完する」ロジックを Reviewer Agent に持たせた。

出力スキーマには title・priority（must/should/nice）・category・options・rationale を含み、ADR（Architecture Decision Record）の下書きとして転用しやすい構造にしている。また、LLM の出力から「過去仕様書由来の論点」を判別するロジックは当初 rationale の末尾マーカー文字列で実装したが、プロンプト変更で silently 壊れるリスクがあるため、スキーマに source: "past_prd" | "agent" という enum フィールドを追加し、バックエンドの post-processing でマーカー文字列による補完を二段構えで行う設計に変更した。

デモ用 PRD での検証では意思決定論点10件・過去仕様書との矛盾2件を抽出し、うち7〜8件が仕様書に明記されていなかった論点だった。プロンプト設計では「役割ベース（専門家として〜）」より「機能ベース（〜を抽出する）＋出力スキーマ＋判断基準」の形式の方が出力が安定することを確認した。監査エージェント開発への示唆として、仕様書の「書かれていないこと」を複数エージェントで並列に検出するアーキテクチャは、監査要件や内部統制の抜け漏れ検出にも同様のパターンが適用できる。

## アイデア

- Past PRD Agent が検出した矛盾を、対応する decision が存在しない場合に Reviewer Agent が新規 decision として生成する設計——過去の組織知を現在の仕様レビューに自動注入する仕組み
- LLM 出力の判別ロジックを自然文マーカーではなくスキーマの enum フィールドで受け取る設計——プロンプト変更による silent break を防ぐ堅牢化パターン
- 役割ベースプロンプト（専門家として〜）より機能ベースプロンプト（機能・出力形式・判断基準を明示）の方が出力が安定するという実験的知見
