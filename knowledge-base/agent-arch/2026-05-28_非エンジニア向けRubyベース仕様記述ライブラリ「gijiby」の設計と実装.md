## 要約

TOKIUMのVPoEが開発した「gijiby」は、RubyとRubyに近い擬似コードを混在させた仕様記述を実行可能なRubyコードへ変換するライブラリ。背景にある問題意識は「AIへの委譲範囲拡大」で、その前提条件としてLLMの出力一貫性（Predictability）を高める必要があるという仮説に基づく。自然言語の曖昧な仕様をAIに渡すと、同一ロジックでも関数定義の有無・ハッシュアクセスの形式などで実装スタイルがばらつく問題を実証している。

gijibyのパイプラインは（1）擬似コードの前処理・決定的置換、（2）Prismによる構文チェック、（3）サンドボックス内実行による実行時エラーチェック、（4）LLMによるエラー行単位の修正、を繰り返す構成。構文チェックにはRubyのPrism gemを利用し、実行時サンドボックスはforkした子プロセス上でKernel#system/exec/spawn等を`remove_method`で無効化し、SocketやFileUtilsなども除去したうえでevalを実行、Marshal/パイプ経由で親プロセスへ結果を返す実装。

LLMへの修正依頼はコード全体でなくエラー発生行のみを対象とし、コード全体は文脈参照用として渡す。これにより無関係な行の書き換えを防ぐ。また、Prismが報告するエラーには根本原因ではなく「余波」のエラー（unexpected 'end'等）が含まれるため、Analyzerクラスでノイズを除去してからLLMに渡す設計。

評価実験では「もし x が 5 より大きい場合 / x を出力せよ / end」という擬似コードを含む仕様をgijiby経由でコンパイルし、5回の試行で安定したRubyコード（if x > 5 / puts x / end）が出力されることを確認。同一仕様をそのままClaude Code Opus 4.6に2回投入した場合には関数定義の有無・アクセス形式の違いが生じた。定量的な一貫性評価指標としてAST（Abstract Syntax Tree）の一致率を用い、gijiby経由では高い一致率を達成した旨が示されている。

監査エージェント開発への示唆：仕様記述の厳密性と柔軟性のバランスを言語レベルで制御するアプローチは、非エンジニアが関与する監査・内部統制領域のAIエージェント開発でも応用可能。曖昧な業務ルールをLLMに直接渡す前に前処理・構造化するレイヤーを挟むことで、エージェントの行動予測可能性を高められる。

## アイデア

- AIへの委譲範囲拡大の前提条件として「Predictability（予測可能性）」を定義し、出力一貫性をASTレベルで定量評価する枠組みを構築した点
- LLM修正をコード全体でなくエラー行単位に限定し、かつPrismのノイズエラーをAnalyzerで除去してから渡すことで修正精度を高める設計
- Rubyのfork+remove_methodによるサンドボックスで任意コード実行リスクを排除しつつ実行時エラー検出を実現した実装アプローチ
