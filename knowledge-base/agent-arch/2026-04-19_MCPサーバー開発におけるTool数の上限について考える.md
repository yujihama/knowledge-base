## 要約

本記事は、MCPサーバー開発者が直面するTool数設計の問題を実践的に整理したものである。筆者は「駅すぱあとAPI MCPサーバー」の開発経験をもとに、「APIパス単位でToolを1つ作る」vs「高レベルなToolをまとめて作る」という設計上のトレードオフを検討している。

MCPプロトコル仕様自体にTool数の上限は定められていないが、実際のAIクライアント側には制限がある。Cursorは従来40個だった上限を約80個に引き上げ、VS Code（GitHub Copilot Chat）では同時に有効化できるToolが最大128個という制限がある。GitHubの公式MCPサーバーはTool数が100個超であり、これらの制限と衝突する典型例として挙げられている。

Tool数が増加することで引き起こされる最大の問題は「コンテキスト汚染（Context Pollution）」である。Scott Spence氏の事例では、12のMCPサーバー（mcp-omnisearch 20 tools、playwright 21 tools、mcp-sqlite-tools-testing 20 toolsなど）を有効化した結果、Tool定義だけで66kトークン以上を消費した。Claude Sonnet 4.5のコンテキストウィンドウ200kのうち実に1/3がTool定義に費やされ、実際のタスク処理に使えるコンテキストが大幅に圧迫された。

Dockerが公開したMCPサーバーベストプラクティスでは、「すべてのAPIエンドポイントを個別のToolにマッピングしない」ことを明示的に推奨している。Tool数が多すぎるとMCPサーバーの複雑化・使用コストの上昇・ユーザー離れを招くと指摘している。

対策として開発者側には、(1) Toolのdescriptionを簡潔に書くこと、(2) 類似Toolをパラメータで分岐させ統合すること（例: google_search/bing_search/yahoo_searchをweb_search({provider})に統合）が提案される。利用者側には必要なMCPサーバー・Toolのみ有効化し、/clearでコンテキスト履歴を定期的にリセットすることが有効。

新たなアプローチとして、Anthropicが2026年1月頃からClaude CodeでTool search toolを提供開始しており、必要なToolをオンデマンドで読み込む仕組みにより全Tool定義を一括ロードせずに済む。ただし全クライアントが対応しているわけではなく、設計品質の問題として根本的なTool数削減が優先される。筆者の結論として「1MCPサーバーあたりのTool数は10個程度、最大20個以内」「descriptionは簡潔に」という具体的な指針が示されている。監査エージェント開発においても、LangGraphベースのエージェントがMCPサーバーと連携する際にはTool数の設計が応答品質やコスト効率に直結するため、この設計原則は直接適用可能である。

## アイデア

- Tool定義だけで66kトークン（コンテキスト全体の1/3）を消費するという具体的な数値は、MCPサーバー設計がLLMのパフォーマンスに与える影響を定量的に示しており、Tool数削減がコスト・品質の両面で重要な最適化ポイントであることを示している
- 類似Toolをパラメータ分岐で統合するアプローチ（google_search/bing_search → web_search({provider})）は、APIをどの粒度で抽象化するかというインターフェース設計の古典的問題をMCPの文脈で再解釈したものであり、RESTful API設計のベストプラクティスと同様の原則が適用できる
- Tool search toolによるオンデマンドTool読み込みは、全Tool定義の事前ロードを避けるRAG的アプローチをツールスキーマ管理に適用したものであり、将来的には動的なTool選択・組み合わせによるエージェントの適応的行動が可能になる方向性を示している
