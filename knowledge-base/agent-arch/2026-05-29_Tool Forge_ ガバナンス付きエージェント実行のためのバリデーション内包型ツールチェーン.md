## 要約

Tool Forgeは、自然言語で記述されたケーパビリティ意図を、ガバナンス制御・サンドボックス検証・カタログ管理されたツールアーティファクトへ変換し、エージェントに対してトークン効率の高いルーティング層を通じて公開するツールチェーンシステムである。

従来のLLMエージェントにおけるツール層は、手書きの統合コードや静的スキーマのリストとして扱われることが多く、ガバナンスや検証の仕組みが欠如していた。Tool Forgeはこの問題に対し、ツールを「カプセル」として定義する。カプセルには、意図・ケーパビリティ契約・実装・依存ポリシー・テスト・ドキュメント・ランタイムバリデーションエビデンス・ライフサイクル状態・クレデンシャルバインディング・ルーティングメタデータが含まれる。

システムの中核となるRouterは、モデルコンテキストにフルカタログスキーマをロードする代わりに、意図スコープ限定のツールセッションを提供する。これにより、ナイーブなフルカタログ公開と比較してタスクフローのツールコンテキストを99.2%削減することに成功した。

ベンチマーク結果として、83件のRouterベンチマークケースで集約マイクロF1スコア0.901を達成。25件のエンドツーエンド生成プローブでは、25/25のツールバンドル生成に成功し、決定論的受け入れチェックに対してマイクロF1が0.940、ライブサンドボックスバリデーションでは25件中23件がパスした。

アーキテクチャ面では、MCP（Model Context Protocol）向けのルーティングモデルを採用しており、ツールの動的発見・スコーピング・提供がMCPプロトコルを通じて行われる。ガバナンス制御として、サンドボックス分離・依存ポリシー・ライフサイクル管理・クレデンシャル管理が含まれる。

監査エージェント開発への示唆として、このシステムが実現する「ツールのガバナンス化」と「コンテキスト効率化」の両立は、エンタープライズ環境でのエージェント展開において極めて重要なアーキテクチャパターンである。特にサンドボックス検証とライフサイクル管理の仕組みは、内部監査エージェントが外部システムやAPIを呼び出す際のリスク管理フレームワークとして直接応用可能。LangGraphベースの監査エージェントにTool Forgeのルーティング概念を組み込むことで、ツール数増加に伴うコンテキスト肥大化問題を抑制しつつ、ツール実行のトレーサビリティを確保できる可能性がある。

残課題として、敵対的ルーティング・広範なAPIグラウンディング・サンドボックス分離・クロスシステム評価が挙げられており、本論文はSoTA主張ではなく初期システムベンチマークとして位置づけられている。

## アイデア

- ツールをコード+スキーマではなく「意図・契約・バリデーションエビデンス・ライフサイクル状態を含むカプセル」として定義することで、ガバナンスとトレーサビリティをツール層に組み込む発想
- フルカタログ公開の代わりに意図スコープ限定のツールセッションを提供するRouterにより、コンテキスト使用量を99.2%削減しながらF1=0.901を維持するトークン効率と精度のトレードオフ解決策
- サンドボックス検証をツール生成パイプラインに統合し、25件中23件がライブ環境での実行検証をパスという形で「実際に動くツール」の自動生成を定量評価する手法
