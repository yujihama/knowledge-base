## 要約

Claude Codeのマルチエージェント機能を使い、個人向けキャリア戦略レポートを自動生成したケーススタディ。単一エージェントに全分析を任せると論点が混在し検証が甘くなるという問題を解決するため、4つの調査エージェント（市場動向・職種とスキル・事例と働き方・ロードマップ）を並列実行し、その後に統合と検証を行う設計を採用した。

各調査エージェントには担当範囲を意図的に狭く設定し（例：「市場動向だけを分析する」「断定しすぎず不確実性も書く」）、出力を「重要な市場変化・狙うべき職種候補・リスク・次の調査に渡す前提」の4項目に限定した。進捗管理はprogress.mdによるシンプルなチェックリスト形式で実施。

最大のポイントは「検証エージェント」の導入である。調査エージェントは本質的に「推進派」になりやすく、初回ドラフトには「活動が順調に進む」「学習が継続する」「新しい機会にすぐ出会える」という楽観前提が重層的に積み上がっていた。検証エージェントには①現実性・②網羅性・③論理整合性・④課題抽出・⑤シナリオ見直しの5軸でレビューさせ、楽観/現実/保守の3シナリオに分解させた。

補正の根拠として明示されたのは「活動時間には上限がある」「意思決定は最短で進まない」「スキル習得には遅延が起きる」「市場評価は本人の努力だけで決まらない」の4点。結果として初回ドラフトの楽観寄りの見立ては大幅に現実的な方向へ補正された。

技術的な躓きとしてWeb検索権限が取れず全エージェントがフォールバックを必要とした事例も報告。この教訓から「Web検索不可の場合は最新情報を断定せずレポートに要確認箇所を明記する」というフォールバック方針をプロンプトに組み込む設計の重要性が示された。

監査エージェント開発への示唆：検証エージェントを「推進役と批判役の役割分離」として設計するパターンは、監査プロセスにおける「作成者と検証者の分離（Maker-Checker）」原則と構造的に一致する。内部監査AIにおいても、リスク評価エージェントが楽観バイアスを持ちやすい点を踏まえ、独立した批判的レビューエージェントを別途配置する設計が有効と考えられる。LangGraphでの実装時は、Agentノードとは別にCriticノードを後段に配置し、前提の強度評価と3シナリオ分解を専担させる構成が参考になる。

## アイデア

- 「推進エージェント」と「批判エージェント」を役割として分離することで、単一エージェントでは除去しにくい楽観バイアスを構造的に補正できるという設計原則
- 3シナリオ（楽観/現実/保守）への分解を検証エージェントに専担させることで、「目標として見る部分」と「意思決定に使う部分」を文書レベルで分離できるという出力設計
- Web検索権限エラーをフォールバックとして処理し「最新データ要確認」を明示的にレポートに残す設計が、AI出力の信頼性担保として機能するという実装上の知見
