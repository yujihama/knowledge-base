## 要約

本書は、AgentシステムをLLMに「溶かさず」、責務境界を明確に保ちながら設計するための実践的なアーキテクチャ論を全7章＋Appendixで展開する。著者mofuteqはLLM/RAG/Agent Runtimeの実装経験をもとに、「Agentが今どこで何をしているか見える」設計思想を軸に据える。

第0章では、DockerをAgent開発から切り離すことで環境依存を減らし、開発サイクルを軽量化するアプローチを示す。第1章では、Vector DBを外すとRAGではなくAgent Runtime（状態管理・実行制御の核）が本質として残ると論じる。第2章では、Agent Runtimeを実装する過程でState Machineの概念と再会し、状態遷移の明示的管理がAgent制御の鍵であることを示す。第3章では、State Machineはブラウザを閉じても復帰できる永続化（Durability）を備えるべきだと主張し、セッション跨ぎの状態保存を設計要件として位置付ける。

第4章は特に実務的で、Agent Frameworkへの過度な依存をやめることで、バグや挙動の問題が「どこで起きているか」を特定できるようになると説く。LangGraph等のフレームワークを使いつつも、ブラックボックス化しない設計姿勢を促す。第5章では、MCPでAgentの機能を際限なく増やすのをやめ、Context Boundary（文脈の境界）を意識することが重要だと論じる。ツール増加による文脈汚染を防ぎ、Agentが扱う情報の責務範囲を絞ることがObservabilityと信頼性向上につながる。

第6章では「Agentは構造で考え、構造を見せずに返す」という設計原則を示す。内部的には状態遷移・ステップ管理を構造化しつつ、ユーザーへの出力は自然言語で返すUI/UXとの分離設計を扱う。第7章「No disclaimer by design」では、Agentが無用な免責文句を出さないよう、プロンプト設計やシステム設計レベルで対処する手法を論じる。Appendixでは契約リスク検知Agentの具体的設計例を示し、内部監査・法務領域への応用可能性を提示する。

監査エージェント開発への示唆として、State Machine＋永続化の設計は監査ワークフローの中断・再開要件と直結する。Context Boundaryの概念は、監査エビデンス収集時のスコープ管理に応用でき、LLMへの過剰な責務委譲を避けることで監査トレースの説明可能性を高められる。

## アイデア

- Vector DBを外すとAgent Runtime（状態管理・実行制御）が本質として残るという逆説的なアーキテクチャ整理は、RAGをAgent設計の文脈で再定義する視点として鋭い
- State Machineにセッション跨ぎの永続化（Durability）を要件として課す設計は、監査ワークフローの中断・再開シナリオと直接対応し実務応用が高い
- MCPでツールを増やすのをやめてContext Boundaryを守るという方針は、ツール爆発によるプロンプト汚染・Observability低下を防ぐ実践的原則
