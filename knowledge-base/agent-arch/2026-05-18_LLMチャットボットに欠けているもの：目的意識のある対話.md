## 要約

現行のLLMチャットボットはMMCU・HumanEval・MATHなどのベンチマークで性能向上を続けているが、一問一答形式の評価ではユーザー体験の実態を捉えられていないという問題提起。著者は「目的意識のある対話（Purposeful Dialogue）」として、旅行計画や心理カウンセリングのような明確なゴールを持つマルチターン会話を定義し、交渉理論の反復交渉モデル（iterative bargaining）との類比でその有効性を論じる。Winograd の発話行為論を援用し、各発話を相手の世界モデルを変化させる意図的行動として位置付ける。

システム構成面では、事前学習（次トークン予測）→チャットテンプレート導入（`tokenizer.apply_chat_template`、`<system>`/`<INST>` タグ）→RLHF（KLペナルティ＋LoRA）という三段階を解説。RLHF はLeCunのケーキ比喩で言う「チェリー」に過ぎず、対話フォーマットが初めて学習データに登場するのがこの段階であることを指摘する。

一貫性評価の実験として、著者らはシステムプロンプトで役割付けされた2つのLMエージェントを最大8ラウンド対話させ、任意ターン後に「話題逸脱」「役割忘却」を検出するフレームワークを構築。評価結果では、対話が長くなるほどシステムプロンプトの遵守率が低下し、GPT-4oやClaude 3.5 Sonnetでも数ラウンド後に役割を逸脱する事例が確認された。

改善アプローチとして、(a) 文脈内での自己リマインド（system promptの繰り返し注入）、(b) 長期記憶を持つパーソナライゼーション（IVA・Siriのような日次インタラクションによる嗜好学習）、(c) ユーザー意図推定と明示的なゴール追跡、の三方向が挙げられる。SWE-benchを例に、コード生成においても単一パス生成ではなくエンジニアとの反復対話が現実の GitHub Issue 解決には必要と論じる。

監査エージェント開発への示唆として、LangGraphで実装するマルチターン監査エージェントでは「役割の一貫性維持」が課題となる。system prompt の定期的な再注入や、各ノードでの目的関数の明示的チェックを設計段階で組み込む必要がある。また、監査手続きはステップが多く長期対話に相当するため、ゴール追跡モジュール（ReAct のThought ループ相当）の精度が成果品質に直結する。

## アイデア

- 2つのLMエージェントを無制限ターン対話させてシステムプロンプト遵守率を定量評価するフレームワーク：任意ターンで「話題逸脱テスト発話」を注入し、相手エージェントの応答から役割忘却を自動検出する設計が再現可能
- 交渉理論の反復交渉モデル（iterative bargaining）を人間-AI対話に適用する視点：一回限りのtake-it-or-leave-it オファーより反復交渉が優れるという経済学知見を、LLMのマルチターン設計の根拠として使える
- RLHF は対話フォーマットが初めて学習データに登場する段階であり、事前学習コーパスとフォーマットの乖離が一貫性低下の根本原因である可能性：この視点はinstruction-tuning データの設計方針に直接影響する
