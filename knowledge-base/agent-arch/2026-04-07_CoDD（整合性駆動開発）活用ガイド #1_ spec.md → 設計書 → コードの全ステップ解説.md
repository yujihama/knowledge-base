## 要約

CoDD（Coherence-Driven Development）は、Markdownで書いた要件定義（spec.md）からAIを用いて設計書群・実装コードを段階的に自動生成するCLIツール（Python製、v1.2.0）。内部ではClaude Code CLIを`claude --print`によるSTDIO経由で呼び出し、プロンプト注入を行う。

中核概念は「wave」と「依存チェーン」。codd plan --initがspec.mdを解析し、設計書の生成順序と依存関係を自動設計する。たとえばSystem DesignはAcceptance CriteriaとADRに`constrained_by`として依存させ、上流の判断が下流に伝播する構造（これが「整合性駆動」の実体）。各waveは前のwaveの出力を入力として使うため、wave 1→2→3→4の順に逐次実行する必要がある。

実際の生成結果として、37行のspec.mdから設計書6本（合計1,353行）・コード102ファイル（6,445行）・ビルド成功（Next.js 15 static export、TypeScript strict）が得られた。設計書の内訳はAcceptance Criteria（154行）、ADR（202行）、System Design（248行）、Component & State Design（283行）、Task Lifecycle Flows（311行）、Implementation Plan（155行）。

コード生成フェーズはcodd implement --sprint 1〜Mで行い、sprint数はcodd plan --sprintsから動的取得してシェルスクリプトのforループで回す。codd validateは各waveの成果物に対してTODO/TBD検出・メタコメンタリー除去・Mermaidダイアグラム存在確認・依存整合性チェックを実施し、品質ゲートとして機能する。

重要な知見として「specの質＝最終成果物の質」が挙げられている。UI要件を書かなかったv1では見た目が地味だったが、ダークテーマ・FAB・トースト通知・キーボードショートカット等を明示したv2では設計書・コードともに正確に反映された。AIは書かれていないことは生成しない、という原則の実証例。

## アイデア

- 依存グラフ（constrained_by）で設計書間の整合性を強制する構造は、LangGraphのノード依存チェーンと同型であり、同じ原理をエージェントのタスク分解・実行計画に適用できる
- codd validateのような品質ゲート（TODO検出・構造チェック）をエージェントパイプラインの各ステップ出力に組み込む「LLM-as-judge中間チェック」パターンとして一般化できる
- claude --printによるSTDIO経由のバッチ呼び出しは、UIなしでLLMをサブプロセスとして組み込む最小構成であり、シェルスクリプト・Pythonスクリプトからエージェントを駆動する際の実装パターンとして参考になる
