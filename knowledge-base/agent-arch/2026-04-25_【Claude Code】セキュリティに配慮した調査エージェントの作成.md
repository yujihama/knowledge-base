## 要約

Claude Codeのサブエージェント機能を用いて、「月曜朝に自動でレポートが揃っている」状態を目指した調査エージェント群の構築事例。1エージェント1責務の原則に基づき、技術トレンド調査・ビジネス動向調査・コーディング・文書作成の4種のサブエージェントを設計。各エージェントは.claude/agents/配下のMarkdownファイルで役割・行動規範・セキュリティポリシーを定義し、成果物を決まったディレクトリ（reports/、drafts/）に出力。親エージェントがそれらを集約するパイプライン構造を採用した。

最大の課題はWebSearchのセキュリティリスク。エージェントが検索クエリを自律生成するため、社内の未公表プロジェクト名がそのまま外部検索エンジンに送信される恐れがある。プロンプトへの指示のみでは「ベストエフォート」に過ぎず、コンテキストが長くなると指示が薄れることが体感として確認された。

そこでClaude CodeのHooks機能、特にPreToolUseフックを活用した禁止ワードフィルターを実装。.claude/settings.jsonでWebSearchツール呼び出し時にbashスクリプトを自動起動するよう設定。スクリプトは標準入力からツールのJSON入力を受け取り、Pythonでクエリ文字列を抽出。禁止ワードリストと照合し、一致した場合はpermissionDecision: "deny"とpermissionDecisionReasonを含むJSONを返してツール実行をブロックする。ブロック時のメッセージはエージェントのコンテキストに渡され、エージェントが自律的にリカバリできる点も確認された。

この経験から「3層防御」の考え方を導出。第1層はプロンプト（エージェント定義ファイル）による言語的指示、第2層はPreToolUseスクリプトによる機械的差し止め、第3層は人間によるレビュー。第1層だけでは穴があり、第2層だけでは過剰ブロックのリスクがある。両者を組み合わせることでエージェントの自律性と安全性のバランスを実現。

監査エージェント開発への示唆として、本手法はLangGraph等で構築するエージェントシステムにも応用可能。外部API呼び出しやDB書き込みなどのツール実行前に機械的な検証レイヤーを挟む設計パターンは、内部監査領域で扱う機密情報の漏洩防止に直接転用できる。ただし禁止ワードリストの継続的な更新運用が必要であり、「セキュリティはシステムではなくプロセス」という認識が重要。

## アイデア

- PreToolUseフックによるツール実行前の機械的ブロックは、プロンプト指示の「ベストエフォート」問題を補完するソフトウェア的ガードレールとして機能し、LLMの本質的な不確実性に対する実用的な対処法
- ブロック時のpermissionDecisionReasonがエージェントのコンテキストに渡される仕様により、単なる遮断ではなくエージェントが自律的にリカバリできる設計になっている点がフィードバックループとして巧妙
- 「プロンプト→Hooks→人間レビュー」の3層防御モデルは、AI安全性の多層防御アーキテクチャとして汎用性が高く、禁止ワード以外（コスト上限、レート制限等）にも拡張可能
