## 要約

コンテキストエンジニアリング（CE）をプロンプトエンジニアリングの延長と捉える誤解を解き、7つの独立した構成要素の組み合わせによるシステム設計として再定義した実践的解説記事。Anthropicの公式ブログが提示する6レイヤー構造（System instructions, semantic context, operational memory, conversational history, retrieval systems, tool access）を踏まえつつ、著者が現場経験から導いた7要素に再整理している。

7要素の役割は以下の通り。①System Prompt：LLMの役割・境界・振る舞いを定義する土台。Claude Sonnet 4の誠実性スコアがSystem Promptの変更だけで0.2→3.7に向上した事例を挙げる。②Few-shot Examples：出力フォーマットを実例で伝える手法。5個超は精度頭打ちになりコンテキストを圧迫するため、量より代表性が重要。③RAG（Retrieval-Augmented Generation）：外部知識ベースをプロンプトに注入する仕組みで、CE改善幅の8割を占めると著者は主張。「何を入れないか」が設計の中心で、社内Slackの全ログをベクトル化すると検索ノイズが増えた事例を紹介。④Tool Use / MCP：外部システムの呼び出しと副作用実行を担う。MCPはAnthropicが2024年末から推進し2026年現在ではClaude Code・Cursorが標準対応。ツール数が20個超になるとコンテキスト汚染が起きるため「minimal viable set of tools」が推奨される。⑤Memory：セッション横断で状態を保持。Anthropicは2026年3月にMemoryツールをデフォルト機能化。短期（Buffer）・中期（Summary）・長期（Vector/Knowledge Graph）の3階層で管理し、「何を忘れるか」の設計が不可欠。⑥Compaction：長大化した会話履歴を圧縮するメタ要素。Claude Codeは5段階プログレッシブCompaction（Budget reduction→Snip→Microcompact→Context collapse→Auto-compact）を実装。重要なルールは必ずSystem Promptに固定し、Compactionに依存しない設計が原則。⑦Agentic Control：他の6要素をいつ・どれだけ使うかを判断する上位レイヤー。LangGraph・Claude Code・Cursorが実装済み。固定パイプラインで先に動かし、判断が必要な枝だけ後から動的化する段階的実装が推奨される。

導入優先順序として、System Prompt整備（数時間）→RAG（1〜2週間）→Few-shot（数時間）→Tool Use/MCP（1〜2週間）→Memory（1〜2週間）→Compaction（API任せ）→Agentic Control（数週間〜）の順が提案されており、土台を固めてから上層を積む設計思想が一貫している。監査エージェント開発においては、RAGによるドメイン知識注入、MCPによる外部システム連携、LangGraphによるAgentic Control実装が直接応用可能な知見。

## アイデア

- RAGとMemoryの役割分担を「静的知識ベースの検索 vs ユーザーとの動的なやり取りの蓄積」と明確に二分する整理は、エージェント設計時の情報格納先判断基準として実用的
- Compactionを独立要素ではなく「他の全要素を縮めるメタ要素」と位置づけ、重要ルールはSystem Promptに固定する設計原則は、長時間稼働エージェントのルール消失バグを防ぐ直接的な対策
- Agentic Controlを「まず固定パイプラインで動かし、判断が必要な枝だけ後から動的化する」段階的実装戦略は、過度な初期設計を避けながらコスト最適化できる実践的アプローチ
