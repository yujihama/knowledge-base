## 要約

RustでLLM駆動のコードレビューCLIツール「agent-reviewer」を実装した事例。Gitの差分を3段階のパイプライン（triage→review→finalize）で処理する設計が核心。triageフェーズでは差分を「レビュー単位」に分割し、各ユニットにLight/Standard/Powerのモデルティアを割り当てる。reviewフェーズでは`join_all`による並列処理を行いつつ、グローバルな`ConcurrencyLimiter`（セマフォ）でAPI同時実行数を`concurrency = 4`のように上限管理する。finalizeフェーズで全レビュー結果をMarkdownレポートに統合する。

アーキテクチャ面では、ReActループを`agent-reviewer-agent` crateに分離し、ファイルシステム/Gitツール実装を`agent-reviewer-tools`、モデルプロバイダ設定を`agent-reviewer-model-provider`として責務を明確に分割したCargo workspaceを採用。設定は`agent-reviewer.toml`でmodel_providers/models/agents/stepsの4層に分離し、同一モデルでもフェーズごとに異なるeffortやmax_tokensを設定可能にした。

ReActループの実装では「最終ターンではsubmit用marker toolのみを渡す」手法により、LLMに必ず構造化結果を返させる仕組みを採用。marker toolは`schemars`クレートでRustの型定義からJSON Schemaを自動生成し、LLMのtool schemaとして渡す。これにより出力形式の変更がRust型変更として管理でき、コンパイル時に構造を追跡できる。

reviewエージェントにはサブエージェントをtoolとして渡す設計を採用。`explorer`（リポジトリ横断調査）と`advisor`（設計方針助言）の2種のサブエージェントを、外側のエージェントからはただのtoolとして扱う。必要時のみ探索を委譲することでコンテキストを節約する。

セキュリティ観点では、エージェントに渡すツールをread-onlyに限定（`read_file`, `git_diff_*`等）し、コードの自動編集を意図的に除外している。プロンプトは`include_str!`でバイナリに埋め込みつつ、設定ファイルで各フェーズごとに上書き可能。CLAUDE.md、AGENTS.md等の既存AIエージェント向け指示ファイルも自動読み込み対応。model_providersにはOpenAI互換APIを指定できるため、OllamaやLM Studioなどローカルモデルも利用可能。

監査エージェント開発への示唆として、「単一の巨大エージェントではなくフェーズ分割+並列処理」「marker toolによる構造化出力の強制」「サブエージェントをtoolとして抽象化」の3パターンは、複数ドキュメントや証憑を並列処理する監査エージェントにそのまま応用できる設計パターンである。

## アイデア

- marker toolをReActループの終了シグナルとして使い、Rustの型からJSON Schemaを自動生成することで、LLMの出力形式をコンパイル時に型安全に管理できる設計
- サブエージェント（explorer/advisor）をtoolのインターフェース（fn tool() -> Tool）として抽象化することで、外側のエージェントがサブエージェントを通常ツールと区別なく扱える構造
- triageフェーズでLight/Standard/Powerのモデルティアを割り当て、変更の重要度に応じてモデルの重さを動的に切り替えるコスト最適化パターン
