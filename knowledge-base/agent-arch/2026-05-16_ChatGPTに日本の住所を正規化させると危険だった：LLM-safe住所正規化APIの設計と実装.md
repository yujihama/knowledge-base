## 要約

日本の住所をLLMやAI Agentに処理させる際の具体的なリスクと、専用正規化APIによる対処法を論じた記事。日本住所は全角・半角・漢数字の混在、ハイフン類似文字（10種類以上）、京都の通り名、札幌の条丁目表記、同名市区（府中市・伊達市が複数都道府県に存在）など、機械処理上の例外が極めて多い。LLMはこれらを「それっぽい補完」で処理しようとするため、静かにデータが壊れる。特に危険なのは、LLMが「わからない住所をわかったふり」する点であり、住所正規化では「表記を直せたのか」「市区町村まで同定できたのか」「郵便番号まで確定できたのか」「事業所・大規模ビルとしてマッチしたのか」という粒度別の信頼度を構造化して返す必要がある。OCRでは漢数字「二」とカタカナ「ニ」の誤認識、音声入力では「番・号・ハイフン」の表記ゆれ、人間入力では都道府県・市区名の省略が頻出する。さらに国土交通省系と日本郵便系で住所表記が異なり（例：袖ケ浦 vs 袖ヶ浦）、業務フォームが完全一致検証をしている場合に正規形不一致で弾かれる問題も存在する。これらを踏まえ、著者はJushoという日本住所正規化APIをOSSとして公開。国土交通省の位置参照情報と日本郵便の郵便番号データを組み合わせ、pref/city/town/banchi/go/post_code/lat/lng/match_typeの構造化JSONを返す。AI Agent統合向けにMCPサーバー（https://api.jusho.dev/mcp）とOpenAPI仕様も提供。推奨アーキテクチャは「LLMが住所文字列を抽出→住所正規化APIが照合・構造化→業務システムに保存」という役割分担。監査エージェント開発への示唆として、CRMや本人確認・契約管理など業務データを扱うAgentでは、LLMの推論に依存せず専用照合ツールを呼び出すパターンが安全性を担保する上で重要であり、同様の設計思想（LLMは抽出・判断、専用APIは照合・構造化）は住所以外の業務データ（法人番号・金融コード等）にも適用できる。

## アイデア

- 「LLMはわからない住所をわかったふりをする」という問題：信頼度・マッチ粒度を構造化して返す専用APIが必要という設計思想は、住所以外の業務データ（法人番号・商品コード等）にも汎用的に適用できる
- 国土交通省系と日本郵便系の2つの住所表記系が並存している事実：正規化した住所をさらに送信先システムの期待する表記体系へ変換する「2段階正規化」が必要なケースがある
- AI Agentのツール設計原則として「壊してはいけない業務データは専用ツールに渡す」という役割分担パターン：LLMの強み（抽出・推論）と専用APIの強み（辞書照合・例外処理）を明示的に分離するアーキテクチャ
