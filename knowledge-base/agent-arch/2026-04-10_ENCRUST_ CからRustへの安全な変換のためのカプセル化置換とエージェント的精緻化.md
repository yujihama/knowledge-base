## 要約

ENCRUSTは、実世界のCプロジェクトを安全なRustへ変換するための2フェーズパイプラインである。既存手法の課題として、(1)メモリ安全性保証のないunsafe出力を生成する、(2)関数単位での孤立した変換によりクロスユニットの型ミスマッチを検出できない、(3)型シグネチャ変更時に呼び出し元の協調更新が必要になる、という3点が挙げられる。ENCRUSTはこれらをABI（Application Binary Interface）保持型ラッパーパターンによって解決する。

**フェーズ1：Encapsulated Substitution（カプセル化置換）**
各関数を2コンポーネントに分割する。①元の生ポインタシグネチャを保持する呼び出し元透過シム（shim）、②LLMがスコープ限定のクリーンなプロンプトで変換対象とする安全な内部関数。この構造により、呼び出し元への協調更新なしに関数ごとの独立した型変更と失敗時の自動ロールバックが可能になる。変換成功後は決定論的な型指向ラッパー除去パスによりシムを削除する。

**フェーズ2：Agentic Refinement（エージェント的精緻化）**
`static mut`グローバル変数、スキップされたラッパーペア、変換失敗箇所など、関数スコープを超えるunsafe構造をLLMエージェントがコードベース全体に対して処理する。ベースライン認識型の検証ゲートにより各中間状態の整合性を担保する。

評価はGNU Coreutilsの7プログラムとLaertesベンチマークの8ライブラリ（計15プログラム）で実施。全15プログラムでunsafe構造の大幅削減を達成しつつ、テストベクタの完全な正確性を維持した。特筆すべきは「ライブスキャフォールド」コンセプトで、各変換ステップを統合コードベースに対してバリデーションすることで、実世界の依存関係複雑性下でもコンパイル可能な出力を生成できる点である。

## アイデア

- ABI保持ラッパーによる「境界適応と関数ロジックの分離」パターンは、レガシーシステムの段階的移行において各ステップを独立してバリデーションできる設計原則として汎用的に応用できる
- 失敗時の自動ロールバックと中間状態のベースライン認識型検証ゲートを組み合わせた構造は、複数エージェントが協調してコード変換を行う際の信頼性保証メカニズムとして参考になる
- 関数レベルの変換（フェーズ1）とホールプログラム推論が必要な構造の処理（フェーズ2）を分離した2段階アーキテクチャは、複雑な変換タスクを局所的問題と大域的問題に分解するエージェント設計パターンの好例
