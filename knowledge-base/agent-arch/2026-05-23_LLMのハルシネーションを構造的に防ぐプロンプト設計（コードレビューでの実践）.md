## 要約

LLMによるコードレビューの信頼性問題を構造的に解決する「検証駆動プロンプト設計（v3）」を提案した実践報告。従来手法（v2）ではfile:行の記載を指示するだけで、LLMが実際にファイルを読まずにもっともらしい行数・評価をハルシネーションするという問題があった。具体的には`browser_use_agent.py`でLLM申告1,592行に対し実測329行（4.8倍乖離）、`__init__.py`では717行vs67行（10.7倍）という事例が確認された。この構造的欠陥の原因は①証拠収集手段の不在、②ファイルアクセス手順の未指示、③行数検証ステップの欠如、④サンプリング戦略の不在の4点と分析。

v3では4ステップのパイプラインを導入する。Step 0ではNetworkXのネットワーク中心性を使い、拡張子重み・パス重み・Import中心性・行数を組み合わせてファイルにスコアを付け、上位20件のみを評価対象とする。Step 1では`wc -l`でスコアリング時の行数を実測し、10%超の乖離がある場合は検証不通過とする。Step 2ではReadツールで対象ファイルを実際に読み込む。Step 3では読み込み済みファイルのみ評価カードを出力し、未読ファイルの評価を明示的に禁止する。評価カードには`loc_verified`・`file_read`・`score`・`centrality`フィールドを必須化した。

3つのPythonプロジェクト（計30ファイル）で3つのLLM（GLM・MiniMax・Claude Opus）を使って検証した結果、信頼性は従来の15〜55%からv3で70〜85%に向上。行数乖離は最大10.7倍からゼロ、未読ファイルの評価件数もゼロになった。また正確な評価により3プロジェクト共通の弱点として「テスタビリティの低さ」が判明し、DI注入・エラー処理改善を含む20項目のリファクタリングにつながった。

設計上の5教訓として「証拠収集手段の先行提供」「サンプリング戦略の必須化」「外部検証の組み込み」「未読=評価不可の明示」「正確な評価は低スコアになる」を挙げる。今後はスコアリング部分のMCPサーバー化により自動パイプライン化を検討している。監査エージェント開発への示唆として、LLMへの指示設計において「出力を信じる」ではなく「信じられる出力を構造的に強制する」設計思想はエビデンス収集・監査判断の信頼性確保にも直接応用可能。

## アイデア

- 「証拠を要求する前に証拠を収集する手段を提供する」という設計原則は、LLMエージェント全般のツール利用設計にも転用できる——ツールを与えずに出力形式だけ指示するプロンプトは構造的にハルシネーションを誘発する
- NetworkXのImport中心性によるファイルスコアリングは、コードレビュー以外でも「どのファイルをコンテキストに載せるか」というRAG・エージェントのコンテキスト選定問題への応用が可能
- 正確な評価に変えるとスコアが下がるという観察は、LLM-as-judgeを使った評価パイプラインで「高スコア=高品質」と解釈することへの警告であり、スコアの絶対値より評価プロセスの検証可能性を重視すべきという設計指針を示す
