## 要約

AIエージェントに長時間の反復作業をさせる際、コンテキスト上限・レート制限・ネットワーク障害などによるセッション中断は不可避である。筆者が開発したワークフローエンジン「flowsmith」は「セッションはいつ死んでもいい」を設計前提とし、チェックポイント機構によって中断後に続きから再開できる仕組みを実装した。

実行状態はすべて state.json という単一ファイルに永続化される。このファイルには flow_id、status、completed_steps、next_step、各ステップの出力ファイルパスなどが記録される。再実行時はこのファイルを参照して completed_steps をスキップし、next_step から再開する。ユーザー操作は「同じコマンドをもう一度叩く」だけでよい。状態更新はステップ完了ごとに行われ、foreach による反復処理では1行完了ごとに更新される。これにより、200件の反復の150件目で死んでも残り50件のみを再実行できる。

設計で最も悩んだのは「途中まで書かれたファイル（半端な成果物）」の扱いだ。節約のため続きを書き足すことも検討したが、書きかけファイルが使用可能かどうかを機械的に検査する手段がなく、破損データが下流に流れるリスクが再実行コストを上回ると判断した。そのため、ステップの状態は「done」か「not-done」の二値のみとし、「半分終わった」という中間状態は設計上存在しない。done の判定条件は厳格で、エージェントが規定形式の結果ブロックを返し、かつファイル存在・必須見出し・件数照合などの機械ゲートを通過した場合のみ。途中で死んだステップは next_step に残り、再開時は頭から作り直す。出力は必ず固定パスへの上書き（overwrite）で、追記（append）は禁止される。

トレードオフとして、ステップ単位の粒度では大きなステップが90%進んで死ぬと全トークンを捨てることになる。これを緩和するのが foreach の行単位チェックポイントで、反復系の重いステップは最大で「行1つ分」しか捨てない。非反復の大きなステップ（例: コード全体の依存関係抽出）への対応は現時点の設計限界として残っている。

本機構は実負荷でも検証済みで、15ステップのレガシー調査フローを JPetStore 6 に対して実行した際にレート制限で2回セッションが死亡。1回目は1ステップ完了時点、2回目は11ステップ完了時点で中断したが、いずれも同じコマンドの再実行で完走。中断込みの合計作業時間は約35分（Claude Code 実測）だった。「賢く救済するより愚直に作り直す方が総コストは安い」という設計思想が実戦で検証された。監査エージェント開発においても、長時間の証跡収集・多数ドキュメントの反復処理にこのチェックポイント設計を適用することで、障害耐性の高いワークフローを構築できる。

## アイデア

- ステップ状態を done/not-done の二値に限定し中間状態を排除することで、「半端な成果物の検査コスト」より「作り直しコスト」を選ぶという割り切りが、AIワークフローの信頼性設計の核心
- 出力を必ず上書き（overwrite）に統一し追記（append）を禁止するという制約が、冪等性（idempotency）を保証する最小コストの実装手段
- 反復処理（foreach）の行単位チェックポイントという粒度の細分化が、ステップ全体の作り直しコストを最小化する現実的な緩和策
