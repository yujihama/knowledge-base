## 要約

Claude Codeに2026年3月からベータ導入された/advisorコマンドと、サブエージェント機能の仕組み・使い分け・組み合わせ方を解説した記事。

アドバイザーは、実行モデル（Executor）が作業中に上位モデルへ自動相談する仕組み。内部的にはanthropicbeta: advisor-tool-2026-03-01ヘッダを付与し、advisor_20260301型のツール定義を/v1/messagesリクエストに注入することで、同一リクエスト内でサーバー側が別推論パスを走らせる。有効なペアはHaiku→Opus、Sonnet→Opus、Opus→Opusの3通りのみ。発動タイミングは「実装着手前」「タスク完了宣言前」「同一エラー繰り返し時」「方針転換検討時」の4つが公式推奨。

公式ベンチマークによれば、Sonnet+Opusアドバイザー構成はSWE-bench MultilingualでSonnet単独比+2.7ポイント、タスクあたりコスト-11.9%。Haiku+Opusアドバイザー構成はBrowseCompで41.2%（Haiku単独19.7%の約2倍）。アドバイザー出力は通常400〜700テキストトークン（thinking込みで1,400〜1,800）。

サブエージェントはメインエージェントとは別コンテキストウィンドウで動く独立インスタンス。組み込みでExplore（Haiku、読み取り専用）、Plan（親モデル継承）、general-purpose（全ツール使用可）の3種。/agentsコマンドで~/.claude/agents/*.mdにYAML frontmatter付きMarkdownとして独自定義も可能。

両者の決定的な違いはコンテキストの扱い。アドバイザーは会話全履歴を共有するため文脈を踏まえた助言が得られる反面、誤った前提を引き継ぐリスクがある。サブエージェントは必要情報のみ渡されるフレッシュな状態で、前提自体を再定義できる点が固有価値。

推奨ワークフローは「アドバイザーで方針確認→Sonnetが実装→サブエージェントがレビュー→修正→報告」の5ステップ。CLAUDE.mdにこのフローを記述することで自動化が可能。

Maxプランの推奨構成は通常利用でSonnet実行+Opusアドバイザー。Opus枠に余裕がある場合はOpus実行+サブエージェント活用が有効（Opus同士のアドバイザーは会話バイアスを引き継ぐため効果薄）。コスト最適化として、Sonnetをmedium effortで動かすとOpusアドバイザーとの組み合わせでデフォルトeffortのSonnet単独と同等の知能が得られ、システムプロンプトに100語以内の簡潔な返答を指示するとアドバイザー出力トークンが35〜45%削減される。

監査エージェント開発への示唆：複数エージェントが連携する監査ワークフロー（証拠収集→リスク評価→レポート生成）において、アドバイザーパターンを応用することで、実行エージェントが判断に迷う局面でより高精度なモデルに自動相談させる設計が実現できる。LangGraphのノード間での判断委譲にも同様のコンセプトを適用可能。

## アイデア

- アドバイザーが「より賢くなる」だけでなくSonnet単独より総コストが下がる点：不要なツール呼び出しや会話の冗長化をOpusが抑制するため、精度向上とコスト削減が同時に達成される
- Opus+Opusアドバイザー構成の限界とサブエージェントの代替価値：同一知能レベルでも会話バイアスを引き継ぐアドバイザーより、新鮮なコンテキストで推論するサブエージェントの方がOpus枠では付加価値が大きいという非直感的な知見
- CLAUDE.mdへのワークフロー記述による自動化：相談→実装→レビューの5ステップをシステムプロンプトに明示することで、ユーザー介入なしに品質保証ループを自律的に回させる設計パターン
