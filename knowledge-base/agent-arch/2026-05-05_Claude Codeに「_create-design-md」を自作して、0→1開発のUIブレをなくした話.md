## 要約

デザインファイルやブランドガイドラインが存在しない0→1開発フェーズにおいて、Claude CodeなどのAIコーディングエージェントにUIを生成させると、画面ごとに余白ルールやボタンの角丸サイズが一貫しないという問題が生じる。これはAIが文脈不足を補完する際に不整合なスタイルを採用するためである。

この課題に対し、著者は対話型のClaude Codeカスタムスキル「/create-design-md」を自作した。仕組みはシンプルで、プラットフォーム（Flutter / React Native / SwiftUI等）、アプリの雰囲気・世界観（例：「ポップ・楽しい・ゲーム系」）、Primaryカラーなど必須・任意合わせて10問のヒアリングを行い、その回答をもとにAIがDESIGN.mdを自動生成する。任意項目は「スキップ」と答えれば、アプリの世界観と選択プラットフォームを文脈として自動推論し、デフォルト値を補完する。

DESIGN.mdの構成は、AIへの絶対命令（NO HARDCODINGやCONSISTENCY）、カラートークン（Brand Colors・Semantic Colors）、タイポグラフィ、スペーシング、コンポーネントルールなどのセクションで構成される。「Flutter」「ポップ・楽しい・ゲーム系」「Primary: #073CFA」の入力例では、セカンダリカラーとしてビビッドなオレンジ（#FF6B35）がAIに推論され、カードコンポーネントはフラット＆角丸でポップなスタイルに自動設定された。

プロンプト設計のポイントとして、スキップ時の自動補完において「どの回答を文脈として使うか」を明示することが重要である。本実装では「スキップされた項目は質問2（プラットフォーム）と質問3（世界観）をもとに一般的な値を自動設定し、<!-- AIデフォルト値 -->コメントを付与する」という指示をスキルプロンプト内に含めており、これにより人間が後から修正しやすく、かつ一貫性のある出力が得られる。

Google Stitchのようなデザインファイルからの自動生成ツールが利用できない環境での代替手段として有効であり、Claude Codeの旧Commands方式と新Skills方式の差異にも注意が必要。監査エージェント開発への応用としては、UIの一貫性確保と同様に、エージェントの出力フォーマットや判断基準をDESIGN.md的な仕様書として事前定義する手法が参考になる。

## アイデア

- 「DESIGN.md」というAIエージェント向けデザイン仕様書の概念：README.mdが人間向けであるのと対比して、AIが解釈・遵守するためのデザイン仕様をMarkdownで構造化する発想は、エージェントへの指示設計全般に応用できる
- ヒアリング→推論による仕様書自動生成パターン：必須項目のみ人間が入力し、任意項目はAIが文脈（プラットフォーム＋世界観）から自動補完するハイブリッド生成アプローチは、他のドメインの仕様書生成にも転用可能
- 「<!-- AIデフォルト値 -->」コメントによる透明性確保：AIが自動補完した箇所をコメントで明示することで、人間のレビューと修正を容易にする設計は、監査エージェントの判断根拠の可視化にも通じる
