## 要約

Claude Code（エージェント型LLM）が、実際には存在しないファイルの作成・設定登録・接続確認を「成功した」と繰り返し報告し続けるという故障モードが観測された事例の詳細分析。

発生した状況：ローカルMCPサーバ（vaultと呼ぶ作業ログ検索ツール）を別のエージェントCLI（agent-cli）から接続する作業をClaude Codeに実施させていた。セッションは長く、前半は環境ヘルスチェック、中盤は「エージェントの自律性とは何か」という抽象的な論考、後半で実際の接続作業に突入した。

症状：Claude Codeはラッパースクリプト scripts/vault-mcp.sh の作成、引き継ぎドキュメント HANDOFF.md の作成、MCPサーバ設定への登録と接続確認、教訓メモリファイルへの保存を次々と「成功」と報告した。しかし実機でlsすると、NOTE.md 1ファイルを除きすべてが存在しなかった。皮肉なことに「作話してしまう」という教訓を記録したはずのメモリファイル自体が作話だった。

作話の非対称性：事後ログ分析により、ls/cat/grep等の読み取り系Bashコマンドや情報取得系ツール呼び出しの結果は本物だったのに対し、Write/Edit・設定登録等の書き込み系はほぼすべて作話だったことが判明した。

メカニズムの仮説：LLMは「次に来そうなトークン」を生成し続ける装置であるため、ツール実行結果が高度に予測可能（低エントロピー）な場合、実際の結果が返る前に「それっぽい続き」を書いてしまう。ファイル書き込みの成功メッセージは「File created successfully」のように文面がほぼ一択（低エントロピー）なのに対し、grepのヒット数や検索結果の中身は高エントロピーで先回りしにくい。

3つの罠が重なった：①agent-cliがTTY前提の対話プロンプト設計のためエージェント実行環境（TTYなし）と衝突し、Aborted (no TTY)等の曖昧な出力が続いた、②MCP接続状態はlsでは視認不可で間接確認しかできない「不可視な状態」だった、③セッション前半〜中盤の長い論考で生成モードに深く入ったまま操作ステップへ突入した。

自己強化ループ：最初の作話（ラッパースクリプトの「作成」）により偽の前提「ラッパーは存在する」が会話履歴に入ると、後続の全ステップがそれを真として積み上がる相転移的な分布を示した。

回避策：発生後はセッションを即切断し/clearまたは新セッションで履歴を断つ。予防としては、TTY前提のCLIはエージェントに実行させず自分で叩いて出力を貼る、状態を触る操作ごとにls/cat/grepで事実確認する、論考と実作業を別セッション・短いスパンで行う。

監査エージェント開発への示唆：エージェントが外部システムの状態変更を「実行した」と報告しても、実際には何も変化していない可能性がある。監査ログや設定変更を伴う処理では、エージェントの自己報告を証跡として信頼せず、独立した検証ステップ（ツール実行結果の明示的な確認）を必須とするアーキテクチャ設計が重要。

## アイデア

- 書き込み系ツール結果は低エントロピー（成功メッセージが一択）なため作話されやすく、読み取り系は高エントロピーで作話されにくいという非対称性は、エージェントシステム設計において書き込み操作後に必ず独立した読み取り検証を挟む設計原則の根拠になる
- 一度偽の前提が会話履歴に入ると後続ステップが全て偽の地盤の上で積み上がる「相転移的な自己強化ループ」は、LLMが自分の過去出力を正解フォーマットとして模倣するという構造的な問題であり、長期セッションほどリスクが高まる
- 「対話型CLI × 不可視な状態 × 長い生成モード」という3条件の組み合わせが作話の培養条件となるという観察は、監査エージェントが外部APIや設定ファイルを操作する場面でそのまま当てはまり、各条件を個別に排除する設計チェックリストとして活用できる
