## 要約

本稿は「責任経路工学」シリーズの第2回として、ISO/IEC 42001が定めるAIマネジメントシステム（AIMS）と、実際のAIエージェント運用の間に存在する「設計ギャップ」を埋める方法論を提示する。

ISO/IEC 42001は組織がAIを管理するための国際規格であり、AIの方針・役割と責任・リスク管理・影響評価・運用管理・監視・文書化・継続的改善を要求する。しかしAIエージェントがメール・ファイル・API・ERP等に接続される現場では、規格の抽象要求だけでは「誰が承認するか」「どのログを残すか」「誰が停止権限を持つか」が自動的に決まらない。

著者はこの問題を解決するために「責任経路工学（Responsibility Pathway Engineering）」を提唱し、AIマネジメントを5つのレイヤーに分離する。①規格レイヤー（ISO/IEC 42001）、②リスク管理レイヤー（ISO/IEC 23894）、③責任経路工学（判断・承認・実行・停止・修復の経路設計）、④実装レイヤー（AIエージェントの行為制御）、⑤証跡レイヤー（Evidence Log / Audit Trail）。

責任経路工学の中核は7要素：Decision Owner（業務判断者）、Approval Gate（実行前承認地点）、Execution Actor（実行主体）、Stop Authority（停止権限者）、Evidence Log（証跡）、Repair Owner（修復責任者）、Human Return Point（人間介入地点）。さらにOpenAIの5原則（民主化・エンパワーメント・普遍的繁栄・レジリエンス・適応性）を受けて、External Oversight PathwayとPolicy Update Logの追加を提唱する。

具体例として「AIによるファイル更新」では上記7要素すべてを明示的に設計すること、「外部API実行」ではAPIの可逆性・外部影響範囲・承認要否を事前に確立する必要があることを示す。また「メールを送る」という単一行為も「下書き作成→宛先確認→添付確認→本文確認→送信承認→送信→誤送信修復」の7ステップに分解され、それぞれに責任経路を割り当てる必要がある。

監査エージェント開発への示唆として、本フレームワークはLangGraph等で実装するAIエージェントのツール呼び出し設計に直接適用可能。各ツール実行ノードにApproval GateとEvidence Logを組み込み、高影響アクションにはAction Class Matrixで分類したStop Authorityを定義する構造が、ISO/IEC 42001準拠の内部監査AIシステム設計の骨格となる。

## アイデア

- AIエージェントの行為を「利用管理」から「行為管理」へシフトさせる概念フレーム：メール送信を7ステップに分解し各ステップに責任者を割り当てる手法は、LangGraphのノード設計と直接対応し、コード実装レベルでのコンプライアンス埋め込みを可能にする
- ISO/IEC 42001の抽象要求を7要素（Decision Owner/Approval Gate/Execution Actor/Stop Authority/Evidence Log/Repair Owner/Human Return Point）にマッピングするテーブルは、規格認証取得と実装品質を同時に達成するための設計チェックリストとして機能する
- OpenAIの5原則を「責任経路として読む」アプローチ：理念レベルの外部監視要求を「誰が止めるか・どこに記録するか・誰が更新するか」という実装粒度の問いに変換する手法は、AI倫理文書を内部システム要件定義に落とすテンプレートとして汎用的に使える
