## 要約

AIエージェントが「提案→承認→実行→ログ記録」というフローを経ても、事後に「誰が止めるべきだったか」が不明になる問題を起点に、「責任経路工学（Responsibility Pathway Engineering）」という設計体系を提唱する記事。

従来の「責任の所在（誰が責任者か）」の固定だけでは、AIエージェント運用における責任の流れを捉えきれない。AIが社内文書を参照して対応案を生成し、人間が承認し、ツールがメールを送信し、後から誤りが判明するケースでは、承認者の認識内容・参照文書の正確性・承認画面の情報提示・送信前の停止権限・誤送信後の修復責任など、複数の責任節点が存在する。これらを「一人の責任者名」に収束させると、事故後に責任が蒸発する。

責任経路工学は、AIを責任主体にするのではなく、AIを経由した判断・行為が人間・組織・制度へ「戻れる経路」を設計・実装・検証・運用する工学的体系と定義される。その実装層として「Responsibility Pathway Layer」を位置付け、最小構成要素として以下の7要素を定義する：Decision Owner（判断帰属者）、Approval Gate（承認ゲート）、Execution Actor（実行主体）、Stop Authority（停止権限者）、Evidence Log（証跡ログ）、Repair Owner（修復責任者）、Human Return Point（人間への回帰点）。

アーキテクチャ上の階層は「Model/Tools/APIs → Agent Runtime/Harness → Responsibility Pathway Layer → Human/Organization」と整理され、モデルの推論やツール実行を制御するハーネスやガードレールとは別に、責任の帰還経路を独立して設計する必要性を強調する。

また「Action Class Matrix」を前処理として位置付け、公開Webページの読み取りと本番DBの更新を同じ「AIの実行」として扱わず、行為クラスごとに異なる責任経路を割り当てる設計を求める。RACI・HITL・Guardrails・Fail-Closed・監査ログといった既存概念を否定せず、それらを「責任の流れ」として束ね直す上位設計対象として位置付ける点が特徴的。

監査エージェント開発への示唆として、LangGraphベースのエージェントにおいても各ノード（ツール呼び出し、LLM推論、承認UI）を責任経路の節点として設計し、Stop AuthorityとRepair Ownerを明示的にコード・設定・運用手順に埋め込む必要がある。次回記事ではISO/IEC 42001との接続、Evidence Log、Stop Authority、Human Return Pointの詳細が扱われる予定。

## アイデア

- 責任を「所在（点）」ではなく「経路（フロー）」として設計対象化する発想は、LangGraphのノード・エッジ設計と直接対応させられる――各エッジに責任帰還メタデータを付与するアーキテクチャが具体的実装として成立する
- Action Class Matrixによる行為分類（読み取り／提案／内部状態変更／外部影響／可逆／不可逆）をエージェントのツール定義に組み込むことで、Guardrailsの発動条件と承認ゲートのトリガーを動的に切り替える仕組みが実現できる
- Stop AuthorityとHuman Return Pointを明示的に実装しない限り、HITL（Human-in-the-Loop）は形式的な承認クリックになり責任は戻らないという指摘は、承認UIの設計要件（差分表示・リスクスコア提示・ロールバック操作）に直接変換できる
