## 要約

著者は日常業務でスクリーンショットやPDF資料の文字起こしに自作ツール「VisionOCR」を利用しており、当初はGemini（マルチモーダルLLM）でOCRを試みた。しかし10ページ程度のPDFを処理した際、中間の数ページが無通告で抜け落ちる現象が発生した。欠落箇所にエラーマーカーは表示されず、出力だけ見ると全ページ成功に見えるため、気づきにくい点が問題である。同一PDFをCloud Vision APIで処理すると全ページ取得できた。

原因の本質的な違いは両者のアーキテクチャにある。Cloud Vision APIは画像から文字を機械的に検出・抽出する「転写エンジン」であり、意味解釈を行わないため文脈による省略が起きない。一方GeminiはLLMとして「確率的に最もらしい応答」を生成する「生成エンジン」であり、長文入力に対しては逐語的な全文転写よりも要約・省略の挙動が働きやすくなる。この性質は他のマルチモーダルLLMにも共通する。

比較検討の結果、長文業務文書の網羅的な転写ではCloud Vision APIが優位（抜け落ちなし）、表のMarkdown化・翻訳との同時実行・レイアウト構造保持ではGeminiが優位という使い分けが適切と結論づけている。手書き・半角カナへの対応はどちらもほぼ同等だが、ハルシネーションリスクはGeminiにのみ存在し、レスポンス安定性もCloud Vision APIが上回る。

著者はこの判断をもとにVisionOCRをCloud Vision APIベースで実装。主要機能として、マウスドラッグによる任意領域選択OCR、複数ページPDF対応、日本語手書き・半角カタカナ認識、テキスト編集機能（句読点・改行・空白の除去）、クリップボードコピー・ファイル保存を備える。動作環境はWindows・Python 3.13以上・Google Cloudサービスアカウント認証。

監査エージェント開発への示唆として、LLMを文書解析パイプラインに組み込む際、OCRフェーズでLLMを使うと長文書類で無音の欠落が生じるリスクがある。内部監査資料や規程文書など完全転写が必要な文書処理では、OCR専用APIで全文抽出した後にLLMで構造化・分析するという二段階設計が安全性・正確性の観点から推奨される。

## アイデア

- LLMの「確率的生成」という特性が逐語転写と本質的に相反することを実証した点：要約・省略はLLMの強みであり弱みでもあり、完全転写タスクには構造的に不向きである
- 欠落が無通告で起きるため出力品質の自己検証が困難という点：ハルシネーション（誤変換）よりも無音の欠落の方が、ダウンストリームタスクでの検出が難しくリスクが高い
- 「新技術が旧技術を全置換する」という思い込みへの反論：タスクの性質（転写 vs 構造化）でツールを使い分けるハイブリッドアプローチが実用上の最適解になる
