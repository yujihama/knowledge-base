## 要約

Claude Opus 4.6 が提供する 100 万トークンコンテキストを使い、172 ファイル・約 18,000 行・24 モジュール構成の実プロジェクト Terraform リポジトリ（約 25 万トークン）を丸ごと読み込ませた実験レポート。従来は「関連ファイルのみ絞り込む」「Agent tool で分割探索」「memory に要点を持たせる」などの工夫が必要だったが、1M コンテキストによりこれらの制約がリセットされた。

試したユースケースは 3 つ。①全ファイル一括読み込みによる IAM ポリシー横断レビューでは、入力 27 万トークン・処理時間約 90 秒で 12 件（Critical 3 件）の問題を検出。特に「モジュール A の IAM ロールがモジュール B のリソースに過剰権限を持つ」というクロスモジュール問題を発見でき、単一ファイル読み込みでは不可能だった分析を実現。②リファクタリング案提示では、module.api と module.worker が共有 DynamoDB テーブルを通じて密結合している点を指摘し、module.shared-resources 新設を提案。③TypeScript アプリコード（約 12 万トークン）と Terraform（25 万トークン）を同時投入（計約 40 万トークン）し、Lambda 関数の IAM 権限と実際の AWS API 呼び出しの乖離を分析。DynamoDB:Scan 権限のみ付与で実装は Query のみという例を 5 件、S3:* 権限で GetObject のみ使用を 3 件、コードから消えた関数への権限残存を 2 件検出。

コスト面ではプロンプトキャッシュの活用が鍵で、キャッシュヒット時はコストが約 0.1 倍になる。TTL は 5 分のためセッションを保持し続けることが重要。注意点として、自動生成ファイル（.terraform.lock.hcl など）やロックファイルは除外した方が精度向上。出力トークン上限は従来と同程度のため、大量変更は分割実行が必要。コンテキスト中盤の情報が軽視される傾向も残存するため、重要な制約は冒頭・末尾に配置することが有効。運用方針として、通常作業は Sonnet + 必要ファイルのみ、クロスモジュール分析やリポジトリ全体リファクタ時のみ Opus 1M を使う使い分けが推奨されている。監査エージェント開発への示唆として、インフラと実装コードを同一コンテキストで横断分析する手法は、アクセス制御レビューや証跡との整合性確認などの監査タスクにそのまま転用可能。

## アイデア

- アプリコード（TypeScript 12万トークン）とインフラ定義（Terraform 25万トークン）を同一コンテキストで読み込み、Lambda の IAM 権限と実装上の API 呼び出しの乖離を自動検出できる点は、最小権限原則の継続的検証を自動化する監査ユースケースに直結する
- プロンプトキャッシュの 5 分 TTL を意識してセッションを維持することで、同一リポジトリへの繰り返し質問のコストを約 1/10 に抑えられる点は、大規模コンテキストの経済的な運用設計として重要
- 1M コンテキストでも「中盤の情報が軽視される」Lost-in-the-middle 問題は残存しており、重要な制約を CLAUDE.md 末尾に再掲するという対策は、長大プロンプト設計の実践的知見として参考になる
