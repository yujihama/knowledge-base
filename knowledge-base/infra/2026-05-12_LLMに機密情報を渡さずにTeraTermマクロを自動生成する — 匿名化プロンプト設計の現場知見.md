## 要約

ネットワーク運用現場でLLMを活用する際の最大の障壁である「実機情報の漏洩リスク」に対し、ゴールデンサンプル方式と呼ぶ設計パターンで解決するアプローチを解説した実践記事。

核心的なアイデアは「秘密を扱うレイヤー」と「ロジックを書くレイヤー」の分離にある。LLMにはRFC 5737で定義されたTEST-NET（192.0.2.0/24）やDUMMY_プレフィックス付きのダミー値のみを含む匿名版テンプレートを生成させ、実機固有のIPアドレス・ホスト名・認証情報はローカルのPythonスクリプトがCSVから読み込んでstring.replace()で後から注入する。LLMのAPIには一切の本番情報が渡らないため、学習データ混入リスクと社内プロキシ・SIEMのログへの残留リスクをゼロにできる。

具体例としてCisco IOSへのSSHログイン＆ログ取得マクロ（TeraTerm .ttl形式）の生成を示す。プロンプトではホストをR1 / 192.0.2.1、ユーザーをadminと明示し、パスワードはpasswordboxで実行時に対話入力させるよう制約する。生成されたマクロはwaitregexで「[#>]\s*$」を判定し、terminal length 0 / show ip interface brief / show running-configを順次送信、結果をC:\logs\R1_YYYYMMDD_HHMMSS.logに保存する構成。ローカル置換スクリプト（replace_template.py）がtargets.csvを読み、1行1デバイス分の実行用マクロをout/ディレクトリに書き出す。

LLMの「親切すぎる補完」対策として、PASSWORD = 'admin123' のようなハードコードを正規表現で検出するguard.pyを実装し、生成物を机上シンタックスチェック→GNS3/EVE-NGなど仮想環境での動作確認→レビューの3段階で検証してから本番投入することを推奨する。

Ansible playbook（vars:でIPを変数化）、expectスクリプト（環境変数TARGET_IP/TARGET_PASSを参照）、PowerShellなど他の自動化手法にも同一の設計パターンが転用可能であり、繰り返し作業の多いNE業務ほど効果が大きい。監査エージェント開発への示唆としては、LLMが生成するコードやSQLのロジック部分と、実際のDBクレデンシャルや監査対象データを分離するアーキテクチャに直接応用できる。特に外部LLM APIを使う監査システムでは、本手法のようなローカル注入パターンがセキュリティ要件の充足と生産性向上を両立させる現実的な設計指針となる。

## アイデア

- ゴールデンサンプル方式：LLMにはRFC 5737 TEST-NETや DUMMY_プレフィックスのダミー値のみ渡し、本番値はCSV+ローカルPythonで後注入するという「ロジック生成」と「秘密管理」の完全分離アーキテクチャ
- guard.pyによる生成物の自動セキュリティ検査：正規表現でパスワード直書き・enable secret・SNMP communityを検出し、LLMの過剰補完（'admin123'等のハードコード）をCIライクに防ぐパターン
- この設計は監査エージェントにも直接転用可能：外部LLMにSQL/コードのロジックだけ生成させ、DBクレデンシャルや監査対象データはローカル処理で注入する構成により、GRC要件と自動化効率を両立できる
