## 要約

Python + Playwrightで構築したKDP（Kindle Direct Publishing）自動出版パイプラインが3日連続で停止した原因を特定・修正した事例報告。環境はmacOS Sequoia 15.x / MacBook Pro M5 32GB / Python 3.12 + Playwright (asyncio)で、launchdによる深夜5時の自動起動スクリプト（kdp_auto_publish.py v3.0）が対象。

根本原因は2つ。1つ目は設定値の流用ミス：スクリプトのcategory_placementフィールドのデフォルト値が「日本の小説・文芸」のままになっており、技術書用のキューJSONにcategory_placementを書き忘れたため、PythonカテゴリーのモーダルUI内で「日本の小説・文芸」というテキストをマッチしようとして失敗し続けた。KDPのUIでは「コンピューター」でも「コンピュータ」でもなく「コンピュータ・IT」（中黒あり）という表記が正しく、実UIで目視確認が必要だった。

2つ目はフォールバック実装のスコープ不備：テキストマッチ失敗時のフォールバック処理が「最初の未チェックのチェックボックスを選択する」という実装で、セレクタのスコープをdocument全体にしていた。モーダルが閉じかけているタイミングでPlaywrightがモーダル外のDOM要素を掴み、クリックしてもモーダルが閉じないゾンビクリックが発生。「次へ」を押しても保存スピナーが永遠に回り続ける状態になっていた。EPUBと表紙のアップロード自体は毎回成功していたため、カテゴリー以降を疑うのに時間がかかった。

解決策は2点。まずキューJSONにcategory_placement: "コンピュータ・IT"を追加する設定修正。次にフォールバックのPlaywright evaluate内のセレクタをdocument全体からaria-hidden="false"のモーダルに限定する実装修正（.a-popover-modal[aria-hidden="false"]もしくは[role="dialog"][aria-hidden="false"]）。モーダルが存在しない場合はmodal_not_foundを即返して中断するガード節も追加した。

デバッグ効率化の観点では、category_placement_fail_050131.pngのようにタイムスタンプ付きスクリーンショットの命名規則を最初から入れておいたことで、停止箇所の特定が5分で完了した。aria-hidden属性を信頼してスコープを絞る手法はPlaywrightで動的モーダルを扱う際の一般原則として有効で、開いていないモーダルを操作しようとする系のバグが全体的に減った。監査エージェント開発への示唆：自動化スクリプトのフォールバック処理でスコープを絞らないと、エラー処理自体が連鎖的なエラーを生み出す。エラーハンドラの操作対象DOM範囲を明示的に制限することは、RPA・UIオートメーション全般に適用できる設計原則。

## アイデア

- aria-hidden='false'でPlaywrightのセレクタスコープをアクティブモーダルに限定することで、ゾンビクリック（閉じかけモーダル外DOM操作）を防げる設計パターン
- フォールバック処理のスコープ不備が「エラー処理がさらなるエラーを産む」連鎖を起こす典型例：汎用フォールバックは意図しないDOM要素を操作するリスクを内包する
- デバッグスクリーンショットにタイムスタンプ付き命名規則（category_placement_fail_050131.png）を最初から組み込むことで、失敗箇所の特定コストを大幅削減できる実践的パターン
