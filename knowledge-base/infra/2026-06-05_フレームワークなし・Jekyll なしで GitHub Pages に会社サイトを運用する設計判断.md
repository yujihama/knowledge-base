## 要約

株式会社には毎年決算公告の義務があるが、官報掲載は費用が発生する。会社法 941 条の特例により、決算公告のみであれば電子公告調査機関の調査が不要なため、自社 URL にページを 1 枚置くだけで掲載費 0 円で自前完結できる。この動機から GitHub Pages（mishima-computing.github.io）を採用し、Jekyll 等の静的サイトジェネレータを一切使わず、素の HTML だけで運用する構成を選択した。

Jekyll を使わない理由は、ページ数が二桁未満の会社サイトでは「大量ページを共通テンプレートから量産する」というテンプレートエンジンの主たるメリットが機能しないためだ。ビルドステップを排除することで、リポジトリの HTML が 1 対 1 でそのまま公開物となり、手元で見ているファイルと配信されるコンテンツが完全に一致する。

ヘッダー・フッター・JSON-LD 構造化データは全 HTML ファイルにベタ書きし、JS による動的インクルード（fetch 差し込み方式）も使わない。その理由は、検索エンジンクローラーや LLM/AI エージェントが生の HTML を直接読む際、JS を実行しないケースが多く、動的注入された DOM 片が欠落するリスクがあるためだ。ベタ書きにすることで、1 回の HTTP リクエストで返る HTML にナビゲーション・構造化データがすべて含まれ、JS 非実行環境でも完全なコンテキストが届く。

一方、ベタ書きの代償として共通部分の修正時に全ページを同時に変更する必要がある。これを Python や PowerShell の正規表現による全 HTML 一括置換スクリプトで対応しているが、エンコーディング指定ミスにより BOM 混入・U+FFFD 置換文字が複数ファイルに波及する事故が実際に発生した。

その対策として GitHub Actions で push のたびに全テキストファイルを走査し、①UTF-8 として厳密にデコードできるか、②UTF-8 BOM が先頭に混入していないか、③U+FFFD が本文に出現しないか、の 3 条件をチェックし、1 つでも引っかかればデプロイを止める CI を導入した（Python 標準ライブラリのみで実装）。「ベタ書き → 一括置換 → 事故の全ページ波及 → CI で機械的に食い止める」という設計が一本の論理として繋がっている。独自ドメインは GitHub Pages のカスタムドメイン機能でいつでも移行可能なため、設立当初の OPEX 抑制を優先して取得していない。

## アイデア

- LLM/AIエージェントが JS を実行せず生 HTML を直接読む前提で、動的インクルードを排除し構造化データをベタ書きする設計判断は、エージェント時代のウェブ設計指針として参考になる
- 「共通化をテンプレートエンジンではなく一括置換スクリプトに委ねる」Code as Data 的発想と、その事故リスクを CI のエンコーディング監査で補完する構成は、シンプルな技術選択とリスク管理の組み合わせとして興味深い
- 会社法 941 条の特例（決算公告のみは電子公告調査不要）を活用して GitHub Pages で法的義務を 0 円で自前完結させるという、法律・インフラ・コスト最適化を横断した実用的アプローチ
