## 要約

Claude Codeがトークンを大量消費する根本原因は、Enterを押すたびに会話履歴全体が毎回再送される仕組みにある。50ラウンドのデバッグセッションでは1ターンあたり5〜10万トークンを送信することも珍しくなく、コストは会話の長さに比例でなく指数関数的に増加する。

著者が提案する6つの習慣は以下の通り。

**習慣1: タスクごとにターミナルを分ける**。異なるタスクを同一セッションで続けると、無関係な過去の会話が毎回送信される「context rot（コンテキストの腐敗）」が発生し、回答品質も低下する。新タスクは新ターミナルで開始し、修正ループが5回以上続いたらセッションをリセットする。

**習慣2: /compactを60%で早めに実行する**。auto-compactは残り20%で発動するが、その時点ではすでに高コストのターンを大量に消費済み。60%到達を「赤信号」とし、`/compact`に「Redisを使用（Memcachedではない）」「/api/ordersの500エラーは未修正」などの保持すべき情報を明示的に指示することで、重要情報の脱落を防ぐ。

**習慣3: 必要な部分だけ渡す**。2000行のファイルを丸ごとコピペせず、`Read file offset=120 limit=50`で対象関数周辺50行のみを取得する。ファイルパスのみ渡してClaudeに自力で読ませる方法も有効。スタックトレースは最初の10行（例外と直後5〜10行）のみ貼付することで、トークンを95%削減しても問題は解決できる。

**習慣4: CLAUDE.mdを200行以内に圧縮する**。CLAUDE.mdは全セッションで毎回読み込まれる固定費。2800トークンのファイルを20セッション/日開くと56,000トークンが「作業前に消費」される。箇条書き化（`Stack: React 18 + TS 5 + Vite + Jest`など）と、Claudeがすでに知っている汎用知識の削除により、2800→950トークン（66%減）を達成した実例がある。

**習慣5: .claudeignoreで不要ファイルを除外する**。dist/、.next/、package-lock.json、*.generated.ts、__snapshots__/等をignoreすることで、中規模TypeScriptプロジェクトでは参照ファイル数が45→18に減少し、コードベース探索フェーズで62%のトークン削減が可能。.gitignoreとは独立した別ファイルとして管理する。

**習慣6: /costとccusageで日次監視する**。`npx ccusage@latest`でバーンレートを可視化し、`/cost`を朝夕2回確認する習慣を持つことで、2025年3月に発生したAnthropicのキャッシュ不具合（通常料金の8〜15倍請求、47時間でMax $200クォータ枯渇）を早期検知できたユーザーは全員このコスト確認習慣を持っていた。

監査エージェント開発への示唆: LangGraphベースの長時間エージェントループでも同様のコンテキスト肥大化が発生しうる。セッション管理戦略（コンテキスト圧縮タイミング、不要ツール出力の除外）を設計段階から組み込むことがコスト制御の鍵となる。

## アイデア

- コンテキスト消費がセッション長に対して指数関数的に増加するという定量的な理解は、エージェントループ設計において「ターン数上限」や「中間圧縮ポイント」を設計パラメータとして明示的に扱う必要性を示唆する
- /compactに保持すべき情報を自然言語で明示指定できる仕組みは、エージェントの「ワーキングメモリ管理」をユーザーが制御できるインターフェースとして興味深く、LangGraphのstate管理と概念的に対応する
- 2025年3月のキャッシュ不具合（サイレント障害で8〜15倍課金）は、外部APIへの依存コストが「観測不能な形で急変しうる」リスクを示しており、コスト監視を運用上の必須要件として設計すべきことを裏付ける
