## 要約

個人開発者がAmazonレビューのサクラ判定Chrome拡張「Sakura Score」を開発中、Gemini API（gemini-2.5-flash）に切り替えた結果、ユーザー0人の状態で約1万2,800円の請求が発生。別途AIエージェントツール「OpenClaw」でも約5,000円の請求があり、合計約1万8,000円を1ヶ月で失った事例の詳細報告。

事故の根本原因は4点。①Ollama（ローカルLLM）時代に設計したクローラーがAmazon商品1,152件のASINを30〜60秒間隔で巡回する設計のまま、API呼び出し先だけをGemini APIに変更したこと。結果として1日約561回、月間約2,000回のAPI呼び出しが発生し、出力トークンは3,174万トークンに達した。②Claude等のAIに「無料の範囲か」を確認して「大丈夫」という回答を得たが、AIはGCPコンソールの実際の課金状態を参照できず、その回答は料金表の知識に基づくものに過ぎなかった。③Google CloudコンソールのRPD（Requests Per Day）表示をレート制限と解釈し「使いすぎない仕組み」と誤解したが、RPDはリクエスト数制限であり課金はトークン量ベースで別途発生する。④Gemini APIの呼び出し経路がagent.js（ローカルクローラー）、Supabase Edge Function、Cloudflare Workerの3経路あり、1つを無効化しても他経路からの呼び出しが継続した。

対策として筆者が最も有効と結論づけたのは「残高0のプリペイドカードをGCP請求先に設定する」戦略。残高がなければ物理的に課金不可となるため、設定ミスやAPI暴走があっても損失が発生しない。アラート設定や予算上限は「正しく設定できていれば」という前提が必要だが、プリペイドカードはその前提を不要にする。その他の推奨対策として、ローカルLLMからクラウドAPIへの移行時のチェックリスト（実呼び出し回数の計測→料金試算→最初の1週間は呼び出し量を1/10→月額上限アラート設定）、およびソースコード全体でのAPI呼び出し経路棚卸し（fetch/axiosでAPIのURLを検索し、kill switchの有無を確認）を提示している。

監査エージェント開発への示唆：LLMを組み込んだ自動化パイプラインでは、バックグラウンドで動作するクローラーやエージェントがユーザー操作なしに大量のAPI呼び出しを発生させるリスクがある。監査エージェントでも同様の設計上の盲点（複数呼び出し経路、ローカルLLMからクラウドへの移行時の量的変化）が存在するため、コスト制御のkill switchと使用量モニタリングを設計段階から組み込むことが重要。

## アイデア

- ローカルLLM（Ollama）とクラウドAPI（Gemini）のコスト構造の根本的な違いが、API呼び出し量の設計思想をそのまま引き継ぐと事故につながるという構造的問題
- AIに「無料か」を確認しても、AIはリアルタイムのGCPコンソール状態にアクセスできないため保証にならないという、AIへの過信リスクの具体例
- 複数の呼び出し経路（ローカルクローラー・Edge Function・Workers）の存在により、1箇所を止めても他経路から継続してAPIが呼ばれるという分散アーキテクチャの盲点
