## 要約

約50名規模の企業でデータ主権を確保しつつ社員全員が使える社内LLM基盤を構築した事例。最大の制約はCGNAT（キャリアグレードNAT）による二重NATで、インバウンドのポート開放が原理的に不可能な環境だった。

Cloudflare Tunnelは候補から除外。理由は①SSEストリーミングがレスポンス終了までバッファされLLMのトークン逐次表示が壊れる、②約100秒のアイドルタイムアウトで長い推論が切断される、③トラフィックが第三者経由となりデータ主権方針に反する、の3点。

最終構成は「AWS EC2（Elastic IP直結）+ Pangolin/Traefik（TLS終端）+ Newt/WireGuard（アウトバウンドトンネル）+ LiteLLM + Open WebUI + 社内vLLM」。EC2はALBを置かずElastic IP直結とし、ALBのidle timeoutによる長応答切断を回避。管理アクセスはSSM Session Managerに集約してインバウンドSSH（22番）をゼロにした。外部入口のコストは月$15〜20程度。

設計の核心は認証の二元化。chat.example.comはGoogle OIDC認証（人・ブラウザ用）、api.example.comはLiteLLM仮想キーのBearerトークンのみ（Claude Code / Clineなどのボット・SDK用）とホスト名で分離。SSOで全経路を包むとOIDCログイン画面をたどれないSDKが動作不能になるため必然的な設計。

LiteLLMはOpenAI形式とAnthropic形式を同時提供し、チーム別にRPM（general:10、dev:20、apps:30）とTPMを設定。pre-call hookでシステムプロンプトを全リクエストに注入し、OpenAI系はmessages先頭にsystem roleを挿入、Anthropic系はトップレベルのsystemフィールドに付加する実装で形式差を吸収。

Open WebUIのリブランディングでは、WEBUI_NAME設定だけでは「(Open WebUI)」接尾辞が強制付与されるため、ビルド時にenv.pyをsedでパッチ。ロゴ変換はマルチステージDockerビルド内でlibrsvg2-binを使いSVG→PNG変換することでホスト依存ゼロを実現。

ハマりポイントとして、コンテナのlocalhostはホストを指さない問題、vLLMが存在しないモデル名を404で拒否する問題、docker compose up -dでconfig.yaml変更が反映されずrestartが必要な問題、DNSホスト名のアンダースコアでLet's Encrypt証明書が発行されない問題、thinkingモデルでmax_tokensが小さいと本文が空になる問題を詳述。

監査エージェント開発への示唆：人とボットの認証経路分離はLangGraphベースのエージェントやAPIクライアントをLLM基盤に接続する際にも同様に必要な設計パターン。LiteLLMのチーム・クォータ管理はエージェントの暴走制御にも応用できる。

## アイデア

- CGNAT制約が逆に「インバウンド0・アウトバウンドのみ」という堅牢な構成を強制し、結果的にセキュリティ上優れたアーキテクチャに帰着した点
- 人用（OIDC）とボット用（APIキー）をホスト名で完全分離する認証二元化パターンは、Claude CodeやClineなどのコーディングエージェントと社内SSO環境を共存させる汎用的な設計として応用可能
- LiteLLMのpre-call hookでOpenAI形式とAnthropic形式の差異を吸収しつつシステムプロンプトを注入する実装は、マルチモデル環境での統一的なポリシー適用の参考になる
