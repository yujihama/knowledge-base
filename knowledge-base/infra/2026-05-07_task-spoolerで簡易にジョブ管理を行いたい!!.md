## 要約

task-spooler（コマンド名: tsp）は、Linuxローカルマシン上で機械学習ジョブを手軽にキュー管理するためのCLIツール。SlurmやKubernetesのような大規模クラスタ管理システムを導入するほどではない個人・小規模環境向けに設計されており、`sudo apt install task-spooler`で即座に利用可能。

基本的な使い方は `tsp <command>` でジョブをキューに投入するだけ。たとえば `tsp bash train.sh` のように学習スクリプトを渡せば、前のジョブが終わり次第自動的に順番実行される。同時実行数の制御には `tsp -S N`（Nを任意の整数に設定）を使う。GPUが1枚しかない環境でのシリアル実行（N=1）に特に有効。

ジョブ管理コマンドは以下の通り: `tsp`または`tsp -l`で一覧表示、`tsp -s <job_id>`でステータス確認、`tsp -k <job_id>`でキャンセル、`tsp -K`でサーバごと停止（全ジョブ破棄）、`tsp -C`で完了履歴の削除（IDカウンターはリセットされないため注意）。

出力追跡は2系統ある。`tsp -t [job_id]`はtail -f相当でジョブ実行中からリアルタイムにストリーミング表示しジョブ終了までブロックする。`tsp -c <job_id>`はcat相当でジョブ完了後に全出力を一括表示する。`tsp -o <job_id>`で出力ファイルパスを取得できるため、外部ツールで加工することも可能。

記事後半ではLinuxの実行権限（パーミッションビット）とshebangの仕組みも解説。`bash script.sh`はbashプロセスに引数としてテキストを渡すため読み取り権限（r）のみ必要。`./script.sh`はカーネルがファイル自体をプログラムとして起動しようとするため実行権限（x）が必要で、さらにshebang（例: `#!/bin/bash`）がないとカーネルがインタープリタを特定できず失敗する。tspでシェルスクリプトを直接渡す場合は`tsp bash script.sh`形式が権限設定不要で手軽。

監査エージェント開発への示唆: 複数の評価実験（LLM-as-judgeのバリアント比較、RAGパラメータのグリッドサーチ等）をローカルRTX 3090環境でシリアルに流す際、task-spoolerは設定ゼロで使えるため実験管理の最初の一手として有効。Slurmほどのオーバーヘッドなくジョブの出力ログを永続保存できる点も、実験の再現性確保に寄与する。

## アイデア

- tsp -S 1 でGPU 1枚環境のシリアル実行を保証できる。SlurmなしでもVRAM競合を防ぎながら複数学習ジョブを安全に直列実行できる
- tsp -t でリアルタイムログをストリーミングしつつ、tsp -o でログファイルパスを取得してtensorboardや外部モニタリングツールと連携する構成が作れる
- bash script.sh と ./script.sh の実行権限の違いという基礎知識が、Dockerコンテナ内やCI環境でのシェルスクリプト実行トラブルの根本理解につながる
