## 要約

Claude Code のグローバル指示ファイル（~/.claude/CLAUDE.md と ~/.claude/rules/）の整理方針についての実践的なメモ。著者が至った結論は「CLAUDE.md には開発哲学のみ、詳細手順は ~/.claude/rules/ に分離する」という役割分担である。

背景として、Claude Code の公式仕様では CLAUDE.md の読み込みは「最初の200行または25KB」に制限されており、それを超えると後半が足切りされる。そのため CLAUDE.md を膨らませることは逆効果で、重要な根本原則が埋もれてしまうリスクがある。

~/.claude/rules/ には YAML frontmatter の `paths` フィールドを使ったスコープ限定機能があり、特定のファイルパターンに一致する作業時のみルールが適用される。これによりトークン・コンテキストの無駄消費を防ぎ、必要な時にのみ必要なルールを注入できる。

著者の CLAUDE.md は大きく3セクションで構成される。①Foundation（Outcome-driven / Backcasting）：アウトカムを「行動・時間・エラー率・価値の状態」と定義し、成果物完成やタスク完了とは区別する。Backcasting では理想の終端から逆算して最小経路を導く手順（Goal→Gap→Path）を定める。②Rules：Response（結論先出し）、Verify（事実は出典引用）、Anti-sycophancy（同意前に検証、正確性優先）、Debug（3つ以上の仮説を立て消去法で特定）の4ルールをテーブル形式で明記。③Completion：機能追加・修正・調査・変更不要の各タスク種別ごとに「必要」「不十分」をテーブルで定義し、根本原因の解決と正常ケースの理解を必須とする。

Anti-sycophancy ルールを設ける動機として、Claude Code が「最小変更に寄りがちな傾向」＝社交的配慮の一形態として LLM が大きな変更をユーザーが嫌がると推定し負担の小さい解に寄る問題を挙げている。実際のファイルはトークン圧縮のため英語で記述されており、Table や List を多用して構造化している。

監査エージェント開発への示唆：Anti-sycophancy ルールと Completion の「症状パッチより根本原因解決」という原則は、監査エージェントが表面的な異常検知にとどまらず根本的なリスク要因を特定・報告するよう設計する際の指針として直接応用できる。また `paths` スコープ限定機能は、監査フロー・データ取得・レポート生成など工程ごとに異なるルールを適用するエージェント設計のアナロジーとして参考になる。

## アイデア

- CLAUDE.md の200行/25KB制限という公式仕様を逆手に取り、ファイル肥大化を防ぐために役割を哲学層（CLAUDE.md）と手順層（rules/）に明示的に分離するアーキテクチャ
- ~/.claude/rules/ の paths フィールドによるスコープ限定機能を活用し、ファイル種別・ディレクトリごとに適用ルールを切り替えることでコンテキスト効率を最大化する設計
- LLM の「最小変更バイアス（社交的配慮）」を Anti-sycophancy ルールで明示的に抑制し、アウトカム優先の行動を Completion テーブルで定義する構造化アプローチ
