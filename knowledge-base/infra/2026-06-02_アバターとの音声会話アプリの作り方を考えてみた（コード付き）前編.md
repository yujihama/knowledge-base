## 要約

Azure Speech SDK を活用したリアルタイム音声会話Webアプリの構築方法を解説する記事の前編。ユーザーの発言をSTT（Speech to Text）でリアルタイムに文字起こしし、LLMに投げて回答を得るまでの部分を扱う。後編ではTTSとAzureのアバターサービスとの連携を扱う予定。

アーキテクチャは2パターンを比較検討している。パターン1は「音声入力→STT→LLM→TTS→音声出力」の分離構成で、各コンポーネントを独立制御でき、カスタマイズ性が高くデバッグしやすい反面、構成が複雑になる。パターン2はマルチモーダルモデルによる一体型で、シンプルだが細かい制御が難しい。現時点での導入事例の多さからパターン1を採用。

全体の通信フローは以下の通り。①クライアントがWebサーバーにHTTPリクエスト→②マイク音声をWebSocket経由でAzure STT（japaneastリージョン）に直接ストリーミング送信→③文字起こし結果をバックエンドのLLM APIに送信→④LLM回答をWebRTC経由でAzure TTS（eastus2リージョン、アバター機能対応）に直接送信しアバターに発話させる。レイテンシ低減のため音声サービスへの接続はバックエンドを介さずクライアントから直接行う設計。

動作確認用の最小構成では、Azure Portalで音声サービスリソースを作成し、microsoft.cognitiveservices.speech.sdk.bundle.jsをダウンロード。SpeechConfig.fromSubscription()でAPI Keyを直接指定し、AudioConfig.fromDefaultMicrophoneInput()でマイク入力を設定。SpeechRecognizerインスタンスのrecognizingイベントで途中結果、recognizedイベントで確定結果を取得し、startContinuousRecognitionAsync()でストリーミング認識を開始する。

本番運用向けにはAPI Keyの直書きを避けるため、Flask（app.py）でトークン発行エンドポイント(/api/token)を実装する方式に切り替える。バックエンドがAzure Cognitive Services の/sts/v1.0/issueTokenエンドポイントにOcp-Apim-Subscription-Keyヘッダー付きでPOSTし、返却されたトークンをクライアントに渡す。クライアント側ではSpeechConfig.fromAuthorizationToken()に切り替える。デプロイはAzure App ServiceへのZIPデプロイを採用。

監査エージェント開発への示唆として、このSTT→LLM→TTSパイプラインのパターンはリアルタイム音声インターフェースを持つ監査支援ツール（ヒアリング自動文字起こし・回答生成等）への応用可能性がある。WebSocketによる低レイテンシストリーミングと、バックエンドAPIへの認証トークン委譲パターンは実装上の参考になる。

## アイデア

- 音声サービスへの接続をバックエンド経由ではなくクライアントからWebSocket/WebRTCで直接行うことで、データ転送量が多い音声ストリームのレイテンシを最小化する設計判断
- API Keyを直書きする動作確認フェーズから、バックエンドがトークンを発行してクライアントに渡す本番方式へ段階的に移行するセキュリティ設計パターン
- TTS+アバター連携可能なAzureリージョンがeastus2に限定されているという制約から、STTとTTSで別リージョンを使い分けるマルチリージョン構成の必要性
