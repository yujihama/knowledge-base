## 要約

本記事は、BYOC（Bring Your Own Credential）の考え方を軸に、ユーザーが保有するLLMモデル利用権限をOAuthで3rd-partyアプリに委譲することの技術的実現可能性を考察したものである。

AWS Bedrockを例に取ると、3rd-partyアプリがユーザーのBedrockにアクセスするには、CognitoやSTS、IAM RoleのARN、AWSアカウントID、リージョン情報をユーザーから取得する必要があり、ユーザー側のセットアップ難易度が非常に高い。IdP（Identity Provider）を3rd-partyアプリ側で用意するパターンでも、IAMロールのAssumeRoleポリシーにIdPのissuer URLを埋め込む作業がユーザーに残り、専門知識が必須となる。Azure OpenAIの場合はEntraIDを活用するが、AOAIインスタンスURLの共有など同様の課題がある。Google CloudのWorkload Identityも構造的には同様で、ProjectIDの提供が必要になる。

一方、LLMモデルベンダ自身がOAuth認可サーバーとなるケースを検討すると、OpenAIはGPT ActionsとMCPのOAuth認証をサポートしているが、いずれも「OpenAIが外部APIを呼ぶ」方向であり、「3rd-partyがOpenAIを呼ぶ」方向のOAuth委譲には対応していない。AnthropicはClaude Code Docsの利用規約で、OAuthは自社ネイティブアプリ向けであり、3rd-partyアプリがOAuthトークンを用いてモデルを呼ぶことを明示的に規約違反と定めている。

2026年5月時点では、主要LLMベンダは3rd-party向けOAuth認可を公開しておらず、実質的にはAPI Keyをユーザーに発行してもらう方式が唯一の現実解となる。

記事後半ではOAuthの優位性自体を再検討している。LLMのBYOCにおいてはスコープの多様性がなく（呼び出し1点のみ）、API Keyをアプリごとに発行すれば利用量の識別も可能であり、不要になれば即時無効化できるというAPI Keyの利点もある。OAuthのRevoke機構はリフレッシュトークンの無効化を経由するため即時性がなく、むしろ管理コストが高い面もある。最終的に、認証方式の選択は「そのサービスがクレデンシャルを適切に管理・保護できるか」と「ユーザー自身が必要時にRevokeできるか」に帰着するという結論を導いている。監査エージェント設計においても、外部API権限の委譲設計（BYOC vs API Key管理）は重要なアーキテクチャ判断点となりうる。

## アイデア

- AnthropicはOAuthを自社ネイティブアプリ（Claude Code等）専用と規約で明記しており、3rd-partyアプリのOAuth利用は規約違反とされている点は、Claude Codeベースのエージェント開発において認証設計の制約として必ず把握すべき事実
- OAuthのRevoke機構は「リフレッシュトークンの次回更新時に無効化」であり即時性がないため、即時停止が必要なセキュリティ要件ではAPI Keyの方が優位という逆転の発想は、監査システムでのアクセス制御設計に応用できる
- LLMモデル呼び出しはスコープが単一（モデルcallのみ）であることから、OAuthの多スコープ管理の利点が活かせず、API Keyで十分なユースケースが多いという指摘は、過剰設計を避けるアーキテクチャ判断の参考になる
