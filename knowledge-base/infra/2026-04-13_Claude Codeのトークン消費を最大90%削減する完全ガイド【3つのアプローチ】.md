## 要約

Claude Codeを本格的に使用すると、Pro/Maxプランでも使用上限に頻繁に達する。その主因は「Claude自身の冗長な出力」「コマンド出力の肥大化」「CLAUDE.mdの過大化」の3つで、合計200,000トークン/30分セッションの約74〜90%が最適化可能。

**アプローチ1: genshijin（原始人口調）プロンプト最適化**
英語圏の「caveman coding」手法を日本語向けに最適化したメソッド。日本語固有の構造的冗長性（敬語体系・助詞連鎖・クッション言葉・ぼかし表現）を排除する。BPEトークナイザでは「設定されています」が5〜6トークン消費するのに対し「設定」なら1〜2トークンで済む。実測では通常モード1,483トークンに対し、genshijinモードで296トークン（80%削減）を達成。CLAUDE.mdに体言止め・短文・冗長表現圧縮のルールを追加するだけで導入可能（5分）。

**アプローチ2: RTK（Rust Token Killer）**
Rust製シングルバイナリのCLIプロキシ。スマートフィルタリング・グループ化・トランケーション・重複排除の4戦略でコマンド出力を60〜90%圧縮する。`rtk init -g`でClaude CodeのPreToolUseフックを設定すると、以降は`git status`等のBashコマンドが透過的にRTK経由に書き換わる。特にテストランナー出力が90%削減されるのが大きく、30分セッションで118,000→23,900トークンに圧縮（80%削減）。Claude Code以外にCursor・Windsurf・Copilot等10種にも対応。

**アプローチ3: CLAUDE.md・設定最適化**
CLAUDE.mdはセッション毎に全行読み込まれるため直接的なコスト要因。200行以内を目安に、詳細説明は別ファイル参照・箇条書き化・不要ルール削除で40%削減可能。`~/.claude/CLAUDE.md`（全プロジェクト共通）と`./CLAUDE.md`（プロジェクト固有）の階層分離も効果的。`.claudeignore`でnode_modules等を除外。さらに`/compact`コマンドをコンテキスト60%時点で能動的に使用する（95%の自動compact待ちは出力品質劣化を招く）。Extended Thinkingはデフォルト31,999トークンを`MAX_THINKING_TOKENS=10000`に制限することで70%削減、サブエージェントを`CLAUDE_CODE_SUBAGENT_MODEL=haiku`に変更するだけで約80%削減できる。

3手法を組み合わせると、同一セッションで約4倍の作業量を実現。トークン削減はコスト節約だけでなく、コンテキストウィンドウへの情報密度向上により応答品質自体も改善される効果がある。

## アイデア

- BPEトークナイザの特性を逆手に取り、日本語の文体制御（敬語・クッション言葉排除）だけで80%のトークン削減を実現するgenshijinアプローチは、LLMの出力制御をコスト工学として捉えた実践的発想
- PreToolUseフックを使ってBashコマンドを透過的にRTKプロキシ経由に書き換える仕組みは、エージェントのツール使用パターンに割り込むミドルウェア設計として監査エージェントのログ圧縮・フィルタリングにも応用可能
- /compactをコンテキスト95%の自動発動ではなく60%時点で能動的に使用する戦略は、コンテキストウィンドウを「消耗品」ではなく「管理リソース」として扱うエージェント設計思想と一致する
