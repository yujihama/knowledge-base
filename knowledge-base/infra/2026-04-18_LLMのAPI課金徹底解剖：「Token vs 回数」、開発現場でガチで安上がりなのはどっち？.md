## 要約

LLM APIの課金モデルには主に2種類ある。「Token課金（従量課金）」はOpenAI・Anthropic・Googleなど世界標準で、入出力のToken数に応じて課金される（$0.001〜$0.01/1k Token）。「回数/リクエスト課金」は中国系BaaSや法人向け定額プランが主で、ペイロード量によらずAPI呼び出し回数で課金される（$0.001〜$0.005/回）。各プロバイダーは「平均1リクエスト＝800Token（入力300＋出力500）」を損益分岐点の基準として料金表を設計しており、平均的な使い方では両モデルのコストはほぼ同等になるよう意図的に調整されている。コスト差が生まれるのは「平均から逸脱したユース」。100Token以下の短文対話が多い場合はToken課金が圧倒的に有利（回数課金は最低単価が固定されるため割高）。8000Token以上のRAG・長文処理が多い場合は回数課金が有利だが、多くのプロバイダーが「1回2000Tokenまで」等の隠れ上限を設けているため注意が必要。回数課金は予算が固定でき稟議を通しやすいが、短文の大量処理ではトークン当たりコストが高騰し、スロットル制限によるサービス停止リスクもある。Token課金はRAG・エージェント等の可変長入力に柔軟対応できるが、バグによる無限ループや悪意あるユーザーによる巨大入力で予算が青天井になるリスクがある。対策としてOpenAI公式の`tiktoken`ライブラリを使ったリクエスト前Token数チェック（ガードレール）が有効で、`cl100k_base`エンコーディングでローカル計算し、閾値超過時にAPI呼び出し自体をブロックする実装例が示されている。監査エージェント開発への示唆：LangGraphやReActベースのエージェントは入力長が不定なためToken課金が適切だが、tiktokenによる事前Token数計算をツール呼び出し前に挟むことでコスト爆発を防ぐ設計が推奨される。

## アイデア

- 各プロバイダーが「平均800Token/リクエスト」を損益分岐点として料金設計しており、平均的使用ではToken課金と回数課金のコストが意図的に均等化されているという業界構造の分析
- tiktokenによるリクエスト前ローカルToken数計算をガードレールとして使い、Token課金の予算爆発リスクをAPI呼び出しより手前でブロックする設計パターン
- 回数課金の「隠れ上限（1回2000Tokenまで等）」がRAGや長文処理の実用性を制約するという、定額制の見えにくいトレードオフ
