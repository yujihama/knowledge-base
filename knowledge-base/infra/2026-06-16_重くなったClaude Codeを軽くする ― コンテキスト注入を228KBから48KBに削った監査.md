## 要約

Claude Codeのセッションが重くなり直近指示を取りこぼす症状が発生。原因を調査した結果、SessionStart時に注入されるコンテキスト総量の肥大化が原因と判明した。重要な切り分けとして、「セッションが重い」原因は有効プラグイン数であり、「指示を取りこぼす・話が混線する」原因は注入総量（バイト数）であることが明確化された。この2つを混同してプラグインを削っても、ボケる問題は解消しない。

注入源の実測手順として、rules/配下・auto-memory（MEMORY.md）・remember履歴（now.md/recent.md/archive.md）をそれぞれwcコマンドでバイト計測する方法を紹介。調査の結果、~/.claude/rules/ecc ディレクトリ単独で約228KB（≒57Kトークン）を占めており、全注入量の中で群を抜く最大ブロックであることが判明した。

rules/配下は@importによる遅延ロードではなくディレクトリ全体がグローバル指示として毎セッションフルロードされる仕様が盲点だった。当該ディレクトリには汎用ルールパック由来でangular/golang/swift/php/ruby/arkts/java/kotlinなど10言語分のコーディング規約が含まれていたが、実際に使用するのはPython/TypeScript/Webの3言語のみだった。

修正方法は非常にシンプルで、使用しない言語のディレクトリをツリー外（~/.claude/.rules-archive/ecc）にmvするだけ。import行の編集は不要で、復元もmvで戻すだけの完全非破壊操作。結果として228KB→48KB（約79%削減、57Kトークン→12Kトークン）を達成し、起動高速化と指示取りこぼしの減少を確認した。

落とし穴として、Obsidianのhot.md（32KB）を削ってもトークンは減らなかった事例を紹介。注入スクリプト側で9000字/2500字と上限カットされており、ファイルサイズに関係なく注入量は約7Kトークンで一定だったため。「ファイルが大きい＝注入が重い」は成り立たず、必ず注入源を実測することが重要。

プラグイン管理においては、settings.jsonのenabled: false（可逆）とclaude plugin uninstall（非可逆）を使い分け、まずdisableで様子を見ることを推奨。MAXプランでもコスト削減以外にコンテキスト窓節約・キャッシュヒット率安定化・直近指示の埋没防止という実質的なメリットがある。

## アイデア

- 「重い」と「ボケる」は原因が別物という切り分け：プラグイン数がレイテンシの原因、注入総量が指示取りこぼしの原因であり、混同すると誤った対処（プラグイン削除）をしてしまう
- rules/ディレクトリはディレクトリ全体のフルロードで動作するため、@importによる遅延ロードと異なり使わない言語のルールが毎回全注入される盲点がある
- ファイルサイズと実際の注入バイト数は一致しない：注入スクリプト側で上限カットされている場合、ディスク上のファイルをいくら削除してもトークン削減にならない
