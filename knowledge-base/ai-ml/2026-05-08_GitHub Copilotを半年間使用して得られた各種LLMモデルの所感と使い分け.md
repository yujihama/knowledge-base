## 要約

北海道札幌の株式会社マーベリックスのエンジニアが、GitHub Copilotを実務に導入してから半年間の経験をもとに、各LLMモデルの特性と最適な使い分けをまとめた実践レポート。

CopilotのAuto（自動選択）機能については、タスク難易度に対して過剰なモデルが選択されトークン消費が想定以上になるとして、手動切り替えを推奨している。

Claude系統（Anthropic）は最も使用頻度が高く扱いやすいと評価。Claude Sonnet 4.6はトークン消費量と出力コード品質のバランスが優秀でメインモデルとして採用されたが、レスポンス速度の遅さとタイムアウトが課題。Claude Opus 4.6は複雑な要件定義や大規模コード解析に有用で論理的な出力が得られる一方、トークン消費量がSonnetの約3倍であり月間制限に達しやすい。Claude Haiku 4.5はトークン消費が少なく、具体的なコード修正指示など軽微な日常作業に最適とされた。

GPT系統（OpenAI）は全体的にレスポンスが早く安定しており、GPT-5はClaudeでタイムアウトするような重い処理でも完遂できる点が優秀。コードの文脈理解や使い勝手ではClaudeに劣ると評価されている。GPT-4.1とGPT-4oは無料枠として実用的で、トークン枯渇時の代替手段として活用。

Gemini系統（Google）はコード生成目的での使用実績がなく、テキスト・画像処理やドキュメント作成補助での活用を今後の検証課題としている。

最終的な運用方針として、メイン作業にSonnet 4.6、軽微修正にHaiku 4.5、大規模解析にOpus 4.6（残量注意）、タイムアウトリスクのある重い処理にGPT-5、トークン枯渇時にGPT-4.1/4oという階層的な使い分けを確立。監査エージェント開発の観点では、複数LLMを役割・コスト・速度に応じて使い分けるオーケストレーション設計の実例として参考になる。

## アイデア

- モデル選択をタスク難易度・コスト・速度の3軸で体系化した実務的フレームワーク：品質重視（Opus）、バランス（Sonnet）、速度/コスト（Haiku/GPT）という階層設計はエージェント内のLLMルーティング設計に直接応用できる
- Claudeのタイムアウト問題とGPT-5のフォールバック戦略：信頼性要件の高い処理でClaudeが失敗した場合にGPT-5へ自動切り替えするフォールバック機構は、マルチLLMエージェント設計における可用性向上のパターンとして有効
- トークンコストを意識したモデルオーケストレーション：Opusのトークン消費がSonnetの約3倍という具体的な比率は、月間予算制約下でのモデルスケジューリングアルゴリズム設計の基礎データになる
