## 要約

JQaRAデータセットを用いたローカルRAGスタック（検索：ruri-v3、生成：qwen3:32b、判定：gemma4:31b）の評価において、「grounded-but-wrong（忠実度≥0.8かつcontext_recall<0.5）」に100クエリ中33件がフラグされた。著者は当初、hybridサーチ（BM25+dense）で改善を試みたが、実装前に天井値計算とランク分布分析を実施した結果、問題の本質がシステムではなくメトリクスの設計にあると判明した。

ID-based context recallの計算式は |retrieved∩relevant_doc_ids| / |relevant_doc_ids| であり、分母に「全関連文書数」を置く。JQaRAはクエリあたりの平均関連文書数が10件超（フラグされた33件では平均16.0件）であるのに対し、検索ではk=5しか取得しない。この場合、達成可能な最大値は5/16≈0.31となり、完璧な検索器でも閾値0.5を超えられない構造になっていた。

hit@5（top-5に関連文書が1件以上含まれるか）で確認すると98/100が合格し、検索自体は正常に機能していた。フラグされた33件をhit@5で再判定するとgrounded-but-wrongは33→0件に消滅した。真の検索失敗（hit@5=0）は2件のみで、いずれもfaithfulness=0.0であり、そもそもgrounded-but-wrongに含まれていなかった。

この罠の本質は単一の設計判断ではなく「組み合わせ」にある：①分母を関連文書数とするID-based recall、②クエリあたり関連文書数が多いデータセット、③小さいk、④分母サイズに関係なく一律の閾値0.5——どれも単独では合理的だが、組み合わさるとシステム品質ではなくデータセット構造を反映した偽の失敗率を生み出す。

教訓として著者は3点を提示する：(1)多答案データセットでは「関連文書が取得できたか」にhit@kを使いproportion recallはk≥典型的関連文書数の場合に予約する、(2)閾値は達成可能な上限に対して正規化する、(3)「完璧な検索器でもフラグが立つか」のoracleチェックを事前に行う。なお、hybridサーチ実験は欠陥のある数字を修正するために走らせることを避け、アーカイブ済みとしている。監査エージェント開発においても評価指標の選択とデータセット特性の整合性確認は重要であり、特に多正解・多根拠が存在するケース（複数証跡から結論を導く監査シナリオ等）ではhit@kベースの評価設計が有効。

## アイデア

- ID-based context recallは分母が関連文書数になるため、多答案データセット（平均関連文書数>k）では構造的に閾値を超えられない——メトリクスとデータセットの「組み合わせ」が偽の失敗率を生む
- 完璧な検索器を仮定した場合の達成可能最大値（ceiling）を事前計算するだけで、メトリクスの設計ミスを実験開始前に検出できる
- faithfulness≥0.8かつcontext_recall<0.5という複合条件フィルタは、評価指標の定義域がデータセットと合っていない場合に全く異なる集合を誤って特定してしまう典型例
