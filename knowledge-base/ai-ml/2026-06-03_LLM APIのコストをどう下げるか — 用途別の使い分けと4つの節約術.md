## 要約

2026年6月初旬時点のLLM API料金を整理し、コスト最適化の実践手法をまとめた記事。経済型モデル（DeepSeek V4 Flash、小米MiMo-V2.5）と旗艦モデルでは出力単価が数十倍〜100倍前後異なり、この差をどう活かすかが核心となる。

**料金の特徴**: DeepSeek V4 FlashとMiMo-V2.5のキャッシュ命中時の入力単価は$0.0028/100万トークンとほぼ無視できる水準。同一システムプロンプトを繰り返し送る構成では実コストに大きく影響する。

**3層モデル選択戦略**: (1)分類・要約・情報抽出・チャット一次応答などの高頻度定型タスクは経済型モデルに回す、(2)コード生成・多段推論タスクのみ旗艦モデルや推論モデルを使用、(3)リアルタイム性不要なバッチ処理はBatch APIに流す。デフォルトを経済型にすることで月コストが体感で大きく変わると報告している。

**4つの実践手法**: 
1. **複雑度ルーティング** — リクエストの7〜8割は経済型で十分。タスク複雑度に応じて振り分けることで品質を保ちつつコスト削減。
2. **Prompt Caching活用** — OpenAI/AnthropicはキャッシュでR50%引き、DeepSeek/MiMoは$0.0028/100万トークン。繰り返し送る部分が多いほど効果大。
3. **Batch API利用** — OpenAI/Anthropicのバッチ実行は50%引き。夜間の集計・分類タスクをまとめて処理することでコストが半減。
4. **推論トークンの計測** — 推論モデルは内部で大量の推論トークンを消費し、最終出力に現れなくても課金される。APIレスポンスのusage.completion_tokensで実消費量を確認し、必要なときのみ推論モデルを使う。

**品質とコストのトレードオフ**: 安いモデルで解けずに再試行を繰り返すと、トークンも時間も消費する。「1回で解ける高いモデル」の方がトータルで安い場合もあり、料金表だけでなく成功率も考慮が必要。監査エージェント開発への示唆として、文書分類・情報抽出などの定型処理は経済型モデルで十分対応でき、複雑な多段推論（LangGraph等のエージェントループ内での判断）にのみ旗艦モデルを充てる設計が費用対効果を最大化する。

## アイデア

- 複雑度ベースのルーティング設計: リクエストの7〜8割は経済型モデルで処理可能という経験則は、マルチエージェントシステムにおけるモデル選択の自動化（難易度スコアリング→モデル割り当て）に応用できる
- キャッシュ命中率の最大化設計: DeepSeek/MiMoの$0.0028/100万トークンというキャッシュ価格は、システムプロンプトの設計（長い固定プロンプト＋短い変動部分の分離）を変える動機になる
- 推論トークンの不可視コスト問題: 思考モード付きモデルはusageに現れる推論トークンで課金されるが、これはエージェントのループ内で推論モデルを使う際の隠れたコスト爆発リスクとして重要
