## 要約

Claude Opus 4.8が提供する「effort」パラメータ（low / medium / high / xhigh / max）の実際の効果を、3つのサーフェス（Claude Code / Cowork / claude.aiチャット）×5レベルの計27runで実測した検証記事。

検証課題は2つ。Task A：SemVer 2.0.0準拠のsemver_compare()をPythonで実装し、隠しテスト15項目で採点。Task B：ベイズ問題（感度99%・偽陽性5%・有病率1%→陽性的中率≈16.7%）を150字以内で回答。

主な結果は以下の通り。（1）正解率はeffortを上げても変わらない：全27runでTask A 15/15パス、Task Bも数値・分解とも正解。（2）出力トークンはlow→maxで約7倍（Task A: 429→3,100、Task B: 166→1,100）、API処理時間は最大6倍に増加。（3）150字制約の遵守はeffortに比例しない：maxのみ3サーフェス全てで遵守したが、low〜xhighはバラツキが大きく、effortを上げれば指示を守るという期待は成立しない。

サーフェス差も顕著。Cowork（エージェント環境）では高effortで自律的な品質保証行動（xhigh相当で回答前に自前の全順序テストを実行して「検証済み」と宣言）が発生。Chatでは低〜highは数式ブロックが一字一句同じテンプレート回答で体感5秒、xhighから挙動が変化。Claude Codeは「コードのみ」指示への忠実度が最も高い。

ultracode（xhigh + workflows、別実行モード）では8サブエージェントが2フェーズ並列実行（独立実装×3＋敵対的テスト×2生成→83ケース検証）。出力トークンはxhigh単発の36倍（58,100）、API時間は約30倍（10分48秒）に対し、隠しテスト成績はlow単発と同じ15/15。ultracodeが買うのは「正しさの検証プロセス」であり、コードの品質自体ではない。

実務的示唆：仕様が明確な小〜中タスクはlow/mediumで十分、指示遵守はプロンプト強調＋出力検証で担保すべき、ultracodeは隠れバグが致命的な場面限定。監査エージェント開発においては、effortによる「答えの正しさ」の向上は期待できないため、ツール呼び出しの精度や制約遵守はシステム側の検証ロジックで担保する設計が重要。またultracodeのような多エージェント並列検証パターンは、監査証跡の信頼性が求められる場面（高リスク判定、例外処理）での活用候補となる。

## アイデア

- effortを上げても正解率は変わらず、変化するのは「答えの確かめ方と作法」（検証プロセスの丁寧さ）である点——LLM性能の評価軸として「正しさ」以外の次元が実在することを実測で示した
- 150字制約のような出力形式の遵守はeffortに単調比例せず、サーフェス×effortの組み合わせで再現性のある違反パターンが出る——制約はプロンプト強調か外部検証で担保すべきという設計原則の実証
- ultracodeの多エージェント並列検証（36倍トークン・30倍時間）は正解率を上げないが「正しさの検証プロセス」を生成する——監査領域で求められる証跡・根拠の生成に特化した活用パターンとして参照価値がある
