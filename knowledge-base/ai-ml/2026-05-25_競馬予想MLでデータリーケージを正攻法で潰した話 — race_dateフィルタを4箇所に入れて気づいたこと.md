## 要約

時系列MLにおけるデータリーケージ（未来情報の混入）を、競馬予想ツールUmaScore（Python + SQLite + LightGBM）の実装を題材に体系的に整理した記事。リーク経路を4つに分類し、それぞれに対する対策を具体的なSQLコードとともに解説している。

【4つのリーク経路と対策】
1. 走力指数（Base_Ability_Index）の集計：過去5走を算出する際、`AND race_date < ?`を付けないと予測対象レース以降の走破タイムが混入する。`ORDER BY race_date DESC LIMIT 5`のようなソート＋LIMIT系クエリは特に注意が必要。
2. 適性スコア（Aptitude_Score）のコース実績集計：会場・距離±200mの空間フィルタだけでは不十分で、時間軸フィルタ（`race_date < ?`）を独立して追加する必要がある。空間軸と時間軸は直交しており、両方の明示が必須。
3. 血統経由の集計：コース実績不足の馬（新馬・転厩等）を種牡馬産駒の成績で補完する際、`past_races`と`horses`をJOINするが、JOIN先の`past_races.race_date`にも`pr.race_date < ?`を付けなければならない。メインルートでフィルタを入れても、fallbackルートには同じガードが自動適用されないため見落とされやすい。
4. 推論時の呼び出し側での渡し忘れ：関数がrace_dateを受け取れても、呼び出し側で渡さなければ全期間集計になる。対策として、パイプライン入口で`races.race_date`を1回だけ取得し、`calculate_base_ability_index`・`calculate_aptitude_score`等の各スコア関数に統一的に渡す設計を採用。

【効果の計測】2025年1月〜2026年4月の16ヶ月、4,409レース・61,036出走でバックテストを実施。フィルタ適用後、EV≥1.5戦略で回収率118.4%（16ヶ月中10ヶ月プラス）に落ち着いた。重要な知見として「フィルタ適用で数値が下がること自体が正常」という点を強調しており、数値が変わらない場合はさらに別経路のリークが残っている可能性を疑うべきとしている。

【一般化できる規律】①集計クエリのWHERE句に必ず基準時刻フィルタを入れる仕組みをCI/lintで担保する ②JOINクエリではテーブル別名（`pr.race_date`等）を明示し、JOIN先の時間軸も独立にフィルタする ③fallbackルートを独立にレビューする ④基準時刻はパイプライン入口で1回だけ取得する。監査AIや内部統制での時系列データ分析（例：過去の監査証跡から将来リスクを予測するモデル構築）においても、同じリーク経路が潜みうる。

## アイデア

- 空間軸フィルタ（会場・距離）と時間軸フィルタは直交しており、どちらか一方だけでは不十分という概念整理は、他の時系列MLタスク（金融、需要予測等）にも直接転用できる
- fallbackルート（血統推定）でのリーク経路が最も見落とされやすいという知見：メインパスでガードを入れても代替パスには自動適用されないという構造的盲点は、複雑なパイプライン設計全般に通じる
- 「フィルタ適用後に数値が下がらなければ特徴量が無効かリークが残っている」という逆説的な品質指標は、モデル評価の健全性チェックとして汎用的に使える発想
