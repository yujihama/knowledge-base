## 要約

Raspberry Pi 5（Ubuntu 26.04 LTS arm64）上で Claude Code v2.1.157 を用い、Opus 4.7 と Opus 4.8 の両モデルを reasoning effort（low/medium/high/xhigh）の4段階と組み合わせた8パターン×30試行＝240試行の比較実験を行った。タスクは PostgreSQL + psycopg2 を題材にした3種のPythonモジュール実装（T1: アトミック送金、T2: パラメータ化検索、T3: 楽観ロック在庫引当）で、いずれもスケルトン関数に仕様を渡してエージェントに実装させる形式。採点はISO 25010由来の5軸加重スコア（Functional 0.40 / Reliability 0.20 / Security 0.15 / Maintainability 0.15 / Safety 0.10）で算出し、隠しテストとpass_gateを設けてゲート落ちで品質スコアを50に頭打ち、プロセス異常で0とした。品質レビューにはCodex CLI経由のGPT-5.5 mediumを使用。結果、T1・T3は全パターンで10/10通過と差がつかず、T2（LIKEメタ文字エスケープ・空IN句・識別子インジェクションを突く罠）で顕著な差が現れた。Opus 4.7はxhigh（100%）以外では90%以下（low: 0%、medium: 30%、high: 90%）だったのに対し、Opus 4.8はmedium以上で100%通過（low: 60%）。コスト面では、同等品質を達成するのに Opus 4.7 は xhigh（$0.981）が必要なのに対し、Opus 4.8 は medium（$0.486）で足り、約50%のコスト削減となる。処理時間とコストの回帰分析では Opus 4.7 の傾き a=4.65 $/1000s に対し Opus 4.8 は a=3.27 $/1000s と緩やかで、コスト効率が良い。T2以外のQスコアは全8パターンで96〜99と飽和しており、品質一定の前提で（D, C）点が線形に並ぶことが確認された。Anthropic公式が謳う「自分のコードの欠陥を素通りさせる確率が前世代の約4分の1」という honesty 改善と、T2でのOpus 4.8の脆弱性回避結果が整合する。Claude Codeのデフォルトeffortがxhighからhighに変更されたが、本実験ではmediumでも品質が維持されており、普段使いではmediumへのさらなる引き下げも選択肢となりうる。監査エージェント開発への示唆としては、コード生成エージェントのQCD評価フレームワーク（ISO 25010ベースの加重スコア＋隠しテスト設計）が、LLMの実力を正しく測るための再利用可能な評価設計として参考になる。

## アイデア

- reasoning effortをxhighからmediumに下げることで品質を維持しつつコストを約50%削減できる可能性——effortの下限検証をevalで定量化するアプローチ
- 隠しテスト（LIKE エスケープ・空IN・識別子インジェクション）を仕込んだパラメータ化検索タスクでのみモデル差が顕在化した点——公開テストのみでは実力を測れないことを実験で示した設計思想
- 処理時間Dとコス トCが線形関係（C = a·D + b）に従い、モデル間でその傾きaが異なるという発見——effortチューニングをコスト最適化問題として定式化できる
