## 要約

M3 Ultra (96GB) + Ollamaで動かすllama3.3:70bに、実際の開発コミットから自動でLoRA学習データを蓄積するパイプライン「M2LoRA」を構築した記事。

【仕組み】git post-commitフックをトリガーとし、(1) llama3.3:70bがdiffをレビュー、(2) Claude Code CLI / Gemini CLI / Codex CLIがそのレビューを並列で採点（SCORE/CORRECTION/REVIEWの構造化出力）、(3) 3者の評価をClaudeが合成して「理想レビュー」を生成、(4) diff→理想レビューのペアをSQLiteに蓄積、(5) 十分溜まったらAlpaca JSONL形式でエクスポートしVAST.ai + Unslothでファインチューン、という流れ。APIキーを追加課金なしで使えるのは、各CLIが既契約の開発ツールとして手元で動いているためで、サブプロセスとして呼び出すだけで済む。

【実装上の工夫】asyncio.Semaphore(2)でCLI同時実行を2本に制限してレートリミット・設定ファイル競合を回避。proc.kill()後のawait proc.wait()でゾンビプロセスを確実に回収（Geminiのレビューで発見されたバグ）。SQLiteはWALモードで連続コミット時の書き込みロックを対策。エクスポート時は平均スコア6未満を除外し低品質データを排除。

【追記：LoRA実施結果（174件で実施）】Stage 1としてCPT（continued pre-training）でFM合成・MIDI 2.0・CoreMIDI等の仕様書3,266チャンクをVAST.ai H100 SXMで35分・$1.70で学習（loss: 1.136→0.457）。結果はGemini評価では平均スコア3.85→5.32・高評価率13.7%→40.8%と改善したが、Claude評価では3.76→3.42と横ばい〜微減。Codexは長時間バックフィル中に93%がNULL（死亡）。また、vanilla期とLoRA期で評価対象のcommitが異なりペア比較ができていないという方法論的問題も残っている。

【監査エージェント開発への示唆】複数LLMの「合議制」によるLLM-as-judge構造は、監査エージェントにおける判断の信頼性向上に直接応用可能。また、git hookによる自動データ収集は監査ログや業務プロセスに組み込んだ継続的学習データ蓄積の設計パターンとして参考になる。採点官の意見不一致（stddev > 2.0）をflaggedとして記録する設計は、不確実性の定量化として有用。

## アイデア

- 開発ワークフロー（git commit）に自動データ収集を組み込むことで、追加コストゼロでプロジェクト固有のLoRAデータが継続的に蓄積される設計パターン
- Claude/Gemini/Codexの3 CLIをLLM-as-judgeとして合議制で使い、スコアの標準偏差で「採点官の意見不一致」を検出してデータ品質フィルタリングに活用する手法
- Stage 1 CPT（仕様書テキストでドメイン知識注入）→ Stage 2 SFT（レビューペアで振る舞い学習）の2段階LoRA戦略
